# file-spliter：ローカルLLM時代のファイル分割ツール

## 背景：実務から生まれたツール

file-spliterは、低スペックPCでも動作するローカルLLMアプリケーションの開発過程から生まれた。開発者が直面した課題は単純だ。

LLMアプリケーションのファイルアップロード機能に容量制限を設けていたところ、ユーザーから「100MBを超える長大なPDFを処理したい」という要望が上がった。しかし、20～30MB程度に分割しないと、低スペック環境ではLLMの処理が重くなってしまう。

そこで必要になったのが、大容量ファイルを指定サイズ以下に分割するツールだ。開発者は「この問題を解決するツールを一緒に提供しないと不便だ」と判断し、file-spliterの開発を決めた。

## 設計：シンプルさと汎用性

file-spliterの設計思想は明快だ：「色々なファイル形式に対応し、非エンジニアにも直感的に使えること」。

ユーザーの操作は限定的だ。分割したいファイルを「分割前」フォルダに置き、アプリケーションで分割サイズを指定するだけ。結果は「分割後」フォルダに出力される。複雑な設定は一切ない。

### ファイル形式による処理の使い分け

興味深いのは、ファイル形式によって分割方式を変えていることだ。

**汎用ファイル（PDF、テキスト、画像など）** は単純なバイナリ分割を採用している。ページ境界などを考慮せず、バイト単位で機械的に分割する。これにより、あらゆるファイル形式に対応可能になり、処理も高速だ。

**音声・動画ファイル** はffmpegを用いて、再生可能な状態に分割する。ここで重要な設計決定が入る。

## 高速モードと精度優先モード：トレードオフの選択肢

音声・動画ファイルの分割には2つのモードが用意されている。

### 高速モード：再エンコードなし

高速モードでは、ffmpegの `copy` コーデックを用いる。既存のコーデック情報をそのままコピーして分割するため、再エンコードは行わない。処理時間は短いが、指定サイズへの精度は落ちる。

```
-c copy  # コーデックをコピー（再エンコードなし）
```

### 精度優先モード：再エンコード

精度優先モードは異なるアプローチを取る。

- **ビデオ**：libx264で再エンコード、ビットレートを計算して指定
- **オーディオ**：AACで再エンコード

さらに、キーフレームの位置を強制指定することで、指定サイズに近い分割点を実現する。処理時間は長くなるが、精度が向上する。

この設計により、ユーザーは「時間に余裕があれば精度を重視」「処理を急ぐなら高速モード」という選択ができる。複雑な技術知識は不要で、ユースケースに応じた判断が可能だ。

## メタデータ：再現性と信頼性

各分割結果は `metadata.json` という形式で記録される。

```json
{
  "original_name": "sample.zip",
  "original_sha256": "abc123...",
  "split_mode": "binary",
  "part_count": 2,
  "parts": [
    { "index": 1, "filename": "sample.part001", "size_bytes": 52428800, "sha256": "..." },
    { "index": 2, "filename": "sample.part002", "size_bytes": 52428800, "sha256": "..." }
  ]
}
```

元ファイルと各断片のハッシュ値を記録することで、分割の正確性を検証可能にし、分割済みファイルの復元や統合時の信頼性を確保している。

## 開発プロセス：仕様承認からの実装

興味深いことに、このプロジェクトはClaudeのコーディング機能（Claude Code）を活用して開発された。開発者の役割は要件定義と仕様承認に絞られ、実装はAIエージェントが担当している。

開発者が伝えた要件は以下の通り：
- ファイルを放り込んで、指定したサイズ以下に分割
- 色々なファイル形式に対応
- 非エンジニアにも直感的に使える
- セットアップが簡単

実装過程で、セットアップの容易性が重要な要件として追加された。PythonとffmpegのインストールはREADMEで詳しく説明され、初心者でも対応できる水準に設定されている。

## 今後：ユーザーフィードバックからの成長

開発者は「自分で使った範囲では必要十分」とコメントしており、機能的には完成度が高い。今後の改善は、実際のユーザーからのフィードバックに基づいて進める予定だ。

「いつでも実務の中に答えがある」という信念のもと、公開後の使用例や問題報告を通じて、新たなニーズや改善点を発見していく計画である。

## 開発の学び：問題定義の重要性

このプロジェクトを通じて、開発者が得た最大の学びは何か。

AIエージェントが実装を引き継ぐ時代においては、「技術的にどう実装するか」よりも「自分は本当は何がしたいのか」という問題定義が重要になるということだ。

さらに掘り下げると、それは要件定義のスキル以前の問題だ。要件を言語化する前提として、「自分をよく知っていることが必須」だということ。自分のニーズを正確に理解し、言語化できるか否かが、AIエージェントの能力を引き出す鍵になる。

エンジニアリングスキルの価値が相対的に低下する一方で、自己認識と問題定義のスキルが高まる。file-spliterはそうした時代転換を体現するプロジェクトとなっている。

## まとめ

file-spliterは、実務の課題から生まれたシンプルで実用的なツールだ。複雑さを避け、ユーザーの直感を優先した設計、ファイル形式に応じた柔軟な分割方式、そしてAIエージェントとの協働による迅速な実装。これらが組み合わさることで、低スペック環境でのLLM活用を支えるインフラが完成した。

今後、多くのユーザーの手に渡る中で、どのような新たなニーズが浮かび上がるか。その実務の声が、file-spliterをさらに進化させていくだろう。
