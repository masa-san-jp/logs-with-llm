# 「ちょうどいい」ツールを8時間で作る──Audio-visual-analyzer 開発秘話

**リポジトリ：** https://github.com/masa-san-jp/Audio-visual-analyzer

## はじめに

GitHub で公開されている Audio-visual-analyzer は、ブラウザで音声・動画ファイルを再生しながら、リアルタイムでビジュアライザーを表示する Web アプリケーションだ。シンプルながら豊富なカスタマイズ機能を備えており、わずか HTML/CSS/JavaScript だけで構成されている。

このプロジェクトを個人開発した開発者に話を聞くと、そこには「ちょうどいい」を追求するための、実に思慮深い設計思想が貫かれていた。

## 動機：毎回のフッテージ作成という課題

開発者の仕事の流れは次の通りだ。AI 音楽生成サービス Suno で曲を生成し、YouTube 向けの MV（ミュージックビデオ）を制作するという音楽制作の趣味を持っている。

ここで直面していた課題が、MV 制作の工程である。従来は、音楽ごとに手作業でビジュアルフッテージを準備する必要があった。静止画の組み合わせにせよ動画素材にせよ、毎回かなりの手間と時間がかかる。ユーザーが長時間見つめる画面が全く動かないコンテンツでは、視聴体験としても成立しない。

この「毎回フッテージを作る」という負担を軽減したいというのが、このツール開発の出発点だった。

## 既存ツール選択の課題

当然ながら、音声ビジュアライザーは既に世の中に多く存在する。しかし既存のツールには課題があった。

- 細かくカスタマイズできるツールは複雑すぎて、セットアップや実際の操作に時間がかかる
- シンプルで使いやすいツールは、自分の曲の雰囲気を十分に表現できない
- 「ちょうどいい」バランスを実現するツールが見当たらない

こうした状況の中で、開発者は「自分で作ってしまった方が早い」と判断した。

## ブラウザで「ぱっと立ち上がる」ことの重要性

Audio-visual-analyzer の最初の要件は、シンプルながら本質的なものだった：**ブラウザで立ち上がることを優先する**。

npm install などの環境構築を必要としない。index.html をブラウザで開くか、簡易サーバーを起動するだけで即座に使い始められる。この決定は、実装の簡潔さを保つだけでなく、開発プロセス全体に大きな影響を与えることになる。

実装 → ブラウザで動作確認 → 改善 というテストサイクルが非常に高速化された。開発の初期段階では、エージェント（AI開発支援ツール）と協働しながら仕様を詰めていく過程で、実際にブラウザで動かしながらのディスカッションが可能になった。

「ここはこう動いているが、こうした方がいいか」という判断が視覚的・実時間で可能になることで、仕様決定が驚くほど早く進む。これはテスト駆動開発の思想にも通じるアプローチだが、ここではツールの選択段階から実現されている。

## パラメーター設計の工夫：階層化による複雑性の削減

一見すると、Audio-visual-analyzer には多くのパラメーターがある。UI を見ると調整項目が並んでいて、機能が豊富に見える。

しかし実装の根底には、賢明な設計思想がある：**「大きな要素」と「小さな要素」の階層化**である。

### 大きな要素（構造的決定）

- アナライザータイプ：棒グラフ か 円形放射 か
- 表現方法：棒 / 波形線 / 点
- 表示モード（棒グラフ時）：通常 / ミラー上下 / ミラー左右
- レイヤー数：1 から 4 まで
- 色系統：色相や彩度の大まかな方向性

### 小さな要素（細部調整）

- 感度、スムージング、線の太さ
- 密度、基準点オフセット、残像強度

これらは単なる機能分類ではなく、UI の「ランダマイズ」ボタンの配置にも反映されている。

- 「アナライザーランダム」：大きな要素をランダムに再生成
- 「色相ランダム」：色周りをランダムに設定
- 「形状ランダム」：小さな要素をランダムに調整

### 設計意図

この階層化には、実装の簡潔性を保つだけでなく、**ユーザーの認知負荷を減らす**という意図がある。

開発者は、曲を聴いて「こういう雰囲気だ」と感じたら、直感的にアナライザータイプを選ぶ。その後、色や細かい形状を微調整する。パラメーターが無秩序に並んでいては、「あれもこれも調整できるけど、何から手をつけよう」という迷いが生じ、創造的な気分が冷めてしまう。

階層化により、ユーザーは明確な順序で決定を進められる。無駄な迷いが減り、実作業に集中できる。

## 開発過程での課題と改善

当初の実装は、やりたい機能を羅列しただけで、ロジックがバラバラだったという。機能の組み合わせによって不具合が発生することもあった。

特に記録されているのが、ファイル読み込み周りのバグである。1 本目のファイル読み込み時に、再生終了のコールバックが正しく設定されず、録画の自動停止が機能しなかった。2 本目以降のファイル読み込み時に、古いリスナーの処理が意図せず実行されることもあった。

こうした課題は、ブラウザでの実際の操作テストを通じて発見された。そしてエージェントとのディスカッションの中で、仕様の提案を受けながら改善が進められた。実装者が「ここで動かすとこんなことが起きている」と共有し、改善策を検討する。この反復が、比較的短期間に安定した実装へと導いた。

## 実運用：1 曲の再生時間で完成

実装が完了した後、開発者は実際にこのツールを使って MV 作成を始めた。流れは以下の通りだ。

1. Suno で音楽を生成
1. Audio-visual-analyzer をブラウザで立ち上げ
1. 音楽ファイルを読み込む
1. 曲の雰囲気に合わせてアナライザータイプを選択
1. 曲を再生しながら、色や細かい形状を微調整
1. **1 曲の再生時間（約 3 分）の間に、ビジュアルの決定が完結**
1. 録画機能で MP4 または WebM で動画をエクスポート
1. 動画編集ソフトで字幕を追加

このワークフローにより、従来と比べて制作時間が大幅に短縮された。もはや毎回フッテージを手作業で用意する必要はなく、ツール一つでビジュアル生成が完結する。

## 開発スタイルの協働モデル

最後に、8 時間という短期間でこのプロジェクトを完成させた開発スタイルについて触れておきたい。

開発者の役割分担は次のようになっていた。

**開発者（人間）の役割：**

- やりたいことを言語化する
- テストして問題点を特定する
- 改善方針を判断する

**エージェント（AI）の役割：**

- 言語化された要件をドキュメント（仕様書）に整理する
- 実装を担当する
- エラー修正時の改善を提案する

この協働では、単なる「人間がエージェントに指示を出す」ではなく、「開発担当と レビュー担当が異なる視点で協働する」という構造になっている。人間はビジネス判断と創造的判断に集中し、エージェントは実装の詳細と効率性を担当する。

これにより、アイデアから実装、テスト、改善という全サイクルが、従来では考えられないほどのスピードで回転した。

## 「ちょうどいい」を追求する意義

Audio-visual-analyzer は、完璧さを目指すのではなく、**自分の用途に「ちょうどいい」を実現することを優先した**ツールである。

- セットアップが簡単 → ブラウザで開くだけ
- パラメーターは豊富だが整理されている → 迷わずに選べる
- 実装は最小限 → HTML/CSS/JavaScript のみ
- 開発時間は短い → エージェントとの協働で効率化

こうした「ちょうどいい」の積み重ねが、実用的で使い続けられるツールを生み出した。

開発者は「みんなこうやってるんじゃない？」とも述べている。つまり、この開発スタイルや思考方法は、特殊なものではなく、これからは当たり前になっていくものかもしれない。

自分の課題に対して、既存ツールではなく「自分にちょうどいい」を作る。エージェントの力を借りて、迅速に実現する。そして実際に使ってみて、改善を重ねる。

Audio-visual-analyzer はそうした開発姿勢の一つの実例を示している。
