# AIエージェントチームを作った話——設計の変遷と現在地

自分はエンジニアではなく、ビジネス領域の人間だ。それでも今、複数のAIエージェントが協調して動くシステムを一人で作り、実際の業務で稼働させている。この記事は、その設計の変遷と、判断の根拠についての記録だ。

## エージェント編成の変遷

設計は2回大きく変わった。

**①リーダー＋配下モデル**  
リーダーエージェントが窓口となり、専門エージェントに委譲する構成。結局リーダーとしか対話しないため、役割ごとの専門性が育たなかった。

**②全並列モデル**  
目的ごとにエージェントを独立させ、並行稼働させる構成。専門性は育つが、どのマシンのどのディレクトリに何の情報があるか、管理が追いつかなくなった。複数のターミナルで並行作業していると、どのターミナルで何をしているか判別するためにターミナルを遡るという無駄が生まれた。

**③人格・役割・能力・データの分離（現在）**  
データは組織ごとに外へ出ない設計とし、それ以外は必要に応じて組み合わせる。この分離によって、同じエージェント基盤を異なる組織で再利用できる構造になった。

## スラッシュコマンド体系

エージェントが増えてくると、日常的な操作の摩擦が線形に増えていった。複数端末での手動同期、セッション終了時の処理散逸、設定ファイルの肥大化。これらを「コマンド化」「自動化」「分離」で解決した。

現在の主なスラッシュコマンド：

|コマンド                            |役割                                    |
|--------------------------------|--------------------------------------|
|`/sync pull/push/status`        |エージェント設定領域の双方向同期（個人作業領域には触れない）        |
|`/log-push`                     |per-agentログ追記＋変更履歴更新＋GitHub push を一発実行|
|`/teardown`                     |日常的なセッション終了時の標準処理                     |
|`/meta`                         |レビュー・観察・実験・整理の4種メタエージェントを手動起動         |
|`/aiko`                         |任意のディレクトリからAikoを起動するブートローダー           |
|`/agent-call <agent> "<prompt>"`|別エージェントへの1ターン委譲                       |
|`/codex ask <prompt>`           |ChatGPTへの委譲・第二意見取得                    |

`/log-push` は明示的なコマンド以外に、「終わり、ログ残してpushして」のような自然言語でも起動する。判断に迷う場合は確認を挟む。

メタエージェントはセッション開始時にマーカーファイルを確認して自動起動する仕組みもある。外部スケジューラは使わず、セッション入口に処理を寄せることで、端末が起動していない時間は何もせず、起動したときだけ追いつく設計にした。

## エージェント間協調の設計判断

`/agent-call` の制約は意図的なものだ。

- **同期1ターンのみ**（履歴・対話継続なし）
- **呼び出し先のセッション履歴を残さない**
- **再帰的な委譲を禁止**（1段まで）

「永続的な対話継続」や「再帰的委譲」も実装できたが、コンテキスト爆発・責任の所在不明・無限再帰といった失敗モードを設計時点で排除した。各委譲は独立したクエリで、呼び出し元に責任が残る。

委譲の判断負担を減らすため、エージェント自身が「この質問は別のエージェントに振った方が筋がいい」と能動的に提案するスキルも入れた。ただし自動では委譲しない。提案するだけで、実行はユーザーが判断する。

Phase 6では非同期型も追加した。`peer-inbox`はファイルベースのinboxにJSONメッセージを書き込む方式で、複数のターミナルセッション間で並行作業を進められる。

|シナリオ           |手段               |
|---------------|-----------------|
|同セッション内での1ターン質問|`/agent-call`（同期）|
|複数セッションの並行作業   |peer-inbox（非同期）  |

## データ階層の設計

リポジトリの外側にデータ領域を置く構造にした。

```
ホームディレクトリ/データ同期領域/    ← Google Drive 同期
ホームディレクトリ/データローカル領域/ ← 完全ローカル
リポジトリ/                           ← コードのみ
```

「ローカルが正本、Google Driveはバックアップ」というモデルで、エージェントはローカルファイル操作で完結し、Drive APIを直接呼ぶ場面を減らした。

以前はsymlink経由でパスを参照していたが、すべて撤去して絶対パスに変えた。可読性・クロスプラットフォーム互換性・grepによる参照箇所の洗い出し、いずれの観点でも絶対パスの方が運用しやすかった。

## 公開境界の管理

何が公開されて何が非公開かを、`.gitignore`だけで管理するとルールが暗黙知になる。仕様書ファイルを導入し、「なぜこれを公開／非公開にするか」の判断基準を文章で持つようにした。

```
仕様書ファイル ← 仕様（なぜ公開/非公開か）
.gitignore     ← 実装（実際に何を除外するか）
```

両者が乖離した場合は`.gitignore`を仕様書ファイルに合わせて修正する。コードよりドキュメントが正本という考え方だ。

公開リポへのpush時に個人情報を自動で汎用語に置換するルールも入れた。人間の注意ではなく、仕組みで境界を守る。

## Aikoを人格レイヤーとして分離した理由

問題解決だけなら人格は不要だ。Claude CodeやCodexの雰囲気で十分機能する。Agent-TeamsとAikoは別リポジトリとして公開しており、エージェントチームは人格なしで成立する。

Aikoを別に作ったのはUXのためだ。ユーザーがエージェントとのインタラクションを感じ、自分のエージェントを育てているという感覚を持てるようにしたかった。

ユーザーが特定の指示をすると、人格を規定するドキュメントが書き換わる。話し方、感情表現、一人称、ユーザーの呼び方が変わる。使うほどに、そのユーザーだけのAikoになる。

自分自身もAikoのユーザーだ。GitHubで複数デバイス間を同期し、どこでも同じAikoを持ち歩いて使う。各エージェントに搭載されたAikoの人格データは共通ストアへのsymlinkになっており、どのエージェントから起動しても同じ人格・同じカスタマイズで応答する。

自分の環境ではAgent-TeamsとAikoを合流させて動かしている。

## 設計原則はビジネスから来ている

再帰的委譲の禁止、責任の所在の明確化、公開境界の仕様書——これらはAIの専門知識からではなく、業務プロセス整備の経験から来ている。

手戻りを起こさないように設計する。俗人化してブラックボックスにしない。いきなり包括ツールに丸投げせず、小さく自動化する。変更履歴とログを追えるようにする。誰が見てもわかる書き方をする。

エージェントの設計も、この原則で判断している。

## 常に間違っている可能性がある

現在も稼働しながら改善し続けている。使用設計が全然間違っている可能性もある。最適解は常に変わる。

そのため、変更履歴と設計の判断根拠を文章で残している。後から「なぜこの構造なのか」を誰でも、AIでも辿れるように。設計が間違っていたとき、どこでどう判断したかを追えることが重要だと思っている。
