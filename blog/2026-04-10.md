- **低スペックPC向けローカルLLM設計** — ochyai/vibe-localを拡張し、8〜16GB RAMの一般事務用PCでもLLM体験を提供するアプリケーション設計を完了  
- **仏教哲学の実装プロトコル研究** — 原始仏教から密教まで4段階の変容プロセスを、現代社会で実装可能な認知変容フレームワークとして体系化

## What I Worked On

今週最も時間を費やしたのは、**AI開発ツールの戦略的活用**についてでした。Claude Code Agent Teamsという新機能を使い込み、複数のAIエージェントを専門性に基づいて束ねて協調させるオーケストレーションの実践を重ねました。単なる自動化ではなく、「デジタル専門家集団の育成」というマネジメントへの転換だと感じています。特に印象的だったのは、エージェント同士が人間の介入なしに現場の異常を検知し、方針を自己修正する場面を目撃できたことです。

**低スペックPC向けローカルLLM設計**では、企業の一般事務用PC、教育現場、児童施設などでも誰もがLLMに触れられる環境を作る設計に取り組みました。当初はGemma 4 E2BとBonsai 8Bの2モデル構成を考えていましたが、フィージビリティレビューの結果、Bonsai 8BのCPU成熟度や導入複雑性の問題でGemma 4 E2B単体に統一することに。8GB環境はテキスト会話中心、16GB環境で全機能推奨という現実的な制約設計に落ち着きました。

並行して、**仏教哲学を現代の認知変容プロトコルとして再構築する**という、一見すると技術とは無関係に見える研究も進めました。これは単なる宗教的関心ではなく、AIや複雑なシステムと向き合う現代の意思決定者にとって、「苦しみの構造解体」という論理的フレームワークが戦略的価値を持つという仮説に基づいています。原始仏教の四諦から密教の即身成仏まで、段階的に認識の深層レイヤーに介入していく4つのプロトコルを整理しました。

## Decisions & Tradeoffs

**Claude Code Agent Teams**では、`in-process`モードを`split panes`より強く推奨する判断をしました。split panesには重大な既知バグ（入力が表示されるだけで処理されない）があり、汎用性を重視するプロフェッショナル現場では採用リスクが高すぎると判断したためです。

**ローカルLLM設計**で最も重要だった判断は、「最大性能より導入しやすさを優先する」という設計原則の徹底でした。Bonsai 8Bは性能面では魅力的でしたが、PrismML独自のllama.cppフォークへの依存、Ollamaレジストリ未収録、2026年3月リリースの新興プロジェクトという安定性リスクを総合的に評価し、除外しました。また、音声入力については「未確定」とするのではなく、Whisper.cpp経由の暫定方針を明確に定めることで、実装時の混乱を回避する判断をしました。

## Progress Since Last Time

代わりに、AIツールの実践的活用方法、ローカルLLMの現実的な社会実装、そして人間の認識変容という根本的な問題に取り組みました。これは意図的なシフトで、ルールを作るだけでなく、実際にAIを使いこなす技術と、それを支える思考フレームワークの両方が必要だと感じたためです。

## What's Next

来週は、今週作成した設計・プロトコルの**実装フェーズ**に入ります。

Claude Code Agent Teamsについては、実際に社内プロジェクトでエージェントチームを稼働させ、5つのトリガー（上司フィードバック、判断を変える発見、失敗からのガードレール、業界パターン抽出、仮説の結果記録）に基づく学習ループを回してみたいと思います。

低スペックPC向けローカルLLMは、ochyai/vibe-localの現行アーキテクチャ確認から始めて、UIスタック、推論ランタイム、mmproj取り扱い方針の確定を優先します。特に音声入力のWhisper.cpp組み込みと、8GB環境でのコンテキスト長制御の実装が技術的なポイントになりそうです。

仏教プロトコルについては、理論から実践への橋渡しとして、実際に「四諦による現状分析」「八正道の行動チェックリスト」「六波羅蜜の日々のコード」を自分自身で回してみて、現代のビジネス環境での有効性を検証してみる予定です。
