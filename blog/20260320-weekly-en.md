# When You Collect Everything and Define Nothing: A Week of Building Systems to Extract What Organizations Won't Say Out Loud

This week was one of those weeks where I started pulling on a thread and ended up unraveling half a sweater. It began innocently enough — poking around Google Workspace CLI architecture — and by Friday I was knee-deep in designing a system to extract organizational tacit knowledge from Calendar, Gmail, and Chat logs using LLM pipelines. Somewhere in between, I also spent time thinking about how to get non-engineers to participate in AI-driven development, investigated a multi-provider AI agent framework called co-vibe, and mapped out which of my four AI tools should do what (and more importantly, what they should *not* do).

The connecting thread, if there is one: how do you get AI to work inside messy, real-world organizational structures without pretending those structures are cleaner than they are?

---

## Highlights of the Week

### "Definition before collection, or you're just hoarding"

This one kept coming back across multiple sessions, like a song stuck in my head. I was designing a pipeline to collect behavioral and decision-making logs from Google Workspace — Calendar events, Gmail threads, Chat messages — and feed them through LLMs for pattern extraction. The idea was to surface the invisible processes that actually run an organization: who really makes decisions, what informal channels matter, which meetings are theater and which are where work happens.

But the AI kept pushing back (politely, as they do) on my instinct to collect first and define later. The core challenge: if you don't operationally define what counts as a "decision" before you start collecting, you end up with a mountain of data you can't label. Calendar entries tell you what was *planned*, not what *happened*. Gmail has the richest decision context but also the most noise. Chat has the highest decision density but the hardest-to-parse threads.

The uncomfortable takeaway: building a dashboard is not the same as understanding anything. Visualization-first projects are the organizational equivalent of rearranging furniture and calling it renovation.

### gws CLI as "Agent DX" — tools designed for AI, not humans

This was the genuinely surprising discovery. Google's gws CLI uses the Discovery Service to dynamically generate commands at runtime, accepts JSON payloads, and offers `gws schema` for runtime schema retrieval. The entire design philosophy positions agents as "well-intentioned but unreliable autonomous actors" — a zero-trust approach where the CLI structurally reduces hallucination by giving the LLM verifiable schemas instead of asking it to guess API shapes.

What struck me is how clearly this marks a transition point. CLI design is no longer primarily about human ergonomics. It's about making tools that an AI agent can operate reliably. The humans are becoming the supervisors, not the operators. Whether that's exciting or terrifying probably depends on what day you ask me.

## Insights and Learnings

**You need at least four weeks of data.** This came out of the behavioral log analysis — anything less and you're capturing anomalies, not patterns. Organizations have rhythms (monthly cycles, sprint cadences, reporting periods) and you need to see at least one full cycle to separate signal from noise.

**The six-axis weighting model for tacit knowledge is promising but fragile.** We defined axes: utterance frequency, role, response speed, meeting density, repetition patterns, and thread depth. Each of these makes sense individually. Together, they risk creating a false precision — a score that looks scientific but is really just a dressed-up guess. I'm proceeding with it anyway, because sometimes you need to build the wrong thing to understand what the right thing looks like.

**"What not to use it for" is a better boundary than "what it can do."** When I mapped Claude Code, Antigravity, Copilot, and Cowork across my workflow, the useful question wasn't capability comparison. It was: where does each tool create more confusion than value? Antigravity's Claude access got scoped down. Cowork got limited to post-completion documentation. The negative space defined the system better than the feature lists.

**The 80-point rule needs a companion rule.** For the AI-driven development proposal, I landed on "accept 80% quality from AI, iterate from there" paired with "no one merges code they can't explain." These sound contradictory — you're lowering the quality bar while raising the accountability bar. But that tension is the point. Without both rules, you get either perfectionism paralysis or black-box spaghetti. Possibly both.

## Summary and Commentary per Discussion

### gws CLI Architecture Deep Dive (March 14)

Spent the session dissecting how gws CLI works under the hood. Discovery Service-based dynamic command generation, MCP compatibility, 100+ agent skills documented in SKILL.md files, AES-256-GCM credential encryption. Also catalogued a lovely collection of OAuth setup pitfalls: forgetting to add test users, hitting the 25-scope ceiling, the usual "it should be simple but isn't" dance with Google's auth systems.

*The CLI has over 100 skills. I have about three. Somehow I'm the one in charge.*

### Development Environment Mapping (March 15)

Assigned my four AI tools across the project lifecycle: design → implementation → review → deliverable distribution. Created a CLAUDE.md file to encode design principles so they persist across sessions. Also sketched a GitHub Actions weekly auto-execution workflow for the collection pipeline.

*Four AI tools, each with its own personality and limitations, coordinated through markdown files. This is either sophisticated orchestration or the world's most elaborate Rube Goldberg machine.*

### Behavioral and Decision Log Collection Design (March 15)

Analyzed what each Google Workspace data source actually gives you versus what you wish it gave you. Designed a five-stage pipeline: define → collect → AI classify → visualize → structuralize. The "define first" principle was the session's main contribution, and also the part most likely to be ignored under deadline pressure.

*We designed a system to extract tacit knowledge from organizations. The first piece of tacit knowledge it revealed: nobody agrees on what tacit knowledge is.*

### Organizational Tacit Knowledge Extraction System (March 15)

Built the four-layer architecture (Raw Collection → Context Assembly → LLM Analysis → Output) with three specialized LLM prompts: implicit process naming, automation possibility scoring, and organizational network analysis. Decided that cross-channel collection across all teams is non-negotiable — sampling from select channels introduces survivorship bias.

*The system is designed to find out who really runs the organization. Management may not love the answers, which is probably a sign we're asking the right questions.*

### AI-Driven Development Company-Wide Proposal (March 15–16)

Framed the engineering shortage not as a hiring problem but as a participation problem. If AI tools can lower the barrier to contributing code, business-side staff become development participants. The 80-point rule and the "explain what you merge" rule form the governance backbone. Pitched this as a cultural shift, not a tool rollout.

*Proposing that non-engineers write code with AI assistance. What could go wrong? Don't answer that — I need to stay optimistic until at least the pilot phase.*

### co-vibe Multi-Provider Agent Investigation (March 17)

Looked into co-vibe's strategy pattern for model selection (auto/strong/cheap), Ollama support for local models, and multi-agent parallel execution. More of a design pattern reconnaissance mission than a direct project input, but the routing logic is relevant to any system that needs to pick the right model for the right task without human intervention.

*A system that automatically decides whether to use the expensive smart model or the cheap fast one. Finally, AI that understands budgeting better than I do.*

## What to Try Next Week

- **Run a 1-week pilot collection** on a single team's Calendar + Chat data, with the operational definitions locked down *before* any data touches an LLM. If the definitions feel wrong after seeing real data, that's useful information — but the sequence matters.
- **Test the six-axis weighting model** against at least two teams to see if the weights need to be team-specific or if there's a reasonable default.
- **Draft the CLAUDE.md** design principles document and see if it actually maintains coherence across three or more Claude Code sessions. If it drifts, I need to know now.
- **Circulate the AI-driven development proposal** to at least two skeptics before the believers. Friendly feedback is nice; hostile feedback is useful.

One more thing I keep thinking about but haven't resolved: the tacit knowledge extraction system is itself an exercise in making the implicit explicit. But some things are implicit for a reason — political dynamics, informal power structures, workarounds that exist because the official process is broken. Surfacing those isn't neutral. It's an intervention. I haven't figured out yet whether that's a feature or a risk.

Probably both. Most interesting things are.
