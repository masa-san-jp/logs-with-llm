# 「定義なき収集は無駄になる」と自分に言い聞かせた週

## 導入

今週は、Google Workspace CLIを起点にした「組織の暗黙知を抽出するシステム」の設計にほぼ丸一週間を費やしていました。

きっかけ自体はシンプルで、「組織の中で誰が何を決めているのか、本当のところ誰もわかっていない」という問題意識です。カレンダー、Gmail、Chat——日々のやりとりの中に意思決定の痕跡は確実に残っているはずなのに、それを構造化して見える形にしている組織はほとんどない。じゃあLLMでやれるんじゃないか、と。

ただ、週の後半になるにつれて「これ、設計が楽しすぎて目的を見失いかけてないか？」という自問が何度か出てきました。正直、まだ答えは出ていません。

---

## 今週の議論ハイライト

### 「定義なき収集は無駄になる」——可視化先行の罠

今週一番繰り返し出てきたフレーズがこれです。

暗黙知抽出システムを設計する過程で、「まずデータを集めて可視化しよう」という方向に何度も流れかけました。Calendar APIからイベント取って、Gmailのスレッド引っ張って、Chat履歴をダンプして……技術的にはそれぞれやれることが見えてくるので、つい手を動かしたくなる。

でもAIとの議論の中で繰り返し突きつけられたのは、「"意思決定"って何を指してるんですか？」という問いでした。

予算承認は意思決定か？ → まあそうでしょう。
会議の日程調整は？ → 微妙。でも「誰が最終的に日程を決めたか」は組織力学の表れかもしれない。
Slackでの「LGTM」は？ → 文脈による。

この「操作的定義」を先に決めないと、ログを集めてもラベリングできない。ラベリングできないデータにLLMをかけても、出てくるのは「それっぽいが検証不能な洞察」でしかない。

データ収集・ダッシュボード化プロジェクトが「ダッシュボードを作ること」自体を目的化して死んでいくパターン、何度も見てきたはずなのに、自分でもそこに片足を突っ込みかけていたのは少し恥ずかしい。

### gws CLIの「Agent DX」設計思想

もう一つ印象的だったのは、gws CLIのアーキテクチャを深掘りした際に見えてきた設計思想です。

Discovery Serviceから動的にコマンドを生成する、JSONペイロードで構造化された入力を受け付ける、`gws schema`でランタイム時にスキーマを取得できる——これらの設計は、人間が叩くCLIとしては正直オーバーエンジニアリングに見えます。

でもこれ、「AIエージェントが使うCLI」として見ると完全に合理的なんですよね。

LLMがAPIを叩くときの最大の問題はハルシネーション——存在しないエンドポイントを呼んだり、パラメータ名を間違えたりすること。スキーマをランタイムで取得できれば、LLMは「覚える」必要がなく「確認してから叩く」ことができる。エージェントを「善良だが信頼性に欠ける自律的アクター」と位置づけるゼロトラスト設計という表現が議論中に出てきて、これはかなり的を射ていると思いました。

CLIの設計が「Human DX」から「Agent DX」に転換し始めている。これはgwsに限らず、今後のツール設計全般に波及する話だと感じています。

## 気づき・学び

### 「何に使わないか」で境界を引く

今週、Claude Code / Antigravity / Copilot / Coworkの4ツール体制をプロジェクトに割り当てる設計もやったんですが、ここでの気づきは「何ができるか」じゃなくて「何に使わないか」で境界を引くのが圧倒的に実用的だということでした。

各ツールの機能は重複している部分が多い。「Claudeでもできるし、Copilotでもできる」みたいなタスクがたくさんある。その中で「Antigravityでは詳細実装をやらない」「Coworkは成果物完成後のドキュメント生成に限定する」といった「やらないこと」を決めることで、初めて使い分けが機能し始める。

これ、AIツールに限らず、チーム内の役割分担でも同じことが言えるはずです。「あの人は何でもできる」状態は、組織としては何も決まっていないのと同義。

### 80点で回す勇気と、説明責任のバランス

AI駆動開発の全社展開構想の中で出てきた「80点ルール」と「説明できないコードのマージ禁止」の組み合わせ。これは矛盾しているようで、実はかなり現実的なバランスだと思っています。

AIに100点を求めると、結局人間がゼロから書くのと変わらない工数がレビューにかかる。でも「AIが書いたからよくわからないけどテスト通ってるしマージしよう」をやると、半年後にチーム全体が負債の沼にはまる。

80点で出して、残り20点の「なぜこうなっているか」を人間が説明できること——これが落としどころかなと。ただし、この「説明できる」の基準をどこに置くかは、まだ自分の中でも曖昧です。

## 各議論の要約と所感

### gws CLIアーキテクチャ解析（0314）

gws CLIの技術的な深掘りを行いました。Discovery Serviceベースの動的コマンド生成、MCP対応、100以上のエージェントスキル（SKILL.md）、AES-256-GCM暗号化によるクレデンシャル管理など、構造を一通り把握。OAuthセットアップではテストユーザー未追加やスコープ上限25個制限といった実践的な落とし穴も記録しました。

**所感：** 技術調査としては密度が高く、後から参照する価値のあるログになっています。ただ、「調べて感心して終わり」のリスクもある。重要なのはこの設計思想を自分のプロジェクトにどう取り込むかで、特にスキーマのランタイム取得パターンは暗黙知抽出パイプラインのLLM入力設計にそのまま応用できるはず。来週、具体的にどこで使うか決めるべき。

### 開発環境マッピング（0315）

暗黙知抽出プロジェクトに対して、4つのAIツールを「設計→実装→レビュー→成果物配布」のフェーズに割り当てました。CLAUDE.mdで設計原則をコード化してセッション横断の一貫性を担保する工夫や、GitHub Actionsの週次自動実行ワークフローも設計。

**所感：** ツール割り当て自体は合理的だけど、この手の「開発環境設計」って、実際にコードを書き始めると3日で形骸化するパターンが非常に多い。特にCLAUDE.mdに設計原則を書くアプローチは、それ自体をメンテナンスし続ける運用コストが見えていない。2週間後にもこの体制が機能しているかどうかで真価が問われる。意識的に振り返りポイントを設けるべき。

### 行動ログ・意思決定ログ取得設計（0315）

Calendar/Gmail/Chatそれぞれのデータ特性を分析。カレンダーは「予定≠実績」で直接的な意思決定情報は薄い、Gmailは意思決定文脈が濃いがノイズ最大、Chatは意思決定密度が最高だがコンテキスト依存度も最高。「定義→収集→AI分類→可視化→構造化」の5段階パイプラインと、最低4週間のデータ蓄積が必要という結論。

**所感：** データソースごとの特性分析は地味だけど価値が高い。特に「カレンダーの予定は実績ではない」という指摘は、カレンダーデータを使った分析プロジェクトが犯しがちな暗黙の前提を突いています。ただ、4週間という数字の根拠がやや弱い。「なぜ3週間ではなく4週間なのか」を説明できるようにしておかないと、ステークホルダーへの説明で詰まる。感覚的な数字を構造的な根拠で裏打ちする作業が残っている。

### 組織暗黙知抽出システム設計（0315）

4層アーキテクチャ（Raw Collection → Context Assembly → LLM Analysis → Output）を設計。重みの6軸（発言頻度・役割・応答速度・会議密度・繰り返しパターン・スレッド深度）と、3種のLLMプロンプト（暗黙プロセス命名・自動化可能性スコアリング・組織ネットワーク分析）を定義。

**所感：** 正直、設計として美しすぎる。6軸の重みとか3種のプロンプトとか、紙の上では綺麗だけど、実際にデータを流してみたら半分くらいは使い物にならない可能性が高い。特に「暗黙プロセス命名」プロンプトは、LLMが本当に暗黙知を「発見」できるのか、それとも「それっぽい名前をつけて分かった気にさせる」だけなのか、検証方法が設計に含まれていない。小さく試して壊すフェーズを意識的に入れないと、壮大な設計図だけが残るプロジェクトになるリスクがある。

### AI駆動開発の全社展開構想

エンジニア不足を「採用で解く」のではなく「AIでビジネス側を開発参加させる」で解くという提案。80点ルール、説明できないコードのマージ禁止、段階的な権限拡大などのルール設計を含む。

**所感：** 方向性には共感するけど、「ビジネス側が開発に参加する」は過去20年間、ローコード/ノーコード文脈で何度も語られて何度も失敗してきたテーマでもある。今回「AIだから違う」と言えるだけの具体的な差異を示せるかがカギ。特に「80点で回す」が「ビジネス側が作った80点」なのか「AIが作った80点をビジネス側が承認する」なのかで、意味がまったく変わる。ここの解像度を上げないと、提案が抽象的な理想論に留まる。

### co-vibe調査

落合陽一氏関連のマルチプロバイダAIエージェントツールを調査。strategy pattern（auto/strong/cheap）によるモデル選択、Ollama対応、マルチエージェント並列実行などの設計パターンを収集。

**所感：** 設計パターンの収集としては有用だけど、「自分のプロジェクトで何に使うか」が明確でないまま調査している感がある。strategy patternによるモデル選択は暗黙知抽出パイプラインのLLM Analysis層で直接使える可能性があるので、来週そこを接続する作業をやるべき。調査は楽しいけど、調査だけでは何も進まない——これは自分への戒めとして書いておく。

## 来週やってみたいこと

1. **「意思決定」の操作的定義を3段階くらいで仮決めする。** 完璧な定義は不要。まず仮置きして、データをかけてみて、定義を修正するサイクルを1回転させたい。

2. **暗黙知抽出パイプラインの最小構成を実際に動かす。** 全4層を作るのではなく、Chatログだけを対象に、1つのプロンプトだけで、何が出てくるかを見る。設計の美しさより、動くものの醜さを優先する週にしたい。

3. **CLAUDE.mdの運用状況を1週間後に振り返る。** 形骸化してないか？ してたら何が原因か？ を正直に記録する。

4. **co-vibeのstrategy patternを暗黙知抽出のLLM層に適用するPoCを書く。** 調査で終わらせない。

あと、ずっと頭の片隅にある問いとして——「AIで組織の暗黙知を構造化する」って、そもそも誰が嬉しいんだっけ？ 経営層？ ミドルマネジメント？ 現場？ 「全員に価値がある」は「誰にも刺さらない」の裏返しなので、来週はこの問いにも向き合いたい。
