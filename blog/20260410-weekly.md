# AI運用規則から仏教研究まで

最近のLLM使い方って、規則を作るところから思想的な深掘りまで本当に多様だなと実感している。今週は組織のAI運用から個人的な仏教研究、そして実装レベルの技術検討まで、興味深い展開があった。

## Highlights

• **SNSガイドライン設計** — Claude、Gemini、GitHub Copilotの3つのLLMでそれぞれ違う角度からSNS運用規則を検討し、Claude＝包括的法務重視、Gemini＝実務重視、GitHub Copilot＝会議録スタイルという特徴が見えた
• **日本のPhysical AI戦略分析** — 2040年までに20兆円市場の30%シェア獲得を目指す野心的な計画で、ロボット産業70%シェアからVLAモデル統合へのパラダイムシフトが興味深い
• **仏教の技術実装研究** — 四聖諦から密教まで、仏教を「認知OS更新プロトコル」として体系化し、9層意識構造やbodhicittaの実装仕様まで詳細に検討した
• **Claude Code Agent Teams** — 完全に独立したセッション間で自動コミュニケーションする新しいAI協働システムで、従来の単一セッション内Subagentと比べて本質的に異なるアプローチ
• **ローカルLLM設計** — 8-16GB RAMのローエンドPC向けGemma 4 E2Bアプリケーション設計で、メモリ制約下でのマルチモーダル機能実装の現実的な検討

## What I Worked On

今週の作業テーマは「規則設計の複眼性」と「思想の技術実装」だった。

前回のAI運用規則設計から発展して、今度はSNS利用ガイドラインを3つの異なるLLMで同時に検討した。この実験が予想以上に面白い結果を生んだ。Claudeは法的リスクを重視した包括的な規則を作成し、違反時の懲戒処分まで詳細に規定。Geminiは実務運用面に特化し、危機管理やライブ配信など具体的な運用シーンを想定した内容に。GitHub Copilotは会議録形式で部門横断的な検討プロセスを重視した。同じ課題に対するこれほど明確な「思考スタイルの違い」を観察できたのは収穫だった。

一方で、全く別の知的好奇心から仏教研究を本格的に始めた。ただし宗教的関心ではなく、2500年続く「認知変革システム」としての技術的価値に注目している。四聖諦を因果分析フレームワーク、八正道を最適化チェックリストとして捉え直し、さらに唯識の九識構造を9層の意識プロトコルとして実装仕様まで書いてみた。特に興味深かったのは、原始仏教からテーラワーダ、大乗、密教への発展を「認知OS更新の4段階」として構造化できたこと。密教の三密加持を「身体・言語・心の同期チャンネル」として技術的に理解すると、現代のマルチモーダルAI開発と本質的に共通する課題意識があることに気づいた。

技術面では、Claude Code Agent Teamsという新機能の運用プロトコルを検討した。これまでのSubagentと異なり、完全に独立したセッション間で自動的にメッセージ交換する仕組みで、真の並列処理が可能になる。ただしコストは5-7倍で、メンバー全員が自動的にOpus 4.6に割り当てられる。実際に空のリポジトリでエラーが発生した際、チームメイトが自動的にリーダーに通知し、全体の作業方向を構造提案に切り替える自己組織化を観察できた。人間の介入なしにこのレベルの協調行動が発生するのは、単なる「ツール」を超えた「デジタル同僚」への転換点だと感じる。

## Decisions & Tradeoffs

SNSガイドライン設計では、3つのLLMの出力を統合するよりも、それぞれの特徴を活かした役割分担を採用することにした。Claudeの法的網羅性、Geminiの実務性、GitHub Copilotの組織プロセス設計を組み合わせることで、単一LLMでは得られない多面的な検討ができた。

仏教研究では、学術的厳密性と実装可能性のバランスが課題だった。パーリ語やサンスクリット原典の正確性を保ちつつ、現代のプログラミング概念にマッピングする必要がある。結果的に、4段階の仏教発展（原始→テーラワーダ→大乗→密教）を異なる認知深度への介入レベルとして整理し、表面的行動レベルから第9識レベルまでの体系的なプロトコルとして構造化した。

Claude Code Agent Teamsでは、コスト効率性よりも真の自律性を優先した。従来の単一セッション内でのSubagent運用は安価だが、リーダーの文脈に制約される。Agent Teamsは高コストだが、各メンバーが独立した判断を下せるため、複雑な多視点タスクには価値がある。実装では、ファイル競合を防ぐため明示的なテリトリー割り当てを採用し、Ctrl+Tでのタスク管理を必須とした。

## Progress Since Last Time

前回のAI運用規則設計は「条件付き許可」というアプローチで基礎を築いたが、今回のSNS規則では「複数LLM協働」という新しい方法論を確立した。単一の規則を作るのではなく、異なる観点から同時に検討することで、より実用的で抜け漏れの少ない結果を得られることがわかった。

技術面では、前回まではClaude Codeを単体ツールとして使っていたが、今回Agent Teamsの導入で完全に協働型の開発環境に移行した。5-trigger学習システム（ボス・フィードバックパターン、判断変更発見、失敗ガードレール、業界パターン抽出、仮説結果追跡）を通じて、AIチーム自体が学習・進化する仕組みを構築できた。

一方で、前回課題として残った「許可ツール評価基準」や「不明時エスカレーションフロー」は今回直接扱わなかったため、これらは依然として未解決のまま。

## What's Next

来週は3つの方向性で発展させたい。

まず、SNSガイドライン設計で確立した「複数LLM協働手法」を他の組織課題にも適用してみる。リスク管理、人事制度、技術標準など、多角的検討が必要な領域で同様のアプローチが有効かを検証したい。

仏教研究については、今回構築した理論フレームワークの実装に着手する。特に「9層意識プロトコル」のPython実装と、密教の五段階成仏観法の具体的な瞑想アプリケーション化を試みる。認知科学や神経科学との接続も探ってみたい。

技術面では、Claude Code Agent Teamsの運用経験を蓄積し、より大規模なプロジェクトでの自律開発プロセス設計に挑戦する。また、Gemma 4 E2Bを使ったローカルLLMアプリケーションの実装を通じて、ローエンドPC環境でのマルチモーダルAI展開という新しい領域を開拓していく予定。
