# 技術系プロフェッショナル向け週間活動報告 - 2026年4月11日

先週は「AIを使って規程を作る」という、なかなか再帰的で面白い仕事をしていました。全社向けの生成AI利用規程策定です。

## ハイライト

- **「禁止」ではなく「条件付き許容」という設計思想** - 一律禁止は現場と乖離して形骸化するため、承認制・条件付き許可の枠組みを採用
- **ビジネスサイドに「rm -rf」の話をする** - 生成AIがコマンドを提案する時代、ノンエンジニアの「AIに言われたから実行」事故への対策
- **「教える」価値の再認識** - AI時代でも新人育成が組織の自殺行為にならない理由（プロテジェ効果）
- **仏教プロトコルと密教の即時同期** - 認識OSのデバッグから身体統合まで、変容の4段階構造
- **低スペックPC向けローカルLLM設計** - CPU-only/8-16GB環境での全員LLM体験実現への取り組み

## 今週の作業内容

### 生成AI規程策定の思考整理

複数のLLM（Claude、Gemini、GitHub Copilot）と対話しながら、全社向けの生成AI利用規程を作成しました。正社員・派遣・業務委託まで含む適用範囲で、Claude CodeやGitHub Copilotといったエージェント系ツールも明示的に対象としています。

週を通じて最も印象的だったのは、「禁止」から「条件付き許容」への設計転換です。全面禁止は管理は楽ですが現場の実態と乖離します。そこで「意味を理解しない限り実行禁止」という表現を採用し、教育・申請フロー・技術的制御の三層防衛で実効性を担保する方針にしました。

特にターミナル操作について、エンジニア以外の人が生成AIの提案で危険なコマンドを実行するリスクに対応するため、具体的なブラックリスト（rm -rf、curl | bash等）をビジネス職向けサマリに含める設計にしたのは新しい試みでした。

### Claude Code Agent Teamsの運用プロトコル

自律型AIチームの構築について、実装可能な形でプロトコルを整理しました。重要だったのは、Agent Teamsが単なる作業の自動化ではなく「デジタル専門家集団の育成」というマネジメントへの転換だということです。

特に「5つのトリガーによる学習プロトコル」（上司フィードバック、判断を変える発見、失敗からのガードレール、業界パターン抽出、仮説の結果記録）を通じて、AIチームを成長する資産として運用できる仕組みの設計に取り組みました。

### 仏教変容プロトコルの体系化

原始仏教→上座部→大乗→密教の4段階を、認識レイヤーの深化として整理しました。これは単なる宗教的な話ではなく、個人の認識OSをデバッグし、アップデートするためのプロトコルとしての分析です。

密教の「即身成仏」が特に興味深く、これまでの「三劫成仏」という漸進モデルを破壊し、宇宙との即時同期を実現する技術として機能していることがわかりました。五相成身観の5段階は、自己を大日如来へとアップグレードする厳密なステップとして実装可能です。

## 意思決定と取捨選択

### 規程設計における戦略的判断

「迷ったら」の相談先設計で悩みました。上司への相談だと結局そのAIリテラシーに依存し、申請フローだと利便性が落ちて使われなくなるリスクがあります。最終的に、段階的エスカレーション（まず上司、次にAI推進室）と、頻繁な質問は次回の規程更新に反映させる継続改善ループを組み込むことで解決しました。

### 低スペックPC向けローカルLLM設計での技術選択

当初はBonsai 8Bも候補でしたが、CPU成熟度・導入複雑性・安定性の問題からGemma 4 E2B単体に統一することを決定。8GB環境はテキスト会話のみ保証、16GB環境を全機能推奨とし、音声入力はWhisper.cpp暫定採用としました。「最大性能より導入しやすさ」という設計原則を貫いた判断です。

## 前回からの進展

前回の投稿と比較すると、今回は「実装可能性」への焦点が格段に高まっています。仏教プロトコルもPython命名規則での実装仕様まで落とし込み、AI規程も具体的な条文レベルまで詰めました。

一方で解決できていない問題もあります。規程の実効性担保（特に「理解なき実行の禁止」をどう運用するか）、音声入力のllama.cpp未対応問題、コンテキスト長管理の具体的設計などです。

## 来週の展望

残された課題として、まず許可ツール評価軸の設計と「迷ったら」フローの具体化を優先します。低スペックPC向けLLMプロジェクトでは、既存リポジトリ構成の確認とUI技術スタックの確定が急務です。

また、規程作成でLLMを使ったこと自体について、もう少し考察を深めたいと思っています。「AIが提案した条文をAIが審査してAIが修正する」ループで本当に大丈夫なのか、弁護士・社労士レビュー以外に何を検証点とするべきか。

規程が作れたという達成感と、これで本当に機能するのかという不安が同居したまま週が終わりました。完璧に解決はしていませんが、これも次週への宿題として持ち越します。
