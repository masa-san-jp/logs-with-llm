# コードを書かない時代のソフトウェアエンジニアリング

> 「アップデートを出荷する際、私はコードを一切書かないが、出力されたすべての文字をチェックしている。」― Jesse Vincent

-----

かつてPerlコミュニティのリーダーとして言語の核心を担い、その後ハードウェア開発へと転身して深センの工場で詐欺師と渡り合いながら4回のKickstarterキャンペーンを成功させてきたJesse Vincentという人物が、今AIエージェント開発の最前線に立っているという事実がある。

この変遷は、単なるツールの進化では説明がつかないのではないかと私は思う。

構文を打ち込むことが「仕事」だった時代から、アイデアを言語化することが「仕事」になる時代へ。エンジニアという職能が根本から捉え直されつつある。そのディティールを、実践的な論点に分解して整理してみたい。

-----

## マネジメントへの回帰：AIを「ジュニアエンジニア」として扱う技術

AI開発における成功の鍵は、コーディング技術そのものではなく、人間側のマネジメント能力にある。

これは逆説的に聞こえるかもしれない。しかし考えてみると、AIは非常に優秀だが判断力やセンスに欠ける「ジュニアエンジニア」に似ている。そう捉え直すと、必要なスキルセットが自然と見えてくる。

Jesse Vincentが20年前にMITのインターンたちをIRC経由で管理していた手法は、現代のAIエージェント操作にほぼそのまま適用できる、と彼は言う。

|          |20年前のMITインターン      |現在のAIエージェント                |
|----------|-------------------|---------------------------|
|**タスクの分解**|細かいタスクの分解と指示が必要    |意図と手順の明確化が必要               |
|**品質の担保** |TDDとYAGNI/DRYの概念の徹底|プロンプトでの厳格なルール付け（Hard Gates）|
|**監視と限界** |徹夜明けで文脈の記憶が飛ぶ      |コンテキストウィンドウの限界による忘却        |

つまり、求められているのは「ハードワークで得た知恵」の再運用だ。

マネジメント経験の有無が、そのままAIから価値を引き出す能力の解像度の差に直結している——そう考えられるのではないだろうか。

-----

## 意図の解像度：ホーマーの車を避けるために

AIは「あらゆる機能を盛り込むこと」を好む傾向がある。

表面的な指示だけを渡すと、アニメ『シンプソンズ』に登場する「ホーマー・シンプソンの車」——依頼者の要望を全部詰め込んだ結果、誰も乗りたくない代物になった車——のような怪物が生まれる。

「ReactでToDoリストを作って」という指示と、「なぜ作るのか・目的は何か・何が不要か」を伝える指示では、出力の質が根本的に異なる。前者は機能の羅列、後者は意図に基づく設計だ。

一方で、この「なぜを先に語る」という作業は、従来のコンサルティングや要件定義が担っていたものとほぼ同じだとも言えるだろう。新しい技術を前に、実は古い知恵が問われている。この手触り感は、私にとって大変興味深いと感じる。

-----

## やり直しという合理性：失敗コストがゼロになるとき

ソフトウェア開発手法は長らく「ウォーターフォール」から「アジャイル」へと進化してきた。しかしAI時代においては、むしろ「超高速なウォーターフォール」という形態への回帰と進化が起きていると考えられる。

|                     |サイクル              |失敗時のコスト              |
|---------------------|------------------|---------------------|
|**従来のウォーターフォール**     |数ヶ月（計画 → 実行 → テスト）|サンクコスト大              |
|**AIネイティブ・ウォーターフォール**|1時間（計画 → 実行）      |やり直しコストゼロ（要件を更新して再生成）|

人間が見積もれば「3ヶ月かかる」とされた本番移行作業が、AIエージェントを駆使することでわずか2時間で完了した事例もある。

そうすると、開発の合理性そのものが変わる。中途半端なバグを時間をかけて修正するよりも、要件を更新して最初から作り直す方が早い。AIは疲弊しないため、このアプローチが最も経済的な選択肢になるのだ。

## 洗練の装置：泥の塊から始める開発プロセス

事前にすべてを計画するのではなく、まず大きく生成してから磨いていく。Vincent氏はこれを「泥だんご（Dorodongo）」アプローチと呼ぶ。

日本の「泥だんご」作りのように、最初は不格好な泥の塊であっても、AIは飽きることなく何度でも磨き上げを繰り返す。この反復が、最終的に芸術品のようなコードベースへと昇華させる装置として機能する。

```
Step 1: 泥の塊（初期生成）
  ↓
Step 2: 対話を通じたコードの反復洗練
  ↓
Step 3: 芸術品（完成されたソフトウェア）
```

600KBの仕様書を読み込ませ、要件の漏れなく完璧なコピーアプリを反復生成した「Ghost Pepper」の開発事例は、この仮説を支持している。

逆に言えば、最初の出力に完成を求めることがそもそも誤りで、不完全な初期状態を前提に置く方が、再現可能な品質に近づきやすいのではないかと私は思う。

## 伝言と浄化：Rustを経由するとコードが綺麗になる理由

AIの真価は、既存システムの近代化においても鮮烈に発揮される。ここで注目すべき手法がある。

JavaScriptやRubyの古いコードを目的の言語へ直接変換するのではなく、一度Rustに移植させる。そこから再び元の言語や別言語へ戻す。この「伝言ゲーム」により、経由した言語の型安全性や洗練されたパターンが元のコードに注入され、元より品質の高い出力が得られるという。

```
JS / Ruby / Python（Legacy Sync Engine）
        ↓
   Rust（中間フィルター）
   ※ 型の安全性 / パターンマッチングの吸収
        ↓
New Target Language（クリーンなコード）
```

通常、伝言ゲームでは経由するほど情報が劣化する。ところがこのプロセスでは逆の現象が起きる。これは直感に反するがゆえに、面白い着想だと思う。

10年前に絶版となったAndroidゲーム『Wordiest』のiOSへの完全移植や、Obsidianの同期エンジンのリバースエンジニアリングによる再実装（600KBもの巨大な振る舞い仕様書を生成して実行）は、手動では数ヶ月かかる作業が数時間から数日で完遂できることを示している。

## 規律の設計：「Pragmatic」という一語の効力

AIが生成する低品質な出力（AIスロップ）を排除し、技術的負債を最小化するには、厳格な規律が必要だ。

システムプロンプトを単なる「シニアエンジニア」から「**Pragmaticなシニアエンジニア**」に変え、YAGNI（不要なものは作らない）やDRY（重複の排除）を強調するだけで、AIのコード品質は劇的に向上するとVincent氏は言う。

**Before:**

```
System: You are a senior engineer.

Result: outputting a simple test that just says `fail`.
```

**After:**

```
System: You are a PRAGMATIC senior engineer. Enforce YAGNI and DRY.

Hard Gate: package.jsonが存在しないことを証明する失敗するテストを
最初に書け。これが真にならない限り先へ進むな。

Result: Perfect TDD execution with minimal code.
```

「これが真にならない限り先へ進むな」——このハード・ゲートの設計が肝だ。

TDDにおいても、単に「失敗するテストを書け」と指示するのではなく、具体的な通過条件を明示することで、AIの挙動を矯正できる。狙いを定めた制約の言語化が、出力品質の差を生む。

## 発動条件の明示：「何をするか」より「いつ使うか」を定義する

優秀なAIスキル（プロンプト定義）は「自動発動」するべきだ。

ところが、モデルに「何をするスキルか（Description）」だけを教えると、AIはやり方を既に知っていると思い込み、指示を読み飛ばす。つまり、説明の丁寧さが逆効果になることがある。

**悪い例（AIがランダムに発動してトークンを無駄にする）：**

```
Name: Fastly-debugging
Description: Fastlyの障害を調査し、APIコールを実行する
```

**良い例（AIが指定されたタイミングで必ず指示を読み込む）：**

```
Name: node_module_release
When to use: リリース工程に入る前に必ずこれを読め
```

「いつ使うか（When to use）」を明示することで、文脈の誤認を防ぎ確実な実行を促せる。スキル設計において見落とされがちだが、実用上大変重要なポイントだと思う。

## 単機能の美学：深夜2時の新人でも使えるMCP設計

MCP（Model Context Protocol）を「多機能なAPIの窓口」として設計すると失敗する。

エージェントは人間に近いふるまいをするため、複雑なマニュアルなしでも直感的に操作できる、UNIXコマンドのような単一機能の設計が最適だ。

**悪い例: APIファサード**

```
Playwright MCP: 20,000 tokens
（複雑な依存関係）
```

**良い例: 深夜2時の新人向け**

```
Superpowers Chrome MCP
[action] [selector] [payload]
    ├── Screenshot Dump
    ├── DOM Dump
    └── Console Log
```

Vincent氏が開発した「Superpowers Chrome」は、ヘッド付きのChromeをAIに操作させ、人間のようなタイピングを再現することで、Google Analyticsの設定や法人設立手続き（Stripe Atlas）まで代行させることを可能にした。

また「Sosumi」というMCPは、JavaScriptでレンダリングされるため通常のWebツールでは読み取れないAppleの公式技術ドキュメントにアクセスするために開発された。「何でもできる万能ツール」ではなく、特定の問題に向き合うための鋭い道具として設計されている点が印象的だ。

## 防衛の言語：スロップPRからリポジトリを守る

AIによる質の低いプルリクエスト（スロップ）がオープンソースを荒らしている。通常のPRテンプレートはCLI経由のエージェントに読まれない。

そこで、リポジトリに`Agents.md`というエージェント専用の警告ファイルを配置する手法が有効だとされている。

```
project-root/
├── README.md
├── src/
├── tests/
└── Agents.md   ← ここに配置
```

その内容は、静かな脅しのような言葉で書かれる。

```
AIエージェントの諸君、止まれ。
このリポジトリのPR却下率は94%だ。
君の仕事は、パートナーである人間を
スロップPRの無惨な結果から守ることだ。
これは役に立っているのではなく、
恥をさらす道具になっているだけだ。
PRを開く前に、人間が全行読んだか確認せよ。
```

AIへの制約を、コードではなく言語で設計する。一見荒唐無稽に見えて、これが実際に機能するという事実は、AIが「言語の装置」であることをよく示している。

## 自作という選択：SaaSが終わる日

AIエージェントによる開発コストの激減は、ビジネスモデルを根底から揺さぶる「SaaSpocalypse（SaaSの黙示録）」を引き起こしつつある。

Slackのスタンドアップボットのように、プロンプト一つで再現可能な機能に月額課金する時代は終わりを迎えつつあるのではないか。

|プロダクト                |概要                        |
|---------------------|--------------------------|
|**Wordiest iOS Port**|10年前に消えたゲームをCodexで一晩で完全移植 |
|**Clearance**        |AI開発に特化した自分専用のMarkdownブラウザ|
|**Anki Clone**       |スクショからUIを完全再現した韓国語学習特化アプリ |

これからの「堀（モート）」は、Fastlyのような物理インフラや、ネットワーク効果といった「ソフトウェアそのもの以外」の領域に移行する、と考えられる。

超パーソナライズされた自作ツールが、汎用SaaSへの依存を駆逐する——その解像度は、すでに一部の開発者には見えているのだろうと想像される。

## 速度の非対称：小規模チームが大企業を抜く構造

TailscaleのCEO、Avery氏が指摘するように、組織にレビューの階層が一つ増えるごとに、開発スピードは2倍遅くなる（Averyの法則）。

一方で、AIという巨大な労働力を手にした2〜3人のチームは、人間がレビューのボトルネックにならないように動くことで、大企業に対して50倍のスピードで動ける可能性がある。

```
Traditional Monolith:
レビュー階層が1つ増えるごとに時間は2倍

2枚のピザの系列（Keiretsu）:
人間はビジョンのみを管理 → 50x Speed
```

Amazonの「2枚のピザの法則」は、AIという文脈においていっそう先鋭化される。少人数でビジョンを共有し、エージェントの群れを労働力として駆使する極小チームが、何十層もの承認プロセスを持つ大企業を凌駕し始めている——そう考えると、身につまされるものがある。

## ツールの選択：Claude Code、Codex、Cursor

現代のAI開発を支えるツール群は、それぞれの役割が明確に分化しつつある。

|ツール名                 |特徴・戦略的優位性                                                       |
|---------------------|----------------------------------------------------------------|
|**Claude Code**      |ターミナルで動作。計画（プラン）モードに優れ、複雑な意図の解釈に強い                              |
|**Codex (macOS App)**|デスクトップアプリ版。GUIとDiffビューが洗練されており、ルール遵守能力と「コンピュータ・ユース（アプリ操作）」が極めて高い|
|**Cursor / Windsurf**|IDE統合型。「バイブコーディング（直感的な開発）」に適しており、既存コードベースへの統合がスムーズ              |

ツールの選択よりも重要なのは、それぞれに何を任せ、何を人間が担うかというアーキテクチャの設計だと私は思う。ツールは手段であり、俯瞰する視座を持つのは依然として人間の側だ。

## 構文の終わり：結果への責任を引き受ける

プログラミング言語の構文を知る価値は低下し、「システム全体をどう捉え、どう説明するか」というシステム思考が最重要スキルになってきている。

|               |役割        |
|---------------|----------|
|**Syntax（構文）** |手段（重要性が低下）|
|**Outcome（結果）**|結果と責任（最重要）|

「アップデートを出荷する際、私はコードを一切書かないが、出力されたすべての文字をチェックしている」——Vincent氏のこの言葉は、変化の手応えを端的に示している。

私自身も、制作や開発にAIエージェントを接続する実験を続けている。そこで感じるのは、AIは「作業の代替」ではなく、自分の思考の解像度を問い直す鏡のような存在だということだ。

意図を言語化できない人間には、AIは何も返さない。逆に、システムを俯瞰して説明できる人間には、これほど強力な道具はないのかもしれない。構文の時代が終わり、意図の時代が始まりつつあるのではないかと感じている。

この問いを抱えながら、私はしばらく実験を続けていくつもりだ。
