# Building the Agent Nest

This week I kept circling the same strange question: if I can already reach my machines from anywhere, what does a *real* AI workspace feel like? The answer turned out to be less about convenience and more about trust, boundaries, and memory — the stuff that makes a system feel alive instead of merely connected.

## Highlights

- I designed a zero-trust remote workspace around **Tailscale + SSH + tmux**, so Claude Code can keep running on my home Ubuntu box while I hop in from Windows, Mac, iPhone, or iPad.
- I split work into role-based agents: a **secretary agent**, a **research agent**, and an **operator agent** that stays on the work machine and never leaks work data out.
- I formalized a shared handoff system with **`handoff.md`**, so every session can be resumed without reconstructing the world from scratch.
- I also pushed the Agent-Workplace architecture toward a more AI-native shape: **JSON/JSONL for machines, Markdown for humans**, with log compression and task sync scripts to keep context lean.
- On the model-switching side, I mapped out four ways to choose between **Claude models and local LLMs** depending on sensitivity, cost, and task type.

## What I Worked On

The biggest thing I worked on was the workspace itself. I stopped thinking of Claude Code as “a tool on my laptop” and started treating it like a persistent resident of my home Ubuntu machine. From there, I can reconnect from my work PC, personal Mac, iPhone, or iPad without punching holes in my router or dragging work data across devices. The decision to use **Tailscale** felt important because it made the whole system feel private by default, and **tmux** became the missing piece that lets a session survive disconnects instead of dying every time I close a terminal.

What I liked most about this design is that it forced me to separate *access* from *state*. Access is just SSH over Tailscale. State is something else entirely: it lives in `handoff.md`, in `current-tasks.md`, and in the discipline of having agents read and write the same small set of files. That tiny habit changes the whole experience. Instead of asking “what was I doing again?”, I can ask the system to tell me.

The other theme was specialization. I’m increasingly convinced that “one agent to do everything” is a trap. The secretary agent can live comfortably on my home machine and handle Google Calendar, Gmail, and chat. The research agent can browse and organize information. The operator agent stays on the work machine and touches only work data. That split feels more honest than pretending one context window should safely span all domains.

I also spent time on the underlying Agent-Workplace architecture. The move toward **`spec.json`**, **`config.json`**, **`current.jsonl`**, and generated summary/recent layers is not glamorous, but it’s the kind of unsexy design that keeps a long-running system from collapsing under its own memory. I’m starting to see logs less as notes and more as a compression pipeline for attention.

## Decisions & Tradeoffs

The biggest tradeoff was security versus convenience. I chose **Tailscale + SSH** over router port forwarding because I wanted zero-trust behavior and cross-platform simplicity, even if it means one more service to maintain.

I also made a hard call to keep **secret keys off GitHub and Google Drive**, even private repos. That constraint is annoying, but it’s the kind of annoyance that protects the whole design.

For persistence, I chose **tmux** over something more abstract because I wanted a battle-tested, inspectable session layer. It’s not fancy; it’s dependable.

On the data side, I accepted that **Markdown is for people, JSON/JSONL is for agents**. That means more conversion tooling, but it pays back in lower token cost, easier parsing, and less “vibes-based” context drift.

## Progress Since Last Time

Compared with last week, the work moved from abstract memory theory into actual workspace architecture. I’m no longer just saying “agents need memory”; I’ve defined where that memory lives, how it gets handed off, and which files are authoritative.

The Claude Code model-switching work also matured. It’s no longer just a list of model preferences — it’s a practical operating system: per-project locking with **direnv**, shell aliases for quick switching, a shared **Makefile** for team consistency, and an interactive selector for ad hoc use.

What’s still unresolved is implementation. I have the map, but not yet the fully wired room: the real `agent-context` repo, the MCP connections, the work-machine setup, and the habit layer that makes `handoff.md` actually stay current.

## What’s Next

Next I want to make the workspace *feel* real in daily use. That means:

- building the actual **private context repo** and wiring the handoff flow end-to-end
- testing whether the **secretary / research / operator** split survives a real week of use
- automating log compaction so the system keeps its memory without bloating its attention
- trying the model-switching patterns on real tasks and seeing when local models are “good enough”
- stress-testing the whole design with a long session, because that’s where the hidden failures usually show up

I’m especially curious whether this setup can become less like a collection of scripts and more like a workspace with personality — something that remembers, but doesn’t hoard.
