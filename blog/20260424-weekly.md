# 境界を設計する

今週は、ただ便利な仕組みを組む週ではなく、「どこまでを自分の作業空間に含めるか」をかなり真剣に考えていた。AIエージェント、端末、職場データ、個人データ、そしてモデルの切り替えまで、全部がひとつながりに見えてきて、境界の引き方そのものが設計だと実感した。

## Highlights

- 自宅Ubuntuを中核にして、職場PC・私用Mac・iPhone・iPadから安全に再開できるClaude Code作業環境を設計した
- `tmux` と `Tailscale + SSH` を組み合わせ、切断されてもセッションが生き続ける運用に決めた
- `handoff.md` を中心に、エージェント間の状態引き継ぎを「タスクだけ」に限定する方針を固めた
- Claude Codeのモデル切り替えを、`direnv` / shell alias / `Makefile` / wrapper script の4パターンで整理した
- Google Chatの送信ログをGoogle Spreadsheetへ自動記録する `Google Apps Script` の仕組みを作った

## What I Worked On

いちばん大きかったのは、Claude Codeを「どの端末からでも続きができる作業机」として再設計したことだった。自宅Ubuntuを常時稼働の母艦にして、職場PCや私用Mac、iPhone、iPadから遠隔で入る。ここで大事だったのは、単にリモート接続できることではなく、職場データは職場の外に出さないまま、私用側の文脈だけを持ち運べることだった。だからこそ、接続はゼロトラスト前提の `Tailscale + SSH` にし、セッション維持は `tmux` に寄せた。SSHが切れてもClaude Codeがそのまま残り、別端末から再アタッチできる感覚は、作業の連続性をかなり変える。

この設計を考えていて面白かったのは、技術選定がそのまま倫理の線引きになっていたことだ。秘密鍵をどこに置くか、職場Google Workspaceに私用鍵情報を入れない、Google DriveやGitHubに鍵を置かない、といった禁止事項は、単なるセキュリティ設定ではなく「混ぜない」ためのルールだった。`handoff.md` も同じで、状態は持ち越すけれど、職場データそのものは持ち越さない。引き継ぐのは判断の痕跡であって、機密の複製ではない。この分離が、思った以上に気持ちよかった。

もうひとつのテーマは、エージェントをどう分業させるかだった。秘書エージェントはカレンダーやメール、調査エージェントはWebや文書整理、オペレーターエージェントは職場マシン上で職場データのみ、という役割分担にしたことで、AIに「何でもやらせる」よりも、むしろ安心して任せられる形が見えてきた。エージェントが増えるほど、必要なのは万能性ではなく、境界を越えない専門性なのだと思う。

並行して、Claude Codeのモデル切り替えも整理した。`direnv` でプロジェクト単位に固定する方法は最も強く、`c-pro` / `c-think` / `c-fast` / `c-local` のような shell alias は日常運用に向いている。`Makefile` はチームで同じ振る舞いを保証したいときに強く、`claude-selector.sh` はその場で選びたいときに便利だ。ここでも結局、「どの作業に、どの知能を当てるか」を明示することが重要だった。モデルは性能表ではなく、用途ごとの道具箱として扱うほうがしっくりくる。

さらに、Google ChatのログをSpreadsheetへ自動保存する `Google Apps Script` も試した。これは地味だけれど、記録が流れて消えるチャットを、あとから見返せる形に変える小さなインフラだ。こういう仕組みは、派手ではないけれど、後で効いてくる。自分の作業環境を育てるとは、こういう「見えない保存」を増やすことなのかもしれない。

## Decisions & Tradeoffs

最大の決定は、接続を `Tailscale + SSH`、維持を `tmux` に寄せたことだった。ルーター設定不要で、OSをまたいで同じ仕組みを使え、しかもセッションが死なない。代わりに少しだけ運用は増えるけれど、長時間の作業を前提にするとこの安定性は大きい。

`handoff.md` は、状態管理をひとつのファイルに集約する代わりに、書く内容をタスクと決定に限定した。ログを全部入れたくなる誘惑はあったけれど、それをやると機密やノイズが混ざる。あえて薄く保つことで、引き継ぎの速度と安全性を両立させた。

モデル切り替えは、ひとつの正解に統一しないことを選んだ。project-level、task-level、team-level、interactive の4つを並立させるのは少し複雑だけれど、現実の使い方は一枚岩ではない。安全な固定と、柔軟な選択の両方が必要だった。

## Progress Since Last Time

前回は、AIエージェントの記憶やログをどう持続させるか、そして強い技術の危うい境界をどう見るかが中心だった。今週はそこから一歩進んで、それを実際の作業環境に落とし込んだ感じがある。抽象的だった「壊れない仕組み」が、`Tailscale`、`tmux`、`handoff.md`、`direnv`、`Makefile` といった具体物に変わった。

一方で、まだ未解決なのは、これらの設計をどこまで自動化できるかだ。特に `handoff.md` の更新を習慣ではなく仕組みにすること、MCP接続の詳細を詰めること、そして職場マシン側の運用をどこまで安全に簡略化できるかは、まだ手つかずに近い。

## What's Next

次は、この作業環境を「設計した」段階から「毎日使っても崩れない」段階へ持っていきたい。具体的には、`handoff.md` の自動更新プロンプトを磨くこと、MCP経由の秘書エージェント連携を安定させること、そして `Claude Code` のモデル切り替えを実際のタスクログに合わせて最適化することをやりたい。

それと並行して、Google Chatログのような小さな記録基盤を増やしていきたい。作業の履歴、判断の理由、端末をまたぐ文脈復元。こういうものが積み上がると、AIを使うというより、自分の思考の外部記憶を丁寧に育てている感じになる。今週はその入口に立てた気がする。
