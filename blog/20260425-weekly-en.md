# Rules, Then Flow

I spent this week moving through a set of decisions that all seemed to point toward the same mechanism: if AI is going to live inside my work, then the surrounding apparatus has to be explicit, small, and reproducible. The logs kept pulling me away from vague ambition and toward structure, and I think that shift matters more than any single tool choice.

## Prompt Files: Shared Memory

What stood out first was the decision to stop typing prompts directly and move them into prompt files. In other words, the goal was not just convenience; it was to make know-how portable, so the hypothesis can survive beyond one person’s hands.

I suspect this is why the `.github/` layout became so important: `copilot-instructions.md`, `rules/`, `prompts/`, `doc_format/`, and `structures/` form a small but legible mechanism. The interplay between `AGENTS.md` and `.github/copilot-instructions.md` also feels deliberate: one points Claude Code toward execution, the other points GitHub Copilot toward review and optimization, while `rules/` stays as the shared source of truth.

That said, the most interesting part is not the folder tree itself but the granularity it enforces. When each rule lives in one place, the system becomes more reproducible, and the friction of changing policy drops. I believe that is the real gain: not “better docs,” but a workspace that can absorb change without dissolving into inconsistency.

## JSON Over Markdown: Machine Texture

The move from Markdown to JSON/JSONL for agent-facing data felt like a quiet but strong insight. Markdown remains for humans, but the agent-side apparatus now prefers structured files such as `spec.json`, `config.json`, `current.jsonl`, `backlog.json`, and `active.json`.

One might wonder whether this is overengineering, but the logs make the mechanism clearer: Markdown becomes costly when analysis needs resolution, while JSON lets scripts and agents read only what they need. The three-layer log design — `current.jsonl`, `_recent.jsonl`, `_summary.jsonl`, plus `archive/` — is especially elegant because it turns memory into a compression pipeline rather than a pile of notes.

If so, the key is not storage volume but access texture. The system keeps the local tactility of recent work while pushing older context into compressed layers, and that seems to reduce token friction without erasing history. I think this is one of those cases where the macro design only makes sense after looking at the micro file boundaries.

## Model Switching: Sensitivity and Cost

The Claude Code model-switching design reads like a practical hypothesis about when cloud, local, and team-wide defaults should diverge. Direnv for per-project switching, shell aliases for personal speed, a shared Makefile for consistency, and an interactive selector for ad hoc choice together form a small ecosystem instead of one universal answer.

On the other hand, the security guidance is what gives the structure its weight. `ANTHROPIC_BASE_URL` has to be cleared when returning to cloud models, `.envrc` files must stay out of Git, and local use still depends on Ollama being present. The mechanism is simple, but the interplay of environment variables, convenience, and leakage risk is where the real design lives.

I suspect this is also why the recommended patterns are split by use case: daily coding, sensitive work, heavy debugging, light cost-saving work, CI/CD automation, and spontaneous choice each get a different path. That feels less like feature sprawl and more like a resolution map for matching task sensitivity to tooling.

## Workspace Roles: Separation by Function

The role split in Agent-Workplace continued to sharpen: Claude Code as the field agent, GitHub Copilot as the lab. I think this separation matters because it keeps the system honest about what each tool is for, instead of pretending one agent should span every domain.

The same logic shows up in the secretary / research / operator framing from earlier work. Access, state, and role are separated; local and global concerns are separated; explicit rules and implicit habits are separated. That decomposition makes the system easier to inspect, and perhaps easier to trust, because each part has a narrower contract.

The Google Chat logger spec and the Google Workspace-linked setup also fit this pattern. Apps Script, a dedicated GCP project, the Chat API, and a timed trigger create a reproducible logging apparatus, while the spreadsheet remains the human-readable surface. That contrast between digital capture and physical-feeling inspection is small, but I think it is telling.

## Management as Practice

The AI-agent development notes kept returning to management rather than typing. The idea that AI behaves like a capable but judgment-poor junior engineer seems useful to me, especially because it reframes the work as explanation, gating, and review.

One might argue that this is simply old project management in new clothing, and perhaps that is partly true. But the logs suggest a more specific mechanism: hard gates, explicit intent, smaller tasks, and repeated review are not overhead here; they are the interface. The “ultra-fast waterfall” and dorodango metaphors both point to the same thing — fast iteration only works if the system can be rebuilt without emotional attachment to the first draft.

I believe this is where the broader insight lands for my own practice. Whether I am shaping prompts, switching models, or compressing logs, I am really designing a workspace that can hold intention, reduce friction, and keep enough texture to remain inspectable. That seems like a useful universalization: good AI systems may not be the ones with the most power, but the ones with the clearest mechanisms for remembering, revising, and handing off.
