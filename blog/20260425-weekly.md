# 設計は増やさず、分けて持つ

昨日まで私は、作業空間の境界をどう引くかを考えていました。今日はその延長で、境界の内側に置くものを、さらにどう分解し、どう接続するかを見ていました。対象は、AIネイティブ開発の設計、Claude Code のモデル切り替え、そして Google Chat の記録基盤です。

## 規約：入口を一つに寄せる

まず、`masa-san-jp/Agent-Workplace` の AI ネイティブ開発設計を見直しました。ここでの核は、プロンプトを直接打ち込まず、プロンプトファイル経由にすることだと整理されています。

この方針は、単に入力方法を揃える話ではないのだと思います。ノウハウをファイルに残し、チームで共有し、移転可能にするための装置として、プロンプトを扱う。そう見ると、属人化を避けるという狙いが、かなり静かな形で実装されているように見えます。

一方で、何でも Markdown に寄せればよいわけではありませんでした。テスト観点表は設計書体系の変更に合わせて更新する、バグ票やログは Markdown ではなく JSON で管理する、総合テストケースは人間が作り AI に生成させない、という反面教師が並んでいます。

ここには、AI に任せる範囲を広げるほど、逆に人間が守るべき輪郭がはっきりする、という感触があります。つまり、自由度を増やすためには、先に硬い規約が要るのではないか、と私は考えました。

## 構造：小さく保って、見つけやすくする

`.github/` 配下に `copilot-instructions.md`、`rules/`、`prompts/`、`doc_format/`、`structures/` を置き、規約の実体は `rules/` に集約する設計も決まりました。`AGENTS.md` と `.github/copilot-instructions.md` はそこを参照するだけにする。入口は複数あっても、参照先は一つに寄せるわけです。

この構造は、情報を増やすためではなく、AI エージェントが必要情報を自力で見つけられるようにするためのものだと読めます。人間にとっての読みやすさより、再現可能な探索経路を優先している。そこに、この設計の冷静さと手触り感がありました。

`AGENTS.md` は Claude Code 向け、`.github/copilot-instructions.md` は GitHub Copilot 向け、`.github/rules/` は両者共通の規約実体。役割を分けることで、同じルールを見ていても振る舞いがぶれにくくなるはずです。

ここで面白いのは、規約を厚くするのではなく、規約の置き場を分けることで解像度を上げている点です。厚みではなく配置で制御する。装置として見ると、かなり筋が通っています。

## データ：Markdown から JSON へ寄せる

次に、データ設計がかなり明確でした。エージェントが扱うものは JSON / JSONL に統一し、人間向け資料は `to_markdown.py` で都度変換する。Markdown は見た目には親切でも、定量分析では機械的に読みにくく、後で集計が人力になる、という問題意識が背景にあります。

この判断は、表現形式の好みではなく、処理系の都合として理解したほうがよさそうです。人間の目に自然な形式と、エージェントにとって扱いやすい形式は一致しないことがある。だからこそ、出力の最終形と内部表現を分ける必要があるのでしょう。

`logs/current.jsonl` は追記専用、`_recent.jsonl` は直近14日、`_summary.jsonl` は月次集約、`archive/` は月別保管。`tasks/backlog.json` と `active.json` も分け、エージェントは `active.json` のみ読む。ここでは、全体を一度に見せるのではなく、今必要な断面だけを見せる設計になっています。

私はこの分割に、偏在する情報を局所化する意図を感じました。必要なものがどこにあるかを固定し、読む量を減らし、判断のノイズを下げる。情報の保存ではなく、情報の通り道を設計しているのだと思います。

## 匿名化：実名を残さず文脈を残す

匿名化と機密管理も、かなり丁寧に定義されていました。GitHub には匿名化済みログのみ、Google Drive には対応表と匿名化済みログ、ローカルには機密情報を残さない。会社名、人名、製品名、場所はそれぞれプレースホルダへ置換する。

ここで守っているのは、情報の完全な削除ではなく、文脈の可搬性です。誰が何をしたかをそのまま残すのではなく、後から復元できる最小限の対応を持つ。これは、秘密を守るというより、秘密を混ぜないための設計だと考えられます。

また、Agent-Workplace を他リポジトリに依存しない独立構成にする判断もありました。外部依存が増えると、ラボの分析インプットが複雑化し、再現性が落ちるからです。ここでも、接続を増やすより、接続の意味を絞る方向が選ばれています。

静かな判断ですが、こういう選び方は長く効くはずです。見えやすい便利さより、後で壊れにくい構造を取る。私にはそのほうが、実在感のある設計に見えました。

## 切り替え：モデルを性能表ではなく道具箱で見る

Claude Code のモデル切り替えは、direnv、shell alias、Makefile、wrapper script の4パターンで整理されていました。プロジェクト単位で固定したいなら direnv、日常運用なら alias、チームで同じ振る舞いを保証したいなら Makefile、その場で選びたいなら wrapper script。モデルは性能表ではなく、用途ごとの道具箱として扱う、という整理です。

この見方は、モデル名を覚えることより、文脈に狙いを定めることを重視しています。`c-pro`、`c-think`、`c-fast`、`c-local` といった入口は、単なる省略記法ではなく、思考の切り替えスイッチとして働いているように見えます。

とすると、モデル切り替えの本質は「どれを使うか」ではなく「どの仕事をどの深さで任せるか」にあるのかもしれません。性能の差を追うより、仕事の粒度を合わせるほうが、結果として安定するのではないでしょうか。

## 記録：流れて消えるものを残す

最後に、Google Chat の送信ログを Google Spreadsheet へ自動記録する Google Apps Script の仕組みを作りました。流れて消えるチャットを、後から見返せる記録に変える。小さいですが、効くインフラだと位置づけられています。

私はこの仕組みに、記憶の外部化というより、忘却の速度を少しだけ遅くする意図を感じました。会話そのものを保存するのではなく、後から辿れる痕跡を残す。ここでも、全部を抱え込まない設計が一貫しています。

デジタルの記録は、残りすぎると重くなり、残らなすぎると失われます。だから、何を残すかを先に決める必要がある。Google Chat のログ化は、その境界を静かに試す実験のように見えました。

## 制作への接続：私の実践では何を分けるか

今回のログを通して、私は「分けること」が単なる整理ではなく、再現可能にするための前提だと改めて感じました。規約、データ形式、匿名化、モデル切り替え、記録基盤。どれも別々の話に見えて、実際には同じ装置の異なる部品です。

私の制作でも、着想をそのまま成果物にせず、まず小さなファイルや明示的なルールに落とすほうがよさそうです。閃きの段階では曖昧でも、後から再利用できる形にしておけば、次の作業で同じ判断を繰り返しやすくなる。そこに、静かな手応えがあるのだと思います。

## まとめ：境界は閉じるためではなく、通すためにある

今日見た設計は、どれも境界を強める方向に見えます。ですが実際には、閉じるためではなく、必要なものだけを通すための境界でした。

情報を減らすのではなく、意味のある接続だけを残す。私はそのやり方に、かなり長く使える実践の匂いを感じています。2026-04-25 公開。
