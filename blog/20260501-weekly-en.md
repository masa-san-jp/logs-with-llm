# Hooks, sync, and markers

A few design choices this week kept circling the same question: how do I make agent work reproducible without turning it into a heavy apparatus? The logs suggest that the answer may be less about adding power than about making the mechanism visible, small, and easy to restart.

## Mechanism: Sync as a bounded system

I suspect the strongest thread is the multi-device sync design for the agent management repository. Instead of treating synchronization as a vague background comfort, I decomposed it into three parts: `agent-sync.sh`, `.claude/settings.json` hooks, and the manual `/sync` command.

That decomposition matters to me because it turns a hidden process into a legible mechanism. `SessionStart` pulls, `SessionEnd` pushes, and the script itself stays strict with `git pull --ff-only`, scope-limited staging, and a fixed `chore(sync)` commit message. In other words, the system keeps its resolution high by refusing to do too much at once.

## Friction: When automation needs restraint

On the other hand, the interesting part is not that the sync works, but that it is designed to fail quietly. The script checks for local changes before pulling, skips unexpected repositories, and exits without blocking the session if something goes wrong.

I think this is a useful hypothesis about automation in general: friction is not always a defect. Sometimes friction is the mechanism that prevents a local convenience from becoming a global mess. The interplay between convenience and safety felt especially clear here, because the apparatus is optimized for narrow, explicit behavior rather than broad magic.

## Markers: Two-phase agent startup

The daily startup design for sub-agents pushed the same idea further. Since Claude Code hooks cannot directly launch sub-agents, the solution became a two-phase mechanism: the hook creates `.pending` markers, and Claude itself reads those markers and starts the relevant agents.

If so, the key insight is that responsibility can be split without loss of reproducibility. The hook does only shell work; the main Claude process does the actual agent invocation. That separation reminds me that a system often becomes more trustworthy when it accepts its own limits instead of pretending to be seamless.

## Local state: Private timing, shared structure

One might wonder whether the `.gitignore` decision is a small implementation detail, but it seems to carry the texture of the whole design. By keeping `.pending/` and `.last-run-*` local to each device, each machine computes its own overdue state independently.

That local/global split feels important: shared Git history provides the global structure, while local timestamps preserve tactility and resolution at the device level. A new clone starts from “all overdue,” which is not elegant in a decorative sense, but reproducible in a practical one. I believe this is the kind of quiet design that makes a multi-device workflow survivable.

## NotebookLM: RAG as a working pipeline

The `notebooklm-mcp-cli` notes extended the same pattern into research. NotebookLM is already a RAG-like apparatus under the hood, but the CLI and MCP server make the mechanism explicit: create notebooks, add sources, query them, and then move across notebooks when the question needs broader resolution.

I was especially drawn to the pipeline framing: source collection, indexing, query, summary. That sequence gives research a reproducible texture, and it also keeps the boundary clear between what I have gathered and what I still need to infer. The constraint list matters here too—authentication, rate limits, non-official API dependence—because a system’s ubiquity is never the same as its reliability.

## Skills and prompts: Small tools, clearer intent

The survey of Claude Code skills pointed in a similar direction. `mattpocock/skills` stood out because it stays small, code-free, and composable, while still addressing common failure modes like misunderstood intent, missing shared language, and codebase drift.

That contrast with heavier frameworks seems useful to me. One path tries to inject an entire operating philosophy; the other offers a set of reproducible, low-friction skills such as `/grill-with-docs`, `/diagnose`, and `/tdd`. I think I am still leaning toward the latter kind of apparatus in my own practice: smaller units, clearer intent, and less hidden interplay.

## War termination: Constraints as self-restraint

The war termination report sits in a different domain, but the mechanism logic is surprisingly consistent. The core contrast between compromise peace and fundamental resolution depends not only on objective damage, but on subjective tolerance, future danger, and the constraints that shape leader judgment.

The UN resolution example makes that concrete. Resolution 678 functioned as a mechanism of self-restraint in the Gulf War, while the absence of a comparable constraint in Iraq removed that brake. In other words, external rules are not just legal texture; they are part of the decision apparatus itself. The same report’s focus on prospect theory, groupthink, and sunk costs also decomposes error into a few recurring elements: reference points, cohesion, and distorted risk perception.

## Practice: Designing for memory and revision

Across these logs, I keep seeing the same hypothesis: good agent work is less about raw capability than about memory, boundaries, and reviewable structure. Whether it is sync hooks, marker files, NotebookLM pipelines, or skill repositories, the useful systems are the ones that can be inspected, reset, and handed off.

That is probably the direction I want to keep practicing in: explicit mechanisms, small reproducible steps, and enough texture that I can still see how the parts interact. The broader lesson feels quiet but durable—complex work becomes manageable when its apparatus is decomposed, named, and made local where it should be local.
