# 装置としての同期と起動

今日は、複数端末間の設定同期と、サブエージェントの日次自動起動を、外部スケジューラなしでどう成立させるかを見ていました。対象は Claude Code の hook と GitHub 自動同期、そして `SessionStart` を起点にした二段階の起動設計です。私はこの一連を、単なる便利機能ではなく、運用のための装置として眺めていたのだと思います。

## 同期：端末ごとの状態をどう揃えるか

まず見ていたのは、複数端末で同じ設定を持つには何を揃え、何を揃えないか、という点でした。`agent-sync.sh` を軸に、`sync`、`pull`、`push`、`status` を分ける設計は、ひとつの処理に見えて実際には役割の分解だと考えられます。

`pull` は `git pull --ff-only` で前に進めるときだけ進み、ローカル変更があれば止まる。`push` は対象スコープだけを stage し、変更があるときだけ commit & push する。つまり、同期は全体を強く一致させるのではなく、壊さずに揃えられる範囲だけを揃える方が再現可能なのではないか、という仮説が立ちます。

一方で、同期の対象を広げすぎると、別の問題が出てきます。`git add .` のような雑な同期は、意図しない差分まで巻き込みやすい。だからこそ、対象ディレクトリを明示して scope-bound に扱う方が、静かな手応えに近い運用になるのだと思います。

## hook：自動化を壊さないための分離

`SessionStart` と `SessionEnd` に hook を置く設計は、見た目には単純ですが、内部ではかなり慎重です。hook の中でいきなりサブエージェントを起動せず、まずマーカーを作るだけに留める。そこから Claude 本体が `pending/` を読んで起動する、という二段階に分けていました。

これは、装置の中で責務を混ぜないための分離だと私は見ています。hook はシェルしか動かせないので、そこで `claude -p "/meta reviewer"` のような再帰を起こすと制御が難しくなる。逆に、シェルは「印を付ける」だけ、Claude 本体は「印を読む」だけにすると、流れが見えやすくなるのではないでしょうか。

この設計には、仕様と実装を分ける感覚がそのまま出ています。`.pending` がスペックで、Agent 起動が実装。つまり、先に状態を外部化しておくことで、起動の判断を再現可能にしているのだと思います。

## 起動：日次レビューをどう待たせるか

サブエージェントの日次自動起動では、クラウド cron、ローカル cron、`SessionStart` hook＋マーカーファイル方式を比較していました。結果として、コストと実現性の観点から最後の方式が選ばれている。ここでも、外部の強い仕組みを足すより、既にある入口を使って成立させる方が筋が良い、という判断が見えます。

`logs/reviews/.last-run-{role}` を見て 24 時間を超えていれば `.pending/{role}` を作る。すると次のセッション開始時に、Claude 本体がそれを拾って起動する。私はこの流れを、時間を使った自動化というより、時間を見張る装置として捉えています。

重要なのは、起動の瞬間を cron に任せるのではなく、作業の入口に寄せている点です。端末が起動していない時間は何もせず、起動したときだけ追いつく。これは完全な自動化ではありませんが、その不完全さがむしろ運用の身体性に合っているのではないか、とも感じました。

## ローカル状態：共有しないことで揃うもの

`.pending/` と `.last-run-*` を `.gitignore` に入れる判断も印象に残りました。普通なら共有したくなる状態ですが、ここでは共有しない方が正しい。端末ごとに独立して動くことで、A 端末の実行が B 端末の判断を誤らせないからです。

これは、偏在する状態をあえて局所化する設計です。共有すれば一見そろって見える一方で、実際には誤認が増えることがある。逆に、ローカルに閉じた transient state にしておくと、各端末が自分の時間で判断できる。私はこのあたりに、運用の解像度を上げるための小さな工夫が集まっていると感じました。

## 実践：私の制作にどう接続するか

私自身の制作でも、何かを自動化するときに、つい「一回で全部やる」方向へ寄せたくなります。ですが今回の設計を見ていると、まず印を付ける、次に読む、最後に動く、という段階化の方が、静かな実在感を持って回るのだと思えました。

特に、hook・マーカー・本体という三層は、私が作る文書管理や記録基盤にもそのまま接続できそうです。入力、判定、実行を分けることで、どこでズレたのかを見つけやすくなる。つまり、装置としての設計は、便利さより先に観察可能性を確保するためにあるのではないだろうか、と考えています。

## 小さな結論：同期は一致ではなく追従

今日の設計で見えてきたのは、同期とは全体を同時に一致させることではなく、必要な範囲だけを追従させることだ、という感覚でした。`ff-only`、scope 限定、`pending` の二段階起動、ローカル状態の分離。どれも派手ではありませんが、再現可能であることを優先したときに残る骨格なのだと思います。

私はこうした仕組みを、静かな熱量で積み上げていきたいです。自分の制作でも、見えないところで動く装置を丁寧に分解し、必要な接続だけを残す方向へ寄せていきます。公開日は 2026-05-01 です。
