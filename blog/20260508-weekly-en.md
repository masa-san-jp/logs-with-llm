# App Server, Barely Hidden

I’m taking this week as a small experiment in making the mechanism visible without turning it into a burden. The logs kept circling around a similar question: how can I keep the apparatus reproducible, local, and inspectable, while still letting it touch real work?

## Codex App Server: Personal, Not General

I think the `codex app-server` design is most interesting where it refuses to become a general service. The scope is deliberately narrow: a single user, a single subscription, a single machine, and stdio-only transport through JSON-RPC 2.0. In other words, the whole system is built around one hypothesis: if I bound the surface area tightly enough, I can keep the texture of use light while still getting a usable mechanism.

That showed up in the setup flow too. `setup.sh` and `setup.ps1` were meant to do the boring work in one command, but the design still kept explicit checkpoints: `codex login status`, `codex login`, temporary `codex app-server` startup, `account/read`, then wrapper generation. The friction here seems intentional, not accidental; it acts as a safety apparatus, especially because `~/.codex/auth.json` is plain-token storage and must stay out of Git and logs.

I suspect this is the same pattern I keep preferring in my own practice: make the path reproducible, but keep the boundary sharp enough that I notice when I cross it.

## API Reality: Spec as Hypothesis

What changed most for me was the reminder that a design spec is only a hypothesis until I hit the actual API. The logs show that the v1 plan and the real Codex CLI diverged in important ways: `thread/create` became `thread/start`, `items` became `input`, and some older methods disappeared and reappeared across versions. That mismatch is not a failure of the idea so much as a reminder about resolution.

I believe the useful move here is to separate three layers: command shape, transport shape, and behavior shape. If those are kept distinct, then API drift becomes legible instead of mysterious. The `codex-client.js` wrapper, the `codex-ask` CLI, and the setup scripts all reflect that decomposition, and the end-to-end checks — `PONG`, streaming counts, and simple arithmetic — gave the design a reproducible floor.

That said, the more important insight may be methodological: direct probing with `node -e`, `strings`, and intentional deserialize errors is not just debugging, but a way of learning the mechanism’s actual granularity.

## Review as a Second-Opinion Apparatus

The `codex-review` skill extends the same logic into critique. It treats Codex as a second-opinion reviewer for Claude Code diffs, and it insists on a strict review prompt, read-only sandboxing, and explicit categories for critical, warning, suggestion, and good findings. I like that it decomposes the review into file, line, issue, and fix; the granularity makes the output more reproducible and less performative.

On the other hand, the skill also refuses to forward findings blindly. The output is supposed to come back through Claude’s own judgment, which feels like an important interplay: external model, local model, then human decision. One might wonder whether that extra step is slower than necessary, but I think it protects the texture of review from becoming a black box.

## Resolve Safety: Power with Boundaries

The `resolve-mcp-safe` work pushed the same mechanism-first habit into a different domain. The design is careful about the difference between safe, standard, and full operations, and it makes the audit trail explicit with JSONL logging, Pydantic validation, allowlists, and AST checks. The apparatus is not trying to be maximal; it is trying to be legible under pressure.

I found the contrast between read-only resources and destructive tools especially useful. `resolve://status`, `resolve://project/current`, and `resolve://timeline/current` are small windows into state, while rendering, deletion, and save operations sit behind stronger gates. That digital/physical contrast — inspect versus alter — feels like a general pattern I want to keep reusing.

## Local Pipelines: Resolution Before Scale

The browser-operation and NotebookLM pipeline specs kept returning to the same theme: start with a small, explicit pipeline, then expand only where the mechanism proves itself. Collection, diffing, structuring, storage, notification, and retrieval are all separate layers, and each layer gets to fail on its own terms. If so, the system can stay local in one place and global in another, instead of pretending those are the same thing.

I think this is also why the Tokyo cinema design felt convincing. GAS handles the steady, repetitive collection; Claude Code helps generate site-specific parsers; Sheets stores the master state; Gmail only speaks when there is a real diff. The ubiquity of the web is not the same as reliability, so the pipeline leans on structure rather than wishful automation.

## Practice: Small Mechanisms, Shared Logic

Across Codex, Claude Code, Resolve, NotebookLM, and the browser pipelines, I keep seeing the same interplay: explicit boundaries, local state where locality matters, and reproducible steps where repetition matters. The vocabulary changes, but the mechanism is similar.

I want to keep building in that direction in my own work: separate transport from behavior, keep destructive actions behind human approval, and prefer small tools whose texture I can still feel. Perhaps that is the quiet lesson of this week — not to make everything simple, but to make the system honest enough to inspect.
