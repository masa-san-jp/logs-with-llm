# Rooms Between Sessions

I spent this week looking at a few different kinds of “between-ness”: between terminals, between systems, and between intention and actual mechanism. The common thread, I think, was not automation itself but the texture of coordination—what becomes reproducible, what stays local, and where friction is worth keeping.

## IPC vs. mailbox: the boundary matters

I started with `peer-inbox`, because I wanted a way for Claude Code sessions in separate terminals to hand work to one another. At first glance, `claude --remote-control` looked like a possible apparatus, but the docs made it clear that it is for mobile/desktop handoff, not local IPC.

That distinction felt important. In other words, the mechanism was not “send a message to another process on the same machine,” but “poll Anthropic’s API from a local process,” which is a very different resolution of the problem. The mailbox pattern from third-party posts also looked tempting, but the official surface did not support it in the environment I checked, so I treated it as hypothesis rather than fact.

## File-based inbox: small, local, reproducible

So the design settled on a file-based inbox under `~/Dev-Data/dev-data-local/<project>/shared/peer-inbox/`. Each peer gets its own `inbox.jsonl`, messages are append-only JSONL, and `read` just marks state without changing the underlying texture of the log.

I suspect this is one of those cases where simplicity is not a compromise but the actual mechanism. Append-only storage removes lock contention, keeps the granularity legible, and makes the whole thing reproducible across sessions without a server, daemon, or DB. That said, it also means the system is honest about its limits: it is a local coordination layer, not true autonomy.

## Attention hooks: notification is not control

The macOS `osascript` notification is a good example of the interplay between signal and action. It can nudge attention—`display notification "new message from $PEER_NAME"`—but it cannot replace the explicit `check` step, and I think that is the right friction to preserve.

One might wonder whether more automation would feel smoother, but I’m not sure smoothness is the same as reliability. The design keeps the mechanism inspectable: a message exists in a file, a peer can check it, and the notification is only an attention hook. On Linux and Windows, the inbox still works; only the tactile alert disappears.

## Conflict detection: passive systems can still be active

In contrast, `TermGuard` lives on the passive side of coordination. It does not exchange messages; it watches for overlap in `cwd`, files, ports, branches, and jobs, then warns when two terminals start to share the same resource.

What I found useful here was the decomposition into three elements: session identity, resource normalization, and change-based notification suppression. PID liveness checks with `os.kill(pid, 0)` keep stale state from accumulating, absolute-path normalization gives the data some resolution, and `last_signature` prevents repeated noise. The result is a small apparatus that feels almost invisible until it matters.

## White-hack as a control loop

The white-hack pipeline had the same shape at a larger scale. It is not an attack system; it is a governance mechanism for surfacing internal risk, turning findings into remediation tasks, and feeding them back into management judgment.

I think the recurring pattern is that a system becomes useful when it can separate inventory, hypothesis, validation, and revalidation. In the white-hack design, Google Workspace external sharing, admin rights, leaver accounts, sensitive-data storage, endpoints, and vendors are all distinct layers. That layered resolution matters because “security” as a word is too broad to inspect; the mechanism only becomes legible when broken into concrete ledgers and weekly/monthly/quarterly loops.

## Grants and pipelines: from search to state

The subsidy and grant pipeline followed the same logic, just in a different domain. Instead of treating grants as isolated applications, I framed them as a portfolio: Phase 0 builds the company master, Phase 1 gathers public calls, Phase 2 screens fit, and the later phases manage submission, post-adoption reporting, and the next cycle.

The interesting part, for me, was the insistence that the organization’s own state comes first. In other words, the mechanism is not “find money on the internet,” but “keep the company, projects, and evidence in a form that can be matched reproducibly against changing opportunities.” That is a local system with global reach: the web is pervasive, but the pipeline still depends on well-structured internal data.

## One practice, many systems

Across these logs, I kept seeing the same interplay: explicit boundaries, small units, inspectable state, and a refusal to confuse convenience with control. Whether the object was a peer inbox, a conflict detector, a white-hack loop, or a grant pipeline, the useful question seemed to be the same: what can I make reproducible without flattening the texture of the work?

That is also the question I want to keep asking in my own practice. If I can preserve the granularity of a mechanism while connecting it to a larger system, perhaps I can build tools that are quiet, local, and still genuinely useful.
