# 装置を分けると、見える

今週は、ひとつの完成形を作るというより、いくつもの装置をどう分けて、どう接続するかを見ていた週でした。外から見ると別々の話題に見えるものが、実際には同じ問いに触れているのではないか、と私は考えています。

## 収益化：作家性をどう装置化するか

まず目に留まったのは、AI × social-media sales agency のモデルでした。Growman でリストを作り、AutoLead AI でDMを送り、ChatGPT や Claude で投稿を分析して再現する、という流れは、営業と制作をひとつの装置として束ねた設計だと読めます。  
ここで重要なのは、単に「便利な自動化」を増やすことではなく、何を再現可能にし、何を人の判断に残すかを狙いを定めることだと私は思います。

一方で、このモデルは広く見せるほど薄まる、という逆の性質も持っていそうです。だからこそ、ニッチを絞り、既にお金が流れている市場に入ることが前提になっていました。広い市場を相手にすると、手触り感のある提案がぼやけ、DMも分析も散ってしまうのではないでしょうか。

私はこの構造を、制作にもそのまま持ち込みたくなりました。作家性を前面に出す場所と、再現性を前面に出す場所を分けること。Instagram と X の役割分担は、そのまま「表現の偏在」をどう扱うかの実験に見えました。ひとつの場で全部を担わせると重くなるので、場ごとに役割を分けるほうが、むしろ静かな手応えが出るのかもしれません。

## 検索：今ある情報にどう触れるか

Hermes Agent と xAI Grok OAuth をつないで、X 検索を直接使う話も、同じく装置の分解でした。検索は検索、推論は推論、と切り分けることで、リアルタイム性を外に引き出しているように見えます。  
`hermes -z` のような非対話型の使い方も含めて、私はここに「問いを投げる装置」と「結果を受け取る装置」を分ける意図を感じました。

つまり、情報収集をひとつの巨大な対話にしない、ということです。ログを残し、スキルとして固定し、必要なときに再呼び出しできる形にするほうが、解像度は上がるのではないだろうか。  
この考え方は、私が日々の実践でメモや手順を残すときの態度ともつながっています。再現可能であることは、冷たさではなく、むしろ次に触れやすい状態を作ることだと感じます。

## 制作物：名札ではなく、構造として残す

Codex custom pet の設計は、`pet.json` と `spritesheet.webp` だけで成立する、かなり明快な構造でした。見た目の豊かさの前に、まず manifest と atlas という二層に分ける。私はこの単純さに、むしろ実在感を覚えました。  
複雑なものほど、内部では単純な部品に分解されていることが多いからです。

さらに `INDEX.md` と `README.md` を分ける提案も印象に残りました。人間向けの説明と、AI向けの入口を分けることで、同じディレクトリでも読む主体ごとに最短経路を変えられる。これは、情報を一枚岩にせず、用途別に分岐させる設計です。  
私はこの分け方を、制作の現場でももっと使えるのではないかと考えています。ひとつの文書に全部を書き込むのではなく、入口・規則・履歴を分けるほうが、後から触るときの身体性が軽くなるからです。

## 安全性：見張る装置を先に置く

white-hack のパイプラインは、攻撃ではなく改善のための装置として組まれていました。資産、権限、リスク、是正、再検証を台帳でつなぎ、Google Workspace を起点に、見つかった問題を管理判断へ戻す。  
ここでは「何が危ないか」を探すことより、「危なさをどの粒度で記録し、どう戻すか」が中心でした。

この設計は、静かな監視に近いと私は思います。通知を鳴らし続けるのではなく、状態変化だけを拾う。過剰に反応するのでなく、変化点にだけ光を当てる。  
安全は追加機能ではなく前提条件である、という考え方は、制作でも同じです。自由に作るためにこそ、境界を先に決めておく必要があるのだと思います。

## 実装：動くものを小さく確かめる

vLLM とスマートフォン SSH の検証では、`gpt-oss-20b` は起動までは進むものの、OpenAI 互換 API としては `content` が返らず、Qwen2.5-7B-Instruct では `--enforce-eager` や `--max-model-len 1024`、`--gpu-memory-utilization 0.50` で正常に動いた、という切り分けがありました。  
私はここに、モデルの良し悪しというより、環境全体の相性を見る姿勢を見ます。ひとつの失敗を「全部だめ」とせず、条件を変えて確かめるのは、かなり実験的な態度です。

同じことは、AI エージェントの ROI を考える文脈にもありました。最初から自律化を目指すのではなく、問い合わせの下書きを作り、人間が承認する。つまり、判断と実行を分ける。  
この分け方は地味ですが、地味だからこそ再現可能です。大きな自動化より、小さな手応えの積み重ねのほうが、あとで装置として育てやすいのではないでしょうか。

## 私の制作へ：分解したものを、また接続する

今週のログを並べてみると、私はずっと「分けること」と「つなぐこと」の往復をしていたのだと分かります。営業と制作、検索と推論、入口と履歴、安全と実行、起動と推論。どれも、ひとつにまとめるより、役割を切り分けたほうが解像度が上がる場面でした。  
そして、その切り分けは冷たくなるためではなく、後でまた接続し直すための準備なのだと思います。

私自身の制作でも、今後はこの分け方をもう少し意識したいです。着想をそのまま作品にしないで、入口・記録・再利用の三層に分けること。そうすれば、静かな熱量を保ったまま、次の実験に移りやすくなるはずです。
