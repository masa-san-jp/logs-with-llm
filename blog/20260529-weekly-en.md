# Systems That Keep History

On a few recent logs, I kept circling back to the same question: what do I preserve, and what do I let be overwritten? The answers were spread across design tools, welfare training, AI wrappers, and database modeling, but they all seemed to point at one habit I want to keep sharpening in my own work.

## Claude Design: Setup Before Speed

Claude Design asks for more preparation than I first expected. It only works on the browser, requires Claude Pro or above, and on Enterprise the admin has to enable it; if I skip those conditions, the rest of the workflow simply does not start.

I think the more interesting part is the setup discipline. For an existing product, the guide recommends gathering 5–10 screenshots, especially the landing page, dashboard, button-heavy screens, error states, empty states, and even mobile views; for a new product, I would need to prepare the brand assets myself, at minimum colors and a font.

## Assets: Screenshots, Colors, and Scope

What stood out to me was the advice to avoid feeding Claude an entire monorepo. Instead, the guide suggests giving only the relevant subdirectory, such as `tailwind.config.js`, `/src/components/`, or `/apps/marketing/`, so the browser does not get weighed down.

In other words, the system works better when I reduce the input to the parts that actually carry design decisions. That feels familiar to me: I suspect good prompting is less about “more context” and more about the right slice of context.

## Handoff: Export as a Boundary

The `Export` button and the `Handoff to Claude Code` option make the boundary explicit. I can send the result either to a local coding agent or directly to Claude Code Web, which means Claude Design is not the place to polish every edge; it is the place to explore and then hand off.

That said, this boundary is also a kind of safety rail. If I try to finish the whole implementation inside the design tool, I may end up fighting the wrong layer of the system.

## Welfare Training: From Personal Tricks to Shared Rules

The disability welfare training log pushed the same idea into a different domain. The core problem was not just busyness, but a culture that stops improvement because everyone treats busyness as an excuse.

I believe that shift matters: the training is not about making one person clever with AI, but about building a shared operating rule. The concrete pieces were clear—allowed and forbidden input lists, allowed and forbidden tools, a shared place for prompts in Notion or Slack, a “ask if unsure” contact, and an incident reporting route.

## Prompt PDCA: One Prompt, Many Hands

The v2 revision made the organizational angle even sharper. There is a plan to appoint one or two AI leads, hold a monthly 15–30 minute prompt review, and standardize prompts so that everyone uses the same version instead of privately improvising.

One might wonder whether this is just process overhead, but the example says otherwise: a new service manager’s individual support plan time dropped from about 4 hours to about 2 hours when the prompt support was designed as a system. The value seems to come from repeatability, not from one-off brilliance.

## Agent-Aiko and nullevi03: Personality or Placement

The comparison between Agent-Aiko and nullevi03 made me think about where a system should live. Agent-Aiko protects and evolves a Claude personality across sessions, with `INVARIANTS`, snapshot-based session recovery, and a gradual `install.sh`; nullevi03 is a minimal Telegram wrapper that keeps Claude running with a `while true` restart loop.

Conversely, the two are not really competitors. Agent-Aiko defines what Claude is, while nullevi03 defines where Claude is used, and the logs suggest they could be combined: Telegram → Claude Code + Channels → Aiko persona.

## State Tables: Keeping Facts, Not Just Outcomes

The database design note made the same pattern concrete in PostgreSQL terms. Instead of storing mutable status columns like `status` or withdrawal flags, it recommends an event table such as `membership_events`, with `event_type`, `occurred_at`, and indexes on `member_id` and time.

I think this is the clearest version of the theme: store facts, derive state. A member’s current status can be computed from the latest event, while the history remains traceable for audits, incidents, and future changes.

What ties these logs together for me is a preference for systems that remember their own path. In my own practice, I want to keep asking whether I am storing a result too early, or whether I should preserve the events that produced it first. That question feels small, but it changes the shape of everything that follows.
