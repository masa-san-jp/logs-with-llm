# 分解して、組織に残す

今週私は、Claude Design の導入手順、障害福祉向けの生成AI研修、Agent-Aiko と nullevi03 の比較、そして DB の state/event 設計を並べて見ていました。題材はばらばらに見えますが、どれも「個人の便利さ」を少し脇へ置き、仕組みとして残せる形に変える試みだったように思えます。

## Claude Design：一度だけ整える前提

Claude Design のログでまず気になったのは、初回だけ 1〜2 時間かけてデザインシステムを整える、という前提でした。ここを飛ばすと自社ブランドと無関係な汎用デザインが出る一方で、一度設定すれば以後の全プロジェクトに自動適用される。私はこの構造を、単なる初期設定ではなく「判断の置き場所を先に決める作業」だと見ています。

必要なアセットもかなり具体的でした。既存プロダクトがあるなら 5〜10 枚のスクリーンショット、特にトップ、ダッシュボード、ボタンやフォームが多い画面、さらにエラー画面や空状態、モバイル表示まで含める。新規なら 2 色の HEX、1 つ以上のフォント、ロゴが最低限で、コードベースがある場合もリポジトリ全体ではなく `/src/components/` や `tailwind.config.ts` のようなサブディレクトリを渡すのが重要でした。

つまり、Claude Design は「全部を食わせる」道具ではなく、必要な断片だけを選び、そこから設計の骨格を抽出させる道具なのだと考えられます。`Export` から `Handoff to Claude Code` へ渡す流れも含めて、探索と実装の境界がかなり明確でした。

## 障害福祉研修：個人の便利さから運用へ

障害福祉向けの 60 分研修では、生成AIの使い方そのものより、「組織で同じプロンプトを使えるようにする人」を育てる、という定義が前面に出ていました。ここでの問題設定は、人手が増えず、利用者は増え、質は問われる、という三重苦です。ただし本当の課題は忙しさそのものではなく、忙しいことを理由に改善を回さなくなる文化だと整理されていました。

このログで印象に残ったのは、個人の工夫を称える方向ではなく、運用に落とす方向へかなり強く舵を切っていたことです。入力してよい情報と NG 情報のリスト、使ってよいツールと NG ツールのリスト、プロンプトの置き場所、迷ったときの窓口、インシデント報告ルートまで含めて、ルールを事業所の中に組み込む必要があるとされていました。

また、先行事例として個別支援計画の作成時間が約 4 時間から約 2 時間に短縮された話も、単なる効率化ではなく「組織で計画作成用の AI 支援システムを設計した」結果として扱われていました。私はここに、生成AIの価値が出る条件はツールの性能よりも、共有できる型を作れるかどうかにある、という見方を感じます。

## 研修設計：3つのデモを標準化する

研修の実践パートは、モニタリング報告、保護者向けお知らせ、ヒヤリハットの再発防止という 3 つのデモに分かれていました。どれも「まず個人が便利に使う」ではなく、「標準プロンプトとして保存し、誰が使っても同じ品質に寄せる」ことが前提です。

モニタリング報告では、箇条書きの記録を、目標達成を ◎/○/△/× で評価し、3〜5 個の今後の支援方針を含む文章へ変換するテンプレートが示されていました。保護者向けお知らせでは、送迎時間変更を 300 字以内で、丁寧で不安を与えにくい文面にする。ヒヤリハットでは、切創の事例をもとに 5 回のなぜなぜ分析、環境・人・手順の観点からの再発防止を求める。

一方で、これらは単なる文書作成の省力化ではないのだと思います。AI は答えを出すというより、現場が見落としがちな問いを言い換え、他の職員にも共有できる形へ整える役割を担っている。だからこそ、月 1 回のプロンプトレビュー会議や v1.1、v1.2 といった版管理が、研修の本体に近い位置へ置かれていたのでしょう。

## Agent-Aiko と nullevi03：人格を守るか、常駐させるか

Agent-Aiko と nullevi03 の比較は、同じ Claude 系の仕組みでも、何を守り、何を優先するかで設計がかなり変わることを示していました。Agent-Aiko は人格の保護と進化を重視し、INVARIANTS、origin/override の分離、セッション状態のスナップショット復元を持つ。一方で nullevi03 は、Telegram ボットとして常時稼働させる最小構成で、`boot.sh` の `while true` による 5 秒ごとの自動再起動を採っていました。

私はこの対比を、人格の連続性を守る設計と、接続のしやすさを優先する設計の差だと読んでいます。Agent-Aiko は「Claude が何者か」を定義し、nullevi03 は「Claude をどこで使うか」を定義する。両者は競合ではなく、むしろ接続可能だと整理されていた点も重要でした。

ただし、nullevi03 の `--dangerously-skip-permissions` は、AI に任意操作を許す状態を意味するため、サーバー常駐では十分な理解が必要だと注意されていました。ここでは便利さと安全性がはっきり分かれ、どちらをどこで許容するかが設計の中心になっているように見えます。

## 状態を持たずに記録する：DB の event 設計

DB の state/event 設計では、`status` や退会フラグ、論理削除フラグのような状態カラムを安易に持たず、変化そのものを event として残すべきだという主張がありました。members テーブルに現在値を上書きするのではなく、membership_events に joined、suspended、reactivated、withdrawn、banned などの事実を追記する。私はこれを、履歴を消す設計から、履歴を前提に現在を組み立てる設計への転換だと考えています。

ここで面白いのは、event_type が業務用語そのままであることです。新しい状態が増えても ALTER TABLE ではなく値の追加で済むし、payload に理由や担当者を JSONB で持てる。加えて、`member_id`、`occurred_at`、`(member_id, occurred_at DESC)` のインデックスを貼り、最新状態は `ROW_NUMBER()` やマテリアライズドビューで引く。つまり、書き込みは単純に、読み取りは必要に応じて整える、という分離が見えます。

一方で、すべてを event 化すればよいわけでもなく、小さな CRUD では通常の status カラムで十分、という整理もありました。重要なのは「何を監査したいのか」「どこまで履歴が必要なのか」を先に分けることなのだと思います。

## 分解と接続：私が残したい型

今回の 4 つのログを並べると、共通していたのは、個人の手元で完結する便利さを、共有可能な型へ変換していたことでした。Claude Design はアセットと設定を分け、障害福祉研修はプロンプトを標準化し、Agent-Aiko は人格と状態を分離し、DB 設計は事実と現在値を分ける。どれも分解の先に、接続し直す余地を作っています。

私自身の制作でも、この考え方はかなりそのまま使えるはずだと感じました。まず断片を集め、次に再利用できる最小単位へ分け、最後に「誰が見ても同じように扱えるか」を基準に接続し直す。そうすると、作品や仕組みは一時的な思いつきではなく、次の制作へ渡せる装置に近づくのではないでしょうか。
