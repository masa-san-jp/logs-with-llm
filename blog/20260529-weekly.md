# 装置を分けると見えるもの

Claude Design の使い方を追いながら、私はずっと「どこで分けると、どこで接続し直せるのか」を見ていました。  
前日までの議論では、営業や検索や安全性をひとつの流れとして束ねるより、役割ごとに装置を分けたほうが解像度が上がる、という感触が残っていたからです。

## デザイン：入口と出口の分離

Claude Design は、左がチャット、右がキャンバスという二分割の画面です。  
この構造だけでも、問いを投げる場所と、形を受け取る場所が分かれている、と私は読みました。

しかも初回にはデザインシステムのセットアップが要ります。ここを省くと汎用的な見た目に寄りやすい一方で、一度設定すれば以後の全プロジェクトに自動適用される。つまり、最初に少しだけ重い準備をしておくことで、その後の再現可能性を高める設計だと考えられます。

この話は、単なるUIの便利さではなく、入口と出口を分けることで制作の流れを整える装置の話でもあるのではないでしょうか。  
私はそこに、静かな手応えを感じました。

## アセット：観察を先に置く

セットアップ前に必要なのは、まず既存の画面や資料です。  
最低でも 5〜10 枚のスクリーンショットが推奨され、トップページ、ダッシュボード、ボタンやフォームが多い画面が優先される。補助としてエラー画面や空状態、モバイル表示もあるとよい、という整理でした。

ここで重要なのは、いきなり生成に入らず、観察を先に置くことだと思います。  
`Cmd + Shift + 4` や `Win + Shift + S` で切り取る行為は、ただの保存ではなく、画面を分解して手触り感のある部品として取り直す作業に近いのではないでしょうか。

一方で、GitHub リポジトリがある場合も、全体を渡すのではなくサブディレクトリに絞るべきだとされていました。  
`tailwind.config.js` や `tailwind.config.ts`、`/src/components/`、`/packages/design-system/`、`/apps/marketing/` のように、狙いを定める範囲を絞るほど、装置は軽くなる。モノレポ全体を渡してしまうとブラウザが重くなる、という注意は、技術的であると同時に、制作態度の比喩にも見えました。

## ブランド：色と文字の仮説

新規で始める場合は、2色の HEX、1つ以上のフォント名、必要ならロゴを用意する。  
まだ決まっていなければ、競合サイトから色を拾うか、初回プロンプトで配色案を提案させることもできる、という案内でした。

私はここで、ブランドを「完成した正解」ではなく、仮説として扱う姿勢を見ました。  
プライマリカラーとニュートラルカラー、フォントの系統、ロゴの有無。これらを先に決めることは、見た目を固定するためというより、後から崩れにくい基準を置くためだと考えられます。

たとえば Geist や Plus Jakarta Sans のようなモダン・テック系、DM Sans や Outfit のようなクリーン・ビジネス系、IBM Plex Sans や Source Sans 3 のような信頼感のある系統。  
こうした選択は感覚だけではなく、どの印象を前景化するかという設計の話でもあります。つまり、文字は情報の器であると同時に、プロダクトの身体性を決める部位でもあるのではないでしょうか。

## プロンプト：一度で作らず、往復で詰める

通常の流れは、仕様や PRD、スクリーンショット、口頭アイデアから新規プロジェクトを作り、デザインシステムを自動適用し、必要なコンテキストを足してプロンプトを入れる、というものです。  
効果的なプロンプトは 150〜300 語が最適で、目的、対象ユーザー、必要な要素、制約の4つを含めるとよいとされていました。

この条件は、長い説明を求めているのではなく、むしろ要素を分けてから結び直すための型だと私は受け取りました。  
目的と対象ユーザーと要素と制約。4つに分けると、何を変えたいのか、何を守りたいのかが見えやすくなる。逆に、全部を一息で書くと、判断の輪郭がぼやけるのではないでしょうか。

修正の仕方も印象的でした。  
チャットは構造や全体変更向きで、インラインコメントは特定コンポーネントの局所変更向き。ここでも、広い指示と細い修正を分けることで、制作の往復を再現可能にしているように見えます。

## 公開：Handoff で実装に渡す

仕上げは `Export` から `Handoff to Claude Code` を選び、ローカルの Claude Code か Web に送る流れです。  
デザインファイル、チャット履歴、README がまとめて渡るため、見た目だけでなく意図まで引き継がれる設計になっていました。

私はこの部分に、制作の出口をきちんと設ける意味を感じます。  
探索の段階で止めて、実装は別の装置に渡す。そうすることで、Claude Design は完成品を閉じる場所ではなく、次の工程へ接続するための中継点になるのだと思います。

エクスポート形式も `.zip`、PDF、PPTX、Canva、standalone HTML と分かれていました。  
ここでも、ひとつにまとめないことが重要です。用途ごとに出口を分けると、同じ素材でも異なる流通に乗せられる。私はその分岐の仕方に、装置としての誠実さを見ました。

## 運用：共有される改善の単位

コスト面では、Pro は週数回で上限に触れやすく、Max 5x や Max 20x のほうが業務利用には現実的だと整理されていました。  
この記述は単なる料金比較ではなく、どの頻度で使うかによって、装置の前提が変わるという話でもあります。

また、デザインシステム未設定で始めること、数値テキストを無確認で本番公開すること、細部まで Claude Design で作り込みすぎること、モノレポ全体を連携すること、デスクトップアプリで開こうとすることが避けるべき点として挙げられていました。  
要するに、探索は探索のままに留め、細部の確定や実装は別工程へ渡す。その線引きが、全体の解像度を保つのだと考えられます。

この整理は、私の制作にもそのまま接続できます。  
着想をいきなり完成物にせず、まず入口・記録・再利用に分けること。スクリーンショットや資料を集めて仮説を立て、チャットで広く詰め、最後に別の実装層へ渡すこと。そうすると、静かな熱量のまま制作を続けやすいのではないかと私は思います。
