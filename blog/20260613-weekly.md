# 複雑な対話とAI支援を「装置」的に再構築する試み  

2026‑05‑30、私は二つのAIプロジェクトの設計ミーティングに参加した。ひとつは自己コンプレックスを徹底的に掘り下げる対話プロトコル、もうひとつは汎用的なAIメンターキットの構築だ。両者は表面的には別領域に見えるが、共通して「人間の思考を機械的に切り分け、再構築する」という根底の問いに向き合っているように思える。

## 概念：対話プロトコルの装置的設計  
`執拗追及型コンプレックス深掘り対話プロトコル仕様書`（v1.0、2026‑05‑30）では、ユーザーが自らの欠陥認識に囚われたまま脱出できないことを防ぐため、**11段階**のフェーズと「回避検出」状態遷移が明文化されている。たとえば「欠乏感」や「恥」については、最低三回の追問が必須で、回避と判断されたら即座に同テーマへ「return‑to‑theme」する指示が入る。  
この設計は「装置」としての対話を意識したもので、**「容赦のなさは核に向かって」**というルールが核心部への到達を保証すると同時に、**「自己破壊的衝動」や「強制参加」**は除外条件として明示的に排除している。  
私の観察では、こうした硬質なフローは「慰め」ではなく「事実の格下げ」を目的としていると考えられる。すなわち、ユーザーが「自分は欠陥品だ」という結論に落ち込むのではなく、**「検証されていない仮説」**という形で認知をリフレームさせることが狙いだと推測できる。  

## 概念：スキルランタイム化したAIメンターキット  
同日、`Personal AI Mentor Kit` のミーティング記録（2026‑05‑30）では、当初の「RAG的参照」から **Claude Code 上の Skill Runtime 型** へと設計方針が転換したことが報告された。具体的には、**5つのコアスキル**（idea_review、writing_review、decision_review、contradiction_finder、next_action_planner）を Markdown/YAML のファイルで定義し、メンターは「人格の模倣」ではなく「判断の型」を呼び出す構造になる。  
重要な決定点としては、**「1領域だけでは評価AIになる」**という懸念から、**「3領域程度の専門性」**を持たせる方針が採られた点だ。これにより、単一視点のバイアスが減少し、ユーザーに対して**「具体的な次の行動」**を提示できるようになると予測できる。  
私の制作実践では、スキル単位での分割が「再利用可能な部品」になると同時に、**「判断の置き場所」**としての役割を明確にする手法として有効だと考えている。

## 概念：緊急時チャットボットの役割分担設計  
`危機対応チャットボット システムプロンプト`（2026‑06‑01）は、単独オペレーターが初期対応を行う際に **「最終判断者の代替ではない」** ことを前提に設計された。プロンプトは **9つの優先目標**（生命安全確認、被害拡大阻止、事実と推測の分離、暫定レベル設定、初回報告作成、記録場所確保、エスカレーション指示、即時実行タスクと保留タスクの区分、次回更新時間設定）を列挙し、質問は最大三つまでに制限される。  
この制約は「モデル性能に依存しない」アプローチを取るという議論と合致し、入力構造と手順設計で品質を担保しようとする試みだと推測できる。さらに、**「事実・未確認情報・推測」**を明確に分ける指示は、情報過多の危機現場での認知負荷を軽減する効果が期待できる。  
私の観点からは、こうした**「役割分担」**の明示は、実務における「装置」的な安全網を構築する上で不可欠であり、AIが「問い続ける」だけでなく**「決定しない」**という姿勢が、組織的なリスク管理に合致すると考える。

## 概念：ビジネスオフィス向けスキルフレームワークの実装  
`Business Office Skills Framework`（2026‑06‑05）は、データサイエンスの設計思想を汎用事務へ移植した体系である。ディレクトリ構造は **AGENTS.md**（薄いルーター）と **skills/*/SKILL.md**（詳細手順）に分割され、**docs/** に組織固有情報、**prompts/** に頻出依頼、**checklists/** に機械的検査が配置されている。  
特筆すべきは **「絶対ルール」**として「正本不変」や「機密情報は許可なく送信しない」ことが `AGENTS.md` に記載され、CI が機能しない事務領域でも**チェックリスト・承認フロー・テンプレ**で機械的検証を代替する方針が取られている点だ。  
この設計は「手順書だけで終わらない」ことを意識し、**「実装コードを書かない」**という Specification‑Driven の原則と共通する哲学を持つと見える。私の制作では、こうした**「分割と再接続」**の手法が、長期的に保守可能な**「装置」**として機能する可能性を示唆している。

### 私の制作への接続  
本稿で触れた四つの設計は、いずれも「人間の判断を部品化し、再構築する」という私の実践テーマと直結している。現在進行中のプロジェクトでは、対話プロトコルのフェーズをスキルランタイムとして抽象化し、緊急対応ボットの質問テンプレートをチェックリスト化することで、**「実験的装置」**を一つのコードベースに統合しようとしている。これにより、実験と実装の境界が曖昧になると同時に、成果物の再利用性が飛躍的に高まるのではないかと期待している。
