# Design Decisions for AI‑Assisted Dialog, Mentoring, and Crisis Systems  

The week’s work focused on formalizing interaction protocols, modular mentor architecture, and deterministic prompting to guarantee coverage and safety across diverse AI‑driven applications.

## Complex Interrogation Protocol: 24‑Hour Action Requirement  
The “執拗追及型コンプレックス深掘り対話プロトコル” (v1.0, 2026‑05‑30) specifies a twelve‑phase dialog that must culminate in a concrete 24‑hour action plan. Acceptance criteria demand explicit entries for deficiency, trigger scene, comparison target, envy, desire, shame, avoidance, core statement, and the actionable item. Phases enforce three iterations per major theme and include a defense‑detection table that maps evasive utterances (e.g., “but”, silence) to forced clarification prompts. The protocol also defines a shortened seven‑question variant for time‑constrained sessions. A key finding was that the protocol treats “high mental load” as a non‑stop condition, requiring continuation until the agent detects functional exhaustion.

## Personal AI Mentor Kit: Skill‑Runtime Architecture  
The mentor design (2026‑05‑30 meeting notes) abandons RAG retrieval in favor of a **Skill‑Runtime** model executed on Claude Code. The repository layout places `skill-registry.yaml` at the core, routing conversation context to one of five MVP skills (`idea_review`, `writing_review`, `decision_review`, `contradiction_finder`, `next_action_planner`). Each skill consumes knowledge cards (principles, questions, anti‑patterns) and outputs a structured response without direct document quoting. Expertise is limited to three domains per mentor to avoid single‑view bias, and the system records missing inputs in a session log after up to three clarification questions. The implication is that modular skill invocation yields deterministic advice while preserving source provenance.

## Self‑Confrontation Deep‑Dive Protocol: No‑Escape Agent Stance  
The high‑intensity self‑confrontation protocol (2026‑05‑30) defines an agent that never offers empathy and never permits the user to terminate on a “defective‑product” conclusion. Defensive behaviors (generalization, minimization, humor) trigger predefined counter‑questions that push the user toward concrete examples. Phase 0 selects a single complex; Phase 1 extracts factual trigger details; Phase 2 maps defenses to deeper probes; Phase 4 reframes the complex as an **unverified hypothesis** rather than a self‑judgment. The protocol mandates continuation despite emotional distress, treating tears or panic as evidence of proximity to the core. This design ensures that the session ends only when the user articulates a testable hypothesis and a first actionable step.

## Weekly Blog Generation: Source‑Card Prompt Constraints  
The `masa-san-jp/logs-with-llm` pipeline (2026‑05‑30) was updated to improve source coverage without increasing model size. New deterministic functions `extract_raw_excerpts()`, `build_source_cards()`, and `build_previous_style_capsule()` populate a “Source Card” section containing path, date, and filtered excerpts (headings, bold, numeric, code fences, 80‑500 char length). The prompt now enforces coverage of at least `min(4, number_of_sources)` logs, prevents dominance, and requires at least one concrete trace per source (e.g., `scripts/generate_weekly_blog.py`). The design relies on structural constraints rather than LLM inference to maintain balanced representation of multiple logs.

## Crisis‑Response ChatBot: System Prompt Safety Rules  
The emergency chatbot specification (2026‑06‑01) defines a system prompt that assists a single responder while explicitly prohibiting decision authority. Required placeholders include company name, first‑report channel, record location, and contact URLs. The prompt enforces a three‑question initial sequence (life safety, damage spread, responder safety) and mandates separation of confirmed facts, speculation, and unverified data. It also lists actions that the bot must never perform (e.g., assigning cause, posting private data, making legal decisions). Incident levels 0‑4 are encoded, and the bot must produce an early, incomplete report within 15 minutes, marking unknown fields as “unconfirmed”. The implication is that deterministic prompting eliminates reliance on model confidence for critical safety decisions.

**Synthesis** – The examined artifacts converge on a common engineering principle: deterministic structure and modularization replace heuristic inference to guarantee safety, coverage, and actionable output. Across dialog probing, mentor advice, self‑confrontation, blog synthesis, and crisis response, the designs encode explicit phase flows, skill registries, or prompt constraints that make system behavior auditable and reproducible. Future work should evaluate cross‑domain applicability of the Skill‑Runtime router and the defense‑detection tables within unified orchestration frameworks.
