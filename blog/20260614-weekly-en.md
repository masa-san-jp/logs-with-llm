# Integrating Persistent Personas, Event Sourcing, and Structured Crisis Workflows  

This week’s logs examine how a layered personality framework, an immutable event‑sourced state model, a skill‑runtime mentor kit, and a disciplined crisis‑response prompt each encode invariants separately from mutable orchestration. The convergence highlights a reusable pattern for AI‑augmented operational systems.

## Agent‑Aiko vs. nullevi03: Layered Personality and Process Management  

Agent‑Aiko implements a three‑layer architecture (distribution, development logs, local‑only) and protects core invariants with `INVARIANTS.md` and read‑only file permissions. Installation relies on `scripts/install.sh`, which preserves existing configuration while copying a template directory containing `persona/origin`, `persona/overrides`, and `hooks/`. Session continuity is achieved via a snapshot file (`session‑state/current.md`).  

nullevi03 adopts a single‑layer design that runs Claude Code in a continuous restart loop using `boot.sh` and the flag `--dangerously-skip-permissions`. It provides Telegram connectivity through the `--channels` plugin but lacks persistence of persona data. The contrast shows that explicit invariant files and staged installers (Agent‑Aiko) enable safe evolution of AI personality, whereas minimal process monitoring (nullevi03) prioritizes availability at the cost of security.

## Event‑Sourced Membership Model: Immutable Audit Trail  

The event‑design document defines a two‑table schema: a static `members` table (id, email, name, created_at) and an append‑only `membership_events` table (id, member_id, event_type, occurred_at, payload JSONB, created_by). Required indexes (`idx_membership_events_member_id`, `idx_membership_events_occurred_at`, composite `(member_id, occurred_at DESC)`) support fast retrieval of the latest state. Current status is derived by selecting the most recent event per member, optionally materialized in `current_membership_status`. Migration guidance includes a one‑time script that converts existing status columns to events and wraps member/event inserts in a single transaction. The design eliminates logical deletes and ensures auditability without schema churn.

## Personal AI Mentor Kit: Skill‑Runtime Architecture  

The mentor kit repository (`personal-ai-mentor-kit/`) structures all assets in Markdown and YAML. Core files include `CLAUDE.md`, `skill‑registry.yaml`, and a `skills/` directory containing five runtime skills (`idea_review`, `writing_review`, `decision_review`, `contradiction_finder`, `next_action_planner`). Each skill follows a template specifying purpose, inputs, procedure, rubric, and output format. Knowledge cards are stored under `cards/` and consulted during skill execution. The system replaces a pure RAG approach with a deterministic skill router that selects the appropriate skill based on user intent, while retaining RAG only for evidence support. This explicit runtime separation makes mentor behavior reproducible and auditable.

## Crisis‑Response ChatBot System Prompt: Structured Incident Workflow  

The system prompt for the internal crisis‑response chatbot enumerates nine ordered checkpoints: confirm life safety, verify ongoing damage, separate facts/guesses/speculation, assign a provisional incident level (0–4), draft an initial report, ensure a record location, notify the decision maker, list immediate stop actions, and schedule the next status‑update. Placeholders for company name, reporting channel, and record repository must be replaced before deployment. Incident levels map to update cadences (e.g., level 2 → hourly updates). The prompt also enforces that the bot never makes final decisions, never deletes evidence, and never shares credentials. By encoding procedural invariants directly in the prompt, the bot provides consistent guidance while delegating authority to human responders.

## Claude Code Telegram Channel Setup: Deployment Constraints  

The Telegram plugin configuration for Claude Code required three corrective actions. First, the `--channels` flag must be included when launching Claude Code (`claude --channels plugin:telegram@claude-plugins-official`). Second, Bun must be reachable via an absolute path; the fix creates `/usr/local/bin/bun` as a symlink to the user‑installed binary. Third, the bot token should be exported globally (`export TELEGRAM_BOT_TOKEN="<token>"`) rather than stored in a project‑local `.env` file, ensuring token availability across directories. The plugin must be installed in **user scope** to avoid project‑specific restrictions. These steps illustrate how deployment scripts must preserve environment invariants to guarantee reliable remote control.

**Synthesis** – Across the examined artifacts, a consistent design principle emerges: encode immutable invariants (personality rules, event records, skill definitions, incident checkpoints, deployment flags) separately from mutable execution layers (restart loops, orchestration scripts, user‑driven workflows). This separation enhances auditability, safety, and extensibility, providing a template for building robust AI‑enabled services that can evolve without compromising core guarantees.
