# 仕組みと対話の交差点

先週は、AI エージェントの安全性、データベース設計、メンターキットの構造、そして Claude Code の Telegram 連携という四つのテーマに同時に向き合いました。各テーマは別個のリポジトリやドキュメントに分かれていたので、全体像をつかむのに時間がかかり、何度も手戻りを経験しました。

## Agent‑Aiko と nullevi03 の安全性と永続化の比較検証

`logs/20260527-comparison-agent-aiko-vs-nullevi03.md` に記された比較表をベースに、実際に両方のリポジトリをクローンして動かしてみました。  
Agent‑Aiko は `persona/` 配下に不変ファイルと上書き可能ファイルを分離し、`--dangerously-skip-permissions` を使用しない点がまず安心材料でした。一方で、セッション状態を保持する仕組みは手動でスナップショットを保存しなければならず、再起動後にコンテキストが失われるリスクを感じました。  
nullevi03 は Telegram プラグイン経由で常時稼働させる設計で、`boot.sh` が 5 秒ごとにプロセスを再起動する単純さは魅力的でしたが、同時に `--dangerously-skip-permissions` が有効になっているため、悪意あるプロンプトがシステムコマンドを実行できる危険性が見えてきました。  
結局、私が求めていた「永続的かつ安全な対話環境」は、どちらか一方だけでは満たせず、Aiko の人格保護と nullevi03 の自動再起動を組み合わせるハイブリッドが最も実用的だと考えるに至りました。実装のイメージは、Aiko の `install.sh` に `--channels` フラグを付与して起動し、再起動時にスナップショットを `membership_events` に保存する形です。

## メンバーシップ管理をイベント駆動に置き換える実装と課題

`logs/20260527-db-state-vs-event-design.md` で提案されている二テーブル構成を、社内の会員管理システムに適用しました。`membership_events` テーブルを作成し、`event_type` に `joined`, `suspended`, `reactivated`, `withdrawn` を列挙しました。  
インサート時にトランザクションで `members` と `membership_events` を同時に書き込むロジックは、既存コードを大幅に書き換える必要があり、テストが足りないことに不安を覚えました。特に、過去に `status` カラムで直接更新していた部分をすべてイベント化したときに、外部システムが期待する `current_status` が取得できなくなるケースがいくつか見つかり、即座にビュー `current_membership_status` を作成して既存の SELECT 文をラップする暫定策を取りました。  
この変更で、監査ログが自動的に残るようになり、削除フラグの曖昧さが解消されたのは大きな手応えでした。ただし、データ量が増えるとインデックスのチューニングが必要になる点は、まだ見通しが甘いと感じています。

## パーソナル AI メンターキットのスキルランタイム設計の方向転換

`logs/20260530-personal-ai-mentor-kit-meeting-notes.md` の会議メモを読み返し、当初予定していた RAG（情報検索）中心の構成から、スキルランタイム方式へと舵を切った経緯を整理しました。  
スキルは `skill-registry.yaml` に列挙し、`idea_review`, `writing_review`, `decision_review`, `contradiction_finder`, `next_action_planner` の五つを実装することにしました。実装段階で感じたのは、各スキルが「判断の型」を抽出する際に、入力プロンプトの構造が統一されていないと結果が散漫になる点です。そこで、スキルごとに「purpose」「trigger」「inputs」「procedure」「rubric」「output format」のテンプレートを Markdown で明文化し、すべてのスキルが同一フォーマットに従うようにしました。  
また、メンターが複数の専門領域を持つことの意義を議論したとき、単一領域だけでは評価 AI に陥りがちだという指摘があり、実際に `idea_review` と `decision_review` を同時に呼び出すテストを行った結果、回答に奥行きが出ていることを確認できました。まだ手元にある知識カードは 20 程度しかなく、カード生成の自動化が次の課題です。

## Claude Code の Telegram プラグイン設定で遭遇した環境依存の障壁

`logs/20260605-claudecode-channels-telegram-setup-troubleshooting.md` では、Apple Silicon Mac における Bun のパス問題が原因でプラグインが起動しないケースを再現しました。  
最初に `claude --channels plugin:telegram@claude-plugins-official` を実行したとき、`--channels` が抜けていたために Telegram が全く認識されませんでした。フラグを付与した後も、`bun` が相対パスで指定されていたためにサブプロセスが実行ファイルを見つけられず、`ENOENT` エラーが出ました。解決策として、`/usr/local/bin` にシンボリックリンクを作成し、絶対パスで `bun` が呼び出せるようにしたことが最も効果的でした。  
さらに、トークンをプロジェクト固有の `.env` に書いたままだとディレクトリを変えるたびに認証が失われることが判明し、`~/.zshrc` に `export TELEGRAM_BOT_TOKEN="…" ` を追記して環境変数で管理するように変更しました。この手順を踏むと、任意の作業ディレクトリからでもプラグインが接続でき、ペアリングコードも安定して取得できるようになりました。  
今回の作業で、ツールチェーンのパス管理が思った以上にシステム全体の挙動に影響することを実感し、今後はインストールスクリプトにパス確認を組み込む方針にしました。

## 次にやりたいこと

今回の四つのテーマを通じて、仕組みの安全性と永続性、データ設計の監査性、メンターのスキル構造、そして開発環境の安定化という観点でそれぞれ前進できたと感じています。一方で、ハイブリッドエージェントの統合テストが未完了であること、イベント駆動 DB のインデックス最適化がまだ手付かずであること、知識カードの自動生成パイプラインが構築できていないこと、そしてプラグイン設定スクリプトの再利用性が低いことが残課題です。次のステップとしては、Aiko と nullevi03 の組み合わせを小規模のデモ環境で実装し、再起動後のセッション復元をイベントテーブルに記録する流れを検証したいです。同時に、`membership_events` のインデックス統計を取得し、クエリプロファイルを基に追加インデックスを検討します。メンターキットでは、既存の 20 枚のカードをベースに、AI で要約生成しつつ人手でレビューするパイプラインを試作し、スキル呼び出しのカバレッジを測ります。最後に、Claude Code のセットアップ手順を `setup.sh` にまとめ、`brew install bun` と同時にシンボリックリンク作成まで自動化すれば、同様の環境構築でつまずくことが減ると期待しています。これらを順に進めながら、全体のフローが一貫した形になるよう調整していきたいと考えています。
