# Delegating Development: AI‑Centric Roles, Local LLM Value, and Edge Deployments  

The week’s logs converge on a single operational premise: the human owner remains the exclusive source of contextual feedback, while all reproducible development tasks are handed to AI agents. The decision records illustrate how that premise is instantiated in ownership policy, competitive‑advantage modeling, hardware‑specific deployment, and session‑persistence tooling.  

## Owner Stance: Language‑Bound Delegation Criteria  

The memo for business owners (2026‑06‑15) defines two binary tests for any development activity. If a correctness condition can be expressed in language (e.g., “sum must be non‑negative” or “output must be sorted”), the condition is fixed in a pre‑check and AI generates within that bound. If the condition cannot be expressed (e.g., “good UX” or “value judgment”), the workflow adopts continuous creation followed by post‑selection. The memo also codifies the only non‑delegable role as “being the first real user” of the product; all implementation, testing, language‑encoding, and contextualisation are delegated to the AI. Four alerts (owner no longer uses the product, synthetic usage dominates, full usage loop delegated, lack of owner reaction) signal loss of unique context. The implication is that ownership of competitive advantage is encoded in real‑time usage signals, not in static logic.  

## Use‑While‑Building Thesis: Merging Consumption with Generation  

The accompanying thesis (2026‑06‑15) reframes the classic waterfall → agile → generative‑AI progression. In the third stage the approval step disappears; the AI consumes the owner’s usage traces directly as training signals. The decisive question reduces to “whose usage produced the response?” Consequently, the human contribution collapses to continuous product consumption, which supplies the only data that cannot be simulated without eroding differentiation. A key finding was that separating creation, testing, and specification from the usage loop is no longer viable when AI can synthesize all three from the same interaction stream.  

## Local LLM Competitive‑Advantage Framework  

The local‑LLM report (2026‑06‑16) introduces a three‑level evaluation matrix. Level 1 covers secure personal use (low‑to‑medium edge). Level 2 adds private data that drives efficiency (medium edge). Level 3 integrates private data, judgment criteria, workflow, and an improvement loop (high edge). Quantitative examples include the OPTiM AI Hospital case, where task time fell from 16.31 min to 7.47 min (‑54 %). Cost‑reduction calculations for GPT‑4 API indicate a potential 80 %–96.6 % saving (1/5 to 1/29 of original cost). The report concludes that local models yield competitive advantage only when they unlock AI for data that cannot leave the premises; otherwise they serve merely as a cost‑saving veneer.  

## ASUS GX10 Codex CLI Deployment Strategy  

The GX10 feasibility notes (2026‑06‑17) specify hardware and software constraints for on‑device AI assistance. The device runs Ubuntu Linux on an ARM v9.2‑A CPU, NVIDIA Blackwell GPU, and 128 GB LPDDR5x memory. Codex CLI is confirmed to support `aarch64`; installation proceeds with `curl https://chatgpt.com/codex/install.sh | sh` followed by `codex --version`. The official Codex GUI app is macOS/Windows‑only; the recommended architecture installs the CLI on GX10, runs the official Codex App on a Mac/Windows host, and connects via SSH (`ssh gx10-codex`). The guide advises disabling root login, using a dedicated non‑sudo user, and tunnelling all traffic through VPN/Tailscale. An experimental Linux desktop build exists but is flagged as fragile due to x64‑centric binaries and Electron native modules. The implication is that reliable edge deployment hinges on minimal native components (CLI) and a hardened SSH bridge rather than full GUI parity.  

## tmux Persistence for Remote AI Sessions  

The tmux guide (2026‑06‑19) outlines a session‑persistence model suitable for the remote‑recovery architecture of Claude Code (2026‑06‑19). Version 3.6b provides stable server‑client separation, allowing long‑running AI tasks to survive SSH disconnections. Typical workflow: `tmux new -s codex` → launch `claude --remote-control …` inside the session → detach with `C‑b d`. Reattachment (`tmux attach -t codex`) restores the exact environment, supporting the “always‑on” supervision layer required for safe recovery. Configuration recommendations (prefix Ctrl‑a, mouse support, vi‑style copy mode) improve ergonomics for developers who alternate between local terminals and remote AI agents. The design demonstrates that a lightweight terminal multiplexer can provide the durability that higher‑level orchestration (systemd, git snapshots) expects without adding external dependencies.  

**Synthesis** – Across ownership policy, AI‑centric development theory, competitive‑advantage modeling, edge hardware deployment, and session persistence, the logs articulate a coherent operational architecture: delegate every reproducible artifact to AI, protect the only source of differentiation—the owner’s real usage—and scaffold the delegation with secure local models, minimal native tooling on edge devices, and robust terminal multiplexing. Future work should quantify the marginal cost of maintaining the “first‑user” feedback loop against the savings realized by local LLM deployment and automated session recovery.
