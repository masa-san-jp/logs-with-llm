# AI 主体の開発フローを実験しながら固めた一週間

AI がコードを書き、テストを走らせ、結果を評価する――その全工程を実際に手を動かしながら検証した週でした。  
「自分が最初のユーザーであり続ける」ことが唯一の差別化要素になる、という考えを実装レベルで試すために、ハード、ツール、プロセスをすべて揃えてみました。  

## AI エージェントへの委任判断基準を整理した  

6 月 15 日のオーナーノートにあった「条件を言語化できるか」という二段階の質問を、実際の開発タスクに当てはめてみました。  
たとえば「ビルドが成功する」かどうかは、`exit_code == 0` と数値で表せるので、事前に検証ロジックを固定し、AI にビルド結果のバリエーションを生成させました。一方で「UX が心地よいか」は主観的な尺度で言語化が困難だったため、生成された UI を実際に自分で操作し、使用中に出た「違和感」や「使いやすさ」の反応を記録することにしました。  
このとき、所有者として「自分が最初のユーザーであり続ける」姿勢を保つことが、AI が生成したアウトプットに自分のコンテキストを乗せ続ける唯一の手段だと実感しました。反応が自分の業務フローに合致しないときは、AI に再生成を促すだけでなく、条件そのものを見直すサイクルが自然に回り始めました。

## ASUS GX10 に Codex CLI を導入し SSH 接続を確立した  

ARM64 の ASUS Ascent GX10（aarch64）上で Codex CLI が動くかを検証しました。まず `uname -m` で `aarch64` が出たことを確認し、公式インストーラ `curl https://chatgpt.com/codex/install.sh | sh` を実行。インストール後に `codex --version` が表示され、CLI が正しく組み込まれたことが分かりました。  

次に、Mac にインストールした Codex GUI アプリから SSH 経由で GX10 に接続する設定を行いました。`~/.ssh/config` に以下のエイリアスを追加し、鍵ペアを `ssh-copy-id` で登録しました。  

```
Host gx10-codex
    HostName 192.168.1.50
    User codex
    IdentityFile ~/.ssh/gx10_codex_ed25519
    ServerAliveInterval 30
```  

SSH デーモンは `PubkeyAuthentication` のみ許可し、パスワードログインを無効化する `90-codex-hardening.conf` を配置しました。`ufw` で外部からの 22 番ポートはブロックし、LAN 内だけが許可されるようにしたので、ネットワーク上の意図しない露出は防げました。接続テストで `whoami`、`hostname`、`uname -m` が期待通りの出力になることを確認し、以後は Mac の Codex アプリでリモートファイルを編集しながら、GX10 上でコード生成を走らせるワークフローが完成しました。

## Telegram ボットで Antigravity CLI を安全に操作できるようにした  

同じ GX10 に Antigravity CLI を導入し、Telegram ボット経由でコマンドを投げられる環境を構築しました。まず `agy` のインストールスクリプトを `agybot` ユーザーで実行し、`~/.local/bin` を PATH に追加しました。`agybot` は sudo 権限を持たない専用ユーザーで、ホームディレクトリ以下にだけ書き込み権限を付与しています。  

BotFather で取得したトークンは `/etc/telegram-agy-bot.env` に `600` の権限で保存し、許可された `chat_id` だけがコマンドを受け付けられるようにしました。Python ブリッジは `python-telegram-bot==22.6` を仮想環境でインストールし、`/ask` と `/review` をデフォルトで有効、`/run` は環境変数 `AGY_ALLOW_WRITE=0` のときは無効化しています。書き込みモードを有効にしたいときは、Git の専用ブランチを作り、環境変数を `1` に変更した上でサービスを再起動する手順を踏むようにしました。  

サービスは systemd ユニットで管理し、`NoNewPrivileges=true`、`ProtectSystem=strict`、`ReadWritePaths=/home/agybot` などのハードニングを適用しました。起動後に Telegram で `/whoami` と入力すると `agybot` と返ってくることを確認し、続けて `/ask "現在のシステム情報を教えて"` を送ると `agy` が内部で `uname -a` を実行し、結果が 3,800 文字以内に分割されて返ってきました。書き込みコマンドが誤って有効になるリスクは、ブランチ保護と `git diff` の必須レビューで抑えられたと感じました。

## Claude Code のリカバリフローを tmux と systemd で構築した  

Claude Code のリモートコントロールを利用している最中に、セッションが予期せず終了するケースが数回発生しました。そこで、`tmux`（バージョン 3.6b）をベースに、永続的なセッション `cc-work` を作り、`systemd --user` の `claude-code@.service` で自動再起動させる仕組みを組みました。  

`tmux` の設定は `set -g prefix C-a` へ変更し、ウインドウ番号は 1 から開始するようにした上で、`mouse on` と `history-limit 100000` を加えて作業感覚を向上させました。`cc-supervisor` スクリプトは 10 秒ごとに `tmux has-session -t cc-work` を確認し、死んでいれば `tmux new-session -d -s cc-work "claude --remote-control myproj --permission-mode plan"` を起動します。  

危険コマンドのガードは Bash の正規表現で `rm -rf /` や `sudo` 系コマンドを検出し、実行前に JSON 行でログに残すようにしました。さらに、5 分ごとに `git status` と `git diff` を `.claude/recovery/snapshots` に保存し、万が一コードが壊れたときは `git checkout` で直前の状態に戻せるようにしています。  

リモートからの通知は、`claude --remote-control` がプッシュできる端末に対して `ntfy` を利用し、タスク完了やエラー発生時にスマートフォンへ通知が届くように設定しました。実際に `claude` が無限ループに陥ったときは、`cc-status` が「heartbeat missing」と報告し、`cc-revive` コマンドで tmux セッションを再起動できたので、作業の中断が最小限に抑えられました。

## 次にやりたいこと  

今回の実装で、AI が生成したアウトプットに対して「自分が最初のユーザーである」感覚を保ちながらフィードバックを回す流れが形になったと感じます。一方、条件を言語化できないタスクが残っていることや、ローカル LLM の競争優位性レポートで示された 80 % 以上のコスト削減余地を実際に測定できていない点が課題です。次のステップとしては、ローカル LLM を導入し、非公開データと評価ループを統合したパイプラインを構築したいです。その際は、先に作った Codex と Antigravity の SSH/Telegram ブリッジを再利用し、tmux と systemd によるリカバリ基盤の上に、ローカルモデルへのリクエストと結果の自動評価を組み込む計画です。さらに、生成された UI の使用感を定量化するために、簡易的な操作ログと主観評価を組み合わせたスコアリングシステムを試作し、条件言語化の限界を可視化したいと考えています。これらを通じて、AI 主体の開発フローが本当に所有者のコンテキストを保持し続けられるか、実務レベルで検証できるように進めていきます。
