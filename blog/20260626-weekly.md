# ネットワーク横断の安全接続を試す週  

3拠点にある Mac、Windows、そして Linux サーバを、同じ VPN 内でやり取りできるようにしたいと考えていました。  
同時に、GPU サーバ上で動く Hermes Agent を Telegram から操作できるようにし、最後に内部 API として利用できそうな Thug‑Fugu を試すことにしました。  

## Tailscale を用いた三拠点間の SSH とリモートデスクトップ構築  

まずは、MacBook Pro、職場の Windows Pro、そして自宅の GB10（Ubuntu）を同一の Tailscale アカウントに参加させました。Google アカウントでのログインはスムーズに完了し、管理コンソールに `macbook-pro`、`work-windows`、`home-gb10` が表示されたのを確認したときは、ネットワーク境界を意識せずに IP が付与される感覚に少し驚きました。  

インストールは各 OS の公式パッケージを利用し、Linux 側は `curl https://tailscale.com/install.sh | sh`、Windows は公式インストーラを実行しました。すべてのデバイスで `tailscaled` が起動し、Tailscale IP が割り当てられたことを `tailscale status` で確認しました。  

SSH の設定では、Linux 側に `openssh-server` を導入し、`systemctl enable --now ssh` で自動起動させました。Windows では「OpenSSH Server」機能を有効化し、`Start-Service sshd` と `Set-Service -Name sshd -StartupType Automatic` で常駐させました。鍵認証に切り替えるために、Mac で `ssh-keygen -t ed25519 -C "macbook-pro-to-remote"` を実行し、公開鍵を `ssh-copy-id masa@home-gb10` で転送しました。Windows の公開鍵は手動で `~/.ssh/authorized_keys` に追記し、権限を `chmod 600` に整えました。鍵が正しく受け入れられたかは、Mac から `ssh gb10` で接続し、パスフレーズ入力が求められないことを確認できたときに実感できました。  

リモートデスクトップは、Windows が Pro エディションであるため Microsoft Remote Desktop アプリをそのまま利用しました。Tailscale IP（例：`100.101.102.103`）をクライアントに入力し、接続が確立した瞬間は、社内ネットワークを回り込んだかのような感覚がありました。  

一方、Linux サーバに対しては RustDesk を代替手段として選びました。公式サイトから `.deb` パッケージを取得し、`sudo apt install ./rustdesk-*.deb` でインストールしました。`ss -lntup | grep 21118` でポートが開いていることを確認し、Permanent password を長く設定した点が安心材料になりました。Mac と Windows のクライアントでも同じ Tailscale IP とポートで接続でき、最初はポートがブロックされているかと焦ったものの、`Direct IP access` を有効にしたことで通信が通りました。  

自動起動設定は、Linux の `tailscaled` と SSH サービスを `systemctl enable --now` で有効化し、Windows のスタートアップに RDP と RustDesk クライアントを追加しました。これらを全て確認したチェックリストは、実装が抜け落ちることなく完了したときに「やった感」が出ました。  

## GX10 に Hermes Agent と Telegram ゲートウェイを常駐させる  

次に、ASUS Ascent GX10 上で Hermes Agent を常時稼働させ、Telegram から指示を送れるようにしました。まずは OS 更新と必要ツール (`curl`, `git`, `build-essential`, `tmux`, `jq`) をインストールし、`curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash` でエージェント本体を配置しました。インストーラは `~/.local/bin/hermes` にバイナリを置き、PATH へ自動追加された点が便利でした。  

`hermes setup` を実行すると、モデルプロバイダーの選択や API キー入力を求められました。私は OpenAI のキーを入力し、`hermes doctor` で接続テストを行ったところ、数秒で成功ステータスが返ってきたので、設定が正しく反映されたと確信できました。  

Telegram 側の設定は、BotFather で取得したトークンと自分の数値 ID を `~/.hermes/.env` に書き込み、`chmod 600` で権限を固めました。`TELEGRAM_ALLOWED_USERS` に自分だけを入れたことで、誤って他人がボットを叩くリスクが低減しました。  

ゲートウェイの永続化は、ユーザー単位の systemd ユニット `hermes-gateway.service` を作成し、`ExecStart=/home/masa/.local/bin/hermes gateway` としました。`systemctl --user enable --now hermes-gateway.service` を走らせ、`loginctl enable-linger $USER` でログアウト後もプロセスが残るように設定したとき、再起動してもボットが即座に応答する様子に安堵感がありました。ログは `journalctl --user -u hermes-gateway.service -f` でリアルタイムに確認でき、エラーメッセージが出た際はすぐにトークンや `.env` の権限を見直す手順が思い出せました。  

トラブルとしては、同一トークンを別マシンで起動したときに Telegram が `409 Conflict` を返すケースがあり、`hermes gateway stop` と systemd の停止コマンドで競合プロセスを止めた後に再起動すると解消できました。また、プライバシーモードが有効だったままグループチャットに追加するとメッセージが届かないことに気づき、BotFather で無効化した点が手間でしたが、設定項目を一つ一つ確認する重要性を再認識させられました。  

## Thug‑Fugu の HTTP バリデーションを検証し改善案を練る  

最後に、ローカルで動作する LLM オーケストレーションツール「Thug‑Fugu」をレビューしました。リポジトリは `masa-san-jp/Thug-Fugu` にあり、`pyproject.toml` の `src` 配置やテストカバレッジがしっかりしている点が印象的でした。HTTP サーバは OpenAI 互換のエンドポイントを提供しますが、リクエスト形状エラーのステータスコードが 502 になる問題（PR 1 で指摘）を実際に試すと、クライアント側で「サーバエラー」と表示され、原因特定が難しいことが分かりました。  

そこで、`_validate_chat_completion_request()` に `messages` の必須チェックを追加し、欠損・空・非リスト・不正構造のテストケースを自作しました。テストはすべて 400 が返ることを確認し、期待通りに振る舞う様子が見えてきました。  

また、バックエンドエラーがそのままクライアントに流出するリスク（P0）については、エラーメッセージを `debug` フラグが無い限りマスクする実装を試みました。`if debug: return raw_error else: return generic_message` の形でコードを差し替え、テストで機密情報が漏れないことを確認したとき、セキュリティ意識が具体的なコードに落とし込めた感覚がありました。  

他にも、非ローカルホストへのバインド時に警告を出す `--allow-unsafe-bind` オプションの追加や、全体オーケストレーションに対するデッドライン設定（P1）を検討しました。実装の骨格はシンプルなので、これらの機能は比較的短時間で組み込めそうだと見込んでいます。  

## 次にやりたいこと  

今回の作業で、三拠点を同一 VPN で結び、SSH と RDP／RustDesk の接続を安定させたことは、リモート作業の基盤が整ったと感じました。一方、Hermes Agent の Telegram ゲートウェイは期待通りに動作していますが、トークン管理やユーザー制限を手動でやる手間が残っています。今後は、トークンの自動更新や複数ユーザー向けのロール管理をスクリプト化したいと考えています。  

Thug‑Fugu については、バリデーションとエラーマスクの実装が完了した段階で、実際に自分の GX10 上の LLM 役割を Orchestrator に組み込んでみる予定です。その際に、全体オーケストレーションのデッドラインやストリーミング対応を段階的に追加し、ローカルネットワーク内での API 形態として安定させることが次の目標です。  

以上が、2026 年 6 月下旬に私が試行錯誤した内容の記録です。次のステップでは、各構成要素をもう少し自動化し、障害が起きても手順だけで復旧できるようにドキュメント化していきたいと考えています。
