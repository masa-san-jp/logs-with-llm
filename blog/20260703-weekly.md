# 同一マシンでClaudeアカウントを分離する試み  

個人的に利用しているClaude Code CLIと、会社のプロジェクトで利用するClaude Code CLIを、同じLinuxノートPC上で同時に走らせたいと考えました。デフォルト設定では認証情報が一つのファイルに上書きされるため、ログインし直すたびに前のアカウントが使えなくなるという壁にぶつかっていました。そこで、認証情報を物理的に分離できる方法を探し、実際に動作させるまでの過程を記します。

## 設定ディレクトリを分けるためのラッパースクリプト作成  

Claude Code CLIは環境変数 `CLAUDE_CONFIG_DIR` が指すディレクトリに `.credentials.json` と `settings.json` を格納します。デフォルトは Linux で `~/.claude/.credentials.json`（ファイルモードは 0600）です。まずは「個人用」と「会社用」の二つのディレクトリを用意し、どちらかを指すようにラッパースクリプトを書きました。

```bash
#!/bin/bash
# ~/.local/bin/claude-company – 会社用アカウント用ラッパー
export CLAUDE_CONFIG_DIR="$HOME/.claude-company"
exec claude "$@"
```

スクリプトに実行権を付与し（`chmod +x ~/.local/bin/claude-company`）、`~/.claude-company` を作成した後に `claude-company auth login` を走らせました。ブラウザが立ち上がり、会社のOAuth認証が完了すると、`~/.claude-company/.credentials.json` が生成されました。これで、`claude` コマンドはデフォルトの `~/.claude/.credentials.json` を、`claude-company` は別ディレクトリのファイルを参照するようになり、相互に上書きされないことが確認できました。

## 環境変数とAPIキーの競合を防ぐ手順  

CLIは認証情報を選択する順序が決まっており、最優先は環境変数 `ANTHROPIC_API_KEY` です。実際に `ANTHROPIC_API_KEY` が設定されたまま別アカウントへ切り替えようとすると、OAuthトークンは無視され、常にAPIキーが使用されてしまいました。そこで、アカウントを切り替えるたびに次のコマンドでキーを削除する手順を組み込みました。

```bash
unset ANTHROPIC_API_KEY
```

この一行をラッパースクリプトの冒頭に入れると、`claude-company` 実行時に自動的にキーが消え、OAuth認証が正しく参照されるようになりました。最初は「キーが残っているだけで切り替えが効かない」ことに戸惑い、`env` コマンドで環境変数を確認したときに初めて原因が見えてきました。

## バックグラウンドプロセスへの影響とログアウトの注意点  

会社のプロジェクトでは、Claude Code CLIをバックグラウンドで常駐させた状態でコード生成を行うエージェントを走らせています。デフォルトアカウントで `claude auth logout` を実行すると、エージェントが参照しているトークンが失われ、即座にエラーが出ました。そこで、ラッパー経由でログアウトする方針に変更しました。具体的には、`claude-company auth logout` と実行すれば、`~/.claude-company/.credentials.json` が削除され、エージェントは引き続きデフォルトアカウントのトークンを使用できるようになりました。

この対応を取るまで、ログアウト操作が全体に波及することに気付かず、何度かエージェントが停止するというトラブルが発生しました。ログアウトの対象が「どのディレクトリのファイルか」を意識しながら操作する重要性が、実体験を通して理解できました。

## macOS の Keychain での制限と代替策  

macOS では認証情報が Keychain に保存され、`.credentials.json` が生成されません。そのため、Linux と同様に `CLAUDE_CONFIG_DIR` を切り替えてもディレクトリ単位で分離できないことが判明しました。代替策としては、アカウントごとに異なる `ANTHROPIC_API_KEY` を発行し、環境変数で切り替える方法を検討しました。キーを切り替えるスクリプトは以下のようにシンプルです。

```bash
#!/bin/bash
# ~/.local/bin/claude-personal-mac – macOS 用個人アカウント
export ANTHROPIC_API_KEY="sk-personal-xxxx"
exec claude "$@"
```

この方法は Keychain の制約を回避できるものの、APIキーの有効期限管理や権限設定に注意が必要です。実際に macOS でテストした結果、キーが正しく設定されていれば `claude auth status` が期待通りに表示されましたが、キー漏洩のリスクを常に意識しなければならない点が課題として残りました。

## 次にやりたいこと  

今回の作業で、設定ディレクトリを分離するラッパーと環境変数のクリア手順が機能し、Linux 上で個人用と会社用の二つのClaude Code CLIセッションを同時に走らせられるようになった点はうまくいきました。一方、macOS では Keychain の構造上ディレクトリ分割ができず、APIキーに依存する代替策に留まっているため、将来的には Keychain に複数のエントリを登録できるツールや、プロファイル切り替えを自動化するスクリプトの開発が必要です。また、バックグラウンドエージェントが参照するトークンの寿命管理や、設定ファイル `settings.json` がアカウントごとに正しく分離されているかの定期的な検証も課題として残っています。次のステップとしては、これらの残課題をまとめたチェックリストを作成し、macOS と Linux の両環境で一貫した運用手順をドキュメント化したいと考えています。
