# 音声と暗黙知をつなぐパイプラインを形にした週

オフィスの会話をそのまま資産に変える仕組みを、今週はハードとソフトの両面で試作しました。  
実際にマイクを設置し、録音データを保存・処理し、知識カードへとつなげるまでの流れを、一つずつ確かめながら進めました。

## Raspberry Pi 5 と ReSpeaker USB マイクの選定と試験

まずは「生音声は捨てない」という基本方針に沿って、録音装置を決める必要がありました。  
社内にある比較表（100〜200 種類）から、音質、24 時間稼働可否、API/SDK の有無、Linux 対応、PoE 供給の可否といった項目を抽出し、コストと保守性を掛け合わせた評価を行いました。  

結果、初期投資が 2〜5 万円程度で済む Raspberry Pi 5 と ReSpeaker USB マイクの組み合わせが、PoC の規模（1〜5 名）には最適と判断しました。実機でのテストでは、USB 接続が安定しないケースが数件あり、電源供給を PoE アダプタに切り替えると解消できました。  
会議室向けには Meeting Owl 3、Poly Sync 60、Yamaha YVC 系列などを候補に挙げ、音質と設置スペースのバランスを見ながら次フェーズの選定材料にしました。  

この選定作業を通じて、単に価格が低いだけでは運用上の負担が増えることに気づき、保守性と AI 連携のしやすさを同等に評価する重要性を実感しました。

## 録音データの保存構造とメタデータ設計

次に、録音ファイルの保存場所と付随情報の管理方法を決めました。  
設計書にあった「flat hierarchy（raw_audio、processed_audio、transcripts …）」を踏襲し、ディレクトリ構造をシンプルに保ちました。ファイル名は `YYYYMMDD-HHMMSS-room-device.wav` の形式に統一し、誰がどこで録音したかが一目で分かるようにしています。  

メタデータは JSON Lines で保存し、`recording_id、room、device、timestamp、checksum、duration、sample_rate、firmware、mic_model` などを必ず記録するようにしました。  
最初は checksum の算出方法が不明確で、同一ファイルでも異なるハッシュが出ることがありましたが、`sha256` に統一したところ安定しました。  

この設計は、後続の AI 処理が「全処理はパイプライン化する」方針と合致し、モデルやプロンプトのバージョン情報をすべての出力に添付できる基盤となります。

## 音声前処理と STT パイプラインの構築

録音された生データに対しては、以下の前処理を自動化しました。  
- 無音区間の検出  
- 音量正規化  
- ノイズ傾向の推定  
- 音声品質スコアの算出  
- 必要に応じた短尺チャンク化  

重要なのは、前処理結果を別ファイルとして保存し、元の WAV は絶対に上書きしない点です。これにより、後からアルゴリズムを改良しても元データを再利用できます。  

STT には Whisper の小型モデルを Raspberry Pi 上で実行し、処理時間は 1 時間の録音につき約 45 分でした。出力 JSON にはセグメントの開始・終了時刻、推定スピーカー（可能な範囲で）、信頼度、モデル名とバージョン、使用したプロンプトをすべて記録しています。CPU 使用率が高くなるため、バッチ処理で夜間に走らせる設計に変更しました。  

この段階で、モデルバージョンが変わった際に同じ生データを再処理できる仕組みが実際に機能することを確認でき、将来の精度向上に備える基盤が整ったと感じました。

## 暗黙知候補カードの生成とレビュー

STT の結果を元に、会話を「質問‑回答」「トラブルシュート」「手順説明」などの単位に分割し、各ユニットに対してビジネス関連度（0‑1 のスコア）とトピック推定を付与しました。  
続いて、これらのユニットから以下のようなタグを付けた知識候補カードを生成しています。  

- ワークアラウンド  
- 判断規則  
- 例外対応  
- 失敗事例  
- 顧客ヒント  
- 暗黙の優先順位  
- 熟練者のコツ  
- 手順のずれ  

カードは JSON 形式で保存し、タイトル、本文、根拠テキスト、信頼度、レビュー要否フラグ、リスクフラグ、対応する音声区間（recording_id とオフセット）を含みます。  

レビューは週に一度、AI がスコア付けした上位 20 件を対象に実施し、所要時間は概ね 30 分です。実際にレビューした結果、スコアが高くても文脈が曖昧なものが散見され、タグ付けの自動化精度を上げる必要があると判明しました。  

このプロセスを通じて、全音声を逐一目を通す必要がないこと、しかしカードの品質を保つための人手がまだ不可欠であることが具体的に分かりました。

## PoC 計画と運用マニュアルの草案

上記の各要素を統合した PoC の計画を立て、以下の項目をドラフトにまとめました。  

- デバイス構成（Pi + ReSpeaker）とネットワーク設計（LAN 経由の固定 IP）  
- セキュリティ・プライバシー方針（録音エリアの掲示、個人情報の自動マスク）  
- 法務上の録音許可取得手順とログ管理方法  
- 運用マニュアルの概略（デバイス起動手順、障害時の再起動フロー、バックアップ方針）  

実装段階で直面した課題は、録音エリア外の会話が混入しやすい点と、PoE 電源が不安定になるとデバイスが再起動してしまう点です。これらは、エリア限定のマイク指向性を高めるか、UPS を導入することで対策できると見込んでいます。

## 次にやりたいこと

今回の作業で、デバイス選定からメタデータ設計、前処理・STT・知識カード生成までの一連の流れが実装でき、特に「生音声は捨てない」という方針がデータの再利用性を高めることを実感できました。一方で、スピーカー分離やリアルタイム処理、複数会議室への同時展開といったスケール面の課題が残っています。次のステップとしては、Poly Sync 60 を用いた会議室パイロットを走らせ、Kubernetes 上に AI 推論サービスをデプロイし、運用マニュアルを完成させることを目指します。また、知識カードのタグ付けアルゴリズムを改良し、レビュー工数のさらなる削減を図りたいと考えています。
