# 運用ガイド

## 1. リポジトリ構成
- `logs/`: 日々の記録。日付付きパスが週次処理の基準になる
- `blog/`: 週次ブログの出力先
- `prompts/`: 会話や執筆のためのプロンプト資産
- `scripts/`: GitHub Actions から呼び出す Python スクリプト
- `scripts/tests/`: 週次自動化スクリプトの単体テスト
- `docs/`: 設計仕様書と運用資料

## 2. 手動確認の基本手順
1. 変更前後で README と `docs/` の整合性を確認する
2. GitHub Actions の入力値・成果物・スケジュールに差分がないか確認する
3. 仕様変更を伴う場合は対応するテストの期待値も確認する

## 3. ローカル検証
```bash
pip install -r requirements.txt pytest
python -m pytest -q scripts/tests
```

## 4. 手動実行時の確認ポイント
### 週次ブログ生成
- 対象期間の日付解釈が意図どおりか
- `blog/` に日本語版と英語版が揃って出力されるか
- `.blog_state.json` が更新されるか

### 週次ドキュメント目標 Issue 生成
- 対象ドキュメントの棚卸し範囲が意図どおりか
- Issue タイトルの形式が維持されているか
- 重複 Issue の抑止が機能するか

## 5. ドキュメント更新ルール
- README には概要と導線、`docs/` には詳細仕様を書く
- GitHub Actions の挙動が変わったら README と対応仕様書を同時更新する
- 認証情報や秘密情報そのものはドキュメントに書かない
