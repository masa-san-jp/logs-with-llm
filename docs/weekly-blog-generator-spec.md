# 週次ブログ生成ワークフロー設計仕様書

## 1. 対象
- GitHub Actions: `.github/workflows/weekly-blog.yml`
- 実行スクリプト: `scripts/generate_weekly_blog.py`

## 2. 目的
`logs/` 配下の直近の記録をもとに、週次のふりかえりブログを日本語版・英語版の 2 本生成し、`blog/` に保存することを目的とします。

## 3. 実行契機
- 定期実行: 毎週金曜日 09:00 UTC
- 手動実行: GitHub Actions の `workflow_dispatch`

## 4. 主な入力
### ワークフロー入力
- `llm_provider`: 利用する LLM プロバイダの選択
- `blog_days`: 何日分のログを集計対象にするか
- `blog_date`: 出力日付の上書き

### リポジトリ入力
- `logs/`: 週次ブログの材料になるログ群
- `blog/`: 既存ブログ。直前回の文脈取得に利用
- `.blog_state.json`: 前回処理時点のコミット情報と日付

### 実行環境入力
- Python 3.12
- `requirements.txt` に定義された依存ライブラリ
- リポジトリ設定側で渡される LLM 接続設定

## 5. 処理フロー
1. リポジトリをチェックアウトし、Python と依存ライブラリを準備する
2. 実行日を基準に対象期間を決定する
3. `logs/` を再帰的に探索し、パス中の `yyyymmdd` 形式の日付から対象ログを選別する
4. 対象ログが見つからない場合は `.blog_state.json` の直前コミット情報を使って差分対象を補完する
5. 対象ログをファイルごとに独立した LLM コンテキストで要約する（ファイル単位要約フェーズ）。
   各ファイルに対して個別に LLM 呼び出しを行い、発生した出来事・意思決定・学びを
   重要な要素を損なわないよう簡潔にまとめた要約を得る。
   PDF のテキスト抽出もこのフェーズで実施する。
   各呼び出しはコンテキストを共有しない完全に独立した処理とする。
   個別の要約呼び出しが失敗した場合でも、その失敗を理由にワークフロー全体は中断しない。
   失敗したファイルは要約結果なしとして扱い、手順 7 の集約対象から除外する。
6. `blog/` から直前の日本語版・英語版ブログを読み込み、独立した LLM コンテキストで
   それぞれ要約する（前回ブログ要約フェーズ）。
   ここでの要約が失敗した場合もワークフロー全体は継続し、当該要約を使わずに後続へ進む。
7. 手順 5 のファイル単位要約と手順 6 のブログ要約のうち、取得に成功した要約のみを集約し、
   圧縮済みコンテキストを組み立てる。
8. 言語ごとに独立した LLM コンテキストでプロンプトを組み立て、ブログ本文生成を依頼する。
   コンテキストには生ログ本文を使わず、手順 7 で得た圧縮済み要約のみを使用する。
   要約フェーズで失敗があっても、生ログ本文を代替入力として本文生成フェーズへ転送してはならない。
   ブログ執筆タスクも要約タスクと同様、他のタスクとコンテキストを共有しない独立した処理とする。
   ワークフロー全体を失敗扱いにするのは、言語別の本文生成またはその出力書き込みが成立しない場合とする。
9. 生成結果を `blog/YYYYMMDD-weekly.md` と `blog/YYYYMMDD-weekly-en.md` に書き出す
10. 実行後のコミット SHA と日付を `.blog_state.json` に保存する
11. `blog/` または `.blog_state.json` に変更があればコミットして `main` に反映する

## 6. 出力
- `blog/YYYYMMDD-weekly.md`
- `blog/YYYYMMDD-weekly-en.md`
- `.blog_state.json` の更新

## 7. 期待する出力内容
- 一人称で書かれた週次ふりかえり
- ハイライト、作業内容、意思決定、前回からの進捗、次の展開を含む構成
- 日本語版と英語版の両方を同一期間のログから生成

## 8. 異常系・フォールバック
- 日付付きログが見つからない場合は git 差分を利用する
- ログが空でもプレースホルダ文面で出力を継続する
- PDF 読み取りに失敗した場合は該当ファイルを空文字扱いで処理継続する
- LLM 呼び出しに失敗した場合はワークフロー全体を失敗として扱う

## 9. 保守観点
- `logs/` の命名規則とブログ出力ファイル名は仕様の中心なので変更時はスクリプトと README を同時更新する
- `.blog_state.json` は差分フォールバックに使うため、コミット対象から外さない
- 生成テキストの構成を変える場合は `scripts/tests/test_generate_weekly_blog.py` の期待値も合わせて見直す
