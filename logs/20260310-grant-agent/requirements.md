# 補助金・助成金 申請支援マルチエージェントシステム 要件定義書

## 1. プロジェクト概要
国や自治体が実施している補助金・助成金事業の情報を自動収集し、自社情報との適格性判定、申請手順の策定、申請書類のドラフト作成までをシームレスに行うマルチエージェントシステムを構築する。
完全自動化ではなく、人間の意思決定と確認を間に挟む「Human-in-the-loop (HITL)」を前提としたオーケストレーションを行う。

## 2. システムアーキテクチャ
本システムは、全体を統括する「オーケストレーター」と、特定のタスクに特化した4つの「エージェント」で構成される。
フレームワークとしては、マルチエージェントの制御に適した LangGraph または CrewAI（Pythonベース）の採用を想定する。



## 3. 機能要件（各エージェントの役割と入出力）

### エージェント1：情報収集エージェント (Information Gathering Agent)
* **役割:** 官公庁や自治体のWebサイトを定期巡回し、最新の補助金情報を取得・構造化する。
* **トリガー:** スケジューラーによる定期実行（週1回以上）、または手動実行。
* **処理内容:**
    1. 指定されたURL（経産省、中小企業庁、自治体サイト等）をクロール。
    2. 新着の公募要領（PDFやHTML）を取得。
    3. LLMまたはパーサーを用いて非構造化データを読み解き、指定のJSONスキーマ（対象者、補助額、対象経費、要件など）に構造化してデータベースに保存する。
* **出力:** 構造化された「補助金マスターデータ (JSON)」。

### エージェント2：適格性判定・レポーティングエージェント (Eligibility Analyzer Agent)
* **役割:** 新規取得した補助金情報と、自社のプロファイル情報を照合し、申請の可否と期待値を判定する。
* **入力:** エージェント1が取得した「補助金マスターデータ」 ＋ 「自社プロファイルデータ（業種、規模、財務状況、事業構想）」。
* **処理内容:**
    1. ハード要件（資本金、従業員数、業種など）のマルバツ判定（ルールベース処理を推奨）。
    2. ソフト要件（事業目的の合致度、対象経費の該当有無）のLLMによるスコアリング。
    3. マッチング結果のサマリーレポート生成。
* **出力:** 適格性スコア、適合理由、懸念事項をまとめた「適格性判定レポート（マークダウン/通知用フォーマット）」。

### エージェント3：プロセスナビゲーター (Process Navigator Agent)
* **役割:** ユーザーが「申請する」と決定した補助金について、公募要領から必要な手続きを抽出し、具体的なタスクリスト（WBS）を生成する。
* **入力:** ターゲットとなる「補助金マスターデータ」の要件・スケジュール部分。
* **処理内容:**
    1. 申請に必要な事前準備（例: gBizID取得、SECURITY ACTION宣言）を抽出。
    2. 必要な書類リスト（例: 決算書2期分、見積書、事業計画書）を抽出。
    3. 締切日から逆算したスケジュール付きの手順書を生成。
* **出力:** 申請完了までの「ステップバイステップ手順書・タスクリスト (JSON / マークダウン)」。

### エージェント4：ドキュメントドラフター (Document Drafter Agent)
* **役割:** エージェント3の手順書に基づく書類作成タスクを受け取り、申請書のドラフト（事業計画の骨子および本文）を作成する。
* **入力:** 「自社プロファイルデータ」 ＋ 「補助金の審査項目（加点要素）」 ＋ 必要に応じて「過去の採択事例（RAG）」。
* **処理内容:**
    1. 審査項目を満たす論理構成（目次案）を生成。
    2. 文字数制限や指定フォーマットに従い、自社の強みや課題を反映した文章を執筆。
* **出力:** 加筆修正の土台となる「事業計画書ドラフト (テキスト/マークダウン)」。

## 4. 人間介在プロセス (Human-in-the-loop: HITL)
完全自動化によるハルシネーションや不適切な申請を防ぐため、以下のポイントで人間の確認・承認フローを必須とする。
* **承認ゲートA:** エージェント2のレポートを受け、人間が「この補助金に申請するかどうか」を判断・承認し、エージェント3をトリガーする。
* **承認ゲートB:** エージェント4が作成したドラフトを人間がレビューし、事実確認（ファクトチェック）と熱意の加筆を行い、最終版を確定させる。

## 5. データモデル定義（要Pydantic実装）
システム内でやり取りする主要なデータモデルの構造。

* `CompanyProfile` (自社情報)
    * company_name, industry_code, employee_count, capital, financial_data (売上, 利益等), current_challenges, proposed_project (やりたいこと)
* `SubsidyMaster` (補助金情報)
    * subsidy_id, title, url, deadline, target_company_size, max_amount, subsidy_rate, eligible_expenses (対象経費リスト), screening_criteria (審査項目)
* `EligibilityReport` (判定レポート)
    * subsidy_id, is_eligible (boolean), match_score (0-100), reasons_for_match, missing_requirements

## 6. 非機能要件
* **LLM API:** Anthropic Claude 3.5/3.7 または OpenAI GPT-4o 等、高精度な論理推論が可能なモデルを使用する。
* **クローラーの制約:** 官公庁サイトへの負荷を考慮し、クロール間隔（sleep）を適切に設定し、`robots.txt` を遵守すること。
* **エラーハンドリング:** PDFのパース失敗時や、要件の解釈が曖昧な場合は、無理に推論せず「人間の確認が必要」というフラグを立てて処理を中断・通知する設計とする。
* **セキュリティ:** 自社の財務情報や事業構想データがLLMの学習に利用されないよう、API連携（オプトアウト環境）を使用する。
