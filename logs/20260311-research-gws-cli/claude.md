## 1. gws CLIの概要と特徴

### 1-1. 基本情報

- **リポジトリ：** https://github.com/googleworkspace/cli
- **パッケージ名：** `@googleworkspace/cli`（npm）
- **最新バージョン：** v0.8.0（2026年3月7日時点）
- **実装言語：** Rust（99.3%）
- **ライセンス：** Apache-2.0
- **⚠️ 重要：Googleの公式サポート製品ではない**

### 1-2. アーキテクチャの特徴

gws は静的なコマンドリストを持たず、Google の **Discovery Service** をランタイムで読み込みコマンドツリーを動的に生成する。Google Workspace が新しい API エンドポイントを追加すると、gws は自動的にそれを取り込む。

処理フローは以下の通り：

1. `argv[1]` でサービスを特定（例：`drive`）
1. サービスの Discovery Document を取得（24時間キャッシュ）
1. `clap::Command` ツリーをドキュメントから構築
1. 残りの引数を再パース
1. 認証 → HTTPリクエスト構築 → 実行

すべての出力は構造化JSONで返される。

### 1-3. 対応サービス

Drive、Gmail、Calendar、Sheets、Docs、Slides、Chat、Forms、Keep、Meet、Tasks、People（Contacts）、Classroom、Admin SDK（Reports / Directory）、Google Workspace Events、Model Armor

### 1-4. AI エージェント機能

- **100以上の Agent Skills（SKILL.md）** が同梱。Drive、Gmail、Calendar等のAPIごと、および高レベルなワークフロー用スキルが含まれる。
- **MCP（Model Context Protocol）サーバーモード**が搭載されており、Claude Desktop、Gemini CLI、VS Code等のMCP対応クライアントから直接 Workspace API をツールとして呼び出せる。
- Gemini CLI Extension として gemini CLI に統合することも可能。

## 2. セットアップと認証

### 2-1. インストール

```
npm install -g @googleworkspace/cli
```

npm パッケージには OS/アーキテクチャ向けのネイティブバイナリが同梱されており、Rust ツールチェーンは不要。GitHub Releases からバイナリを直接取得することも可能。

### 2-2. 認証方式

|優先度|認証方式        |設定方法                                        |
|---|------------|--------------------------------------------|
|1  |アクセストークン直接指定|`GOOGLE_WORKSPACE_CLI_TOKEN` 環境変数           |
|2  |認証情報ファイル    |`GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE` 環境変数|
|3  |暗号化済み認証情報   |`gws auth login` で設定                        |
|4  |プレーンテキスト認証情報|`~/.config/gws/credentials.json`            |

認証情報はAES-256-GCMで暗号化されてOSキーリングに保存される。

### 2-3. 初期設定フロー

- **`gws auth setup`**：GCPプロジェクトの作成・API有効化・ログインを一括実施。`gcloud` CLIが必要。
- **`gws auth login`**：OAuth ログイン。スコープを指定して実行（例：`gws auth login -s drive,gmail,calendar`）

### 2-4. 重要な制約

- テストモード（未検証）のOAuthアプリは**約25スコープが上限**。`recommended`プリセット（85スコープ以上）は失敗する。
- テストユーザーへの自アカウント追加が必要。未追加の場合「Access blocked」エラーが発生。
- OAuth クライアントは**Desktop app**タイプで作成すること。

## 3. 週次レポート自動化：ユースケース別の実現可否

### 3-1. カレンダー情報の取得（日時・参加者・タイトル・説明）

**✅ 完全に実現可能**

`gws calendar events list` で以下のパラメータが利用可能：

- `calendarId`：対象カレンダー（`primary` で自分のカレンダー）
- `timeMin` / `timeMax`：取得期間（RFC3339形式）
- `singleEvents: true`：繰り返しイベントを個別展開
- `orderBy: startTime`：開始日時順ソート

取得可能なフィールド：タイトル（summary）、開始/終了日時、参加者（attendees）のメールアドレス、説明（description）、場所、会議URL等。

### 3-2. 複数スペースにわたる自分の発言の取得

**⚠️ 実現可能だが手順が必要**

Chat API のメッセージフィルターは **`create_time`（日付範囲）と `thread.name`（スレッド指定）のみ**対応しており、送信者によるフィルターはAPIレベルで直接サポートされていない。

そのため以下の手順が必要：

1. `gws chat spaces list` で参加中のスペース一覧を全量取得
1. 各スペースに対し `gws chat spaces messages list` を期間フィルター付きで実行
1. 取得したJSONを `jq` で `sender.name` または `sender.displayName` でフィルタリング

スペース数が多い場合はシェルのループ処理が必要。

### 3-3. 指定スペースの会話履歴（スレッド含む）の取得

**✅ 実現可能**

- スペース全体のメッセージは `createTime` フィルターで期間指定して取得できる
- **スレッドフィルター：** `thread.name = "spaces/SPACE_ID/threads/THREAD_ID"` で絞り込み可能
- **制約：** 1クエリに指定できるスレッドは1件のみ。全スレッドを網羅するには、スレッド一覧を先に取得してからループする設計が必要

### 3-4. 複数報告書を統合した包括レポートの作成

**✅ Claude Code / MCP 連携で最も効率的に実現可能**

`gws mcp -s calendar,chat,drive` でMCPサーバーを起動し、Claude Desktopや他のMCP対応クライアントから自然言語でクエリを発行できる。

## 4. アカウントアクティビティ：セキュリティ監視

### 4-1. Reports API で取得可能なセキュリティイベント

|イベント種別    |applicationName|主要 eventName                              |
|----------|---------------|------------------------------------------|
|ログイン監査    |login          |login, logout, login_failure              |
|管理者操作     |admin          |CREATE_USER, DELETE_USER, GRANT_ADMIN_ROLE|
|OAuthトークン |token          |AUTHORIZE, REVOKE                         |
|SAML認証    |saml           |SAML_ACCESS 等                             |
|Driveアクセス |drive          |view, edit, download, share               |
|モバイルデバイス  |mobile         |DEVICE_REGISTER_UNREGISTER_EVENT          |
|Chromeブラウザ|chrome         |login, remove_user, unsafe_site_visit 等   |

Reports API の最大保持期間は**180日間**。

### 4-2. 2025年10月〜の拡張

2025年10月に Admin 監査ログのイベント名・タイプ・量が拡張された。既存のクエリ・アラート・レポートには修正が必要な場合がある（旧イベントと新イベントは一定期間並行提供）。

2025年12月には Reports API のフィルタリング機能が強化され、ラベルとリソースによる詳細な絞り込みが可能になった。

### 4-3. 即時アラートが必要な高優先イベント

- 深夜帯（22:00〜6:00）の管理者ログイン
- 同一ユーザーの連続ログイン失敗（5回以上/10分）
- 海外IPアドレスからのアクセス（通常業務地域外）
- 新規OAuthアプリへの認可（管理者未承認アプリ）
- 管理者ロールの付与・剥奪
- 大量ダウンロード（1時間で100ファイル超）
- 外部ドメインへの共有設定変更

### 4-4. gws admin-reports スキルの概要

`gws admin-reports` は Google Workspace Admin SDK の監査ログ・使用状況レポートにアクセスする。主要メソッド：

- `activities list`：特定のアプリケーションのアクティビティ一覧を取得
- `activities watch`：アカウントアクティビティのプッシュ通知を開始
- `activities stop`：通知チャンネルの停止
- `customerUsageReports get`：全ユーザーの使用状況レポートを取得
- `userUsageReports get`：ユーザー別の使用状況レポートを取得

## 5. 生成AI（Gemini）活用状況の把握

### 5-1. Gemini 監査ログの提供開始

**2025年7月**より、Gemini in Workspace Apps のすべてのユーザー活動が Reports API から取得可能になった（`applicationName=gemini_in_workspace_apps`）。同時に、管理コンソールの監査・調査ツールおよびセキュリティ調査ツールにも反映されるようになった。

### 5-2. 取得可能な情報

- ユーザーがWorkspaceアプリ内でGeminiを使って実行したアクション（例：コピーの正式化、背景生成、会話のサマリ）

- 使用された機能（「Help me write」「Remove image background」「Help me organize」など）

### 5-3. 分析できる指標

|指標         |活用目的                          |

|パワーユーザーの特定 |社内AIチャンピオンの発掘・ベストプラクティス普及     |

## 6. ドキュメント生成・組織活動の可視化

### 6-1. Drive アクティビティレポートの概要

`applicationName=drive` の Reports API により、ファイルの作成・編集・共有・ダウンロード・閲覧がすべて記録される。取得可能なフィールド：

- `actor.email`：操作ユーザー
- `id.time`：タイムスタンプ
- `events[].name`：イベント種別（create / edit / view / download / share / copy 等）
- `events[].parameters[]`：`doc_id`、`doc_title`、`doc_type`、`owner`、`visibility` 等

### 6-2. 可視化できる活動指標

|可視化項目                     |把握できること                              |
|--------------------------|-------------------------------------|
|日別・週別ドキュメント作成数推移          |繁忙期・プロジェクト開始時期の特定                    |
|ユーザー別生産量（create イベント数）    |高生産性メンバーの把握                          |
|ファイルタイプ分布（doc_type 集計）    |Docs/Sheets/Slides/Formsなどドキュメント文化の傾向|
|外部共有率（visibility=external）|データ流出リスクの定量評価                        |
|組織ユニット（OU）別活動量            |部署間の生産性比較                            |

### 6-3. 注意事項

- すべての Drive アクティビティが記録されるわけではない（ログされないイベントが存在）
- ファイルの所有者がサポート対象エディションのライセンスを持つ場合のみ高度な監査イベントが記録される

## 7. ハブドキュメントの特定と可視化

### 7-1. ハブ度スコアの設計

「仕事の起点・ハブ」になっているドキュメントを定量化するための複合スコア指標：

|指標         |推奨ウェイト|取得イベント                      |
|-----------|------|----------------------------|
|ユニーク閲覧ユーザー数|40%   |drive audit: view           |
|編集参加ユーザー数  |30%   |drive audit: edit           |
|共有・リンク送信数  |20%   |drive audit: acl_change     |
|コピー・ダウンロード数|10%   |drive audit: copy / download|

### 7-2. 実装フロー

1. 過去30日間の Drive アクティビティを全量取得（`--page-all` オプション）
1. `doc_id` をキーに view / edit / share / copy イベントを集計
1. ユニークユーザー数・総アクション数でスコアリング
1. TOP50ドキュメントをリスト化（タイトル・オーナー・URL付き）
1. 週次でスコア変動をトラッキングし、急上昇ドキュメントをアラート
1. Drive Files API でメタデータ（最終更新日・共有設定）を補完

## 9. 長時間労働リスク：時間外活動の可視化

### 9-1. データソースと取得方法

Reports API の各タイムスタンプ（`id.time`）を時間帯別に集計することで、深夜・早朝・休日の活動パターンを検出できる。

|データソース     |applicationName   |取得できる時間外活動     |
|-----------|------------------|---------------|
|Googleドライブ |drive             |深夜のファイル作成・編集・閲覧|
|Gmail      |gmail             |深夜の送受信・下書き作成   |
|Googleカレンダー|calendar          |深夜の予定作成・承諾     |
|Google Meet|meet              |深夜の会議開催・参加     |
|Google Chat|chat（usage report）|深夜のメッセージ送信     |

### 9-2. 検出すべきパターン

- 22:00〜翌6:00 の活動（深夜帯）
- 土日の活動時間が平日を超えるパターン
- 継続4週間以上の深夜活動
- 週60時間超の活動量が疑われるユーザー

### 9-3. 推奨対応

- 月次レポートで「深夜活動TOP20ユーザー」をマネージャーへ共有
- 継続4週間以上の深夜活動ユーザーには人事・上長への自動通知
- Gemini の時間外利用状況も組み合わせてオーバーワーク度を評価
- データは個人識別より**チーム単位の集計を優先**し、プライバシーに配慮

## 10. 追加ベストプラクティス

### 10-1. オフボーディング自動化

退職者アカウントの対応漏れはセキュリティリスクの主因。gws CLI で以下の処理を自動化できる：

- アカウントのサスペンド（`admin directory users update` で `suspended: true`）
- 所有ファイルの後任者への移管（`drive files list` + `permissions create`）
- OAuthトークンの一括失効
- グループからの除外

### 10-2. ライセンス最適化

|チェック項目                 |目的          |
|-----------------------|------------|
|90日間ログインなしのアカウント抽出     |不要ライセンスの削減  |
|Gemini利用ゼロのライセンス保有者レポート|コスト対効果の評価   |
|サービス利用率レポートで実利用率を可視化   |アプリ別の費用対効果  |
|グループ所属が0件のユーザー特定       |組織構造の整合性確認  |
|重複アカウント（別名・エイリアス）の検出   |統合によるライセンス削減|

### 10-3. DLP（データ損失防止）モニタリング

|リスクシナリオ       |検出方法                                              |対応アクション            |
|--------------|--------------------------------------------------|-------------------|
|個人情報の外部共有     |drive: visibility=external ＋ doc_title に「個人情報」等を含む|共有設定の即時取り消し・当事者通知  |
|大量ダウンロード      |download イベント集計（1時間で100件超）                        |アカウント一時ロック・セキュリティ調査|
|未知アプリへのOAuth認可|token: AUTHORIZE ＋ 未承認アプリ                         |トークン失効・アプリ承認フロー強制  |
|管理者権限の不正使用    |admin ロール付与の即時監視                                  |変更の差し戻し・インシデント起票   |

### 10-4. Google Meet 会議効率化の可視化

Google Meet utilization report から会議文化の課題を発見できる：

- 平均会議時間と参加人数の分布（過大な会議の特定）
- 会議中のカメラOFF率（エンゲージメント低下のシグナル）
- 繰り返し会議（週次・隔週）の総消費時間を部署別に算出
- 会議なし時間帯（フォーカスタイム）の確保状況
- 社外との会議比率（営業・CS等の外部活動量の把握）

### 10-5. Vault × Reports API によるコンプライアンス対応

- 調査対象ユーザーのDrive/Gmail全活動をVaultで保全 → Reports APIで時系列再現
- 保全ポリシーの対象アカウント変更を admin audit で自動記録
- 定期的なコンプライアンスレポート生成（四半期・年次）
- GDPR・個人情報保護法対応のデータ所在確認

### 11-3. MCP連携によるAIエージェント化

`gws mcp` コマンドで MCP サーバーを stdio 経由で起動し、Claude Desktop 等の MCP 対応クライアントから直接 Google Workspace API をツールとして呼び出せる。これにより「先週のセキュリティインシデントをまとめて」「Gemini利用率が低い部署を教えて」といった自然言語クエリが直接実行可能になる。

サービスはカンマ区切りで指定（例：`gws mcp -s calendar,admin-reports,drive,chat`）。各サービスは約10〜80ツールを追加するため、クライアントのツール上限（一般的に50〜100）を考慮して必要最小限に絞ることが推奨される。

### 11-4. Chat API 固有の制約

- **メッセージフィルター：** `create_time`（日付範囲）と `thread.name`（スレッド指定）のみ対応。送信者フィルターは非対応
- **スレッド指定：** 1クエリにつき1スレッドのみ
- **Chat 利用：** Google Workspace アカウント必須。個人の `@gmail.com` アカウントは非対応
- **Reports API の Chat メトリクス：** 2024年時点で48の新メトリクスが追加。`customerUsageReports.get()` と `userUsageReports.get()` が利用可能

## 12. 制約・注意事項

### 12-1. gws CLI の制約

|制約          |内容                                            |
|------------|----------------------------------------------|
|公式サポート外     |Googleの公式サポート製品ではない。v1.0未満で破壊的変更のリスクあり        |

|Chat送信者フィルター|APIレベルで送信者絞り込みができず、取得後に処理が必要                  |
|スレッド一覧      |スレッドIDを事前に把握する必要がある                           |

### 12-2. Reports API の制約

- **保持期間：** 最大180日間
- **対象：** サポートエディションのライセンスが必要（全イベント取得に管理者権限が必要）
- **ログされないイベント：** 一部の Drive 操作（Item content prefetched など）は条件付きでのみ記録
- **タイムラグ：** データが利用可能になるまでに遅延が生じる場合がある

### 12-3. プライバシー・法的考慮事項

- Drive アクティビティレポートは**合法的な目的のみ**で使用可能（利用規約に明記）
- 個人の活動データの収集・利用は就業規則・プライバシーポリシーに準拠する必要がある
- 長時間労働監視などの用途ではチーム単位の集計を優先し、個人特定・監視的な利用を避けることが推奨される
- 日本では個人情報保護法に基づくデータ取り扱い規定を確認すること
