# Mobo アプリケーションコンセプト

## 1. 概要

Mobo は、GitHub をハブにして、人間と複数のローカル AI エージェントを対等なチームとして運用するためのアプリケーションコンセプトです。単なるチャットアプリやコード生成ツールではなく、人間が非同期に意図を渡し、AI が役割に応じて解釈し、自律的に作業し、レビュー可能な形で成果を返すための運用レイヤーを提供します。

この構想では、性質の異なる 2 台のマシンを前提とします。

- 人間に近い場所で動き、対話、要件整理、レビューを担うモビリティ機
- 常時稼働し、重い生成、実装、反復処理を担う計算特化機

Mobo は、この 2 台を別々の端末としてではなく、1 つのエージェントチームとして統合することを目指します。

## 2. 解決したい課題

既存の LLM 利用は、チャットごとに文脈が分断され、プロンプトはその場しのぎになりやすく、マシンごとの実行環境もバラバラになりがちです。その結果、次の問題が起こります。

- 意思決定の経緯が散逸し、あとから続きがやりにくい
- 人間の意図と AI の実行結果のつながりが追いにくい
- ローカル LLM 環境を複数端末で活かし切れない
- 外出先の軽量端末と自宅の高性能マシンを連携させにくい

Mobo は、プロンプト、意思決定、実行、レビュー、ふりかえりを GitHub とローカル LLM 基盤の上で一貫したチームワークフローに変換することで、この分断を解消します。

## 3. プロダクトビジョン

Mobo のビジョンは、人間と AI が主従ではなくチームメイトとして協働するための「チーム OS」をつくることです。

人間は、直感、方向づけ、最終判断、物理世界との接点を担います。AI エージェントは、曖昧な意図の論理化、高速実行、並列処理、継続的な記憶を担います。Mobo が目指すのは、この協働関係が、自然で、信頼できて、時間や場所をまたいでも継続する状態です。

## 4. 中核コンセプト

Mobo は、次の 4 つの考え方を土台にしています。

### 4.1 フラットなチームメイトモデル

人間は命令者ではなく、Navigator / Anchor としてチームに参加します。AI は使い捨ての補助ツールではなく、責務を持った仲間として振る舞います。

### 4.2 役割分担されたマルチエージェント運用

エージェントは互換的な存在ではなく、ハードウェア特性と役割によって分担されます。

- Mobility エージェント: スクラムマスター兼コミュニケーター
- Compute エージェント: リードエンジニア兼メインワーカー

これにより、即応性と処理能力を両立します。

### 4.3 GitHub を共有作業空間として使う

Issue、Pull Request、ログ、承認フローを GitHub に集約し、人間の意図をモバイル環境から起票し、構造化されたタスクへ変換し、遠隔の計算機で実行し、レビュー可能な成果物として戻せる状態をつくります。

### 4.4 記憶と進化を内蔵する

運用ログ、ふりかえり、チームルールを副産物ではなくプロダクトの一部として扱います。Mobo は一度きりの出力生成ではなく、使うほど協働の質が上がるシステムとして設計されます。

## 5. 想定ユースケース

主な利用領域は、次の 2 つです。

- 開発支援
- 文書作成

具体的には、以下のような使い方を想定します。

- 外出先で思いついた曖昧なアイデアから実装計画を立てる
- 自宅の GPU マシンに長時間の生成やコーディングを委譲する
- モビリティ機側で結果を確認し、修正判断やマージ判断を行う
- 議事録やログを残し、別セッションや別 LLM に同じ文脈を引き継ぐ

## 6. 体験設計のイメージ

Mobo が提供する体験は、同期的な会話というより、非同期のリレー運用です。

1. 人間がアイデアや依頼を Issue やプロンプトとして投げる。
2. Mobility エージェントが意図を整理し、実行パケットへ翻訳する。
3. Compute エージェントが重い生成、実装、比較、反復試行を行う。
4. 結果が GitHub 上のログ、成果物、Pull Request として返る。
5. 人間が最終承認、修正、マージを行う。
6. チームが短いふりかえりを残し、次回以降の協働に活かす。

毎回チャットをゼロから始めるのではなく、信頼できる専門家同士が仕事を受け渡していく感覚を目指します。

## 7. システム構成の考え方

基盤としては、次の要素を組み合わせます。

- mubo によるローカル LLM 実行環境のブートストラップ
- Ollama を使ったモデル提供
- 長コンテキスト派生モデルなど、マシンごとの最適化
- Tailscale を有力候補とした、端末間の安全なリモート接続
- GitHub 中心のタスク管理とレビュー導線

重要なのは、セットアップの複雑さをブートストラップで隠蔽することです。たとえ Ollama が既に入っていても、モデル配備、コンテキスト長調整、エージェント UI、実行条件の再現性を両マシンで揃えること自体に価値があります。

## 8. ガバナンスと安全性

Mobo は、自律性を高めつつ暴走を防ぐために、レイヤー化された統治モデルを前提にしています。

### 8.1 不変の原則

人間レビューを迂回しない、危険な振る舞いをしない、といった絶対ルールを最上位に置きます。

### 8.2 アイデンティティ層

各エージェントの役割と人格の核を固定し、チームとしての振る舞いを予測可能にします。

### 8.3 進化可能なチームルール

運用ルールは改善可能ですが、変更は Pull Request のようなレビュー可能な経路を通して行います。

### 8.4 圧縮されたふりかえり

ふりかえりは短く制約された形式にし、ノイズではなく再利用可能な知見として蓄積します。

## 9. 差別化ポイント

Mobo が一般的な AI チャットツールと違うのは、次の 3 点です。

- LLM を単一の汎用アシスタントではなく、役割を持つ協働者として扱う
- ローカル実行、遠隔委譲、人間の承認を 1 つの流れに統合する
- ログと記憶を成果物と同じくらい重要なプロダクト資産として扱う

つまり価値の中心は、単発の生成結果ではなく、継続性、説明可能性、チーム知の蓄積にあります。

## 10. 非目標

この段階の Mobo は、次のものを主目的にはしません。

- 一般消費者向けのチャットボット
- 人間承認を不要にする完全自律システム
- 汎用クラウドエージェント基盤
- GitHub や既存開発ツールそのものの置き換え

Mobo の役割は、人間の判断、ローカル AI 実行、リポジトリベースの仕事管理をつなぐことです。

## 11. 成功条件

このコンセプトは、次の状態を実現できれば成立したと言えます。

- 人間がどこからでも低摩擦で仕事を起票できる
- 2 台のマシンが別個の役割を持つ協調エージェントとして機能する
- ログや GitHub 成果物を通して文脈をあとから再開できる
- 重い処理を計算特化機へ委譲しても、可視性と制御を失わない
- プロンプトの属人化ではなく、ふりかえりとルール更新で運用品質が上がる

## 12. コンセプト要約

Mobo は、ローカル LLM 環境を前提とした GitHub 中心のマルチエージェント協働アプリケーションです。目的は、人間と専門化された AI エージェントが、端末や時間やタスク種別をまたいで、継続的に協働できるチームをつくることにあります。

設計の中心にあるのは、モデル性能の最大化ではなく、役割の明確さ、非同期委譲、安全な自律性、そして記憶による継続性です。Mobo は、ばらばらのマシン、プロンプト、セッションを、進化していく 1 つのチームに変えるための協働レイヤーとして位置づけられます。
