# Google Workspace CLIを活用した行動ログ・意思決定ログ取得設計

## 前提：gws CLIの特性理解

`gws`はGoogle Discovery Serviceからリアルタイムにコマンドを構築するRust製CLI。Calendar, Gmail, Chat, Drive, Sheetsを単一コマンドで操作可能で、**全出力がJSON**という点がスクリプト連携・AI処理に強い。

```bash
npm install -g @googleworkspace/cli
gws auth setup  # Google Cloud ProjectのOAuth設定
gws auth login -s calendar,gmail,chat
```

---

## あなたの案の批判的検証

### 案1：Googleカレンダーの1週間分の行動ログ

**評価：有効だが不完全**

カレンダーは「予定された行動」であり「実際の行動」ではない。ミーティングのキャンセル・変更・時間延長・飛び込み対応は取れない。意思決定の文脈（なぜそのミーティングが入ったか）も欠落する。

→ **補完策**：後述するGmailのやり取りとの時系列マッチングが必要。

### 案2：自分のアカウントの発言ログ（Gmail）

**評価：最も意思決定文脈が濃いが、ノイズも最大**

Gmailには指示・承認・却下・依頼・報告が混在する。「どのような指示をしたか」を抽出するには送信済みメールのみを対象にし、かつAIによるラベリングが必要。生ログをレポートにしても読めない。

→ **問題点**：スレッド構造を無視した単発メッセージの取得では文脈が死ぬ。

### 案3：特定のチャットスペースログ

**評価：意思決定ログとして最も密度が高い**

Google Chatのスペース（特にプロジェクトチャンネル）は、短期間に意思決定・承認・却下・ネクストアクションが集中する。ただしスペースIDの特定が前提条件。

## 根本的な問題提起：「可視化から着手」は正しいか？

**部分的に間違っている。**

可視化はゴールではなく手段であり、先に「何を意思決定と定義するか」を決めないと、ログは取れても意思決定ログにはならない。

暗黙知の外化という目的に対して、可視化フェーズの前に必要なことがある：

| フェーズ | 内容 |
|---|---|
| **0. 定義** | 「意思決定」の操作的定義（承認行為？方針変更？タスクアサイン？） |
| **1. 収集** | gws CLIでログを機械的に取得 |
| **2. 分類** | AIで意思決定タイプをラベリング |
| **3. 可視化** | パターンと頻度の可視化 |
| **4. 構造化** | 暗黙プロセスをフローチャート・判断木として記述 |

可視化から着手するのは「2. 分類」の結果が出てから。生ログの可視化は意味をなさない。

## 推奨アーキテクチャ

### データソースと取得コマンド

#### カレンダー（行動ログ）

```bash
# 1週間分のイベント取得（今日から7日間）
gws calendar events list \
  --params '{
    "calendarId": "primary",
    "timeMin": "2026-03-15T00:00:00Z",
    "timeMax": "2026-03-22T00:00:00Z",
    "singleEvents": true,
    "orderBy": "startTime",
    "maxResults": 100
  }' \
  --page-all > calendar_week.json
```

取得できる情報：主催者、参加者、所要時間、繰り返し頻度、会議説明文（アジェンダ）

#### Gmail（意思決定・指示ログ）

```bash
# 送信済みメールのみ（自分の指示・承認行為）
gws gmail users messages list \
  --params '{
    "userId": "me",
    "labelIds": ["SENT"],
    "q": "after:2026/03/15 before:2026/03/22",
    "maxResults": 200
  }' \
  --page-all > sent_messages_ids.json

# メッセージ本文の取得（スレッド形式）
gws gmail users threads list \
  --params '{
    "userId": "me",
    "q": "after:2026/03/15 before:2026/03/22 in:sent",
    "maxResults": 100
  }' > sent_threads.json
```

#### Google Chat（意思決定・ネクストアクションログ）

```bash
# スペース一覧の取得
gws chat spaces list \
  --params '{"pageSize": 20}' > spaces.json

# 特定スペースのメッセージ取得
gws chat spaces messages list \
  --params '{
    "parent": "spaces/SPACE_ID",
    "pageSize": 100,
    "filter": "createTime > \"2026-03-15T00:00:00Z\""
  }' \
  --page-all > chat_messages.json
```

### AI処理パイプライン（推奨）

生ログをそのままレポートにするのではなく、以下のAI処理を挟む。

```
[gws取得JSON]
    ↓
[Claude API / Gemini API]
    ↓ プロンプト設計が核心
[意思決定タイプ分類]
  - 承認/却下
  - 方針決定
  - タスクアサイン
  - 情報共有（意思決定なし）
  - 問題提起
    ↓
[構造化出力]
  - 意思決定ログ（日時・内容・関係者・決定タイプ）
  - ネクストアクション抽出
  - 繰り返しパターンの検出
    ↓
[Sheets書き込み or レポート生成]
gws sheets spreadsheets values append
```

---

### 自動化・AI促進への接続

暗黙知の外化という目的から逆算すると、最終的に必要なのは：

1. **繰り返し意思決定パターンの検出** → 自動化候補の特定
2. **判断基準の言語化** → AIエージェントへの委譲可否の判断
3. **承認フローのマッピング** → ボトルネック人物・工程の特定

このためには最低4週間分のログが必要。1週間では季節変動・週次/月次の固定業務と非定常業務の区別ができない。

## 実装優先順位（修正提案）

| 優先度 | タスク | 理由 |
|---|---|---|
| **1** | 意思決定の操作的定義を決める | これなしに収集しても分類不能 |
| **2** | gws認証セットアップ + 主要3ソースのデータ収集スクリプト作成 | 機械的・再現性のある収集基盤 |
| **3** | Claude APIでの分類プロンプト設計と検証（少量データ） | ラベリング精度の確認 |
| **4** | 4週間データでのパターン分析 | 統計的に意味のある量 |
| **5** | Sheets自動書き込みによるダッシュボード化 | 可視化はここで初めて登場 |

## 注意事項

- **スコープ制限**：OAuth未検証アプリは約25スコープまで。`gws auth login -s calendar,gmail,chat` で必要なものだけ指定する
- **Chat APIの制限**：Google Chat APIはWorkspace Business/Enterprise以上が必要なケースあり。個人アカウントでは取得できないスペースがある
- **プライバシー・ガバナンス**：他者の発言ログを取得・AI処理する場合、組織のデータポリシー確認が必要
- **レート制限**：Gmail APIはユーザーあたり250quota units/秒。`--page-delay 500` を入れると安全
