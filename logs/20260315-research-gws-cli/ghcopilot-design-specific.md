# Google Workspace CLI を活用した行動ログ・意思決定ログ可視化
## ユースケース設計仕様書（草案）

- 対象リポジトリ: `googleworkspace/cli`
- Repository ID: `1171026502`
- 対象CLI: `gws`
- 作成日: 2026-03-15
- 文書目的:
  Google Workspace 上に分散した行動ログ・発話ログ・議論ログを収集し、
  組織内で暗黙的に運用されている意思決定プロセスや業務プロセスを可視化・構造化し、
  将来的なAI活用・自動化に接続するためのユースケース設計の草案を示す。

---

## 1. 背景

多くの組織では、重要な意思決定や業務推進が、明文化されたワークフローではなく、
以下のような非構造なコミュニケーションに依存しています。

- カレンダー上の会議の連なり
- Google Chat での相談・確認・承認
- 特定個人の発言や調整による暗黙の意思決定
- 会議後に明示されないまま進行するネクストアクション

この状態では、以下の課題が生じます。

- 何がどこで決まったか追跡しづらい
- 意思決定の根拠が再利用されない
- 属人的な判断・調整に依存する
- AI や自動化に必要な「文脈」が構造化されていない

---

## 2. 本取り組みの目的

`gws` を用いて Google Workspace 上のデータを収集し、以下を実現することを目的とします。

1. **行動ログの可視化**
   - 誰が、いつ、何に時間を使っているかを把握する

2. **意思決定ログの抽出**
   - 誰が、どこで、どのような指示・判断・合意形成を行ったかを記述可能にする

3. **ネクストアクションの構造化**
   - 議論から実務に接続するアクションを抽出し、運用改善や自動化候補を発見する

4. **LLMに渡せる文脈の整備**
   - 生ログではなく、要約・重み付け・関係性を含む構造化コンテキストに変換する

---

## 3. 基本方針

本取り組みでは、いきなり完全自動化を目指すのではなく、以下の順序で進めます。

1. 観測可能化
2. 可視化
3. 意味づけ
4. LLMによる解釈
5. 運用介入
6. 自動化

### 補足
「まずは可視化から始める」という方針は妥当です。  
ただし、可視化だけでは改善に直結しないため、初期フェーズから以下を意識する必要があります。

- 何を「意思決定」とみなすか
- 何を「ネクストアクション」とみなすか
- どのログがノイズで、どのログが重要か
- 誰を対象にし、どの範囲でPoCするか

そのため、初期段階では  
**可視化 + 意思決定/アクション抽出の仮説検証**  
として設計するのが望ましいです。

---

## 4. 対象ユースケース一覧

本仕様書では、以下の3ユースケースを対象とします。

- **UC-01**: Googleカレンダーの1週間分の行動ログを取得し、活動レポートを作成する
- **UC-02**: 特定個人の1週間分の発言を取得し、指示・判断・意思決定促進行為をレポートする
- **UC-03**: 特定のGoogle Chatスペースのログを取得し、意思決定ログとネクストアクションをレポートする

---

# 5. ユースケース詳細

---

## UC-01: Googleカレンダーの1週間分の行動ログ可視化

### 5.1 目的
個人または特定ロールの活動実態を、カレンダーイベントから可視化する。  
会議の量や種類、集中しているテーマ、参加者構成を把握し、
「何に時間を使っているか」「どの活動が業務上の重心になっているか」を見える化する。

### 5.2 想定利用者
- 経営層
- マネージャー
- 業務改善担当
- AI/自動化推進担当

### 5.3 入力データ
- Google Calendar イベント
  - タイトル
  - 開始日時 / 終了日時
  - 主催者
  - 参加者
  - 説明文
  - ロケーション / Meet URL
  - 繰り返し設定

### 5.4 利用CLI
- `gws calendar events list`

### 5.5 取得条件
- 対象期間: 直近7日
- 対象カレンダー: `primary`
- 必要に応じて、私用予定や非公開予定を除外
- 必要に応じて定例会議を識別

### 5.6 想定コマンド例
```bash
gws calendar events list \
  --params '{"calendarId":"primary","timeMin":"2026-03-08T00:00:00Z","timeMax":"2026-03-15T23:59:59Z","singleEvents":true,"orderBy":"startTime"}' \
  --fields "items(id,summary,description,start,end,attendees,organizer,location,recurrence)"
```

### 5.7 出力
#### 人間向けレポート
- 日別会議件数
- 総会議時間
- テーマ別会議比率
- 参加者数分布
- 社内/社外会議比率
- 集中時間の欠損状況
- recurring meeting 比率

#### LLM向け中間構造
- `activity_summary`
- `meeting_clusters`
- `time_allocation_by_topic`
- `frequent_counterparties`

### 5.8 期待される発見
- 特定人物が調整役に偏っていないか
- 意思決定のための会議が過剰でないか
- 重要テーマが会議名の揺れで分散していないか
- 定例化されているが標準化されていない業務がないか

### 5.9 留意点
- カレンダーは「予定」であり実績ではない
- タイトルだけでは意思決定の強さは判断できない
- 非公開予定の扱いに配慮が必要

### 5.10 評価指標
- 会議分類の妥当性
- 活動テーマ抽出の納得度
- レポートの有用性評価
- 改善候補会議の発見件数

## UC-02: 特定個人の1週間分の発言から意思決定促進ログを抽出

### 6.1 目的
特定個人が組織内でどのような指示、問い、承認、論点整理を行っているかを抽出し、
暗黙的なリーダーシップ行動や意思決定促進行為を可視化する。

### 6.2 想定利用者
- 本人
- 上長
- 組織開発担当
- AIエージェント設計担当

### 6.3 入力データ
- Google Chat メッセージ
- 必要に応じて Gmail 送信メッセージ
- 投稿日時
- 投稿先スペース
- スレッド情報
- 返信関係

### 6.4 利用CLI
主対象:
- `gws chat spaces messages list`

補助候補:
- `gws gmail users messages list`
- `gws gmail users messages get`

### 6.5 分析観点
発言を以下の観点で分類する。

- 指示
- 依頼
- 判断
- 承認
- 却下
- 問いかけ
- 論点整理
- 優先順位付け
- エスカレーション
- ネクストアクション指定

### 6.6 想定コマンド例
```bash
gws chat spaces messages list \
  --params '{"parent":"spaces/SPACE_ID"}' \
  --fields "messages(name,text,createTime,sender,thread)"
```

必要に応じて複数スペースから収集後、送信者でフィルタする。

### 6.7 出力
#### 人間向けレポート
- 総発言数
- 発言先スペース数
- 指示/判断/問いかけ比率
- トピック別発言量
- 意思決定への寄与が高いスレッド一覧
- 周囲の反応率
- 行動タイプ別代表発言

#### LLM向け中間構造
- `decision_promoting_messages`
- `instruction_patterns`
- `topic_weight_map`
- `leadership_signal_summary`

### 6.8 期待される発見
- 本人が「決定者」なのか「交通整理役」なのか
- どの論点で頻繁に介入しているか
- 判断を明示的に行っているか、暗黙に誘導しているか
- 反復的な指示がテンプレート化可能か

### 6.9 留意点
- 発言量の多さは意思決定力と一致しない
- 非言語的な影響力は取得できない
- 個人監視として受け取られない設計が必要
- プライバシー・労務・ガバナンス上の合意が必要

### 6.10 評価指標
- 分類精度（指示/判断/依頼など）
- 本人・マネージャーの納得度
- 反復指示パターンの抽出件数
- 自動化候補フローの発見数

## UC-03: 特定チャットスペースの意思決定ログとネクストアクション抽出

### 7.1 目的
プロジェクトやチームの議論が行われる Google Chat スペースから、
意思決定事項、未解決論点、担当者付きアクションを抽出し、
実務運営の可視化と再利用可能な意思決定知識の蓄積を行う。

### 7.2 想定利用者
- プロジェクトマネージャー
- チームリーダー
- オペレーション改善担当
- AIエージェント設計担当

### 7.3 入力データ
- 特定スペースのメッセージログ
- スレッド構造
- 投稿者
- 投稿日時
- 添付リンク
- 関連ドキュメント参照情報（取得可能な範囲で）

### 7.4 利用CLI
- `gws chat spaces messages list`

将来的な補助:
- Google Drive ファイル参照
- Google Calendar 会議との突合

### 7.5 分析観点
- 何が論点だったか
- 何が決定されたか
- 誰が決めたか
- 誰が次に何をやるか
- 期限があるか
- 未解決論点は何か
- 議論がループしていないか

### 7.6 想定コマンド例
```bash
gws chat spaces messages list \
  --params '{"parent":"spaces/SPACE_ID"}' \
  --fields "messages(name,text,createTime,sender,thread,annotations)"
```

### 7.7 出力
#### 人間向けレポート
- トピック一覧
- 決定事項一覧
- 未決事項一覧
- ネクストアクション一覧
- 担当者未設定タスク一覧
- 長期化スレッド一覧
- 再議論頻出テーマ一覧

#### LLM向け中間構造
- `decision_log`
- `open_questions`
- `next_actions`
- `topic_timeline`
- `stakeholder_map`

### 7.8 期待される発見
- 決定と実行の間で抜け落ちる工程
- 担当不明のまま放置されるタスク
- 繰り返し議論される未標準化テーマ
- スペースごとの意思決定様式の違い

### 7.9 留意点
- Chat だけでは結論が閉じていない場合がある
- 実際の決定が会議・口頭・Docs 上で行われることもある
- スレッド運用が弱い場合は文脈復元が難しい
- 「誰が決めたか」は推定になる可能性がある

### 7.10 評価指標
- 決定事項抽出精度
- ネクストアクション抽出精度
- 未決論点の再現率
- プロジェクト運営改善の実感

# 8. 共通設計要件

## 8.1 データ処理要件
- 取得データは JSON で保存する
- 生ログと整形後データを分離して保持する
- 正規化済み中間テーブルを作成する
- LLM投入前に、要約単位・トピック単位に再構成する
- 個人情報・秘匿情報のマスキング方針を定める

## 8.2 LLM活用要件
LLMには生ログをそのまま渡さず、以下の段階で処理する。

1. データ取得
2. 整形
3. トピック抽出
4. 重み付け
5. 意思決定候補抽出
6. 要約
7. 人手確認
8. ナレッジ化

## 8.3 可視化要件
最低限、以下を可視化対象とする。

- 時系列
- トピック別頻度
- 意思決定数
- アクション数
- 未解決件数
- 関与人物
- 発言密度
- 再議論頻度

## 8.4 ガバナンス要件
- 対象者の同意
- 分析目的の明確化
- 評価/監視目的との分離
- アクセス制御
- ログ保持期間の定義
- 個人データ匿名化方針

# 9. 導入ステップ案

## Phase 1: 小さく観測する
- 対象: 1人 + 1スペース + 1週間
- 目的: データ品質確認

## Phase 2: 可視化する
- カレンダー/チャットの基本ダッシュボード作成
- 目的: 活動・議論の偏りを把握

## Phase 3: 意味づけする
- 意思決定・指示・アクション抽出
- 目的: LLMが扱える構造に整える

## Phase 4: 運用へ接続する
- 週次レポート
- 会議後サマリ
- スペース別アクションボード

## Phase 5: 自動化する
- 定例レポート自動生成
- 未完了アクション通知
- 意思決定テンプレート提案
- 業務フロー標準化支援

# 10. 結論

この3ユースケースは、単なるログ収集ではなく、  
**組織内に埋もれている暗黙知を、AIが扱える構造化コンテキストへ変換する起点** となる。

重要なのは以下の3点である。

1. ログ取得そのものではなく、意思決定構造の再記述が目的であること
2. 可視化は入口であり、最終目的は標準化・自動化・AI支援であること
3. 個人監視ではなく、組織知の抽出と再利用に設計思想を置くこと
