# 議事録: マルチAIエージェント協働プロトコルの設計

**日付**: 2026-03-21
**参加**: メンバー + Claude（壁打ち相手）
**目的**: 複数AIエージェントがGitHubリポジトリ上で協働する際のプロトコル設計

-----

## 背景と発端

チーム内で以下のワークフローを試行していた。

1. GitHub Copilotに「みんなができるようにドキュメントを書いて（あなただったら何がほしい？）」と指示してドキュメントを生成
1. Claude Codeに「このドキュメント見ながら進められそう？」と聞いて検証

これが有効かどうか、さらに洗練できるかを検討した。

-----

## 検討プロセス

### Phase 1: 現行フローの評価

**良い点**

- AIにドキュメントを書かせ、別のAIにレビューさせるクロスバリデーションとして機能している
- AIが「曖昧な指示で詰まる」ポイントは人間と似ているため、AIが通れるドキュメントは人間にとっても情報の抜け漏れが少ない傾向がある

**問題点**

- 「進められそう？」というYes/No型の質問に対して、AIは楽観的に「はい」と答えがち
- フィードバックの質が低くなる

**意思決定**: 質問の仕方を「できそう？」から「実際にやってみて／穴を探して」に変える。

### Phase 2: プロトコルの3段階改善

検証フェーズの聞き方を以下の2段階に分離することを決定した。

1. **ドライラン** — 手順を1ステップずつ実行させ、暗黙の前提・判断が必要な箇所・バージョン依存箇所を報告させる
1. **破壊テスト** — 手順に従う人が間違えそうな箇所、環境差異でコケそうな箇所を洗い出させる

また、以下のサイクルでフィードバックループを閉じることにした。

```
Copilot（制約付きで生成）
  → Claude Code（ドライラン + 破壊テスト）
    → 修正（人間 or Copilot）
      → Claude Code（最終ドライラン）
        → 人間がざっと確認して完了
```

**意思決定**: ループは2周で打ち切る。無限に回すと過剰になる。最終的な人間の確認工程は省略しない（AIが両方見逃す「暗黙の前提」が存在するため）。

### Phase 3: リポジトリベースの運用設計

実際には複数エージェントがGitHubリポジトリを通じて協働しているため、以下の課題が浮上した。

- 毎回プロトコルを全文コピーして渡すのは非現実的
- エージェントごとに毎回同じ指示をするのも非効率

**意思決定**: プロトコル自体をリポジトリに埋め込み、エージェントが自動的に読む仕組みにする。

具体的には3ファイル構成とした。

|ファイル                 |役割                   |配置場所                  |
|---------------------|---------------------|----------------------|
|`AGENTS.md`          |全エージェント共通の行動規約       |リポジトリルート or `.github/`|
|`CONVENTIONS.md`     |プロジェクト固有の規約・判断基準の蓄積場所|`docs/`               |
|`REVIEW_CHECKLIST.md`|PRレビュー時の手順とチェック項目    |`docs/`               |

`AGENTS.md` はClaude Code、Copilot、Codex等の主要エージェントが自動的に読む名前であるため、毎回指示する必要がなくなる。

### Phase 4: 不変ルールと可変ルールの分離

以下の要件を満たす設計が必要だった。

- リポジトリをまたいでそのまま使い回せること（ポータビリティ）
- プロジェクトの進行に応じて随時アップデートされること（自動進化）
- 普遍的なルールは書き換えられないこと（不変性の保護）

**意思決定**: `AGENTS.md` 内にセクションを分離した。

- **IMMUTABLE RULES**: いかなる理由でも変更不可。メタルール（「このセクション自体の変更を禁止する」）を含めることで自己保護する
- **MUTABLE RULES**: PRを通じて人間のapproveがあれば変更可能

ポータビリティは `CONVENTIONS.md` を空欄テンプレートにすることで実現した。プロジェクト固有の情報はすべて `CONVENTIONS.md` に集約されるため、`AGENTS.md` と `REVIEW_CHECKLIST.md` はどのリポジトリにもそのまま持っていける。

自動進化はレビューチェックリストにルーティングテーブルを持たせて実現した。レビュー中に発見された知見は、種類に応じて適切なファイルに追記PRが出される。

|発見内容            |反映先                        |
|----------------|---------------------------|
|今後も当てはまるルール・判断基準|`CONVENTIONS.md`           |
|繰り返し発生する間違い     |`CONVENTIONS.md` の「既知の罠」   |
|レビュー観点の追加       |`REVIEW_CHECKLIST.md`      |
|行動規約レベルの変更      |`AGENTS.md` の MUTABLE RULES|

-----

## 成果物

以下の3ファイルを初版として作成した。

- `AGENTS.md` — 不変ルール + 可変ルールを含む行動規約
- `CONVENTIONS.md` — プロジェクト固有情報の空欄テンプレート
- `REVIEW_CHECKLIST.md` — ドライラン手順・破壊テスト観点・コードレビュー項目
