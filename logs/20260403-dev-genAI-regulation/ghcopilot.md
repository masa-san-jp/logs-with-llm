# 生成AI利用規程（就業規則別紙）案

## 第1条（目的）
本規程は、会社における生成AIの適正かつ安全な利用に関する基本事項を定め、業務効率化および業務品質向上を図るとともに、情報漏えい、誤情報の利用、著作権侵害、不正なファイル操作、システム障害その他会社に不利益を及ぼす行為を防止することを目的とする。 ([ipa.go.jp](https://www.ipa.go.jp/jinzai/ics/core_human_resource/final_project/2024/generative-ai-guideline.html))

## 第2条（適用範囲）
1. 本規程は、役員、正社員、契約社員、派遣社員、アルバイトその他会社の業務に従事するすべての者に適用する。  
2. 本規程は、文章生成、要約、翻訳、検索補助、議事録作成、画像生成、コード生成、表計算支援、ファイル操作支援、ターミナル操作支援その他生成AIを利用する一切の行為に適用する。  
3. 会社が承認した生成AIサービスのみならず、個人契約・無償版・試用版その他名称を問わず、業務に関連して利用するすべての生成AIサービスに適用する。

## 第3条（定義）
本規程において、次の各号に掲げる用語の意義は、当該各号に定めるところによる。  
1. **生成AI**：利用者の入力に基づき、文章、画像、音声、プログラム、コマンドその他の出力を生成するシステムをいう。  
2. **会社データ**：会社が保有、管理又は業務上取り扱う文書、顧客情報、営業情報、契約情報、財務情報、人事情報、技術情報、ソースコード、設定情報、ログ、ファイル名、フォルダ構成、画面情報その他一切の情報をいう。  
3. **ローカルファイル**：会社貸与端末又は私物端末に保存されたファイル、フォルダ、キャッシュ、ダウンロードデータその他端末内のデータをいう。  
4. **ターミナル操作**：コマンドライン、シェル、PowerShell、コマンドプロンプト、スクリプト実行環境その他これに類する方法による端末又はシステムへの命令実行をいう。  
5. **承認済みAI**：会社がセキュリティ、契約、ログ管理、データ保護等を確認し、業務利用を認めた生成AIサービスをいう。  
6. **高リスク操作**：ファイルの作成・更新・削除・移動・一括変換、権限変更、外部送信、システム設定変更、ターミナル実行その他会社資産へ直接影響を及ぼす操作をいう。

## 第4条（基本原則）
社員は生成AIを利用するにあたり、次の原則を遵守しなければならない。  
1. **人間最終責任原則**：生成AIの出力をそのまま正しいものとして扱ってはならず、最終判断および最終責任は利用者本人および管理者が負う。  
2. **最小入力原則**：業務遂行に必要最小限の情報のみを入力する。  
3. **機密保護原則**：会社データ、個人情報、秘密情報を無断で入力、アップロード又は連携してはならない。  
4. **検証原則**：重要な出力は、法令、契約、社内規程、一次情報、担当部署の確認等により検証する。  
5. **権限適合原則**：利用者の権限を超えるデータ取得、ファイル操作、システム操作を行ってはならない。  
6. **記録可能性原則**：重要な利用は、後日追跡可能な形で記録されなければならない。 ([ipa.go.jp](https://www.ipa.go.jp/jinzai/ics/core_human_resource/final_project/2024/generative-ai-guideline.html))

# 利用区分

## 第5条（許容される一般利用）
次の各号に掲げる用途は、所属部門の通常業務として利用できる。  
1. 文案作成、要約、議事録草案作成  
2. 翻訳、言い換え、表現改善  
3. アイデア整理、論点抽出、構成案作成  
4. 公開情報に基づく調査補助  
5. 社内公開済みテンプレート文書の改善提案  
6. 表計算関数、資料構成、業務手順の草案作成  
7. プログラミング補助、ただし別途定める技術利用ルールに従う場合

## 第6条（原則禁止される利用）
次の各号に掲げる利用は、原則として禁止する。  
1. 個人情報、要配慮個人情報、マイナンバー、採用選考情報、評価情報、給与情報等の入力  
2. 顧客情報、取引先情報、未公開の営業情報、契約条件、秘密保持契約の対象情報の入力  
3. 未公開の経営情報、財務情報、事業計画、M&A関連情報の入力  
4. ソースコード、秘密鍵、アクセストークン、パスワード、証明書、接続情報の入力  
5. 社内限定ファイル、Google Drive上の非公開資料、共有ドライブ内情報の無断投入  
6. 生成AIの出力を確認せず、そのまま社外提出物、顧客説明、契約文書、法務文書、対外公表物に使用すること  
7. 他者の著作物・商標・営業秘密を侵害する態様での生成AI利用  
8. 差別、ハラスメント、誹謗中傷、違法行為、就業規則違反その他不適切な指示又は出力取得  
9. 会社の承認なく、生成AIにローカルファイル又はターミナルへの直接操作権限を与えること  
10. 会社の承認なく、外部AIエージェントにGoogle Workspace、Drive、Gmail、Calendarその他業務システムを接続すること。 ([ppc.go.jp](https://www.ppc.go.jp/news/careful_information/230602_AI_utilize_alert/?trk=public_post_comment-text&utm_source=openai))

# データ入力ルール

## 第7条（入力情報の分類）
生成AIへの入力可否は、次の区分に従う。  
1. **入力可**：公開情報、一般的ノウハウ、自ら作成した非機密の草案、匿名化・抽象化済み情報  
2. **条件付き入力可**：社内一般情報、社内テンプレート、匿名化済みの業務データ、統計化済みデータ  
3. **入力禁止**：個人情報、秘密情報、未公開情報、認証情報、契約上第三者提供が禁止された情報、法令上保護される情報

## 第8条（匿名化・抽象化）
社員は、やむを得ず業務情報を生成AIに入力する場合、氏名、会社名、メールアドレス、電話番号、住所、取引条件、案件名、ファイル名、URL、ID等を削除又は置換し、個人又は案件を特定できないようにしなければならない。  
特にGoogle Workspace上のデータを題材にする場合は、原本を貼り付けるのではなく、要旨又は抽象化した論点のみを入力するものとする。 ([ppc.go.jp](https://www.ppc.go.jp/news/careful_information/230602_AI_utilize_alert/?trk=public_post_comment-text&utm_source=openai))

# ローカルファイル・ターミナル操作

ここがご要望の中心だと思うので、少し実務寄りに強めに作っています。

## 第9条（ローカルファイルの原則）
1. 会社の業務データは、原則としてGoogle Workspaceその他会社指定のクラウド環境で管理し、ローカル保存は必要最小限とする。  
2. 生成AIを利用してローカルファイルを直接読み込み、変更し、削除し、移動し、又は生成物を書き込む行為は、会社が承認した仕組みを通じてのみ行うことができる。  
3. 社員は、生成AIに対し、ローカルディレクトリ全体への包括的アクセス権を付与してはならない。  
4. ダウンロードフォルダ、デスクトップ、ドキュメント、共有同期フォルダその他業務データが混在しやすい場所を、生成AIに無制限に参照させてはならない。  
5. 生成AIにより生成されたファイルは、保存前に利用者本人が内容、保存先、共有範囲および上書き有無を確認しなければならない。

## 第10条（意図しないローカル操作の防止）
社員は、次の各号を遵守しなければならない。  
1. 自動実行機能、自律エージェント機能、連続実行機能を用いる場合は、**確認画面なしにファイル変更・削除を実行しない設定**を用いること。  
2. ファイル操作を伴うAI利用は、可能な限り**読み取り専用モード**又は**プレビュー後承認モード**を利用すること。  
3. 一括処理前には、テスト用フォルダ又は複製データで事前検証を行うこと。  
4. 上書き、削除、移動、リネーム、一括変換を行う場合は、復元可能なバックアップ又はバージョン履歴を確保すること。  
5. 不明なコマンド、長いコマンド、難読化されたコマンド、パイプやリダイレクトを多用するコマンドは、その意味を理解せず実行してはならない。  
6. AIが提案したコマンドは、1行ずつ意味を確認し、必要に応じて情報システム部門又は管理者に相談すること。

## 第11条（意図的にローカルファイルを操作して業務効率化する場合）
1. 社員が生成AIを用いてローカルファイルの整理、変換、集計、命名統一、文書整形、コード修正その他の業務効率化を行う場合は、事前に所属長又は会社指定の承認権者の承認を得なければならない。  
2. 前項の承認を受けるには、少なくとも以下を明らかにしなければならない。  
   (1) 対象ファイルの種類および保存場所  
   (2) 含まれる情報の機密性  
   (3) 実施する操作内容  
   (4) 使用するAIツール名  
   (5) バックアップ方法  
   (6) ロールバック方法  
   (7) 実施者  
   (8) 実施日時  
3. 機密性が高いファイル又は多数ファイルに対する一括操作は、承認済み環境、承認済みツール、承認済み手順でのみ実施できる。  
4. AIによる処理結果を本番データへ反映する前に、サンプル検証及び利用者による目視確認を行わなければならない。  
5. 承認なき自動削除・自動移動・自動上書き・自動同期設定は禁止する。

## 第12条（ターミナル操作の原則）
1. 生成AIが提案したターミナルコマンドは、利用者の自己責任で確認し、理解した上で実行しなければならない。  
2. ターミナル操作は、会社が認めた端末、権限、ネットワーク、対象環境に限る。  
3. 本番環境、顧客環境、共有サーバ、管理者権限を要する環境でAI提案コマンドを実行する場合は、原則として事前承認を要する。  
4. AIに対し、対話の流れのみでターミナルの連続実行権限を与えてはならない。  
5. ターミナル操作を行う場合は、実行者、日時、対象、実行コマンド及び結果が追跡可能であることを要する。

## 第13条（直接操作を許容する条件）
生成AIによるローカルファイル又はターミナルの直接操作は、次の条件をすべて満たす場合に限り認める。  
1. 会社が承認したツールであること  
2. 対象範囲が限定されていること  
3. 読み取り専用又は段階承認付きであること  
4. 監査ログが保存されること  
5. バックアップ又はバージョン履歴が確保されていること  
6. 個人情報・秘密情報・認証情報を含む領域を対象外としていること  
7. 実施者および責任者が明確であること

# 全社員向けの運用ルール

## 第14条（ビジネス職を含む全社員の注意義務）
営業、企画、管理、採用、経理、法務、広報その他技術職以外の社員も、生成AI利用に伴う情報漏えい、誤記載、著作権侵害、差別的表現、誤送信その他のリスクを理解し、出力の真偽・妥当性・社外開示可否を確認しなければならない。  
議事録、メール、提案書、説明資料等は生成AIと親和性が高く、企業利用でも多い用途ですが、その分、誤情報や機密混入を起こしやすいため、最終確認を必須とする。 ([jil.go.jp](https://www.jil.go.jp/kokunai/blt/backnumber/2026/01_02/blm_special.html))

## 第15条（社外送信・公開）
生成AIの出力を社外へ送付、公開、納品、配布又は営業活動に使用する場合は、通常の社内承認手続に従うものとし、生成AIが作成したことのみをもって確認を省略してはならない。

## 第16条（権利侵害の防止）
社員は、生成AI出力について、著作権、商標権、肖像権、営業秘密その他第三者の権利を侵害しないよう確認しなければならない。  
特に、画像、コピー、提案文、設計案、コード等は既存成果物との類似性に注意しなければならない。

## 第17条（教育）
会社は、全社員に対し、生成AIの適切な利用、情報分類、入力制限、出力検証、ローカルファイル操作、ターミナルリスク等に関する教育を実施する。  
管理職および高リスク利用者には追加教育を実施する。 ([ipa.go.jp](https://www.ipa.go.jp/jinzai/ics/core_human_resource/final_project/2024/generative-ai-guideline.html))

## 第18条（事故時の報告）
社員は、次の各号に該当する事象を知ったときは、直ちに所属長及び会社指定窓口に報告しなければならない。  
1. 機密情報、個人情報、認証情報を誤って入力又は送信したとき  
2. AIによりローカルファイルの削除、改変、移動その他想定外の操作が行われたとき  
3. AI提案コマンドによりシステム障害、データ消失、権限変更、外部通信等が生じたとき  
4. 著作権侵害、誤情報公開、顧客苦情その他事故が疑われるとき

## 第19条（監査）
会社は、必要に応じ、生成AIの利用履歴、入力内容、出力内容、接続設定、ログ、承認記録その他本規程遵守状況を確認できるものとする。

## 第20条（違反時の措置）
本規程に違反した場合、就業規則、情報セキュリティ規程その他社内規程に基づき、指導、利用停止、懲戒その他必要な措置を講ずることがある。

## 第21条（主管部門）
本規程の主管部門は、情報システム部、法務部、人事部又は会社が別に定める部門とする。

## 第22条（施行）
本規程は、2026年○月○日から施行する。

# 別表3：注意すべきターミナルコマンド一覧

以下は**全社員向け注意喚起**として規程に添付しやすい形です。  
「禁止」ではなく、**原則として単独判断で実行禁止**、もしくは**意味を理解しない限り実行禁止**と書くのが実務的です。

## 1. 削除系
- `rm`
- `rm -rf`
- `del`
- `rmdir`
- `Remove-Item -Recurse -Force`

**注意点**  
ファイルやディレクトリを削除する。`-r` `-f` `-rf` や再帰削除は特に危険。  
共有フォルダ、同期フォルダ、マウント先で実行すると被害が広がる。

## 2. 移動・上書き系
- `mv`
- `move`
- `cp`
- `copy`
- `xcopy`
- `robocopy`
- `rsync`
- `scp`

**注意点**  
意図しない上書き、階層移動、同期、複製漏えいを起こす。  
`rsync --delete` は特に危険。

## 3. 権限変更系
- `chmod`
- `chown`
- `icacls`
- `takeown`

**注意点**  
アクセス権を変更し、漏えい・利用不能・監査不能を招く可能性がある。

## 4. 実行・取得系
- `curl | sh`
- `wget ... | sh`
- `bash <(curl ...)`
- `Invoke-WebRequest` と `iex` の組合せ
- `powershell -ExecutionPolicy Bypass`
- `sh`, `bash`, `zsh`, `python`, `node` での不明スクリプト実行

**注意点**  
外部から取得した内容をそのまま実行するため、極めて危険。  
AI提案ではこの種のコマンドが混ざることがある。

## 5. ディスク・システム操作系
- `dd`
- `diskpart`
- `format`
- `mkfs`
- `fdisk`
- `mount`
- `umount`
- `shutdown`
- `reboot`
- `systemctl`
- `service`

**注意点**  
端末、ディスク、サービス停止など重大な影響がある。

## 6. 一括変換・置換系
- `find ... -exec`
- `sed -i`
- `perl -pi`
- PowerShell の `Get-ChildItem | % { ... }`
- 一括 rename スクリプト

**注意点**  
大量ファイルに一度に反映され、誤りが広範囲化する。

## 7. 圧縮・展開・転送系
- `tar`
- `zip`
- `unzip`
- `7z`
- `gcloud storage cp`
- `aws s3 cp/sync`
- `gsutil cp/mv/rsync`

**注意点**  
クラウド転送・一括展開で情報流出や意図しない公開を招く。  
Google Workspace前提の会社では、Drive以外の外部ストレージ転送は特に管理対象にすべき。

## 8. バージョン管理系
- `git add .`
- `git commit`
- `git push`
- `git reset --hard`
- `git clean -fd`
- `git checkout --`
- `git restore`
- `git rebase`
- `git filter-repo` / 履歴改変系

**注意点**  
コードだけでなく設定ファイル・秘密情報・誤ファイルを含めて公開・消去するおそれがある。

## 9. 管理者権限系
- `sudo`
- `su`
- 管理者権限 PowerShell / cmd
- `runas`

**注意点**  
影響範囲が端末全体・システム全体に広がる。  
AI提案で安易に使ってはいけない。

# 別表4：全社員向けの短い運用ルール（配布用サマリ）

規程本文とは別に、実際にはこの短縮版を配ると浸透しやすいです。

## やってよい
- 議事録のたたき台作成
- メール文面の改善
- 公開情報の要約
- 匿名化した情報での相談
- 文書構成やアイデア出し

## やってはいけない
- 顧客情報や個人情報を貼る
- 契約書や未公開資料をそのまま入れる
- APIキーやパスワードを入れる
- AIの提案コマンドを意味不明のまま実行する
- AIにローカルフォルダ全体を自由に触らせる
- AIにDrive/Gmailを勝手に連携する

## 迷ったら
- 抽象化して入力する
- 出力を必ず確認する
- 一括操作は承認を取る
- 削除・上書き前にバックアップする
- 不明コマンドは情シスへ確認する

# この規程案のポイント
この案の肝は、**「ローカルやターミナルを全面禁止しない」代わりに、以下で統制する**ことです。

1. **原則クラウド管理**
2. **ローカル直接操作は承認制**
3. **AIの直接操作は承認済みツールのみ**
4. **読み取り専用・プレビュー承認・ログ保存を条件化**
5. **一括処理はバックアップ必須**
6. **全社員向けに入力禁止情報を明確化**
7. **技術職以外にもそのまま適用できる表現にする**

これは、企業で生成AI活用が文書作成や議事録など幅広い職種に広がっている一方、使用ルールやガイドライン整備が重要になっているという実態にも合っています。 ([jil.go.jp](https://www.jil.go.jp/kokunai/blt/backnumber/2026/01_02/blm_special.html))
