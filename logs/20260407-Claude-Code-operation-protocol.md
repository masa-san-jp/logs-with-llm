# Claude Code プロフェッショナル運用・導入プロトコル

## 自律型開発環境の構築と最適化

-----

## 1. イントロダクション：エージェント型開発へのパラダイムシフト

現代のソフトウェア開発において、エンジニアの最大の課題は「複雑性の増大」とそれによる「認知的負荷（Cognitive Load）」の増大です。Claude Codeは、単なるコード補完やチャットボットの域を超え、自律的にファイルを読み取り、コマンドを実行し、問題を解決する**「エージェント型開発環境」**へとパラダイムをシフトさせます。

### エージェント型開発の本質的価値

本プロトコルにおける「エージェント」の定義は、エンジニアの「指示（Intent）」を理解し、現在のコードベースを「探索（Explore）」し、実行可能な「計画（Plan）」を立て、自律的に「実装（Implement）」および「検証（Verify）」を行う能力を指します。これにより、エンジニアの役割は「自らコードを書く」という戦術的作業から、**「戦略的意図を設計し、AIのアウトプットを検証・承認する」**上位レイヤーへと移行し、デベロッパーベロシティ（開発速度）を非連続的に向上させます。

### プロトコルの目的

Claude Codeは極めて強力ですが、その自律性は「コンテキスト管理」と「適切な指示設計」という制約に依存します。本ドキュメントは、組織がClaude Codeを導入する際、一貫した成果を出すための**「再現可能な運用プロトコル」**として策定されています。

-----

## 2. システム環境の最適化とアーキテクチャの選択

戦略的な環境選択は、開発の安全性と効率性の基盤となります。プロジェクトの要件に応じ、最適なインターフェースと実行環境を定義します。

### インターフェースの比較分析：CLI vs デスクトップ版

デスクトップ版は単なるラッパーではなく、用途の異なる3つのエンジンを提供します。

|機能 / インターフェース|CLI (ターミナル)  |デスクトップ版 (Code/Cowork)    |
|-------------|-------------|-------------------------|
|主要エンジン       |インタラクティブCLI  |Chat / Cowork / Code の3種 |
|視覚的レビュー      |テキストベース      |ビジュアルDiff表示 & インラインコメント  |
|環境分離         |–worktree フラグ|Git Worktree による自動分離     |
|自動化・検証       |スクリプト連携      |埋め込みブラウザによるUI検証 (Preview)|
|マルチモーダル      |非対応          |画像/PDFの添付、スクリーンショット対応    |
|OS統合         |限定的          |Computer Use (画面操作・アプリ制御)|

### 戦略的レイヤー：Codeタブ vs Coworkタブ

- **Codeタブ (Interactive)**: ローカルファイルに直接アクセスし、エンジニアがリアルタイムで各変更をレビュー・承認する外科的な開発に適しています。
- **Coworkタブ (Autonomous)**: クラウドVM上の隔離された環境で動作する自律型バックグラウンドエージェントです。長時間かかるリサーチ、依存関係の更新、ドキュメント作成などを並行して実行させる際に、戦略的なレバレッジを発揮します。

### 環境構築の必須要件

- **OS**: macOS 11以降（CoworkはApple Silicon必須）、Windows 10以降
- **Gitの必須性**: ローカル実行およびWorktree分離のためにGit（およびGit LFS）が不可欠です
- **設定の同期**: CLIとデスクトップ版は、.claude/settings.json や CLAUDE.md などの構成ファイルを共有します

## 3. コンテキストウィンドウの戦略的運用管理

Claudeのパフォーマンスを左右する最大の変数は「コンテキスト管理」です。コンテキストが飽和すると、モデルは指示を「忘却」し、エラー率が上昇します。

### リソース管理プロトコル

#### 1. Extended Thinking（拡張思考）の制御

複雑な論理推論には `MAX_THINKING_TOKENS` の設定が不可欠です。Opus 4.6等では適応型推論が機能しますが、コストと精度のバランスを最適化するため、不要な思考は抑制し、難易度の高いタスク（リファクタリング、バグ調査）でのみ最大化するよう設定します（`MAX_THINKING_TOKENS=0`で無効化可能）。

#### 2. 情報密度の最適化（/compact と /btw）

- **/compact**: セッションが長引いた際、重要な決定事項のみを要約して履歴を凝縮し、精度を回復させます
- **/btw**: コンテキスト履歴に残したくない一時的な質問に使用し、リソースの「汚染」を防ぎます

#### 3. チェックポイントと /rewind

リスクの高い変更を行う前に、`Esc + Esc` または `/rewind` を使用してチェックポイントを確認します。失敗したアプローチが履歴に残ると精度が下がるため、早期に「巻き戻す」ことが戦略的に重要です。

## 4. 標準実行プロトコル：EPIVサイクルと検証戦略

Claudeにコードを直接書かせるのではなく、以下の「EPIVサイクル」を回すことが品質担保の鉄則です。

### 1. Explore（探索）

Planモードで開始し、コードベースを質問します。Claudeに「インタビュー」をさせ、要件の曖昧さを解消します。

### 2. Plan（計画）

具体的な実装設計を作成させ、`Ctrl+G` で内容を直接レビューします。この段階で、人間が設計の整合性を担保します。

### 3. Implement（実装）

NormalモードまたはAuto acceptモードへ移行します。小規模な修正であれば、計画をスキップして直接実行することも許容されます。

### 4. Verify（検証）：最高レバレッジの追求

本プロトコルでは、人間が唯一のフィードバックループにならない仕組みを構築します。

- **テスト主導**: 実装前にテストを書かせ、その実行結果をClaude自身に検証させます
- **視覚的検証 (Visual Verification)**: デスクトップ版のPreview機能とChrome拡張機能を用い、スクリーンショットの比較やDOMの検査を自動化します

### 戦略的「Writer/Reviewer」パターン

大規模なタスクでは、セッションA（Writer）で実装を行い、独立したコンテキストを持つセッションB（Reviewer）でそのコードをレビューさせることで、バイアスを排除した品質向上を実現します。

## 5. 知識基盤の構築：CLAUDE.md と拡張機能

「一度教えたことを二度教えない」ために、プロジェクト固有のルールを外部化します。

### CLAUDE.md の設計指針：情報の取捨選択

指示の「埋没（Submergence）」を防ぐため、以下の基準で記述を管理します。

**✅ 含めるべき:**

- 独自のビルド/テストコマンド
- 特定の命名規則
- プロジェクト特有のアーキテクチャ上の決定
- デプロイ手順

**❌ 除外すべき:**

- 一般的な言語規約
- コードから容易に類推できる事実
- 詳細すぎるAPIドキュメント（URLリンクに留める）

### 決定論的プロセスの実装 (Hooks/Skills)

- **Hooks**: eslint の自動実行など、編集後に必ず行うべきプロセスを `.claude/settings.json` に記述し、ヒューマンエラーを物理的に排除します
- **Skills**: 再利用可能なワークフロー（例：`/review-security`）を `.claude/skills/` に定義し、複雑な手順を決定論的に再現可能にします

## 6. スケーリングと並行開発の高度な制御

プロジェクトの規模に応じて、Claudeの出力をスケールさせるための高度な技術を適用します。

### Git Worktree による並行セッション管理

デスクトップ版では、新しいタスクごとに自動的に独立したWorktreeが作成されます。

- **.worktreeinclude の活用**: `.env` などの gitignored ファイルを新しいWorktreeに自動的に持ち越すために、このファイルの設定を徹底してください。これにより、環境構築のダウンタイムをゼロにします

### 非インタラクティブモードとファンアウト（Fan-out）

`claude -p` フラグを用いたCI/CD統合や、大量のファイルに対する一括修正を行います。

- **安全性の担保 (–allowedTools)**: 非インタラクティブ実行時は、Claudeが使用できるツールを制限し、予期せぬインフラ操作や破壊的変更を防止する「信頼境界（Trust Boundary）」を定義します

## 7. リスク管理とアンチパターンの回避

エージェントの自律性に伴うセキュリティと品質のリスクを、組織的に管理します。

### 5大失敗パターンの特定と解決

|失敗パターン       |リスク内容                  |解決プロトコル                   |
|-------------|-----------------------|--------------------------|
|キッチンシンク・セッション|無関係なタスクの混在による精度低下      |`/clear` による定期的リセット       |
|無限修正ループ      |失敗した履歴の蓄積による混乱         |`/rewind` で失敗前に戻り、プロンプトを再考|
|指示の埋没        |肥大化した CLAUDE.md による命令無視|記述を「ルールの憲法」に絞り、剪定を継続      |
|探索不足の実装      |設計思想に反するパッチワーク的修正      |Planモードの強制と `Ctrl+G` レビュー |
|検証の欠如        |動作しないが「それらしい」コードの混入    |テスト/リンター/視覚的検証の自動化        |

### セキュリティ：Computer Use とアクセスティア

画面操作を許可する際、プロトコルは以下の**「アクセスティア」**に基づき権限を定義します。

- **View-only (ブラウザ等)**: 画面の視認のみ可能
- **Click-only (IDE/ターミナル)**: クリックやスクロールは可能だが、タイピングやショートカットは禁止
- **Full control**: 全操作を許可

組織的な導入においては、Ask モード（各操作の個別承認）から開始し、信頼が確認されたドメインにおいてのみ Auto または Bypass（サンドボックス環境限定）を適用してください。

## 総括

Claude Codeを単なるツールではなく、「信頼できるデジタル同僚」へと進化させる鍵は、厳格なプロトコル運用とエンジニアの直感の融合にあります。本プロトコルをベースに、各組織のコンテキストに合わせた微調整を継続することで、開発プロセスは真の意味で「自律型」へと進化します。
