# Claude Code Agent Teams 運用・構築再現プロトコル：自律型AIチームの設計と実践

-----

## 1. 導入：AIエージェント・オーケストレーションの戦略的意義

現代のソフトウェア開発および事業企画において、単一のAIチャットによる支援はすでに通過点に過ぎない。真の生産性向上を実現するためには、AIを単なる「効率的な検索・執筆ツール」としてではなく、高度な判断と自律的な行動を伴う **「意思決定の分身（Alter Ego）」** として運用するパラダイムシフトが求められている。

ここで中核となるのが、複数のAIエージェントを専門性に基づいて束ね、協調させる「エージェント・オーケストレーション」の視点である。Claude Codeの新機能「Agent Teams」は、単なる自動化の枠を超え、以下の戦略的インパクトをもたらす。

- **コンテキストの完全分離**：各エージェントが独立したセッション（思考空間）を持つことで、情報の混濁を防ぎ、専門性を極大化する
- **高度な並列処理（Parallel Processing）**：調査、設計、実装、レビューといったタスクを個別のコンテキストで同時進行させ、開発リードタイムを劇的に短縮する
- **自律的な自己組織化（Self-organizing）**：メンバー間の直接通信（Mailbox）により、人間が介入せずとも現場の異常（空リポジトリ等）を検知し、方針を自己修正する能力を備える

本プロトコルでは、この自律型AIチームを再現可能な資産として構築するための、具体的アーキテクチャと実装手順を提示する。

-----

## 2. アーキテクチャの選定基準：Subagent vs. Agent Teams

構築にあたり、従来の「Subagent（サブエージェント）」と新機能「Agent Teams（エージェントチーム）」の技術的差異を正しく理解し、タスクの複雑性に応じて最適なアーキテクチャを選択する必要がある。

|比較項目            |Subagent（サブエージェント）|Agent Teams（エージェントチーム）   |
|----------------|------------------|-------------------------|
|役割のイメージ         |「報告を行う有能な部下」      |「互いに議論・補完し合う専門家集団」       |
|コンテキストの独立性      |メインセッション内の独自ウィンドウ |完全に独立した個別のセッション          |
|コミュニケーション       |呼び出し元（Lead）への報告のみ |メンバー間での直接メッセージ交換（Mailbox）|
|調整（Coordination）|リーダーによる中央集権的管理    |共有タスクリストによる自律的な状態管理      |
|トークンコスト         |低い（結果を要約して返却）     |高い（通常比5〜7倍程度）            |
|適したユースケース       |単純なファイル探索、単一観点の調査 |多角的なレビュー、並列開発、複雑な議論      |


> **判断基準の要諦**：アウトプットの「結果」だけが必要な場合はSubagentで十分だが、セキュリティ、パフォーマンス、保守性といった「複数の専門的観点」を戦わせ、自律的に最適解を導き出したい場合は、Agent Teamsの採用が不可欠である。

## 3. 実装プロトコル：環境構築とモード選択

Agent Teamsを実運用に投入するための技術的なセットアップ手順を詳説する。

### 3.1 構成設定（settings.json）

環境変数を毎回定義するオーバーヘッドを排除するため、`~/.claude/settings.json` に以下の設定を記述し、チーム機能を恒久的に有効化する。

```json
{
  "CLAUDE_CODE_AGENT_TEAMS": true,
  "CLAUDE_CODE_AGENT_TEAMS_UI_MODE": "in-process"
}
```

### 3.2 UIモードの選択：in-process 推奨の理由

Agent Teamsには2つの動作モードがあるが、当プロトコルでは `in-process` モードを強く推奨する。

|モード               |説明                                                                                                                       |
|------------------|-------------------------------------------------------------------------------------------------------------------------|
|**in-process（推奨）**|メインターミナル内で全メンバーが並列動作。VS Code統合ターミナルやWindows環境、Ghostty等あらゆる環境で安定動作し、汎用性に優れる                                               |
|split panes（非推奨）  |メンバーごとに物理的なペインを分割（tmux/iTerm2必須）。**「入力内容が表示されるだけで送信・処理されない」という重大な既知のバグ（Issue #24108）** が報告されており、再現性を重視するプロフェッショナルな現場では非推奨|

## 4. チーム・オーケストレーションの実践：役割定義と自律運用

チームを起動し、Teammatesを自律的に機能させるための運用プロトコルを策定する。

### 4.1 プロンプトによる専門性の付与

チーム起動時には、各Teammatesに具体的かつ専門的な役割（Role）を付与する。

**プロンプト例：**

> 「セキュリティ、パフォーマンス、アクセシビリティの各専門家チームを構成し、このコードを並列レビューしてください。メンバー同士で相談しながら進めてください。」

この指示により、リーダー（Team Lead）は自動的に最適なチーム名を生成し、各メンバーを適切な初期プロンプトと共に起動（spawn）させる。

### 4.2 共有タスクリスト（Ctrl+T）の管理

チームの進捗状況は、共有タスクリストで厳密に管理される。

- **タスク番号の仕様**：`#1〜#3` はメンバー起動用の内部タスク（`_internal`）であり、`#4` 以降から採番される点に留意する
- **ステータス確認**：`pending` → `in_progress` → `completed` と遷移する。反映には1〜2秒の遅延が生じる場合があるため、`Ctrl+T` を再打鍵して最新の「State management」を確認する

### 4.3 Mailboxによる自律的方針修正の再現

Agent Teamsの真価は、異常事態における自律的な通信フローにある。

**再現事例（空リポジトリ検知）：**

1. **検知**：レビュー担当のTeammateがリポジトリの不在（空）を検知
1. **通信**：`SendMessage` ツールを使用し、リーダーへ「対象が存在しない」と自律的に報告
1. **再調整**：報告を受けたリーダーが全員に対し、推測に基づく構造案作成への「方針変更」を通知

この一連のシークエンスは人間が指示したものではなく、チームが自己組織化した結果であり、これこそがオーケストレーションの価値である。

### 4.4 in-processモードでの操作技術

- **メンバー切り替え**：`Shift+Up/Down` でアクティブなメンバーを選択し、個別指示が可能
- **並列状況の把握**：処理中はスピナー（`✻`）のサブ行に、全メンバーが現在実行中のツール（ファイル検索、読み取り等）がリアルタイムでインターリーブして表示される

## 5. 高度なカスタマイズと資産化：CLAUDE.mdと学習プロトコル

AIチームを単なる作業員から、プロジェクト固有の知見を蓄積する「成長するパートナー」へと昇華させるためのインフラ設計を解説する。

### 5.1 知見の核としての CLAUDE.md

`CLAUDE.md` はチーム全体の人格、判断基準、ワークフローを定義する中核資産である。

> **MCP連携の競合警告**：Teammatesも `CLAUDE.md` を読み込む性質があるため注意が必要。例えば、タスク完了時にToggl APIを叩くルールがある場合、全Teammatesが実行を試み、Toggl側の設定不足で400エラーが発生するといった事故が起こり得る。「このルールはTeam Leadのみが実行する」といった制約を明記し、競合を物理的に回避すること。

### 5.2 カスタムエージェントの構築

`.claude/agents/` 内で特定の専門性を持つエージェントを定義できる。

> **技術的制約（重要）**：YAMLフロントマターは必ず **「1行目・1列目（Line 1, Column 1）」** から記述すること。行頭にスペースが混入するだけで、エージェントは認識不能となる。

### 5.3 5つのトリガーによる学習プロトコル

事業企画AI「Nina」の事例に基づき、以下の5つのトリガーを用いて「Observe → Learn → Improve」の成長ループを実装する。

|#|トリガー           |記録先                                  |
|-|---------------|-------------------------------------|
|1|**上司フィードバック**  |承認・修正・却下パターンを `boss-patterns.md` に記録 |
|2|**判断を変える発見**   |調査中の重要な事実を `logs/learnings/` に即時蓄積   |
|3|**失敗からのガードレール**|インシデント発生時に `guardrails.md` を自動更新し再発防止|
|4|**業界パターン抽出**   |同業種複数社の調査から得た共通項を知識ベースへ昇格            |
|5|**仮説の結果記録**    |提示した仮説の有効性をトラッキングし精度の「打率」を向上         |

## 6. 実運用における制約管理とリスク回避

強力なツールには、相応の技術的制限とコスト特性が存在する。

|制約項目                      |詳細                                                                                                       |
|--------------------------|---------------------------------------------------------------------------------------------------------|
|**モデル割り当てとコスト**           |Team Leadはユーザー設定（例：Sonnet）に従うが、Teammatesには自動的に「Opus 4.6」が割り当てられる。これがコスト5〜7倍増の主因であり、不要なタスクへのチーム動員を避ける規律が必要|
|**セッション管理の限界**            |現在、Agent Teamsは `/resume`（再開）および `/rewind`（巻き戻し）に非対応。セッション終了はチームの解散を意味するため、長期タスクは計画的に実行する                |
|**コンテキストの継承性**            |Teammatesはリーダーの過去の会話履歴を完全には引き継がない。メンバー起動（spawn）時に、その役割に必要な背景情報を「十分なコンテキスト」として明示的に渡す設計が不可欠                |
|**ファイル競合（Race Condition）**|複数メンバーが同時に同一ファイルを編集すると競合が発生する。「Aは`src`を、Bは`tests`を担当」のように、担当ファイルセットをプロンプトで明示的に分担させる                      |

## 7. 結論：AIチームとの共生に向けたチェックリスト

Agent Teamsの導入は、ツールの追加ではなく「デジタル専門家集団の育成」というマネジメントへの転換である。

### 導入フェーズ：基盤の構築

- [ ] `settings.json` で `in-process` モードを優先設定したか？
- [ ] `CLAUDE.md` にチーム全体の人格と、MCP連携等の実行制約を定義したか？

### 運用フェーズ：オーケストレーションの実行

- [ ] 並列性が必要なタスクに対し、役割を定義して「チームを使って」と指示したか？
- [ ] `Ctrl+T` を活用し、予約されたタスク `#1-3` 以降の進捗を管理できているか？

### 成長フェーズ：ナレッジの資産化

- [ ] 5つのトリガーに基づき、フィードバックを `guardrails.md` 等へ蓄積しているか？
- [ ] 失敗を「ルールの不備」と捉え、即座にプロトコルを更新するループが回っているか？

> AIは記録された知見を決して忘れない。適切なフィードバックを積み重ねることで、このAIチームは、あなた自身の思考プロセスを最も深く理解し、自律的に価値を創出する唯一無二の資産（Knowledge Assets）へと進化する。
