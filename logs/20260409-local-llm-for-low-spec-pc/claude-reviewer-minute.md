# 議事録: 低スペックPC向けローカルLLMアプリ 設計仕様策定

- プロジェクト: `ochyai/vibe-local` 拡張設計
- 作成日: 2026-04-09
- 対象: 開発チームへの引き継ぎ用

-----

## 1. 背景・目的

GPU非搭載・CPU-only・8〜16GB RAM の一般事務用PCでも、誰もがローカルでLLMを体験できる環境を作る。  
対象は企業事務部門・学校・児童施設・福祉拠点など、ITリテラシーが必ずしも高くない現場。

設計原則として「最大性能より導入しやすさと継続利用しやすさを優先する」を掲げた。

-----

## 2. 検討の経緯

### Step 1: Draft v1 の作成

以下の構成で初期仕様書（Draft v1）を作成した。

- 2プロセス構成（Input Processor / Chat Engine）
- 対象モデル: `Gemma 4 E2B`（Input Processor）と `Bonsai 8B`（Chat Engine）の2モデル構成
- 8〜16GB RAM での全機能動作を想定

### Step 2: Draft v1 フィージビリティレビュー

レビューの結果、以下の3点に実現リスクが集中していることが判明した。

#### 問題1: Bonsai 8B の CPU サポートが未成熟

- CPU 向け公開ベンチマークが存在しない
- PrismML 独自の llama.cpp フォークが必須であり、メインストリームに未マージ
- Ollama レジストリ未収録
- 2026年3月リリースの新興プロジェクトで本番安定性が未検証
- 「導入容易性」という設計原則に反する

**決定: Bonsai 8B は採用しない。**

#### 問題2: 8GB RAM での同時起動が困難

メモリ試算の結果、Gemma 4 E2B（約5GB）＋ Bonsai 8B（約1.5GB）＋ OS（約2〜3GB）で合計9〜10.5GBとなり、8GBマシンでスワップが発生することが判明した。

**決定: 8GB 環境の保証スコープをテキスト会話のみに限定する。全機能は16GBを推奨とする。**

#### 問題3: アーキテクチャ上の未定義事項

- プロセス間通信方式が未定義
- 推論ランタイムが未定義
- モデルと役割（Input/Chat）のマッピングが未定義
- 音声処理バックエンドが未定義

これらを仕様書に明記するよう指摘した。

### Step 3: Draft v2 の作成

v1 レビューを受けて以下の変更を行った。

|変更点          |内容                                  |
|-------------|------------------------------------|
|モデル統一        |Bonsai 8B を除外し、Gemma 4 E2B 単体に統一    |
|8GB 保証スコープ明確化|テキスト会話のみ保証。マルチモーダルは16GB推奨と明記        |
|プロセス間通信      |localhost HTTP を第一候補として明記           |
|進捗通知         |非同期進捗通知を非機能要件として明記                  |
|音声バックエンド     |未確定として §15 に明示（「単体で扱うか補助バックエンドを使うか」）|
|2系統の定義       |論理的分離と物理的同時常駐を区別して記述                |

### Step 4: Draft v2 フィージビリティレビュー

v1 の主要リスクは解消されたと評価した上で、新たに以下3点を指摘した。

#### 問題4: 音声入力が llama.cpp で現時点未動作

- Gemma 4 E2B はアーキテクチャ上、音声入力をネイティブサポートする
- しかし llama.cpp の音声実装は 2026年4月現在まだ開発中であり、実際には動作しない
- v2 の §15 では「未確定」扱いだが、「動かない」ことが判明しているため対応方針の決定が必要

**推奨決定（未確定）: Phase 3 の音声入力は Whisper.cpp との併用を暫定方針とし、llama.cpp の音声対応が安定した段階でネイティブ入力に差し替える設計とする。**

#### 問題5: 画像入力に mmproj ファイルが必要

- llama.cpp で Gemma 4 E2B の画像入力を使う場合、モデル本体 GGUF に加え、マルチモーダルプロジェクタ（mmproj）ファイルが別途必要
- 初回セットアップの導線（ダウンロード対象・配置方法）に直接影響する
- 仕様書に未記載であり、実装時に詰まるリスクがある

**推奨決定（未確定）: §5.3 または §15 に mmproj の取り扱い方針を追記する。**

#### 問題6: UI技術スタック未定義のままであり低スペックPCへの影響が大きい

- Electron を使う場合、追加メモリが 150〜300 MB 発生する
- Tauri であれば 20〜60 MB 程度に抑えられる
- 既存の `ochyai/vibe-local` のスタックを最優先で確認する必要がある

**推奨決定（未確定）: リポジトリ構成確認を最初のタスクとし、UIスタックを早期に確定させる。**

加えて以下2点も追記を推奨した。

- **コンテキスト長管理**: 8GB 環境での長会話による OOM を防ぐため、アプリ側でコンテキスト上限を設定可能にし、超過時はユーザーに通知する
- **初回セットアップ UX**: Gemma 4 E2B（Q8_0）は約5GBあり、ダウンロード進捗表示・失敗時リトライ・オフラインインストール対応を検討する

## 3. 確定した意思決定

|#  |決定事項                                                     |理由                              |
|---|---------------------------------------------------------|--------------------------------|
|D-1|採用モデルを Gemma 4 E2B 単体に統一する                               |Bonsai 8B の CPU 成熟度・導入複雑性・安定性の問題|
|D-2|Input Processor と Chat Engine は論理的分離を維持するが、物理的同時常駐は前提にしない|8GB 環境でのメモリ制約                   |
|D-3|8GB RAM 環境の動作保証はテキスト会話のみとする                              |メモリ試算の結果、同時起動が困難                |
|D-4|16GB RAM 環境を全機能推奨とする                                     |D-3 の裏返し                        |
|D-5|プロセス間通信の第一候補を localhost HTTP とする                         |実装単純性・デバッグ容易性・将来の分離デプロイ対応       |
|D-6|前処理中および応答中の非同期進捗通知を必須要件とする                               |低スペック PC でのフリーズ誤認を防ぐ            |
|D-7|推論ランタイムは導入容易性を最優先で1系統に絞る（具体的なランタイムはリポジトリ確認後に確定）          |複雑性を増やさない設計原則                   |

## 5. 実装フェーズと未確定事項の対応関係

```
Phase 1（テキストチャット）
  └─ 前提: リポジトリ確認・ランタイム確定・UIスタック確定

Phase 2（添付・画像）
  └─ 前提: mmproj 取り扱い方針確定・8GB 逐次処理設計

Phase 3（マイク・カメラ）
  └─ 前提: 音声バックエンド方針確定（Whisper.cpp 暫定採用 or 待機）

Phase 4（アーカイブ）
  └─ 前提: Phase 1〜3 の安定動作確認

Phase 5（軽量化・拡張整備）
  └─ llama.cpp 音声対応の状況確認・Bonsai 8B 再評価（任意）
```

## 6. 参考情報

### Gemma 4 E2B の主要スペック

|項目          |値                      |
|------------|-----------------------|
|実効パラメータ数    |2B（埋め込み込みで 5.1B）       |
|コンテキスト長     |最大 128K トークン           |
|CPU-only 速度 |5〜10 tokens/sec（Q4量子化時）|
|推奨量子化       |Q8_0 または Q4_K_M        |
|RAM（Q4_K_M） |約 3〜5 GB               |
|RAM（Q8_0）   |約 5〜6 GB               |
|マルチモーダル     |テキスト・画像 ✅ / 音声 ✅（モデル仕様）|
|llama.cpp 画像|✅ 安定（mmproj 必要）        |
|llama.cpp 音声|❌ 2026年4月現在未対応         |
|Ollama 対応   |✅                      |
|ライセンス       |Apache 2.0             |

### Bonsai 8B を採用しない根拠（記録用）

- CPU 向け公開ベンチマークが存在しない
- PrismML 独自 llama.cpp フォークへの依存（メインストリーム未マージ）
- Ollama レジストリ未収録
- 2026年3月リリースの新興プロジェクト（安定性未検証）
- テキスト専用モデルのため Input Processor には使用不可
- 「導入容易性」という設計原則に反する

将来、CPU サポートの安定化・Ollama 対応・十分なコミュニティ検証が揃った段階で Chat Engine への追加採用を再評価する余地はある。
