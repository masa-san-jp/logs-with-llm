# 低スペックPC向け ローカルLLM / バイブコーディング拡張 設計仕様書

- 対象リポジトリ: `ochyai/vibe-local`
- 版: Draft v3
- 日付: 2026-04-09

## 1. 概要

本書は、`ochyai/vibe-local` をベースに、**低スペックPCでもローカルでLLM体験および軽量なバイブコーディング体験を実現するアプリケーション**へ拡張するための設計仕様を定義するものである。

想定利用環境は、企業の一般事務用PC、教育現場、児童向け施設、福祉・地域活動拠点などであり、**GPU非搭載・CPUのみ・メモリ8〜16GB程度の一般的なPC**でも、誰もがLLMを試し、繰り返し使いながら習得できる環境を目指す。

なお、本版ではフィージビリティレビューを踏まえ、**対象モデルを Gemma 4 E2B 単体に統一**し、さらに **音声入力の暫定方式、画像入力時の mmproj 要件、スペック別に使える/使えない機能の明示思想** を設計へ取り込む。

---

## 2. 背景と狙い

### 2.1 背景

多くの職場では一般事務向けPCしか利用できず、会社契約のクラウドLLMを利用できない場合がある。  
また、教育現場や児童向け施設では、GPU搭載機や有償クラウドサービスを前提とした環境整備が難しいことが多い。

その結果、多くの人がLLMに触れる機会を得られず、日常的に試しながら学ぶ環境が不足している。

### 2.2 狙い

本プロジェクトの狙いは、**CPUのみ・8〜16GB RAM程度のPCでも、全員がローカルでLLMを体験し、試行錯誤しながら使いこなせる環境を実現すること**である。

これにより、以下を目指す。

- LLM未経験者でも気軽に触れられる
- 試行回数を増やし、習熟しやすくする
- 教育・福祉・一般業務の現場で導入しやすくする
- 将来的なLLM人材育成の土壌をつくる

### 2.3 設計原則

- 最大性能よりも**導入しやすさ**と**継続利用しやすさ**を優先する
- クラウド依存を避け、**ローカル完結**を重視する
- UIは単純で分かりやすく、初学者にも扱いやすい構成とする
- マルチモーダル入力の前処理と会話生成を論理的に分離する
- モデル構成は当面 **Gemma 4 E2B 単体**に絞り、複雑性を増やさない
- 8GB環境では機能保証範囲を限定し、16GB環境を全機能推奨とする
- **ユーザーに対して、PCスペックごとに「できること / できないこと」を明示する**
- その明示は、アプリUIおよび README の両方で行うことを前提とする

## 3. スコープ

### 3.1 対象範囲

- シンプルな1対1チャットUI
- Gemma 4 E2B によるローカル会話
- 画像入力
- 音声入力
- マイク入力
- Webカメラ入力
- ファイル添付
- 会話アーカイブ出力機能
- CPU-only低スペックPC向け最適化
- 将来のモデル追加に備えた拡張可能設計
- スペック別利用可能機能の明示

### 3.2 重要なアーキテクチャ方針

高度な自律エージェント機能は対象外とする。

また、画像・音声・添付ファイルがある場合は、**入力段でLLMが推論しやすい形式へ変換する処理系**を設け、チャット応答処理とは分離する。

ただし本版では、**別モデルを使うのではなく、Gemma 4 E2B 単体を用途に応じて使い分ける**。

つまり、役割は以下のように整理する。

1. **Input Processor（論理役割）**
   - Gemma 4 E2B を用いて、画像や添付ファイルをテキストまたは構造化コンテキストへ変換する
   - 音声入力については、暫定的に Whisper.cpp によりテキスト化した結果を受け取る

2. **Chat Engine（論理役割）**
   - Gemma 4 E2B を用いて、会話・応答生成・議事録整形を行う

### 3.3 対象外

- 高度な自律エージェント
- ターミナル操作
- ディレクトリ参照
- ローカルファイルシステムの自由な走査
- 自動コード修正の連続実行
- 外部SaaSとの強い連携
- マルチユーザー管理
- Gemma 4 E2B へのネイティブ音声入力の初期対応

## 4. 想定利用環境

### 4.1 ハードウェア前提

- CPUのみ
- メモリ 8GB〜16GB
- 一般的な事務用ノートPCまたはデスクトップPC
- 必要に応じて内蔵Webカメラ
- 必要に応じてマイク

### 4.2 動作保証の考え方

#### 8GB RAM 環境
- **最低限のテキスト会話機能を保証対象**とする
- 音声入力は Whisper.cpp の軽量モデル構成で条件付き対応とする
- 画像入力、Webカメラ入力、重い添付ファイル処理は「利用できる場合があるが保証外」または「制限付き」とする
- 長い会話ではコンテキスト上限により履歴トリミングが必要
- 同時常駐は避け、逐次ロード/逐次処理を基本とする

#### 16GB RAM 環境
- テキスト会話
- 添付ファイル処理
- 画像処理
- 音声入力
- Webカメラ静止画処理
- アーカイブ機能

を含む**全機能の推奨環境**とする

### 4.3 利用現場

- 一般企業の事務部門
- 学校・学習施設
- 児童向け施設
- 福祉・地域活動拠点
- ITに不慣れな利用者が多い環境

## 5. モデルおよびランタイム方針

## 5.1 採用モデル

初期採用モデルは以下の1つのみとする。

- `Gemma 4 E2B`

### 採用理由

- 軽量量子化時にCPU-only環境でも現実的な動作可能性がある
- 画像入力に対応可能である
- 入力前処理と会話生成の両方に同一系統で対応しやすい
- 初期フェーズで実装対象を絞ることで、導入性と保守性を高められる

## 5.2 役割マッピング

| 論理役割 | 使用構成 | 説明 |
|---|---|---|
| Input Processor | Gemma 4 E2B / Whisper.cpp | 画像・添付ファイルは Gemma 4 E2B、音声は Whisper.cpp による文字起こしを暫定採用 |
| Chat Engine | Gemma 4 E2B | 会話、要約、命名、議事録整形 |

### 補足
- 2つの論理役割は維持する
- ただし、物理的には**同一Gemma 4 E2Bの逐次利用**を基本とする
- 8GB環境では同時常駐を避ける
- 音声入力は初期段階では Gemma 4 E2B に直接渡さない
- 将来、Gemma 4 E2B の音声ネイティブ入力が安定したランタイムで実用化された段階で差し替え可能とする

## 5.3 推論ランタイム方針

初期の推論ランタイムは、**導入容易性を最優先して1系統に絞る**。

### 初期方針
- Gemma 4 E2B を安定して扱える単一ランタイムを採用する
- 導入フローを複雑にしない
- 将来のランタイム追加は抽象化層で吸収する

### 要件
- CPU-onlyで動作可能
- localhost経由でアプリ本体と連携可能
- モデルロード状態取得が可能
- 進捗通知または状態通知が可能
- 画像入力に必要な **mmproj ファイル** を扱えること

### mmproj 取り扱い方針
Gemma 4 E2B の画像入力を有効化する場合、**モデル本体に加えて mmproj ファイルが必要**となる。  
そのため、初回セットアップおよび実行時には以下を考慮する。

- ダウンロード対象は「モデル本体」と「mmproj」の2つになる
- 画像機能は、両方が正しく配置されて初めて有効化される
- UI上で「画像対応準備完了 / 未完了」を表示する
- README にも、画像入力には追加ファイルが必要であることを明記する

※ 実際に Ollama / llama.cpp / llamafile などのどれを採用するかは、対象リポジトリの現状構成確認後に確定する。

## 6. 機能要件

## 6.1 シンプルなチャット機能

アプリケーションは以下を提供する。

- 1対1の会話UI
- 会話履歴表示
- モデル選択
- メッセージ送信
- 応答生成中の表示
- 生成停止
- セッション単位の履歴保持
- コンテキスト長上限の設定

### コンテキスト管理要件
- アプリ側でコンテキスト長上限を設定可能にする
- 初期値は低スペックPC向けに実用的な範囲へ制限する
- 上限の目安は 8K〜32K の範囲で設定可能とする
- 上限超過時はユーザーに通知する
- 古い履歴の自動トリミング、またはユーザー主導のリセットを提供する

---

## 6.2 マルチモーダル入力機能

以下の入力種別をサポートする。

- テキスト
- 画像ファイル
- 音声ファイル
- マイク録音
- Webカメラ静止画
- 添付ファイル

### 処理方針

画像・音声・添付ファイルを含む場合は、まず **Input Processor** の処理に回す。

Input Processor は、それらを **Chat Engine が扱いやすいテキストまたは構造化コンテキスト**へ変換する。

#### 変換例

- 画像 → 内容説明テキスト
- 音声 → Whisper.cpp による文字起こし
- PDF / 文書 → 要約・抽出テキスト
- Webカメラ画像 → 視覚内容の説明
- 複数添付 → 統合された構造化コンテキスト

### 実装上の重要方針

- Input Processor と Chat Engine は**論理的に分離**する
- ただし初期実装では、Gemma 4 E2B を**逐次利用**し、同時常駐を前提にしない
- 音声処理は Whisper.cpp を暫定実装とする
- 将来 Gemma 4 E2B の音声ネイティブ入力に置き換えられるよう、音声前処理インターフェースを抽象化する
- 重い前処理中は必ずUIに進捗を表示する
- ユーザーに「処理中」「要約中」「文字起こし中」などの状態を明示する

---

## 6.3 ファイル添付機能

ファイルは **UI経由でのみ** 添付可能とする。

### 要件

- UI上からファイル選択できる
- 必要に応じてドラッグ&ドロップ対応
- 送信前に添付一覧を確認できる
- 添付削除ができる
- ファイル種別に応じた前処理を行う

### 制約

- ローカルファイルは **UIにアップロードして使うのみ**
- ターミナル操作はできない
- ディレクトリを直接参照できない
- アプリケーションがユーザーのローカルファイルを自由に走査してはならない

---

## 6.4 マイク入力機能

アプリケーションは以下を提供する。

- 録音開始
- 録音停止
- Whisper.cpp による音声→テキスト変換
- 必要に応じて送信前の認識結果確認
- 変換結果を Chat Engine に渡して会話へ反映

### 暫定実装方針
- Phase 3 では **音声 → Whisper.cpp → テキスト → Gemma 4 E2B** の構成を正式採用する
- Gemma 4 E2B へのネイティブ音声入力は初期対象外とする
- 将来、利用ランタイム側で音声入力が安定したら差し替える

---

## 6.5 Webカメラ入力機能

アプリケーションは以下を提供する。

- 利用可能なカメラ一覧表示
- PC本体の内蔵Webカメラ選択
- 静止画撮影
- 撮影内容の確認
- Input Processor への送信

### 初期方針

- 連続映像処理ではなく静止画撮影のみ
- 初期段階では常時映像解析は行わない
- 画像処理には Gemma 4 E2B 本体に加え mmproj の存在確認を必須とする

---

## 6.6 アーカイブ機能

### 概要

UI上に **アーカイブボタン** を設ける。  
押下時に、現在のチャット会話履歴から **Markdown形式の議事録** を生成し、`log` ディレクトリへ保存する。

### 保存仕様

- 保存先ディレクトリ: `log/`
- ファイル形式: `.md`
- ファイル名先頭: 当日日付 `yyyymmdd`
- 残りのファイル名: 会話内容に応じて Gemma 4 E2B が命名

### 例

- `20260409_ローカルLLM導入検討.md`
- `20260409_画像入力テスト記録.md`

### 議事録内容

生成されるMarkdownには、少なくとも以下を含める。

- タイトル
- 作成日時
- 使用モデル
- 会話要約
- 主な論点
- 結論 / 次アクション
- 必要に応じて添付一覧
- 元会話ログまたは構造化ログ

### UX方針

- タイトルは Gemma 4 E2B が生成する
- 保存前にユーザーがプレビュー・編集できる構成が望ましい
- 保存失敗時はリトライ可能にする

## 7. 非機能要件

## 7.1 配布・運用

- GitHubを通じて配布・展開する
- 改修要望、バグ報告、改善提案はGitHubで受け付ける
- ドキュメントや既知の制約もGitHub上で管理する
- README にスペック別利用可能機能表を掲載する

## 7.2 性能

- CPU-only環境で動作可能であること
- 8GB RAM環境ではテキスト会話を主保証範囲とする
- 16GB RAM環境では全機能を推奨利用範囲とする
- UIがフリーズしにくいこと
- モデルロード状態が可視化されること
- 前処理中は進捗または状態を非同期通知すること
- 長い会話ではコンテキスト長上限によりメモリ使用量を抑制すること

## 7.3 保守性

- 入力前処理と会話生成を分離する
- モデル固有実装を抽象化する
- UIロジックと推論ロジックを分離する
- 設定駆動で将来モデル拡張可能にする
- 音声入力はバックエンド差し替え可能な抽象インターフェースにする

## 7.4 安全性・プライバシー

- ローカルファイルの自由アクセスは禁止
- ユーザーが明示的にアップロードしたファイルのみ扱う
- マイク/カメラ権限は明示的に要求する
- データは原則ローカル内で完結させる
- 無断で外部送信しない

## 7.5 スペック別利用可能機能の明示

アプリケーションおよび README は、ユーザーに対し **PCスペックごとの利用可能範囲** を分かりやすく示すこと。

### 例
- 8GB RAM: テキスト会話中心、音声は条件付き、画像は非推奨
- 16GB RAM: テキスト、音声、画像、添付、アーカイブを推奨
- 画像入力には mmproj が必要
- 音声入力は暫定的に Whisper.cpp ベース

この明示は、導入前後の期待値調整を目的とする。

## 8.2 プロセス構成方針

性能とUX向上のため、主要処理は **論理的に2系統** に分ける。

### 系統1: Input Processor
責務:

- 画像入力処理
- 音声文字起こし
- 添付ファイル理解
- マルチモーダル解釈
- 中間表現生成

### 系統2: Chat Engine
責務:

- 会話履歴管理
- 応答生成
- アーカイブ文書生成
- ファイル名命名
- ストリーミング応答

### 重要補足
- 本版では、2系統は**必ずしも別モデル・同時常駐プロセスを意味しない**
- 低メモリ環境では、Gemma 4 E2B をオンデマンドで逐次利用する
- 音声処理は Whisper.cpp を独立前処理として扱う
- 必要であれば内部的には別ワーカーや別サービスとして分けるが、同時メモリ常駐は前提にしない

---

## 8.3 プロセス間通信方針

Input Processor と Chat Engine を物理的に分離する場合、通信方式は **localhost HTTP** を第一候補とする。

### 採用理由
- 実装が比較的単純
- デバッグしやすい
- 将来の分離デプロイにも対応しやすい
- UI側から状態取得しやすい

### 代替案
- IPC
- stdio ベース通信

ただし初期実装では、アプリ内部サービス呼び出しで完結させてもよい。  
外部化する場合でも、インターフェースはHTTP相当の責務分離を意識して設計する。

## 9. モデル設計

## 9.1 役割定義

### ModelDescriptor
`ModelDescriptor` は**モデル固有属性**を表す。  
一方で Input / Chat の使い分けは、**呼び出しコンテキスト上の役割**として扱う。

つまり、`role` は「このモデルが専用に固定された役割」ではなく、**その呼び出し時点でどの用途に使うかを示す実行コンテキスト情報**として扱う。

### Input Role
- Gemma 4 E2B を使用
- 画像、添付ファイルから意味を抽出する
- 音声は Whisper.cpp が文字起こしした結果を受け取る
- 出力はテキストまたは構造化中間表現とする

### Chat Role
- Gemma 4 E2B を使用
- 会話、要約、命名、議事録整形を担当する

## 9.2 メタデータ構造案

### `ModelDescriptor`

- `id`
- `name`
- `supportsText`
- `supportsImage`
- `supportsStreaming`
- `minMemoryGb`
- `recommendedMemoryGb`
- `cpuOnlySupported`
- `requiresMmproj`

### `InvocationContext`

- `role` (`input` / `chat`)
- `contextLimit`
- `stream`
- `performanceMode`

### `InputProcessingResult`

- `sourceType`
- `summary`
- `transcript`
- `structuredContext`
- `warnings`

### `ArchiveDocument`

- `title`
- `date`
- `modelInfo`
- `summary`
- `body`
- `attachments`

## 10. データフロー

## 10.1 テキストのみ

1. ユーザーがテキストを入力する
2. Chat Engine に送信する
3. Gemma 4 E2B が応答を生成する
4. UIに表示する

## 10.2 画像・添付あり

1. ユーザーが入力を添付する
2. Input Processor に送る
3. Gemma 4 E2B が中間テキストまたは構造化コンテキストを生成する
4. 会話文脈と統合する
5. Chat Engine が Gemma 4 E2B で応答を生成する
6. UIに表示する

## 10.3 音声入力あり

1. ユーザーがマイク録音または音声ファイルを入力する
2. Whisper.cpp が文字起こしを行う
3. 必要に応じて整形・確認する
4. Chat Engine が Gemma 4 E2B で応答を生成する
5. UIに表示する

## 10.4 アーカイブ

1. ユーザーがアーカイブボタンを押す
2. 現在の会話履歴を取得する
3. Chat Engine がMarkdown議事録を生成する
4. Gemma 4 E2B がファイル名タイトルを提案する
5. `log/yyyymmdd_*.md` に保存する
6. 保存結果をユーザーに通知する

- ターミナル操作不可
- ディレクトリ参照不可
- 自動ファイル収集不可
- クラウド依存を前提にしない
- 初期段階では高負荷な連続映像処理は対象外
- Gemma 4 E2B を唯一のLLM対象モデルとする
- 音声入力は初期段階では Whisper.cpp 経由とする
- 画像入力には mmproj を必須とする

## 14. 受け入れ基準

初期目標達成条件は以下とする。

- CPU-only環境で起動・利用できる
- 8GB RAM環境でテキストチャットが利用できる
- 16GB RAM環境で画像・音声・添付・カメラ・アーカイブを含む全機能が利用できる
- UI経由で添付を扱える
- 画像/音声/カメラ入力が前処理を経て会話に反映される
- ターミナル操作およびディレクトリ直接操作ができない
- アーカイブボタンで `log/` にMarkdownを保存できる
- 保存ファイル名が `yyyymmdd_` で始まる
- タイトルが Gemma 4 E2B で命名される
- 前処理中および応答中に進捗がUI表示される
- README または UI 上で、スペック別にできること / できないことが明示されている
- 将来的なモデル追加が大幅なUI改修なしで可能である
