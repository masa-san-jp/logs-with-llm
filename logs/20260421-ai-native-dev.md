## 1. 起点：NTT Data 記事のレビュー

### 参照記事

「設計書・コード・テストを全部AIに書かせて半年間開発してみたよ」  
（Zenn / NTT DATA TECH / 茂呂範 / 2026-04-17）

### 記事から得た主要な示唆

**採用する考え方**

- プロンプトを直接入力させず、プロンプトファイル経由にすることで属人化を防ぐ
- ノウハウをプロンプトファイルに蓄積し、チームで共有・移転可能にする
- AIエージェントが必要な情報を自力で見つけられる構造にする
- 1ファイルを小さく保ち、コンテキストウィンドウを節約する

**反面教師にする点**

- テスト観点表は設計書体系の変更に合わせて必ず更新する
- バグ票・ログはMarkdownではなくJSONで管理する（集計・分析のため）
- 総合テストのケースは人間が作る（AIにケース生成を任せない）

## 2. AIネイティブ開発の再現方針

### 基本構造

```
.github/
├── copilot-instructions.md   ← AIへの永続的指示
├── rules/                    ← 規約の実体（Single Source of Truth）
├── prompts/                  ← 工程別プロンプトファイル群
├── doc_format/               ← 成果物フォーマット定義
└── structures/               ← ディレクトリ構造定義
```

### 設計原則

- **規約の実体は `rules/` に集約する**。`AGENTS.md` も `copilot-instructions.md` も `rules/` を参照するだけにする。これにより、規約変更は `rules/` 1箇所だけ触れば済む
- プロンプトは工程ごとにファイル化し、直接入力を禁止する
- 新機能・新エージェントの追加は必ずプロンプトファイル経由で行う

## 3. Agent-Workplace の設計決定

### 3.1 ツールの役割分担

|役割         |ツール           |責務                 |
|-----------|--------------|-------------------|
|フィールドエージェント|Claude Code   |実行・生成・ログ記録         |
|ラボ         |GitHub Copilot|ログ分析・エージェント最適化・レビュー|

**決定理由**  
Claude Codeは実行サイクルを回し、Copilotはその結果を分析して改善提案を行う。両者が同じ規約（`rules/`）を参照することで矛盾を防ぐ。

### 3.2 ファイル構成の役割分担

|ファイル                             |読む主体          |記載内容                           |
|---------------------------------|--------------|-------------------------------|
|`AGENTS.md`                      |Claude Code   |実行・生成タスクの指示。規約は `rules/` を参照と明記|
|`.github/copilot-instructions.md`|GitHub Copilot|レビュー・最適化の指示。規約は `rules/` を参照と明記|
|`.github/rules/`                 |両者            |規約の実体。ここだけ更新すれば両者に反映される        |

### 3.3 エージェントの運用サイクル

```
【作業開始】
Claude Code
  → Google Drive から対応表・前回ログを取得し文脈を復元
  → GitHub から最新エージェント定義を取得
  → 作業実行

【作業終了】
Claude Code
  → 固有名詞を匿名化
  → 対応表を Google Drive に保存
  → 匿名化済みログを GitHub・Google Drive の両方に同期

【週次】
GitHub Copilot（ラボ）
  → ログを分析し改善提案を Issue または PR で提示
  → Claude Code が次週開始前に反映
```

## 4. データ設計の決定

### 4.1 形式の統一方針

|対象          |形式                    |理由            |
|------------|----------------------|--------------|
|エージェントが扱うデータ|JSON / JSONL          |構造化・最小トークン    |
|コード         |Python / JS           |変更なし          |
|人間が読む資料     |`to_markdown.py` で都度変換|エージェントに渡す形式と分離|

**決定理由**  
当初 Markdown で管理していたが、定量分析時に機械的に読み込めず人力集計が必要になるという問題（記事の失敗事例）を避けるため、エージェントが扱うものはすべて JSON/JSONL に統一した。

### 4.2 ログ管理の3層構造

```
current.jsonl      ← 当月ログ（追記専用）
_recent.jsonl      ← 直近14日（日次自動更新）
_summary.jsonl     ← 過去ログ集約（月次自動更新）
archive/           ← 月次アーカイブ
```

**決定理由**  
6ヶ月運用するとログが膨大になりトークンを圧迫する。`_recent.jsonl` と `_summary.jsonl` を自動生成することで、エージェントが読む量を約85%削減できる。エージェントは `archive/` を直接読まない。

### 4.3 タスク管理の2ファイル構成

```
backlog.json   ← 全タスク（正のデータ）
active.json    ← in_progress / blocked のみ（自動生成）
```

**決定理由**  
エージェントが毎回 `backlog.json` 全件を読むのはトークンの無駄。`active.json` を `sync_tasks.py` で自動生成し、エージェントはこちらだけ読む。

### 4.4 スキーマ定義

#### `logs/current.jsonl`（1行1エントリ）

```json
{
  "ts":        "YYYY-MM-DD",
  "agent":     "エージェント名",
  "task":      "タスク名",
  "status":    "done | in_progress | blocked | cancelled",
  "summary":   "1行サマリー（匿名化済み）",
  "issues":    ["課題1"],
  "decisions": ["決定事項1"]
}
```

#### `agents/{name}/config.json`

```json
{
  "name":          "エージェント名",
  "role":          "役割（1文）",
  "skills":        ["skill-name"],
  "input_schema":  "agents/{name}/input/schema.json",
  "output_schema": "agents/{name}/output/schema.json",
  "rules":         ["ログはcurrent.jsonlに追記する"]
}
```

## 5. 匿名化・機密管理の決定

### 5.1 方針

- **GitHub（公開可）**: 匿名化済みログのみ
- **Google Drive（社内）**: 対応表 + 匿名化済みログ（端末間共有）
- **ローカル**: 機密情報を残さない

### 5.2 匿名化ルール

|種別 |置換形式                         |
|---|-----------------------------|
|会社名|`[COMPANY_A]`, `[COMPANY_B]`…|
|人名 |`[PERSON_A]`, `[PERSON_B]`…  |
|製品名|`[PRODUCT_A]`, `[PRODUCT_B]`…|
|場所 |`[PLACE_A]`, `[PLACE_B]`…    |

### 5.3 対応表の管理

対応表は Google Drive の `agent-mappings/` にのみ保存する。  
端末を跨いで作業する場合は Google Drive 経由で対応表を取得し、文脈を復元する。

### 5.4 Google Drive 連携

`skills/drive-io.py` としてスキルを実装する。認証はサービスアカウントを使用。  
`credentials/service-account.json` はローカルのみ（`.gitignore` で除外）。

-----

## 6. 独立性の確保

**決定事項**: Agent-Workplace は他リポジトリ（logs-with-llm 等）に依存しない。

**理由**: エージェントプロジェクトの独立性を担保するため。外部リポジトリとの連携はラボの分析インプットを複雑化させ、再現性を損なう。
