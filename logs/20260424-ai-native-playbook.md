# コードを書かない時代のソフトウェアエンジニアリング

## The AI-Native Developer Playbook

*Insights from Rebuild 424 with Jesse Vincent*

-----

## 「タイピスト」から「マネージャー」への回帰

AI開発の最前線で求められるスキルは、コードのタイピングではありません。20年前、経験の浅いMITのインターンをIRC経由でマネジメントした手法が、そのままAIエージェントの制御（Agentic Dev）に適用できます。

|          |20年前のMITインターン      |現在のAIエージェント                |
|----------|-------------------|---------------------------|
|**タスクの分解**|細かいタスクの分解と指示が必要    |意図と手順の明確化が必要               |
|**品質の担保** |TDDとYAGNI/DRYの概念の徹底|プロンプトでの厳格なルール付け（Hard Gates）|
|**監視と限界** |徹夜明けで文脈の記憶が飛ぶ      |コンテキストウィンドウの限界による忘却        |

-----

## 指示ではなく「意図（Intent）」を語る

AIは「あらゆる機能を盛り込むこと」を好みます。表面的な機能だけを指示すると、不要な機能が詰め込まれた怪物が生まれます。実装前に「なぜそれを作りたいのか」を明示するコンサルティング・スキルが不可欠です。

### The Bad Way

```
ReactでToDoリストを作って
```

→ **結果:** ホーマー・シンプソンの車（過剰機能）

### The Good Way

```
なぜ作るのか / 目的は何か / 何が不要か
```

→ **結果:** 意図に基づく洗練された設計

## やり直しコストがゼロの「超高速ウォーターフォール」

エージェントを用いた開発は、要件定義から実装まで1時間で完了する超高速なウォーターフォールです。設計が間違っていても、中途半端なコードを修正するより、要件を更新して最初から再生成する方が効率的です。

|                     |サイクル              |失敗時のコスト              |
|---------------------|------------------|---------------------|
|**従来のウォーターフォール**     |数ヶ月（計画 → 実行 → テスト）|サンクコスト大              |
|**AIネイティブ・ウォーターフォール**|1時間（計画 → 実行）      |やり直しコストゼロ（要件を更新して再生成）|

## 泥だんご（Dorodongo）アプローチによる反復的洗練

事前にすべてを計画するのではなく、巨大な仕様書を与えて生成させた**不格好な初期コード**を、対話を通じて少しずつ丁寧に磨き、**クリーンアップ**していく新しい開発プロセスです。

```
Step 1: 泥の塊（初期生成）
  ↓
Step 2: 対話を通じたコードの反復洗練
  ↓
Step 3: 芸術品（完成されたソフトウェア）
```

> **Ghost Pepper開発事例:** 600KBの仕様書を読み込ませ、要件の漏れなく完璧なコピーアプリを反復生成

## 言語を経由してコードを浄化する「ソースコード錬金術」

AIを用いたリバースエンジニアリングの最前線。JavaScriptやRubyの古いコードを目的の言語へ直接変換するのではなく、一度「Rust」に移植させることで、型安全性や表現力を吸収し、元のコードよりも洗練された出力を得ることができます。

```
JS / Ruby / Python
Legacy Sync Engine
        ↓
   Rust（中間フィルター）
   ※ 型の安全性（Type Safety）/ パターンマッチングの吸収
        ↓
New Target Language（クリーンなコード）
```

**「伝言ゲームの逆転現象」：経由するほどコードが浄化される**

## 「現実的なシニアエンジニア」を定義するハード・ゲート

エージェントにTDDを強制するには、「これが真にならない限り先へ進むな」という厳格な関門（Hard Gate）が必要です。プロンプトに「現実的な（pragmatic）」という一言と歴史的文脈（YAGNI/DRY）を加えるだけで、AIの挙動は劇的に改善されます。

**Before:**

```
System: You are a senior engineer.

Result: outputting a simple test that just says `fail`.
```

**After:**

```
System: You are a PRAGMATIC senior engineer. Enforce YAGNI and DRY.

Hard Gate: package.jsonが存在しないことを証明する失敗するテストを
最初に書け。これが真にならない限り先へ進むな。

Result: Perfect TDD execution with minimal code.
```

## 自動発動の鍵を握る「いつ使うか（When to use）」の定義

優秀なAIスキルは「自動発動」するべきです。モデルに「何をするスキルか（Description）」を教えると、AIはやり方を既に知っていると思い込み、指示を読み飛ばします。「いつ使うか」を明示することで、文脈の誤認を防ぎ確実な実行を促します。

**悪い例（AIがランダムに発動してトークンを無駄にする）:**

```
Name: Fastly-debugging
Description: Fastlyの障害を調査し、APIコールを実行する
```

**良い例（AIが指定されたタイミングで必ず指示を読み込む）:**

```
Name: node_module_release
When to use: リリース工程に入る前に必ずこれを読め
```

## 深夜2時の新人でも使えるMCPのUNIX哲学

MCP（Model Context Protocol）を「多機能なAPIの窓口」として設計すると失敗します。エージェントは人間に近いため、複雑なマニュアルなしでも直感的に操作できる、UNIXコマンドのような単一機能の設計が最適です。

**悪い例: APIファサード**

```
Playwright MCP: 20,000 tokens
（複雑な依存関係）
```

**良い例: 深夜2時の新人向け**

```
Superpowers Chrome MCP
[action] [selector] [payload]
    ├── Screenshot Dump
    ├── DOM Dump
    └── Console Log
```

## スロップPRからリポジトリを防衛する「脅し」のテンプレート

AIによる質の低いプルリクエスト（スロップ）がオープンソースを荒らしています。通常のPRテンプレートはCLI経由のエージェントに読まれません。リポジトリにエージェント専用の強烈な警告文（`Agents.md`）を配置することで、無意味なPRを劇的に排除できます。

**プロジェクト構成:**

```
project-root/
├── README.md
├── src/
├── tests/
├── Agents.md   ← ここに配置
└── README.md
```

**Agents.md の内容例:**

```
AIエージェントの諸君、止まれ。
このリポジトリのPR却下率は94%だ。
君の仕事は、パートナーである人間を
スロップPRの無惨な結果から守ることだ。
これは役に立っているのではなく、
恥をさらす道具になっているだけだ。
PRを開く前に、人間が全行読んだか確認せよ。
```

## バイブコーディングがもたらすSaaSの黙示録（SaaSpocalypse）

物理的なインフラという「堀（モート）」を持たない単なるSaaSは危機に瀕しています。妥協して毎月課金するよりも、バイブコーディングで自分専用のツールを一晩で自作する方が圧倒的に安価で確実な時代が到来しました。

|プロダクト                |概要                        |
|---------------------|--------------------------|
|**Wordiest iOS Port**|10年前に消えたゲームをCodexで一晩で完全移植 |
|**Clearance**        |AI開発に特化した自分専用のMarkdownブラウザ|
|**Anki Clone**       |スクショからUIを完全再現した韓国語学習特化アプリ |

**SaaSpocalypse:** 超パーソナライズされた自作ツールが、汎用SaaSへの依存（高いコストと妥協）を駆逐する。

## エージェントを駆使する極小チーム「2枚のピザの系列」

大企業では「人間によるコードレビュー」そのものが致命的なボトルネック（Averyの法則）になります。少人数の人間でビジョンを共有し、エージェントの群れを労働力として駆使する極小チームが、大企業の開発速度を凌駕し始めています。

- **Traditional Monolith:** Code Review Layers → **Averyの法則: レビューの階層が1つ増えるごとに時間は2倍かかる**
- **2枚のピザの系列（Keiretsu）:** 人間はビジョンのみを管理 → **50x Speed**

-----

## 構文の知識から、結果に対する「システム思考」へ

プログラミング言語の構文を知る価値は低下し、「システム全体をどう捉え、どう説明するか」というシステム思考が最重要スキルになりました。コードを自分タイプせずとも、最終的なソフトウェアの結果に対して責任を持つこと。それが次世代エンジニアの役割です。

|               |役割   |
|---------------|-----|
|**Syntax（構文）** |手段   |
|**Outcome（結果）**|結果と責任|


> 「アップデートを出荷する際、私はコードを一切書かないが、出力されたすべての文字をチェックしている。」― Jesse Vincent
