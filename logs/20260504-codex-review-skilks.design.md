# Codex Review

Claude Code が生成・修正したコードを、OpenAI Codex CLI 経由で別モデルにレビューさせるためのskill。同じモデルで自己レビューしても見落としが出やすいため、別ベンダーのモデルを当てて差分検出する用途を想定している。

## 前提

- `codex` コマンドがインストール済み (`npm i -g @openai/codex` 等)
- 対象リポジトリは git 管理下にあると仮定 (差分取得のため)
- 認証は以下のいずれか (後述)

未インストールの場合はユーザーに案内し、続行確認を取ること。

## 認証モードと課金の違い

実行前に必ずユーザーの認証状況を確認する。モードによって課金体系が異なる。

|認証方法             |対象ユーザー                                  |課金                    |セットアップ                                          |
|-----------------|----------------------------------------|----------------------|------------------------------------------------|
|ChatGPT サブスクリプション|Plus / Pro / Business / Edu / Enterprise|**プラン内に含まれる (追加課金なし)**|`codex login` でブラウザ認証 → `~/.codex/auth.json` に保存|
|OpenAI API キー    |API ユーザー / CI 環境                        |**API 使用量で従量課金**      |`CODEX_API_KEY=sk-...` を環境変数に設定                 |

```bash
# 現在の認証状態を確認
cat ~/.codex/auth.json 2>/dev/null | jq '{auth_mode, last_refresh}' || echo "未認証"
```

**`codex exec` はデフォルトで保存済み認証を再利用する。**
CI や自動化でサブスクリプション認証を使いたい場合は `~/.codex/auth.json` を環境にコピーする方法があるが、このファイルはアクセストークンを含むため**パスワードと同等に扱うこと**。

→ 自動化環境では API キー (`CODEX_API_KEY`) を使うのがシンプルかつ安全。サブスク認証はインタラクティブ利用向け。

## 実行モードの選択

Codex を呼び出す経路は3つある。状況で使い分ける。

|用途            |コマンド                       |特徴                                        |
|--------------|---------------------------|------------------------------------------|
|単発レビュー (デフォルト)|`codex exec`               |stdoutに結果を流すだけ。最も単純で安定                    |
|永続セッション/対話継続  |`codex app-server` + クライアント|TUIや別プロセスから接続。複数ターンを保持                    |
|MCP 経由        |`codex mcp`                |Codex 自体をMCPサーバとして起動し、他エージェントから tool として呼ぶ|

**迷ったら `codex exec` を使う。** app-serverやMCPはユーザーが明示要求した時だけ。

## 標準ワークフロー (codex exec)

### 1. レビュー対象の特定

ユーザーが対象を指定していなければ、以下の優先順で決める:

1. 直前のターンで Claude Code が編集した差分 (`git diff HEAD`)
1. ステージ済み差分 (`git diff --cached`)
1. 直近コミット (`git diff HEAD~1`)
1. ユーザーが明示したファイル/ディレクトリ

判断がつかなければ `ask_user_input_v0` で確認。

### 2. レビュープロンプトの組み立て

レビューは観点を絞らないと冗長になる。以下のテンプレを使う:

```
あなたは厳格なコードレビュアです。以下の差分をレビューしてください。
出力は日本語の Markdown で、次のセクションに分けてください:

## 🔴 Critical (バグ / セキュリティ / データ破壊リスク)
## 🟡 Warning (動作はするが問題のある実装、パフォーマンス、エッジケース漏れ)
## 🟢 Suggestion (可読性、命名、設計改善の提案)
## ✅ Good (意図的に良いと判断した点。1〜3個)

各指摘には:
- 該当ファイル名と行番号
- 問題の説明
- 推奨される修正コード (短く)
を含めてください。

問題がないセクションは「該当なし」と明記してください。
褒めだけのレビューや「全体的に良いです」みたいな曖昧な総評は禁止です。

--- 差分 ---
<DIFF>
```

### 3. 実行

```bash
git diff HEAD | codex exec --sandbox read-only \
  "$(cat <<'EOF'
<上記プロンプトのテンプレ。差分はstdinから読む旨を明記>
EOF
)" > /tmp/codex-review.md
```

ポイント:

- `--sandbox read-only` を必ず付ける。レビューに書き込み権限は不要
- 大きい差分 (>2000行) は事前に `git diff --stat` で警告し、ファイル単位の分割レビューを提案する
- `--json` を付ければJSONLでパース可能だが、人間に見せるなら通常出力で良い

### 4. 結果の提示

Codex の出力をそのまま流すのではなく、以下を行ってからユーザーに渡す:

1. Critical / Warning の件数を冒頭サマリで提示
1. Critical 指摘について、Claude Code 自身の判断 (同意/反論/要追加調査) を1行ずつ付ける
1. 修正に着手するかをユーザーに確認

セカンドオピニオンの価値は「外部視点 + Claude の自己判断との突き合わせ」にある。Codexの出力を鵜呑みで転載しないこと。

## app-server モード (オプション)

複数ファイルを連続レビューする / 対話的に深掘りしたい時のみ:

```bash
# サーバ起動 (別ターミナル)
codex app-server

# 動作確認
codex debug app-server "review this snippet: ..."
```

実用上はほぼ `codex exec` で足りる。ユーザーが「セッション継続」「会話形式で」と明示した時のみ切り替える。

## エラー対処

|症状                        |対処                                         |
|--------------------------|-------------------------------------------|
|`codex: command not found`|インストール案内: `npm i -g @openai/codex`         |
|`Not authenticated`       |`codex login` を促す。CI環境なら `CODEX_API_KEY` 設定|
|差分が空                      |git status を確認し、レビュー対象を再確認                 |
|タイムアウト                    |差分を分割。`--model` でより小さいモデルを指定               |
|レビュー結果が浅い                 |プロンプトに「特に <観点> を厳しく見て」を追記して再実行             |

## やらないこと

- Codex の指摘を一切の批判的検討なしにユーザーへ転送する
- レビュー目的で `--sandbox workspace-write` や `--yolo` を使う (読み取り専用で十分)
- 数千行の差分を一発で投げる (分割すること)
- API キーやトークンを含むファイルをそのまま渡す (`.env` 等は除外する)
