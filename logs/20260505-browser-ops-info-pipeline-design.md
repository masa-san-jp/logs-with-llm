# 情報収集・配信パイプライン 設計仕様書

## 1. 概要

### 目的

Web上の関心領域（映画館上映、展覧会、ワークショップ等）の情報を定期的に収集し、自分専用のニュースソースとしてプッシュ配信する仕組みの設計指針をまとめる。

### 設計方針

- **層構造**：収集・処理・保存・通知・実行管理の5層に分離し、各層で独立に選択肢を入れ替え可能にする
- **コスト最小**：手元のリソース（Google Workspace、Claude Max、ローカル環境）を活用し、有料APIに依存しない
- **段階的拡張**：最小構成から始め、対象領域を増やしながら同じ枠組みを使い回す
- **正確性優先**：自動化の利便性より情報の正確性を上に置く

-----

## 2. 5層パイプラインモデル

```
┌──────────────────────────────────┐
│ Layer 5: 実行管理（スケジューラ）       │
├──────────────────────────────────┤
│ Layer 4: 通知                     │
├──────────────────────────────────┤
│ Layer 3: 保存・差分検出              │
├──────────────────────────────────┤
│ Layer 2: 処理（パース・フィルタ）       │
├──────────────────────────────────┤
│ Layer 1: 収集（情報源アクセス）         │
└──────────────────────────────────┘
```

各層は**インターフェース（入出力データ形式）を定義**することで、実装の入れ替えを容易にする。

-----

## 3. 各層の選択肢と選定基準

### Layer 1: 収集（情報源アクセス）

#### 情報源の分類

|種別          |特徴                         |例                         |
|------------|---------------------------|--------------------------|
|静的HTML      |サーバレンダリング、`UrlFetchApp`で取得可|単館系映画館、個人ギャラリー            |
|動的サイト（JS必須） |クライアントレンダリング、ヘッドレスブラウザ要    |チケット予約サイト、最新の美術館サイト       |
|RSS / Atom  |公式提供。最も扱いやすい               |ニュースサイト、ブログ               |
|iCal        |カレンダー形式。イベント情報に強い          |大学イベント、勉強会                |
|SNS（X等）     |RSS化が必要                    |全領域で速報性あり                 |
|メールマガジン     |Gmailに届く                   |美術館、劇場                    |
|イベントプラットフォーム|API/RSSあり                  |Connpass、Peatix、Doorkeeper|

#### 取得手段の選択肢

|手段                  |適する情報源           |コスト |制約           |
|--------------------|-----------------|----|-------------|
|GAS `UrlFetchApp`   |静的HTML、RSS、iCal  |無料  |JS非対応、6分実行制限 |
|RSSHub（公開/自前）       |SNS、API無しサイトのRSS化|無料  |公開インスタンスは不安定 |
|Gmail受信＋フィルタ        |メルマガ             |無料  |配信側に依存       |
|Brave Search MCP    |横断検索・補完調査        |無料  |自動化には不向き、手動主体|
|Cowork / Claude Code|ヘッドレス取得・複雑な収集    |プラン内|対話/手動実行が基本   |
|ローカルPlaywright      |JS必須サイト          |無料  |実行環境の常時起動が必要 |

#### 選定基準

1. **公式RSS / iCal があれば最優先で採用**
1. **静的HTMLは GAS で直接取得**
1. **SNSは RSSHub 経由でRSS化してGASで取得**
1. **JS必須サイトは原則対象外**。どうしても必要ならローカルPlaywrightかCoworkで対処
1. **メルマガが配信されている対象は購読し、Gmailラベルで集約**

-----

### Layer 2: 処理（パース・フィルタ）

#### 選択肢

|方式                 |精度 |開発コスト     |実行コスト |用途                   |
|-------------------|---|----------|------|---------------------|
|サイト固有パーサー（正規表現/DOM）|高  |中（サイト毎に必要）|ゼロ    |HTML構造が安定したサイト       |
|RSSの構造化フィールド利用     |高  |低         |ゼロ    |RSS/Atomフィード         |
|キーワードフィルタ          |中  |低         |ゼロ    |SNSや雑多な情報源の絞り込み      |
|LLMによる抽出（ローカル）     |中〜高|低         |計算リソース|構造が不定型な情報源、開発時の補助    |
|LLMによる抽出（対話的）      |高  |低         |プラン内  |初期セットアップ、サンプル抽出ロジック生成|

#### 選定基準

- **構造が安定しているなら固有パーサー**：実行時に外部依存がなく安定
- **構造が変わりやすい・多様 → ローカルLLM**：Ollama / LM Studio で抽出
- **開発時の試行錯誤は対話型LLMで**：Claude Code / Cowork でパーサー雛形を生成
- **キーワードフィルタは併用前提**：SNS等のノイズの多い情報源では必須

#### データ正規化スキーマ（共通）

異なる情報源・処理方式を経たデータを統一フォーマットに変換することで、Layer 3以降を共通化できる。

```json
{
  "source": "情報源識別子（例: eurospace, mot_museum）",
  "source_type": "website | rss | sns | email",
  "category": "screening | exhibition | workshop | lecture | other",
  "title": "イベント名",
  "venue": "会場",
  "start_date": "ISO8601",
  "end_date": "ISO8601",
  "url": "詳細URL",
  "description": "概要テキスト",
  "tags": ["..."],
  "fetched_at": "ISO8601",
  "raw_text": "原文（任意）"
}
```

### Layer 3: 保存・差分検出

#### 選択肢

|保存先                   |特徴                  |適性         |
|----------------------|--------------------|-----------|
|Google Sheets         |可視化・手動編集が容易、GASネイティブ|全般。最初の選択肢  |
|Notion                |構造化・タグ管理が強い、ビュー切替可能 |領域横断ダッシュボード|
|Google Drive上のJSON/CSV|軽量、バージョン履歴あり        |スクリプト中心の運用 |
|ローカルSQLite            |高速、複雑なクエリ可          |自宅PC常時運用なら |

#### 差分検出の基本ロジック

```
1. 一意キー（source + url + start_date 等のハッシュ）を計算
2. 既存レコードと比較
   - 新規 → 「新着」フラグで通知対象に追加
   - 既存・内容変更 → 「更新」フラグで通知対象に追加
   - 既存・同一 → 何もしない
3. 終了したイベントは別シート/タグへアーカイブ
```

#### 選定基準

- **領域追加・横断閲覧の頻度が高い → Notion**
- **シンプル運用・GASとの親和性 → Google Sheets**
- **両者を併用するパターンも有効**（Sheetsを一次保存、Notionをビュー）

-----

### Layer 4: 通知

#### 選択肢

|手段                  |特徴               |向く頻度  |
|--------------------|-----------------|------|
|Gmail（自分宛）          |フィルタ・検索・ラベル運用しやすい|週次〜日次 |
|Google Chat（Webhook）|即時性、スマホ通知        |リアルタイム|
|Slack（Webhook）      |既存ワークスペース活用      |リアルタイム|
|Telegram Bot        |個人通知に最適、無料       |リアルタイム|
|Googleカレンダー追加       |予定として可視化         |確定情報のみ|
|Notionデータベース更新      |通知よりはダッシュボード     |任意    |

#### 設計上のポイント

- **「届ける情報」と「ためておく情報」を分ける**：すべてを通知すると埋もれる
- **しきい値設定**：差分件数・キーワード一致時のみ通知
- **領域別にチャネルを分ける選択肢**：映画はGmail、ワークショップはGoogle Chatなど

-----

### Layer 5: 実行管理（スケジューラ）

#### 選択肢

|実行環境                           |特徴                         |コスト    |
|-------------------------------|---------------------------|-------|
|GAS時間トリガー                      |Workspaceに含まれる、最大15分間隔〜週次まで|無料     |
|GitHub Actions（Schedule）       |任意の言語、外部サービス連携が柔軟          |無料枠あり  |
|Cloud Run Jobs（Cloud Scheduler）|スケーラブル、Workspaceと統合        |無料枠あり  |
|ローカルcron                       |完全制御                       |常時起動が必要|

#### 選定基準

- **GASで完結するなら GAS時間トリガー** が最も摩擦が少ない
- **Playwright等が必要なら GitHub Actions**
- **頻度は領域ごとに最適化**（後述）

## 4. 適用ケース別の推奨構成

### 4.1 映画館の上映情報

|層 |採用案                          |
|--|-----------------------------|
|収集|各館公式サイト（静的HTML） + X（RSSHub経由）|
|処理|サイト固有パーサー + キーワードフィルタ（SNS）   |
|保存|Google Sheets                |
|通知|Gmail（週次サマリー）                |
|実行|GAS週次トリガー                    |

**ポイント**：構造が安定しているため固有パーサーが有効。SNSは舞台挨拶等の補完情報源。

### 4.2 展覧会の情報

|層 |採用案                           |
|--|------------------------------|
|収集|美術館公式RSS（あれば） / 公式サイト / メルマガ受信|
|処理|RSSフィールド利用 + メールパース           |
|保存|Notion（会期管理に向く）               |
|通知|Gmail（月次の会期開始ダイジェスト）          |
|実行|GAS週次トリガー                     |

**ポイント**：会期が長期のため頻繁なチェックは不要。「会期開始◯日前」「終了◯日前」の通知が有用。Notionで会期カレンダービューを作ると俯瞰しやすい。

### 4.3 ワークショップ・勉強会

|層 |採用案                                         |
|--|--------------------------------------------|
|収集|Connpass / Peatix の API・RSS、関連団体のメルマガ       |
|処理|RSS/APIレスポンスから直接抽出 + キーワードフィルタ（参加意欲のあるキーワード）|
|保存|Google Sheets                               |
|通知|Gmail（即時 or 日次）+ Googleカレンダー候補追加            |
|実行|GAS日次トリガー                                   |

**ポイント**：先着順・少人数枠が多いため即時性が重要。確定参加でなくても候補としてカレンダー登録すると判断しやすい。

### 4.4 領域横断ダッシュボード

複数領域を統一フォーマット（Layer 2のスキーマ）でNotionに集約し、`category`でビューを切り替える。週次のGmailは「全領域の新着サマリー」として一通に統合する。

## 5. 共通実装パターン

### 5.1 GAS基本骨格

```javascript
function runPipeline() {
  const sources = loadSourceConfig();   // Sheetsから情報源リスト取得
  const collected = [];

  sources.forEach(src => {
    try {
      const items = collect(src);       // Layer 1
      const normalized = process(items, src); // Layer 2
      collected.push(...normalized);
    } catch (e) {
      logError(src, e);
    }
  });

  const diffs = detectDiff(collected);  // Layer 3
  if (diffs.length > 0) {
    notify(diffs);                      // Layer 4
  }
}
```

### 5.2 情報源設定シート（例）

|id         |name         |category  |url      |source_type|parser        |active|
|-----------|-------------|----------|---------|-----------|--------------|------|
|eurospace  |ユーロスペース      |screening |https://…|website    |parseEurospace|TRUE  |
|mot        |東京都現代美術館     |exhibition|https://…|rss        |parseRss      |TRUE  |
|connpass_ai|Connpass AIタグ|workshop  |https://…|rss        |parseRss      |TRUE  |

設定をシートに外出しすることで、コード変更なしに対象を増減できる。

### 5.3 エラーログシート

|timestamp|source   |error|severity|
|---------|---------|-----|--------|
|2026-…   |eurospace|404  |warn    |

毎回の実行で書き込み、週次でレビュー。連続失敗した情報源は自動非アクティブ化する仕組みを入れると保守が楽になる。

## 7. 拡張・運用ガイド

### 新領域追加の手順

```
1. 情報源を3〜5個リストアップ
2. 各情報源の種別を判定（HTML/RSS/SNS/メール）
3. 収集手段をLayer 1の選択肢から選ぶ
4. 処理方式をLayer 2から選ぶ（共通スキーマに変換）
5. 既存パイプラインに情報源設定を追加
6. 1週間試運転し、誤検知・取りこぼしを観察
7. キーワード・パーサー・しきい値を調整
```

### 運用上の注意

- **利用規約の確認**：スクレイピング・SNS取得は対象サイトのToSに従う
- **アクセス頻度の節度**：週1〜日1の範囲に留める。短時間に大量リクエストしない
- **キャッシュの活用**：同一データの再取得を避ける（差分検出と組み合わせる）
- **個人利用の範囲を超えない**：取得したデータの再配布はしない

### 設計の見直しタイミング

- 情報源が10を超える → 設定の構造化（カテゴリ別シート分割等）
- パーサー保守コストが高い → ローカルLLMによる抽出への移行検討
- 通知が埋もれる → しきい値・チャネル分離の見直し
- 取得失敗が増える → RSSHubセルフホスト等への切り替え検討

## 付録：参考インターフェース定義

### 収集関数の共通シグネチャ

```javascript
/**
 * @param {SourceConfig} config - 情報源設定
 * @returns {RawItem[]} - 取得した生データ配列
 */
function collect(config) { ... }
```

### 処理関数の共通シグネチャ

```javascript
/**
 * @param {RawItem[]} items
 * @param {SourceConfig} config
 * @returns {NormalizedItem[]} - 共通スキーマ準拠
 */
function process(items, config) { ... }
```

この2つのシグネチャを守れば、情報源・処理方式を独立に差し替えられる。
