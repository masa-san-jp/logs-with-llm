**バージョン**: 1.0  
**ステータス**: Draft  
**最終更新**: 2026-05-08

---

## 1. 目的

| 項目 | 内容 |
|------|------|
| 主実装環境 | Claude Code |
| レビュー・軽微修正役 | Codex CLI（常時稼働） |
| 人手介入の方針 | 最小化。低リスク変更は Codex が自律処理する |
| ユーザーゴール | タスクが人手なしで前進し続けること |

---

## 2. 設計方針

| レイヤー | 役割 | 優先度 |
|----------|------|--------|
| CLI | 実働の主軸。スクリプト・hook から非対話実行 | **最優先** |
| GitHub | レビュー結果の受け皿・変更の可視化 | 高 |
| App | 例外対応・複雑差分の目視・最終承認 | 補助 |
| SDK / サーバー | 独自常駐ワークフローが必要になった時のみ導入 | 必要時のみ |

## 3. システム構成

### 3.1 コンポーネント一覧

```
┌──────────────────────────────────────────────────────────┐
│                    ユーザー (最小介入)                      │
└───────────────────────┬──────────────────────────────────┘
                        │ エスカレーション時のみ
          ┌─────────────▼─────────────┐
          │        Claude App          │
          │  (複雑差分確認・手動承認)   │
          └─────────────┬─────────────┘
                        │
┌───────────────────────▼──────────────────────────────────┐
│                     GitHub                                 │
│         PR生成 / レビューコメント / 変更履歴               │
└───────────┬───────────────────────────┬──────────────────┘
            │                           │
┌───────────▼───────────┐   ┌───────────▼───────────┐
│      Claude Code       │   │       Codex CLI        │
│  実装・変更・試行錯誤   │──▶│  レビュー・分類・修正   │
└───────────────────────┘   └───────────────────────┘
```

### 3.2 各コンポーネントの責務

#### Claude Code
- ローカル実装の主担当
- 変更作業・試行錯誤・ファイル編集
- 変更差分を Codex CLI に受け渡す起点

#### Codex CLI
- 非対話実行を前提とした自律レビューエージェント
- スクリプトまたは Git hook から呼び出される
- 差分の受信 → 分類 → 自動修正 or エスカレーション

#### GitHub
- Codex レビュー結果の永続化先
- PR コメント・レビューラベル・変更ログを格納
- 追跡可能性の担保

#### Claude App
- 高リスク差分の人間確認インターフェース
- 自動処理の例外フロー専用

## 4. 動作要件

### 4.1 レビュー起動条件
- 新規変更（コミット・ステージング）を検知したら自動起動

### 4.2 レビュー優先カテゴリ（上位から順に処理）

| 優先度 | カテゴリ | 説明 |
|--------|----------|------|
| 1 | セキュリティ | 認証・認可・インジェクション・機密情報露出 |
| 2 | バグ | 明確な誤動作・null 参照・境界値エラー |
| 3 | 回帰 | 既存テストが通らなくなる変更 |
| 4 | 設計の破綻 | 責務の逸脱・依存の循環・過剰結合 |
| 5 | 型 / テスト不足 | 型なし変更・テスト未記述のロジック追加 |

### 4.3 処理分岐ルール

```
変更検知
    │
    ▼
リスク判定
    │
    ├── 低リスク ──▶ Codex が自動修正 ──▶ 再レビュー ──▶ 問題なし ──▶ PR 反映
    │
    └── 高リスク ──▶ 要約と根拠を出力 ──▶ 人間にエスカレーション ──▶ 承認後に適用
```

---

## 5. 自動化フロー（詳細）

```
Step 1: Claude Code で実装
    └─ ローカルファイルを変更・コミット

Step 2: 差分を Codex CLI に渡す
    └─ git diff または patch ファイルを入力として渡す

Step 3: Codex がレビューし、指摘を分類
    └─ カテゴリ・リスクレベル・根拠を構造化出力

Step 4: リスク判定に基づき分岐
    ├─ 低リスク → Step 5へ
    └─ 高リスク → Step 7へ

Step 5: Codex が自動修正を適用

Step 6: 修正後に再レビュー
    ├─ 問題なし → Step 8へ
    └─ 問題あり → Step 5へ（上限回数まで再試行）

Step 7: 人間にエスカレーション
    └─ 要約・根拠・リスク分類を GitHub PR コメントに投稿
    └─ 承認 → 適用 / 却下 → スキップ

Step 8: PR に反映
    └─ 変更ログを GitHub に記録（追跡可能な状態で残す）
```

---

## 6. 自動修正対象（低リスク定義）

以下の条件をすべて満たす変更を低リスクとして自動処理する。

| 区分 | 具体例 |
|------|--------|
| Typo 修正 | 変数名・コメント・文字列リテラルの誤字 |
| 安全なリファクタ | 変数抽出・関数分割（既存テストをパスするもの） |
| テストに沿った小修正 | 既存テストの期待値に合わせた実装修正 |
| ドキュメント修正 | コメント・README・文言の修正 |
| Import 整理 | 未使用 import の削除・並び替え |
| Lint / Format | フォーマッタ・lint ルールへの自動適合 |

---

## 7. 人間承認が必要な対象（高リスク定義）

以下のいずれかに該当する変更は自動適用を停止し、エスカレーションする。

| 区分 | 具体例 |
|------|--------|
| 認証・認可・課金・削除 | セッション処理・権限チェック・決済・DELETE 系 |
| DB スキーマ変更 | マイグレーション・カラム追加・削除・型変更 |
| 外部 API 仕様変更 | エンドポイント変更・認証方式変更・レスポンス構造変更 |
| テストなしのロジック変更 | テストが存在しない関数・クラスへの変更 |
| 広範囲の挙動変更 | 複数モジュールに波及する変更・共通ユーティリティ変更 |
| 推測ベースの修正 | 根拠が不確かな変更・不確実性の高い推論による修正 |

## 8. 受け入れ基準

| # | 基準 | 検証方法 |
|---|------|----------|
| AC-1 | Codex が変更後に自律的にレビューを実行できる | 変更トリガー後のレビューログ確認 |
| AC-2 | 簡易修正は人手なしで完了する | 低リスク変更の自動適用ログ確認 |
| AC-3 | 高リスク変更は自動適用されない | 高リスクケースのエスカレーション確認 |
| AC-4 | 自動処理された変更が追跡可能である | GitHub PR / ログの記録確認 |
| AC-5 | ユーザーが「止まっている」と感じる頻度が低い | エスカレーション発生率のモニタリング |

## 9. 運用上の導入優先順位

```
Phase 1: CLI 導入
    └─ Codex CLI のセットアップ
    └─ git hook / スクリプトによる自動起動
    └─ ローカルでの差分レビュー動作確認

Phase 2: GitHub 接続
    └─ PR へのレビューコメント自動投稿
    └─ 変更ログの GitHub への記録
    └─ エスカレーション通知の設定

Phase 3: App の補助利用
    └─ 複雑差分の目視確認フロー整備
    └─ 手動承認インターフェースの確認

Phase 4: SDK / サーバー拡張（必要性が明確になってから）
    └─ 独自常駐ワークフローの設計・実装
```
