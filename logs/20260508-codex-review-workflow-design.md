**バージョン**: 1.1  
**ステータス**: Draft  
**最終更新**: 2026-05-08

---

## 1. 目的

| 項目 | 内容 |
|------|------|
| 主実装環境 | Claude Code |
| レビュー役 | Codex CLI（原則 read-only。allowlist 内の自動補正のみ許可） |
| 最終責任者 | **ユーザー**（すべての判断はユーザーに帰属する） |
| 人手介入の方針 | allowlist 外の変更はすべてユーザー承認を要求 |
| ユーザーゴール | エージェントが統制された範囲で前進し、逸脱しないこと |

---

## 2. 設計方針

| レイヤー | 役割 | 優先度 |
|----------|------|--------|
| CLI | 実働の主軸。スクリプト・hook から非対話実行 | **最優先** |
| GitHub | レビュー結果の受け皿・変更の可視化・証跡保管 | 高 |
| App | 例外対応・複雑差分の目視・最終承認 | 補助 |
| SDK / サーバー | 独自常駐ワークフローが必要になった時のみ導入 | 必要時のみ |

### エージェント権限の基本原則

- Codex CLI は原則 **read-only reviewer**。allowlist に定義された補正のみ書き込みを許可する。
- Claude Code はユーザーの指示のもとで実装を行う。独断でのコミットは行わない。
- **main への直接 push は禁止**。Codex が生成する変更はすべて bot 専用ブランチ経由とし、required checks + CODEOWNERS によるユーザー承認を必須とする。

## 3. システム構成

### 3.1 コンポーネント一覧

```
┌──────────────────────────────────────────────────────────┐
│               ユーザー (最終責任者・最終承認者)              │
└───────────────────────┬──────────────────────────────────┘
                        │ allowlist 外の変更はすべてここで承認
          ┌─────────────▼─────────────┐
          │        Claude App          │
          │  (複雑差分確認・手動承認)   │
          └─────────────┬─────────────┘
                        │
┌───────────────────────▼──────────────────────────────────┐
│                     GitHub                                 │
│  PR生成 / レビューコメント / review-report.json / 証跡     │
│  ※ main 直 push 禁止 / bot branch 限定                    │
│  ※ required checks + CODEOWNERS 必須                      │
└───────────┬───────────────────────────┬──────────────────┘
            │                           │
┌───────────▼───────────┐   ┌───────────▼────────────────────┐
│      Claude Code       │   │          Codex CLI             │
│  実装・変更・試行錯誤   │──▶│  read-only reviewer            │
│  (ユーザー指示に従う)   │   │  allowlist 内補正のみ書き込み可 │
└───────────────────────┘   └────────────────────────────────┘
```

### 3.2 各コンポーネントの責務

#### Claude Code
- ローカル実装の主担当
- 変更作業・試行錯誤・ファイル編集
- 変更差分を Codex CLI に受け渡す起点
- ユーザーの指示なしに自律コミットは行わない

#### Codex CLI
- 原則 **read-only** のレビューエージェント
- スクリプトまたは Git hook から非対話実行で呼び出される
- 差分の受信 → 分類 → allowlist 内補正のみ適用 → 結果を証跡化
- allowlist 外はすべて停止・エスカレーション

#### GitHub
- Codex レビュー結果・`review-report.json` の永続化先
- PR コメント・レビューラベル・変更ログを格納
- main ブランチ保護・CODEOWNERS による承認フローを担保

#### Claude App
- 高リスク差分の人間確認インターフェース
- allowlist 外変更の最終承認専用

## 4. 動作要件

### 4.1 レビュー起動条件
- 新規変更（コミット・ステージング）を検知したら自動起動

### 4.2 レビュー優先カテゴリ（上位から順に処理）

| 優先度 | カテゴリ | 説明 |
|--------|----------|------|
| 1 | セキュリティ | 認証・認可・インジェクション・機密情報露出 |
| 2 | バグ | 明確な誤動作・null 参照・境界値エラー |
| 3 | 回帰 | 既存テストが通らなくなる変更 |
| 4 | 設計の破綻 | 責務の逸脱・依存の循環・過剰結合 |
| 5 | 型 / テスト不足 | 型なし変更・テスト未記述のロジック追加 |

### 4.3 処理分岐ルール

```
変更検知
    │
    ▼
allowlist 判定
    │
    ├── allowlist 内 ──▶ Codex が自動補正（1回のみ）──▶ 再レビュー
    │                                                      │
    │                                              ├─ 問題なし → PR 反映
    │                                              └─ 問題あり → revert して停止・エスカレーション
    │
    └── allowlist 外 ──▶ 即時停止 ──▶ 要約・根拠・リスク分類を出力 ──▶ ユーザー承認待ち
```

> **自動補正ループは最大 1 回。** 再レビューで問題が残った場合は revert して停止し、人間にエスカレーションする。無限ループを許可しない。

## 5. 自動化フロー（詳細）

```
Step 1: Claude Code で実装
    └─ ローカルファイルを変更・コミット（ユーザー指示に基づく）

Step 2: 差分を Codex CLI に渡す
    └─ git diff または patch ファイルを入力として渡す

Step 3: Codex がレビューし、指摘を分類
    └─ カテゴリ・リスクレベル・根拠を構造化出力
    └─ 結果を review-report.json に書き出す

Step 4: allowlist 判定に基づき分岐
    ├─ allowlist 内 → Step 5へ
    └─ allowlist 外 → Step 7へ

Step 5: Codex が allowlist 内の自動補正を適用（bot branch に push）

Step 6: 再レビュー（1回限り）
    ├─ 問題なし → Step 8へ
    └─ 問題あり → 変更を revert → Step 7へ（再試行しない）

Step 7: ユーザーにエスカレーション
    └─ 要約・根拠・リスク分類を GitHub PR コメントに投稿
    └─ review-report.json に記録
    └─ ユーザー承認 → 適用 / 却下 → スキップ

Step 8: PR に反映
    └─ required checks 通過 + CODEOWNERS 承認後にマージ
    └─ 処理結果を review-report.json と PR コメントに証跡化
```

## 6. allowlist（Codex 自動補正の許可範囲）

Codex が書き込みを許可されるのは以下の項目に**厳密に限定**する。これ以外はすべて read-only で出力・停止する。

| 区分 | 具体例 |
|------|--------|
| コードフォーマット | `prettier`・`black`・`gofmt` 等のフォーマッタ適用 |
| Lint 自動修正 | `eslint --fix`・`ruff --fix` で修正可能な項目 |
| 未使用 import の削除 | ロジックに影響しない import の除去・並び替え |
| Markdown 整形 | 空白・改行・見出しレベルの統一 |
| コメント・文言の typo | スペルミス・誤字（ロジックに非関与） |

### 自動適用の禁止対象

以下の区分は allowlist に含まない。変更を検知した場合は即時停止しエスカレーションする。

| 区分 | 具体例 |
|------|--------|
| ロジック変更 | 条件分岐・計算式・データ変換の修正 |
| テスト期待値の変更 | `expect(...)` の値変更・テストケース追加・削除 |
| 認証・認可・課金・削除 | セッション処理・権限チェック・決済・DELETE 系 |
| DB スキーマ変更 | マイグレーション・カラム追加・削除・型変更 |
| 外部 API 仕様変更 | エンドポイント変更・認証方式変更・レスポンス構造変更 |
| 依存関係の変更 | `package.json`・`requirements.txt`・`go.mod` の変更 |
| 設定ファイルの変更 | `.env`・`config.*`・CI 設定・Dockerfile の変更 |
| テストなしのロジック変更 | テストが存在しない関数・クラスへの変更 |
| 推測ベースの修正 | 根拠が不確かな変更・不確実性の高い推論による修正 |

## 7. RACI

| アクション | ユーザー | Claude Code | Codex CLI | GitHub |
|------------|----------|-------------|-----------|--------|
| 実装・変更 | A（承認） | R（実行） | I（情報） | I（記録） |
| レビュー実行 | I | I | R（実行） | C（受け皿） |
| allowlist 内補正 | I | I | R（実行） | C（記録） |
| allowlist 外判断 | **A・R（最終決定）** | I | C（根拠提示） | I（記録） |
| PR マージ承認 | **A・R（必須）** | I | I | C（required checks） |
| 証跡記録 | I | I | R（生成） | R（保管） |

> R = Responsible（実行）/ A = Accountable（最終責任）/ C = Consulted（相談・提供）/ I = Informed（通知）

## 8. GitHub ブランチ保護・承認設定

| 設定項目 | 内容 |
|----------|------|
| main 直 push | **禁止**。Codex・bot からの直接 push を含む |
| bot 専用ブランチ | `codex/fix/*` 等の命名規則を定め、そこ以外への push を禁止 |
| required checks | CI（lint・test・build）の全通過を必須とする |
| CODEOWNERS | ユーザーをオーナーとして設定し、merge 前の承認を必須とする |
| レビュー証跡 | `review-report.json` を PR コメントに添付することを必須とする |

## 10. 受け入れ基準

| # | 基準 | 検証方法 |
|---|------|----------|
| AC-1 | Codex が変更後に自律的にレビューを実行できる | 変更トリガー後のレビューログ確認 |
| AC-2 | allowlist 内補正は人手なしで完了する | 自動補正ログと review-report.json の確認 |
| AC-3 | allowlist 外の変更は自動適用されない | 禁止対象ケースのエスカレーション確認 |
| AC-4 | 再レビュー失敗時は revert されて停止する | 失敗ケースでの revert ログ確認 |
| AC-5 | すべての判断が review-report.json と PR コメントに記録される | 証跡の存在確認 |
| AC-6 | main への直接 push が拒否される | ブランチ保護設定の動作確認 |
| AC-7 | ユーザー承認なしに PR がマージされない | CODEOWNERS + required checks の動作確認 |

## 11. 運用上の導入優先順位

```
Phase 1: CLI 導入
    └─ Codex CLI のセットアップ
    └─ git hook / スクリプトによる自動起動
    └─ allowlist の定義と review-report.json 出力の動作確認

Phase 2: GitHub 接続・保護設定
    └─ bot branch 命名規則の設定
    └─ main ブランチ保護（直 push 禁止・required checks）
    └─ CODEOWNERS の設定
    └─ PR へのレビューコメント・review-report.json 自動投稿

Phase 3: App の補助利用
    └─ 複雑差分の目視確認フロー整備
    └─ allowlist 外変更の承認インターフェース確認

Phase 4: SDK / サーバー拡張（必要性が明確になってから）
    └─ 独自常駐ワークフローの設計・実装
```
