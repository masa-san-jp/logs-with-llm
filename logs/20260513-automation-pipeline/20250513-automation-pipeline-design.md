# 既存業務20%自動化による週40時間→35時間短縮パイプライン設計

## 1. 目標定義

### 最終目標

既存業務のうち、**時間消費が大きく、定型化可能で、品質劣化リスクが低い業務から20%を自動化対象化**し、実際の労働時間を以下の状態にする。

```text
現状：40時間 / 週
目標：35時間 / 週
削減：5時間 / 週
削減率：12.5%
```

重要なのは、**業務の20%を自動化すること**ではなく、最終的に**週5時間分の実労働を消すこと**。

そのため、設計上は次のように考える。

```text
自動化対象業務：週25時間分
自動化による実削減率：20%
削減時間：25時間 × 20% = 5時間
```

または、

```text
全業務40時間のうち20% = 8時間分を自動化対象化
ただし、確認・例外処理・修正・運用負荷を差し引き、
純削減5時間を達成する
```

---

# 2. 基本方針

## 2.1 自動化対象にする業務

優先して対象化するのは、以下の業務。

* 毎週・毎月繰り返される
* 入力情報がある程度決まっている
* 判断基準を言語化できる
* 人がやると時間がかかるが、AIやスクリプトで下処理できる
* 完了物の品質チェックが可能
* 属人性が高すぎず、標準化できる
* ミスしても即重大事故になりにくい

例：

* 情報収集
* 議事録作成
* 報告資料の下書き
* メール文案作成
* データ集計
* スプレッドシート更新
* タスク抽出
* 進捗確認
* FAQ対応
* 稟議・申請書類の下書き
* 補助金・助成金情報の一次整理
* 月次報告の草案作成
* Google Chatやメールのログ整理

---

## 2.2 自動化対象にしない業務

初期段階では、以下は自動化しない。

* 最終意思決定
* 人事評価の確定
* 懲戒・解雇・異動判断
* 法務・労務・会計上の最終判断
* 顧客や社員への重要な個別対応
* 高度な交渉
* 経営判断そのもの
* センシティブ情報を含む外部送信が必要な処理
* 責任所在が曖昧になる処理

これらは、**AIが補助し、人間が判断する領域**として残す。

# 4. フェーズ別設計

## フェーズ1：業務棚卸し

### 目的

現場が何に時間を使っているかを可視化する。

### 実施内容

全業務を以下の粒度で洗い出す。

```text
業務名
担当者
発生頻度
1回あたり所要時間
週あたり合計時間
入力情報
成果物
使用ツール
関係者
判断の有無
ミスした場合の影響
自動化できそうな部分
```

### 粒度の目安

「経理業務」では粗すぎる。

適切な粒度は以下。

```text
請求書を受領する
請求書の内容を確認する
支払予定表に転記する
不備があれば担当者に確認する
会計ソフトに入力する
上長に承認依頼する
支払処理を行う
```

業務を**動詞単位**に分解する。

---

## フェーズ2：時間・頻度・負荷の計測

### 目的

感覚ではなく、時間削減効果が大きい業務を特定する。

### 計測する指標

```text
週あたり所要時間
月あたり所要時間
年間所要時間
発生頻度
担当者数
手戻り回数
確認待ち時間
属人化度
心理的負荷
ミス発生頻度
締切プレッシャー
```

### 重要指標

最重要は以下。

```text
週あたり削減可能時間 = 現在の週次所要時間 × 自動化可能率
```

例えば、

```text
週10時間かかる業務 × 50%自動化可能 = 週5時間削減候補
```

この時点で、週35時間化の主候補になる。

---

## フェーズ3：自動化候補の抽出

### 自動化候補の分類

業務を以下の5種類に分類する。

```text
A. 完全自動化できる業務
B. AIが下書き・下処理し、人間が確認する業務
C. テンプレート化・標準化で短縮できる業務
D. 業務フロー変更で不要化できる業務
E. 自動化せず、人間がやるべき業務
```

### 狙うべき順番

最初に狙うべきは、AではなくBとC。

```text
B. AI下処理型
C. テンプレート化型
D. 業務削除型
A. 完全自動化型
```

理由は、完全自動化は設計・保守・例外対応が重くなりやすいから。

初期成果を出すには、**人間の前処理・下書き・集計・整理をAIやスクリプトに任せる**方が速い。

---

## フェーズ4：自動化インパクト評価

各業務を以下の観点で評価する。

```text
時間削減効果
頻度
実装しやすさ
業務リスク
品質確認のしやすさ
標準化のしやすさ
横展開可能性
現場の受容性
データ取得のしやすさ
運用保守の軽さ
```

### 優先順位スコア

以下の式で評価する。

```text
優先度スコア
= 削減見込時間 × 発生頻度 × 横展開可能性
  ÷ 実装難易度 ÷ 業務リスク
```

より実務的には、次の順で選ぶ。

```text
1. 週あたり削減時間が大きい
2. 毎週発生する
3. 判断基準を言語化できる
4. 成果物を人間が確認できる
5. 失敗時の影響が限定的
6. 他部署にも横展開できる
```

# 5. 自動化対象業務の代表パターン

## 5.1 情報収集・調査業務

### 現状

```text

資料を読む
要点を抜き出す

```

```text

AIが比較表を作成
人間が確認・判断
必要に応じて追加調査
```

### 削減効果

```text
調査時間を30〜60%削減しやすい
```

```text
決定事項・未決事項・タスク抽出
担当者・期限を整理

```

### 削減対象

```text
議事録作成
タスク整理
共有文面作成
進捗確認
```

---

## 5.3 定例報告

### 現状

### 自動化後

```text
データを自動取得
差異を自動検出
AIがコメント草案作成
報告資料のドラフト作成
人間が確認・補足
```

### 狙う成果

```text
週次・月次報告の作成時間を半減
```

### 現状


```text

返信案作成
必要な確認事項を抽出
```

# 6. 週40時間から35時間に落とすための設計

## 6.1 削減対象時間の分解

目標は週5時間削減。

削減候補を以下のように分解する。

```text
議事録・会議後処理：週1.0時間削減
報告資料作成：週1.0時間削減
メール・チャット整理：週1.0時間削減
調査・情報収集：週1.0時間削減
転記・集計・確認作業：週1.0時間削減
合計：週5.0時間削減
```

このように、1つの大きな自動化で5時間削るのではなく、**複数の定型業務から1時間ずつ削る**方が現実的。

---

## 6.2 削減の考え方

```text
業務時間削減 = 
作業そのものの削減
+ 探す時間の削減
+ 書く時間の削減
+ 確認する時間の削減
+ 手戻りの削減
+ 会議時間の削減
+ 待ち時間の削減
```

特に重要なのは、単純作業の自動化だけではない。

```text
探さない
迷わない
書き直さない
確認待ちを減らす
会議で決めきる
同じ説明を繰り返さない
```

この部分が週5時間削減に効く。

# 7. パイプラインの運用設計

## 7.1 週次運用

毎週1回、30分で以下を確認する。

```text
1. 今週発生した非効率業務
2. 時間がかかった業務
3. AIで下処理できそうな業務
4. テンプレート化できそうな業務
5. 自動化した業務の削減実績
6. 新たな手戻り・例外処理
7. 来週実装する小改善
```

### 週次会議の成果物

```text
自動化候補リスト
削減時間見込
実装担当
期限
検証方法
```

---

## 7.2 月次運用

毎月1回、60分で以下を確認する。

```text
1. 削減時間の実績
2. 週40時間→35時間に近づいているか
3. 業務品質は落ちていないか
4. 現場負荷は本当に下がっているか
5. 自動化が逆に負荷を増やしていないか
6. 横展開できる業務はあるか
7. 廃止できる業務はあるか
```

### 月次判断

月次では、次の3分類で判断する。

```text
継続：効果が出ている
改善：効果はあるが運用負荷が高い
廃止：自動化したが逆に面倒になった
```

# 14. 失敗しやすいポイント

## 14.1 自動化そのものが目的になる

失敗例。

```text
AIを使った
ツールを導入した
スクリプトを作った
テンプレートを作った
```

しかし、労働時間が減っていなければ意味がない。

判断基準は常に以下。

```text
週5時間削減に貢献しているか
```

---

## 14.2 現場に入力負荷を押し付ける

業務改善のために、現場へ詳細な記録を求めすぎると失敗する。

避けるべきこと。

```text
毎日細かく業務時間を入力させる
複雑な改善シートを書かせる
AI利用報告を毎回書かせる
新しいツールを何個も使わせる
```

望ましい設計。

```text
既存のチャット
既存のカレンダー
既存のメール
既存のスプレッドシート
既存の会議メモ
```

から、できるだけ自動でログを取る。

---

## 14.3 自動化で確認作業が増える

AIの出力確認に時間がかかりすぎると、逆に非効率になる。

防止策。

```text
AIに任せる範囲を限定する
出力形式を固定する
チェックリストを作る
判断基準を明文化する
人間の確認ポイントを減らす
```

---

## 14.4 業務を残したまま自動化する

本来は不要な業務を、わざわざ自動化してはいけない。

順番は以下。

```text
1. やめる
2. 減らす
3. 標準化する
4. 自動化する
5. AI化する
```

最初からAI化しない。

# 15. 最小構成で始める場合

最初から大規模なシステムを作らず、以下で始める。

```text
Googleスプレッドシート：業務棚卸し・削減時間管理
Googleドキュメント：業務テンプレート・議事録
Google Chat：改善ログ・現場メモ
Google Drive：資料保管
ChatGPT / Gemini：要約・下書き・分析
Apps Script：定型処理の自動化
```

# 16. 初期に作るべき5つの仕組み

## 1. 業務棚卸しシート

全業務を時間・頻度・自動化可能性で整理する。

## 2. 自動化候補リスト

削減見込時間順に並べる。

## 3. 週次レビュー議事録

毎週、削減実績と次アクションを決める。

## 4. AIプロンプトテンプレート集

よく使う業務をAIに渡せる形にする。

例：

```text
議事録作成プロンプト
メール返信案作成プロンプト
月次報告コメント作成プロンプト
補助金情報整理プロンプト
業務改善案作成プロンプト
```

## 5. 削減時間ダッシュボード

以下を見える化する。

```text
今週の削減時間
累計削減時間
業務別削減時間
部署別削減時間
自動化済み業務数
次に着手する業務
```

# 17. 週35時間化の最終運用イメージ

## Before

```text
会議に出る
議事録を書く
メールを読む
資料を探す
報告資料を作る
数字を転記する
進捗確認する
同じ説明を繰り返す
締切前に慌てる
```

## After

```text
会議内容はAIが整理
議事録・タスクは自動生成
メール・チャットは要約される
報告資料は半自動生成
数字は自動集計
進捗はダッシュボードで確認
判断が必要なものだけ人間が見る
不要な会議と報告は廃止
```

---

# 18. このパイプラインの設計原則

```text
自動化する前に、業務を分解する
業務を分解する前に、時間を測る
時間を測る前に、成果物を定義する
成果物を定義する前に、目的を確認する
```

最終的には、以下の状態を目指す。

```text
人間がやる仕事：
判断、設計、対話、責任、例外処理、意思決定

AI・システムがやる仕事：
収集、整理、要約、下書き、分類、転記、集計、チェック、リマインド
```

---

# 19. まず着手すべき実行順序

```text
1. 週40時間の内訳を業務単位で棚卸しする
2. 週5時間以上削れそうな候補を10個出す
3. 削減見込時間順に並べる
4. リスクが低いものから3つ選ぶ
5. 2週間だけ試験導入する
6. 実際の削減時間を測る
7. 効果があるものだけ標準化する
8. 効果がないものは廃止する
9. 毎週改善候補を追加する
10. 月次で週35時間に近づいているか確認する
```

# 20. 結論

このプロジェクトは、単なるAI導入ではなく、以下の順番で進めるべき。

```text
業務可視化
↓
時間計測
↓
業務分解
↓
不要業務の廃止
↓
標準化
↓
AI・スクリプトによる自動化
↓
削減時間の実測
↓
週35時間運用への再設計
```

最初のゴールは、いきなり全社自動化ではない。

最初に達成すべき状態はこれ。

```text
2週間で自動化候補を見える化する
1か月で週1〜2時間削減する
3か月で週3〜4時間削減する
6か月で週5時間削減し、週35時間運用を定着させる
```
