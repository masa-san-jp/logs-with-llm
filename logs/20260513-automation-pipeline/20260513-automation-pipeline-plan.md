# 既存業務20%自動化・週40時間→35時間化パイプライン構築計画

## 1. 計画の目的

この計画の目的は、単にAIや自動化ツールを導入することではなく、以下を実現すること。

```text
現在：週40時間労働
目標：週35時間労働
削減目標：週5時間
削減率：12.5%
```

そのために、既存業務のうち、**自動化・標準化・廃止・短縮が可能な業務を継続的に発見し、実装し、効果測定するパイプライン**を構築する。

---

# 2. 構築方針

## 基本原則

```text
1. いきなり自動化しない
2. まず業務を可視化する
3. 次に時間を測る
4. そのうえで、やめる・減らす・標準化する・自動化する
5. 削減時間を必ず実測する
6. 効果が出たものだけ標準化する
7. 効果が出ないものは廃止する
```

## 判断順序

```text
自動化より先に、業務削減を検討する。

1. その業務は本当に必要か
2. 頻度を減らせないか
3. 成果物を簡素化できないか
4. テンプレート化できないか
5. AIで下処理できないか
6. スクリプト・システムで自動化できないか
```

# 5. Phase 1：業務棚卸し・時間可視化

## 目的

どの業務を自動化すれば週5時間削減できるかを判断するために、業務と時間を可視化する。

## 実施内容

対象者の業務を、以下の単位で洗い出す。

```text
業務名
発生頻度
1回あたり時間
週あたり時間
入力情報
作業内容
成果物
使用ツール
判断の有無
手戻りの有無
自動化可能性
```

## 業務分解の粒度

悪い例。

```text
経理業務
採用業務
報告業務
会議対応
```

良い例。

```text
請求書を受領する
内容を確認する
支払予定表に転記する
不備を担当者に確認する
会計ソフトに入力する
承認依頼を送る
支払予定を報告する
```

## ここで重要なこと

業務を「大分類」で見ると、自動化できないように見える。

しかし、業務を**動詞単位**に分解すると、AIやスクリプトに任せられる部分が見えてくる。

| 判断基準の明確さ | AIやテンプレートに落とせるか    |
| 横展開可能性   | 他部署でも使えるか          |
| 現場負荷     | 新しい入力や運用が増えすぎないか   |

## 優先順位の考え方

```text
優先度が高い業務 =

× 実装が簡単
× リスクが低い
× 横展開できる
```

```text
1. 議事録作成
2. 会議後のタスク整理
3. 定例報告の下書き
```

# 9. 週5時間削減の設計

## 削減目標の内訳

```text
議事録・会議後処理：週1時間
定例報告：週1時間
メール・チャット対応：週1時間
調査業務：週1時間
転記・集計：週1時間

合計：週5時間
```

この構成により、週40時間から35時間への短縮を狙う。

## 重要な考え方

1つの巨大な自動化で5時間削るのではなく、**5種類の業務から1時間ずつ削る**。

その方が実装リスクが低く、失敗時の影響も限定できる。

# 12. KPI設計

## 最重要KPI

```text
週あたり実労働時間
週あたり削減時間
週35時間化の達成率
```

## 補助KPI

```text
自動化候補数
実装済み自動化数
AI下書き利用回数
テンプレート利用率
削減された会議後処理時間
削減された報告作成時間
削減された転記時間
手戻り件数
ミス件数
現場満足度
```

## 見てはいけないKPI

以下だけを見ると失敗する。

```text
AI利用回数
ツール導入数
プロンプト作成数
自動化件数
```

これらは手段であって、成果ではない。

判断基準は常に以下。

```text
本当に週の労働時間が減ったか
品質が落ちていないか
現場の負荷が増えていないか
```

# 15. 6か月実行計画

## 1か月目：可視化と初期PoC

```text
Week 1：
- 対象部門・対象者を決める
- プロジェクト憲章を作る
- KPIを決める
- 業務棚卸しシートを作る

Week 2：
- 業務棚卸しを実施する
- 週40時間の内訳を把握する
- 時間消費が大きい業務を抽出する

Week 3：
- 自動化候補を10〜20件出す
- 優先順位を決める
- 最初のPoCを3件選ぶ

Week 4：
- 議事録、定例報告、メール要約などを試験導入
- 削減時間を測定する
```

## 2か月目：PoC拡大

```text
- PoC対象を5件程度に拡大
- AIプロンプトテンプレートを整備
- Googleスプレッドシートで削減時間を管理
- Apps Scriptで簡単な自動化を実装
- 週1〜2時間削減を目指す
```

## 3か月目：標準化

```text
- 効果が出た自動化を標準業務に組み込む
- 使われなかった仕組みを廃止する
- 業務手順書を作成する
- チェックリストを整備する
- 週3時間削減を目指す
```

## 4か月目：横展開

```text
- 他チーム・他部署へ展開する
- 部門別の自動化候補を収集する
- 横展開できるテンプレートを共通化する
- 週3〜4時間削減を目指す
```

## 5か月目：業務廃止・会議削減

```text
- 自動化だけでなく、不要業務を廃止する
- 定例会議を統合・短縮する
- 報告資料を簡素化する
- 確認フローを減らす
- 週4〜5時間削減を目指す
```

## 6か月目：週35時間運用の定着

```text
- 週35時間化の達成状況を確認
- 部門ごとの標準運用に落とし込む
- 削減時間ダッシュボードを定着させる
- 月次レビューで継続改善する
- 週5時間削減を安定化させる
```

# 17. 最初に着手すべき3つ

## 1. 議事録・会議後処理

理由。

```text
頻度が高い
成果物が明確
AIとの相性がよい
失敗リスクが低い
削減時間を測りやすい
```

## 2. 定例報告

理由。

```text
毎週・毎月発生する
型が決まっている
数値とコメントの構造化がしやすい
テンプレート化しやすい
横展開しやすい
```

## 3. メール・チャット要約

理由。

```text
毎日発生する
認知負荷が高い
返信案作成をAIに任せやすい
重要度分類ができる
確認時間を削減できる
```

# 18. 完了条件

このパイプライン構築は、以下の状態になれば初期完成とみなす。

```text
1. 業務棚卸し台帳がある
2. 自動化候補台帳がある
3. 削減時間管理台帳がある
4. 週次レビューが回っている
5. 月次レビューが回っている
6. 最低3件の自動化・標準化が実運用されている
7. 週あたり1〜2時間以上の削減実績がある
8. 効果測定に基づいて、継続・改善・廃止を判断できている
```

最終完成は以下。

```text
週5時間削減が3か月連続で維持されている
週35時間運用が例外ではなく標準になっている
品質低下や手戻り増加が発生していない
現場が自動化候補を自発的に出せる状態になっている
```

# 19. この計画の要点

```text
1. 最初から全社展開しない
2. まず1チーム・1か月で試す
3. 業務を動詞単位に分解する
4. 自動化より先に、廃止・短縮・標準化を検討する
5. AIは完全自動化ではなく、下処理・要約・下書きから使う
6. 成果指標はAI利用数ではなく、削減時間にする
7. 週5時間削減は、5つの業務から1時間ずつ削る
8. 毎週改善し、毎月標準化判断を行う
```
