# 業務棚卸しの省力化レベル設計

## 結論

業務棚卸しで失敗する主因は、**現場に「自分の業務を言語化して書き出す」という高負荷な作業を求めること**にある。

そのため、業務棚卸しは次の順番で省力化するべき。

```text
人に聞く
↓
人に書かせる
↓
既存資料から抜き出す
↓
カレンダー・チャット・メール・ドキュメントから推定する
↓
AIが業務候補を生成する
↓
人間は確認・修正だけ行う
↓
継続的に自動更新される業務マップにする
```

最初から完全自動化を狙うより、実務上は以下の構成が最も現実的。

```text
推奨構成：
レベル3：既存資料ベース棚卸し
＋
レベル4：チャット・カレンダー・メール等の行動ログ推定
＋
レベル5：AIによる業務カード自動生成
＋
人間は週1回15分だけ確認
```

# 2. レベル0：完全人力ヒアリング型

## 概要

担当者に直接聞いて、業務を洗い出す方式。

```text
何をしていますか？
どれくらい時間がかかりますか？
誰から依頼されますか？
成果物は何ですか？
どこが面倒ですか？
```

## メリット

```text
文脈を深く理解できる
暗黙知を拾いやすい
感情的負荷や心理的負荷も把握できる
```

## デメリット

```text
時間がかかる
聞く側のスキルに依存する
担当者がうまく言語化できない
網羅性が低い
聞かれる側が面倒に感じる
```

## 向いている用途

```text
重要業務の深掘り
属人化業務の把握
管理職・キーパーソンへの確認
```

## 使い方

これは最初の全量棚卸しには使わない。

使うべき場面は、AIやログから抽出された候補について、**確認・補正する場面**。

```text
悪い使い方：
最初から全員に細かくヒアリングする

良い使い方：
AIが出した業務候補を見せて、違う部分だけ直してもらう
```

# 3. レベル1：簡易アンケート型

## 概要

Googleフォームやスプレッドシートで、最低限の項目だけ入力してもらう方式。

ただし、詳細な棚卸し表を書かせると失敗するため、入力項目は極限まで絞る。

## 入力項目

```text
1. 今週、一番時間を使った業務は何ですか？
2. その業務は何時間くらいでしたか？
3. その業務の成果物は何ですか？
4. その業務で面倒だったことは何ですか？
5. AIや自動化で減らせそうな部分はありますか？
```

これ以上細かくしない。

## メリット

```text
導入が簡単
全員から情報を集めやすい
定量化しやすい
```

## デメリット

```text
入力が面倒
記憶に依存する
細かい業務は漏れる
実態よりも「印象に残った業務」に偏る
```

## 省力化ポイント

毎日ではなく、**週1回3分**にする。

```text
毎日入力：負担が大きく失敗しやすい
週1回入力：継続しやすい
月1回入力：記憶が曖昧になりすぎる
```

# 4. レベル2：ワークショップ型

## 概要

対象チームを集め、60〜90分で業務を一気に洗い出す方式。

## 進め方

```text
1. 直近1週間の業務を思い出す
2. 付箋・スプレッドシートに業務名を書く
3. 業務を分類する
4. 時間が大きいものに印をつける
5. 面倒なものに印をつける
6. 自動化できそうなものに印をつける
7. 上位10件だけ詳細化する
```

## メリット

```text
短時間で全体像が出る
チーム内で認識が揃う
業務の重複や無駄が見えやすい
```

## デメリット

```text
参加者の時間をまとめて取る必要がある
声の大きい人の意見に偏る
準備とファシリテーションが必要
```

## 省力化ポイント

全業務を洗い出そうとしない。

最初は以下だけでよい。

```text
時間が大きい業務
繰り返し発生する業務
手戻りが多い業務
面倒だと感じる業務
自動化できそうな業務
```

# 5. レベル3：既存資料抽出型

## 概要

人に書かせるのではなく、すでに存在する資料からAIで業務を抽出する方式。

## 対象資料

```text
会議資料
議事録
業務マニュアル
手順書
報告書
稟議書
申請書
日報
週報
月次報告
スプレッドシート
過去のメール文面
Google Drive内のドキュメント
```

## 抽出する情報

```text
業務名
担当者
発生頻度
入力情報
作業内容
成果物
関係者
使用ツール
判断ポイント
繰り返し性
自動化可能性
```

## メリット

```text
現場の入力負荷が小さい
すでにある情報を活用できる
AIと相性がよい
業務の抜け漏れを減らせる
```

## デメリット

```text
資料化されていない業務は漏れる
古い資料に引っ張られる
実態と違う可能性がある
```

## 実装イメージ

```text
1. Google Drive内の対象フォルダを決める
2. 関連資料をAIに読み込ませる
3. 業務候補を抽出する
4. 業務カード形式に変換する
5. 人間が確認・修正する
```

## AI抽出プロンプト例

```text
以下の資料から、実際に発生している業務を抽出してください。

出力項目：
- 業務名
- 業務の目的
- 担当者または関係者
- 発生頻度
- 入力情報
- 作業内容
- 成果物
- 使用ツール
- 判断が必要な箇所
- 自動化できそうな部分
- 業務削減の仮説

注意：
- 資料に明記されている内容と、推測を分けてください
- 業務はできるだけ動詞単位で分解してください
- 自動化候補は、AI下書き・テンプレート化・転記自動化・通知自動化・廃止候補に分類してください
```

# 6. レベル4：行動ログ推定型

## 概要

人に業務を書かせず、日常業務の痕跡から業務を推定する方式。

## 対象ログ

```text
Googleカレンダー
Gmail
Google Chat
Google Driveの作成・更新履歴
Google Docs
Google Sheets
会議メモ
タスク管理表
申請・稟議ログ
問い合わせ履歴
```

## 推定できること

```text
会議に使っている時間
定例業務の周期
誰とよくやり取りしているか
どの資料を頻繁に更新しているか
どの業務が締切前に集中しているか
どのチャットが依頼・確認・承認に使われているか
どの資料が成果物になっているか
```

## メリット

```text
現場入力がほぼ不要
実態に近い
継続的に取得できる
隠れた業務負荷を発見しやすい
```

## デメリット

```text
プライバシー・同意・権限設計が必要
ログだけでは業務の意味が分からない場合がある
ノイズが多い
最初の分類設計が必要
```

## 抽出ロジック例

```text
カレンダー：
会議時間、定例会議、参加者、会議名から業務カテゴリを推定

メール：
依頼、確認、承認、報告、督促、問い合わせを分類

チャット：
タスク依頼、意思決定、相談、進捗確認、手戻りを抽出

ドキュメント：
作成・更新された資料から成果物を推定

スプレッドシート：
更新頻度の高いシートから定例業務を推定
```

## 重要な考え方

行動ログ推定型では、最初から正解を出そうとしない。

目的は、**業務棚卸しのたたき台を自動生成すること**。

```text
AIが80点の業務一覧を作る
↓
人間が違う部分だけ直す
↓
次回以降の推定精度が上がる
```

# 7. レベル5：AI業務カード生成型

## 概要

AIが各種ログ・資料・メモから、業務を「業務カード」として自動生成する方式。

## 業務カードとは

業務棚卸し表の細かい入力欄ではなく、1業務1カードで整理する形式。

```markdown
# 業務カード

## 業務名
週次売上報告の作成

## 業務目的
経営会議で売上進捗と課題を確認するため

## 発生頻度
毎週

## 現在の作業
- 売上データを確認する
- 前週との差異を確認する
- コメントを書く
- 報告資料に転記する

## 入力情報
- 売上管理シート
- 前週報告資料
- 店舗別コメント

## 成果物
- 週次売上報告資料

## 関係者
- 経営管理
- 事業責任者
- 店舗責任者

## 自動化候補
- データ取得の自動化
- 差異検出の自動化
- コメント草案のAI生成
- 報告資料テンプレート化

## 削減仮説
週60分のうち、30分削減可能

## 人間が確認すべき点
- 異常値の理由
- 経営判断に関わるコメント
- 事業部への伝え方
```

## メリット

```text
現場はゼロから書かなくてよい
確認・修正だけで済む
自動化候補まで一緒に出せる
業務改善につながりやすい
```

## デメリット

```text
AIの推測が混じる
根拠資料との紐付けが必要
最終確認は人間が必要
```

## 実装イメージ

```text
1. AIが資料・ログから業務候補を抽出
2. 業務カードを自動生成
3. 根拠となる資料・ログを添付
4. 担当者が「合っている / 違う / 不足」を選ぶ
5. 修正後、業務台帳に登録
```

# 8. レベル6：継続観測型

## 概要

業務棚卸しを一度きりの作業にせず、毎週自動更新する方式。

## 仕組み

```text
日常業務ログを取得
↓
AIが業務候補を更新
↓
新規業務・増加業務・減少業務を検出
↓
自動化候補を再評価
↓
週次レビューで確認
```

## 毎週自動で出すレポート

```markdown
# 今週の業務変化レポート

## 1. 時間が増えた業務
- 業務名：
- 増加理由の仮説：
- 関係資料：
- 改善余地：

## 2. 新しく発生した業務
- 業務名：
- 発生背景：
- 継続業務化する可能性：
- 自動化候補：

## 3. 手戻りが多い業務
- 業務名：
- 手戻りの原因：
- 関係者：
- 改善案：

## 4. 自動化候補
- 業務名：
- 削減見込：
- 実装案：
- 優先度：
```

## メリット

```text
棚卸しがイベントではなく仕組みになる
業務変化を継続的に追える
新しい非効率を早く発見できる
```

## デメリット

```text
ログ連携が必要
分類ルールのメンテナンスが必要
個人監視と受け取られない設計が必要
```

## 重要な設計原則

これは「人を監視する仕組み」ではなく、**業務プロセスを観測する仕組み**として設計する。

そのため、出力は個人評価ではなく、業務改善に限定する。

```text
避けるべき出力：
誰が遅いか
誰が返信していないか
誰が生産性が低いか

出すべき出力：
どの業務が詰まっているか
どの確認フローが重いか
どの資料作成が繰り返されているか
どこで手戻りが発生しているか
```

# 10. 最も現実的な推奨構成

## 推奨は「完全自動化」ではなく「AIたたき台生成＋人間確認」

最初から完全自動化を狙うと、ログ整備・権限・プライバシー・分類精度・運用負荷で止まる。

したがって、現実的には以下の構成がよい。

```text
1. 既存資料からAIが業務候補を抽出
2. カレンダー・チャット・メールから業務実態を補足
3. AIが業務カードを生成
4. 人間は確認・修正だけ行う
5. 週次で自動更新
6. 月次で自動化対象を決める
```

# 11. 実務導入の3段階

## 第1段階：半日で始める最小構成

### 目的

人力棚卸しをやめ、AIがたたき台を作る状態にする。

### やること

```text
1. 直近1か月の会議資料・議事録・報告書を集める
2. AIに業務候補を抽出させる
3. 業務カード形式に変換する
4. 担当者が確認する
5. 上位10業務だけ削減候補にする
```

### 現場負荷

```text
資料提供：10分
確認：30分
合計：40分程度
```

---

## 第2段階：1週間で行動ログを加える

### 目的

資料に出てこない実務を拾う。

### やること

```text
1. カレンダー予定を業務カテゴリに分類
2. チャットから依頼・確認・報告を抽出
3. メールから定例対応・問い合わせを抽出
4. Google Driveの更新資料から成果物を推定
5. AIが業務カードを補完する
```

### 現場負荷

```text
初期同意・対象範囲確認：15分
業務カード確認：30分
合計：45分程度
```

---

## 第3段階：1か月で継続観測にする

### 目的

業務棚卸しを一度きりではなく、毎週自動更新する。

### やること

```text
1. 毎週ログを取得
2. AIが新規業務・増加業務・手戻り業務を検出
3. 自動化候補を更新
4. 週次レビューで15分確認
5. 月次で実装対象を決定
```

### 現場負荷

```text
週次確認：15分
月次確認：30〜60分
```

# 12. 棚卸しを省力化するための設計原則

## 原則1：最初から網羅しない

最初に全業務を完璧に棚卸ししようとすると止まる。

最初は以下だけでよい。

```text
時間が大きい業務
毎週繰り返す業務
手戻りが多い業務
心理的負荷が高い業務
自動化できそうな業務
```

---

## 原則2：業務名を書かせず、候補を選ばせる

人にゼロから書かせるのではなく、AIが候補を出し、人間は確認する。

```text
悪い設計：
あなたの業務を全部書いてください

良い設計：
AIが抽出した業務候補のうち、正しいものにチェックしてください
```

---

## 原則3：業務棚卸し表ではなく、業務カードにする

巨大なスプレッドシートは入力負荷が高い。

最初は1業務1カードにして、読みやすくする。

```text
業務棚卸し表：
一覧性は高いが、入力が面倒

業務カード：
文脈が分かりやすく、確認しやすい
```

---

## 原則4：時間は厳密に測らず、最初はレンジでよい

最初から正確な時間を求めると入力負荷が増える。

```text
15分未満
15〜30分
30〜60分
1〜2時間
2時間以上
```

この程度でよい。

---

## 原則5：業務改善ではなく「削減候補の発見」に目的を絞る

業務棚卸しの目的は、業務をきれいに整理することではない。

目的はこれ。

```text
どの業務を削れば週5時間減るかを見つけること
```

したがって、棚卸し対象は全業務ではなく、**削減余地のある業務**に絞る。

# 13. 業務棚卸しの最小データ項目

最初から細かくしない。

最低限は以下で足りる。

```markdown
# 最小業務カード

## 業務名

## 発生頻度

## おおよその所要時間

## 成果物

## 面倒な点

## 自動化できそうな部分

## 人間が判断すべき部分
```

この7項目でよい。

不要な項目を増やすと、棚卸し自体が業務負荷になる。

# 15. 自動化対象に直結する業務分類

AI抽出後、業務を以下の5種類に分ける。

```text
A. 廃止候補
そもそも不要、目的が曖昧、誰も使っていない

B. 簡素化候補
頻度・粒度・資料の作り込みを減らせる

C. 標準化候補
毎回同じなのに、人が考えている

D. AI補助候補
要約、下書き、分類、比較、コメント生成が可能

E. システム自動化候補
転記、集計、通知、リマインド、ステータス更新が可能
```

この分類にすると、業務棚卸しがそのまま自動化計画に接続できる。

# 20. 実装時の注意点

## 1. 個人監視にしない

業務棚卸しAIは、個人評価に使わない。

```text
目的：
業務プロセスの改善

禁止：
個人の生産性評価
返信速度の監視
勤務態度の推定
```

## 2. 対象ログを限定する

最初から全メール・全チャットを対象にしない。

```text
対象：
業務改善に関係するスペース
定例会議の議事録
共有フォルダ内の資料
業務用メールの一部サンプル

対象外：
私的やり取り
人事・労務・健康・相談系のセンシティブ情報
評価・懲戒・個別トラブル関連
```

## 3. 推測と事実を分ける

AIの出力には、必ず根拠区分を持たせる。

```text
事実：
資料・ログに明記されている

推測：
AIが文脈から推定した

要確認：
人間の確認が必要
```

## 4. 全件正確にしようとしない

目的は正確な業務図鑑ではない。

目的は、**削減できる業務を発見すること**。

# 24. 最終的な目指す姿

業務棚卸しは、現場に重い入力を求める作業ではなく、以下のような流れに変える。

```text
業務の痕跡を集める
↓
AIが業務候補を作る
↓
AIが自動化候補を分類する
↓
人間は確認だけする
↓
効果が高いものから実装する
↓
毎週自動更新する
```

最終形はこれ。

```text
業務棚卸しをする
のではなく、
業務棚卸しが常に更新されている状態にする
```

この状態になれば、業務改善は「棚卸しで止まる」のではなく、**常に自動化候補が流れてくるパイプライン**になる。
