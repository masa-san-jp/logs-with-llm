# 業務自動化パイプライン 構築計画
**目標：週40時間 → 35時間（5時間削減 ＝ 既存業務の20%自動化）**

---

## フェーズ概要

| フェーズ | 内容 | 期間目安 |
|---|---|---|
| 1 | 業務棚卸し・計測 | 1〜2週間 |
| 2 | 自動化候補の選定・優先順位付け | 1週間 |
| 3 | PoC（小規模検証） | 1〜2週間 |
| 4 | 本番実装・テスト | 2〜4週間 |
| 5 | 運用・改善 | 継続 |

---

## フェーズ1：業務棚卸し・計測

### 1-1. タスクリストの作成
- 1週間分のすべての作業を記録する（時間ログツール：Toggl / Clockify / Excelでも可）
- 記録項目：タスク名、所要時間、頻度、使用ツール、インプット/アウトプット形式

### 1-2. 時間の可視化
- タスクごとの週次換算時間を集計
- 上位20%のタスクが全体の何時間を占めるか確認（パレート分析）

### 1-3. タスクの性質分類
各タスクを以下の2軸で分類する：

```
【繰り返し度】高い ← → 低い
【判断複雑度】低い ← → 高い
```

**自動化しやすい象限**：繰り返し度が高く、判断複雑度が低いタスク

---

## フェーズ2：自動化候補の選定・優先順位付け

### 2-1. 自動化適性スコアリング
各タスクを以下の基準で1〜5点評価し、合計点で優先順位を決める：

| 評価軸 | 内容 |
|---|---|
| 時間インパクト | 削減できる時間の大きさ |
| 自動化難易度（逆転） | 実装が容易か（5=簡単） |
| リスク | 誤動作した場合の影響（5=影響小） |
| 即効性 | 短期間で実装できるか |

### 2-2. 5時間削減の組み合わせ設計
- スコア上位タスクを積み上げ、合計5時間/週に達する組み合わせを特定
- 1つの大きなタスクではなく、複数の小さなタスクの組み合わせでも可

### 2-3. 技術アプローチの仮決め
タスクの性質に応じてツールを仮選定：

| タスクの種類 | 推奨アプローチ |
|---|---|
| アプリ間のデータ連携 | Zapier / Make / n8n |
| ファイル・スプレッドシート操作 | Python（openpyxl / pandas） |
| ブラウザ上の繰り返し操作 | RPA（UiPath / Power Automate） |
| 定型文書の作成 | テンプレート＋スクリプト |
| 通知・リマインダー | Slack Bot / メール自動化 |

---

## フェーズ3：PoC（概念実証）

### 3-1. 最優先タスク1〜2件に絞って試作
- 本番環境には手を付けず、テスト環境またはサンプルデータで実施
- 目的：「本当に自動化できるか」「想定外の例外はあるか」を確認する

### 3-2. 例外ケースの洗い出し
- 自動化スクリプトが失敗するエッジケースを意図的に試す
- 例外をどう扱うか（スキップ・通知・手動介入）を定義する

### 3-3. 費用対効果の再確認
- PoC結果をもとに実装コスト（時間・費用）を再見積もり
- ROIが見合わないタスクはリストから除外

---

## フェーズ4：本番実装・テスト

### 4-1. 実装順序の決定
- リスクが低く、効果が高いものから順に実装
- 1タスクずつ完結させてから次に進む（並行実装はしない）

### 4-2. 実装の標準構成
各自動化モジュールに以下を含める：

```
- トリガー定義（いつ動くか）
- 処理ロジック（何をするか）
- エラーハンドリング（失敗時の挙動）
- ログ出力（何が起きたか記録）
- 通知（完了・失敗を人間に伝える）
```

### 4-3. 並行稼働テスト
- 自動化と手動作業を2週間並行して実施
- 結果が一致するか検証してから手動作業を停止

### 4-4. ドキュメント作成
- 自動化の仕様・操作方法・トラブルシューティングをまとめる
- 自分以外でも維持管理できる状態にする

---

## フェーズ5：運用・改善

### 5-1. 効果測定（実装後2〜4週間）
- 実際に削減された時間を計測（目標：5時間/週）
- 目標未達の場合、原因分析と追加施策を検討

### 5-2. 定期メンテナンスの設定
- 月1回：自動化スクリプトの動作確認
- 四半期1回：新たな自動化候補がないか業務棚卸しを再実施

### 5-3. 次の20%に向けた拡張
- 今回の自動化で生まれた余力（5時間）の使い方を定義
- さらなる削減が必要な場合、フェーズ1に戻って繰り返す

## 典型的な失敗パターンと対策

| 失敗パターン | 対策 |
|---|---|
| 一度に大きく自動化しようとする | 1タスク完結を徹底し、小さく積み上げる |
| 例外処理を後回しにする | PoC段階で必ず例外ケースを試す |
| ドキュメントを省く | 実装と同時に書く（後回しにしない） |
| 効果測定をしない | 実装前後の時間ログを必ず比較する |
| ツールありきで考える | 業務棚卸しを先に完了させてからツールを選ぶ |
