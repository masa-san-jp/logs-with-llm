# 自組織ホワイトハックパイプライン構築計画

## 0. 構築方針

このパイプラインは、攻撃技術を実行する仕組みではなく、**自組織のセキュリティ・ガバナンス上の弱点を、許可された範囲で安全に顕在化し、是正し、経営判断に接続する統制改善システム**として構築する。

設計軸は次の4つ。

1. **経営管理主導**

   * 情報システム部門だけに閉じない
   * 経営・事業部門・個人情報管理・委託先管理を含める

2. **安全な検証**

   * 本番業務・顧客データ・要配慮個人情報に影響を与えない
   * 無許可の侵入・攻撃的検証は行わない

3. **台帳駆動**

   * 資産台帳
   * 権限台帳
   * リスク台帳
   * 是正タスク台帳
   * 検証履歴台帳

4. **継続運用**

   * 単発診断ではなく、週次・月次・四半期で回す

NIST Cybersecurity Framework 2.0 は、サイバーセキュリティを **GOVERN / IDENTIFY / PROTECT / DETECT / RESPOND / RECOVER** の6機能で整理しているため、本計画でも「統治・識別・防御・検知・対応・復旧」を基本構造にする。([NIST出版シリーズ][1])
また、経済産業省のサイバーセキュリティ経営ガイドライン Ver.3.0 は、経営者がリスク認識、体制構築、予算・人材確保などをリーダーシップをもって進めることを求めているため、経営管理部門が主導する設計と整合する。([経済産業省][2])

# 1. ゴール定義

## 最終ゴール

```md
自組織のセキュリティ・ガバナンスリスクを、
事故発生前に意図的・安全に顕在化し、
是正タスクとして管理し、
再検証し、
経営判断に接続する継続的パイプラインを構築する。
```

## 90日後の到達状態

```md
## 90日後に完成しているべき状態

- ホワイトハック憲章がある
- 対象範囲・禁止事項・承認フローが明文化されている
- 資産台帳・SaaS台帳・権限台帳・リスク台帳・是正タスク台帳がある
- Google Workspaceを中心に初回検証が完了している
- 重大リスクが経営報告されている
- 是正タスクが責任者・期限付きで管理されている
- 月次セキュリティ・ガバナンス会議が始まっている
- 四半期ごとの検証テーマが決まっている
```

# 4. Phase 1：ホワイトハック憲章を作る

## 目的

検証活動が暴走しないように、**実施権限・禁止事項・承認者・停止条件**を明文化する。

## 作成物

```md
## ホワイトハック憲章

### 1. 目的
自組織のセキュリティ・ガバナンスリスクを、事故前に安全に発見し、是正し、経営判断に接続する。

### 2. 基本原則
- 防御目的に限定する
- 正式な承認範囲内で行う
- 本番業務に影響を与えない
- 個人情報・要配慮個人情報を不用意に扱わない
- 発見事項は責任追及ではなく、統制改善に使う
- 是正完了まで追跡する

### 3. 承認者
- 経営責任者
- 経営管理責任者
- 情報システム責任者
- 個人情報管理責任者
- 対象部門責任者

### 4. 禁止事項
- 無許可診断
- 本番データ破壊
- サービス停止を伴う検証
- 不正ログイン試行
- 権限外データ閲覧
- 検証結果の無制限共有
- 攻撃手順の社内拡散

### 5. 緊急停止条件
- 本番業務への影響が出た場合
- 個人情報漏えいの恐れがある場合
- 対象外システムに影響が及んだ場合
- 想定外の障害が発生した場合
```

# 9. Phase 6：リスク評価基準を作る

## 評価軸

```md
## リスク評価軸

### 重大度
- 事業停止につながるか
- 個人情報・要配慮個人情報に影響するか
- 金銭被害につながるか
- 法令違反につながるか
- 行政対応・取引停止につながるか
- 信用毀損につながるか

### 発生可能性
- すでに設定不備が存在するか
- 日常業務で起こりやすいか
- 外部に露出しているか
- 多数の社員が関与しているか
- 委託先・外部環境に依存しているか

### 検出可能性
- ログで検知できるか
- アラートがあるか
- 誰が確認するか
- 何日以内に気づけるか
- 手作業確認に依存していないか

### 是正難易度
- 設定変更で直せるか
- 業務フロー変更が必要か
- 規程改定が必要か
- 予算が必要か
- 外部委託が必要か
```

## 優先度判定

```md
## 優先度

### P1：即時対応
- 個人情報漏えいの恐れがある
- 管理者権限が過大
- 退職者アカウントが残存
- 外部公開状態の機密情報がある

### P2：30日以内対応
- 権限棚卸し不足
- 外部共有管理不足
- ログ確認体制なし
- バックアップ復旧未確認

### P3：90日以内対応
- 規程未整備
- 教育不足
- 委託先確認不足
- 台帳未整備

### P4：次年度計画
- MDM導入

- SIEM導入
- 外部診断拡大
```

# 10. Phase 7：是正管理フローを作る

## 是正フロー

```text
発見
 ↓
リスク台帳登録
 ↓
重大度評価
 ↓
暫定対応判断
 ↓
恒久対応設計
 ↓
責任者・期限設定
 ↓
部門対応
 ↓
再検証
 ↓
完了判定
 ↓
経営報告
```

## 是正タスクの粒度

```md
## 悪いタスク例

- Google Driveを整理する
- 退職者アカウントを確認する
- セキュリティを強化する

## 良いタスク例

- 2026年4月末時点の退職者12名について、Google Workspace、会計SaaS、勤怠SaaS、業務SaaSのアカウント残存有無を確認し、残存があれば5営業日以内に停止する
- 外部共有されているGoogle Driveファイルのうち、個人情報を含む可能性があるファイルを抽出し、オーナー確認後、不要共有を解除する
- 特権管理者権限を持つアカウントを棚卸しし、常時利用する必要がないアカウントは権限を削除する
```

# 20. 外部専門家を入れるタイミング

外部専門家は最初から常時入れる必要はない。ただし、以下の場合は早期に入れる。

```md
## 外部専門家を入れるべき条件

- 外部公開Webサイトや顧客向けサービスがある
- 要配慮個人情報を大量に扱う
- ランサムウェア対策が未整備
- バックアップ復旧確認ができていない
- Google Workspaceの設定に不安がある
- 情報システム担当が1名または兼任である
- ベンダー管理DBに自社でアクセスできない
- 重大リスクを経営に説明する必要がある
```

外部から脆弱性報告を受ける仕組みを整える場合、CISAのVDP Platformのように、外部の善意ある報告を受け取り、対応・協力する考え方が参考になる。ただし、最初は公開バグバウンティではなく、社内・委託先・取引先向けの報告窓口から始めるのが現実的。([CISA][4])
