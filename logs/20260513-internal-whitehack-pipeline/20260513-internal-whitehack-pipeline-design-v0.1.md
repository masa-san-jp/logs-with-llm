# 自組織ホワイトハックパイプライン設計 v0.1

## 1. 目的

自組織のセキュリティ・ガバナンス上のリスクを、外部攻撃や重大事故として顕在化する前に、**管理された検証・演習・監査によって意図的に発見し、是正し、経営判断に接続する仕組み**を作る。

ここでいう「ホワイトハック」は、攻撃手順の実行ではなく、以下を含む防御目的のパイプラインとして扱う。

* 情報資産・業務プロセス・権限の棚卸し
* リスク仮説の作成
* 許可された範囲での脆弱性診断・設定確認・アクセス権確認
* インシデント対応演習
* 委託先・SaaS・Google Workspace等の統制確認
* 是正タスク化
* 再検証
* 経営報告

NIST Cybersecurity Framework 2.0 は、サイバーセキュリティを **GOVERN / IDENTIFY / PROTECT / DETECT / RESPOND / RECOVER** の機能で整理しており、今回のパイプラインはこの構造に沿って設計すると経営管理・現場運用・監査に接続しやすい。([NIST出版シリーズ][1])

---

## 2. 基本方針

### 2.1 やること

* 自社が正式に管理権限を持つ範囲だけを対象にする
* 事前に対象・期間・権限・禁止事項を明文化する
* 本番業務停止・個人情報漏えい・過剰な攻撃再現を避ける
* 技術リスクだけでなく、稟議・権限・委託先・運用逸脱・教育不足も検証する
* 発見したリスクは必ず是正責任者・期限・再検証方法まで落とす

### 2.2 やらないこと

* 許可のないシステム・外部サービスへの侵入試行
* 実運用アカウントの不正利用
* 本番環境への負荷攻撃
* 社員を欺く過度なソーシャルエンジニアリング
* 要配慮個人情報・顧客データを使った危険な検証
* PoCコードや攻撃手順の社内拡散
* 「見つけた」で終わる診断

## Phase 1. 資産・業務・データ棚卸し

ホワイトハックの対象は「システム」だけではない。中小企業・サービス業では、むしろ以下のような領域が事故原因になりやすい。

* Google Driveの共有設定
* 退職者アカウント
* 店舗・拠点での端末管理
* 個人スマホ・私物端末
* SaaSの管理者権限
* スプレッドシートの公開範囲
* 顧客情報・要配慮個人情報の保管場所
* 委託先・ベンダーの管理範囲
* 口頭・チャット・メールでの例外運用
* バックアップと復旧手順

METIのサイバーセキュリティ経営ガイドライン Ver.3.0 も、サプライチェーンを介した被害拡大、ランサムウェアによる事業停止リスク、経営者のリーダーシップを重視しているため、技術対策だけでなく経営・委託先・業務継続を含めるべき。([経済産業省][2])

## Phase 2. リスク仮説作成

棚卸し結果をもとに、検証すべきリスク仮説を作る。

### リスク仮説の形式

```md
## リスク仮説テンプレート

- リスクID：
- リスク名：
- 想定される事故：
- 起点：
- 影響を受ける情報資産：
- 影響を受ける業務：
- 想定原因：
- 現在の統制：
- 統制の弱点：
- 検証方法：
- 検証時の禁止事項：
- 重大度：
- 発生可能性：
- 検出可能性：
- 優先度：
- 是正責任者：
```

### 例

```md
## リスク仮説例

### R-001：Google Driveの外部共有過多

- 想定される事故：
  - 顧客情報・経営資料・申請書類が外部に共有されたまま放置される

- 起点：
  - 社員が一時的に外部共有したファイル
  - 退職者が作成した共有フォルダ
  - 部門横断プロジェクト用フォルダ

- 想定原因：
  - 外部共有ルールが曖昧
  - 共有期限が設定されていない
  - 定期レビューがない
  - 管理者が共有状況を一覧で確認していない

- 検証方法：
  - Google Driveの外部共有ファイル一覧を取得
  - 機密区分ごとに共有範囲を確認
  - オーナー不明・退職者所有・リンク共有ファイルを抽出

- 是正：
  - 外部共有の原則禁止または承認制
  - 共有期限設定
  - 月次レビュー
  - 機密フォルダの管理者移管
```

## Phase 3. 検証計画作成

検証は、攻撃的な手法から始めない。以下の順番で安全に進める。

```text
設定確認
 ↓
権限確認
 ↓
ログ確認
 ↓
机上演習
 ↓
限定的な実地テスト
 ↓
外部専門家による診断
```

### 検証カテゴリ

```md
## 検証カテゴリ

### A. ガバナンス検証
- 情報セキュリティ規程があるか
- 個人情報管理規程があるか
- 権限付与・削除の承認フローがあるか
- 退職者アカウント削除が期限内に行われているか
- SaaS導入時の審査があるか
- 委託先のセキュリティ確認があるか

### B. Google Workspace検証
- 管理者権限の人数
- 2段階認証の強制状況
- 外部共有設定
- 共有ドライブの管理状態
- 退職者・休職者アカウント
- Gmail転送設定
- OAuthアプリ連携
- 監査ログの確認体制

### C. 端末検証
- OS更新状況
- EDR / antivirus 状況
- 画面ロック
- ディスク暗号化
- ローカル管理者権限
- 紛失時のリモートワイプ
- 私物端末利用

### D. データ管理検証
- 要配慮個人情報の保管場所
- アクセス権
- ダウンロード可否
- 印刷可否
- 外部送信可否
- 削除・保管期限
- バックアップ

### E. インシデント対応検証
- 誰が第一報を受けるか
- どのチャネルで報告するか
- 初動判断者は誰か
- 顧客・行政・取引先への報告判断
- 証跡保全
- 復旧手順
- 広報対応

### F. 委託先・ベンダー検証
- DB管理範囲
- 障害時の連絡体制
- 脆弱性対応SLA
- ログ提供可否
- バックアップ責任
- データ返却・削除条件
- 再委託の有無
```

OWASP SAMM は、組織のソフトウェアセキュリティ戦略をリスクに合わせて構築・改善するための成熟度モデルで、開発・取得を含むライフサイクル全体を対象にできる。自社開発が少ない場合でも、SaaS選定・外部委託・Webサイト運用の評価軸として使える。([OWASP Foundation][3])

### 4.2 脆弱性報告の受け皿

外部公開Webサイトや顧客向けサービスがある場合は、将来的に **脆弱性報告窓口** を整備する。CISA は、脆弱性開示ポリシーにより、善意のセキュリティ研究者からの報告を受け、リスク低減につなげる考え方を示している。([CISA][4])

ただし、最初から公開バグバウンティにする必要はない。中小企業では、まず以下の順番が現実的。

```text
1. 社内報告窓口
2. ベンダー・委託先からの報告窓口
3. 取引先からの報告窓口
4. 外部公開用の脆弱性報告ポリシー
5. 必要に応じて外部診断・バグバウンティ
```

## Phase 5. リスク評価・優先順位付け

発見事項は「危険そう」で処理しない。以下の軸で評価する。

```md
## リスク評価軸

### 重大度
- 事業停止につながるか
- 個人情報・要配慮個人情報に影響するか
- 金銭被害につながるか
- 法令・契約違反につながるか
- レピュテーション毀損につながるか

### 発生可能性
- すでに設定不備が存在するか
- 類似事例があるか
- 攻撃対象として露出しているか
- 社員の通常業務で起こりやすいか
- 委託先・外部環境に依存しているか

### 検出可能性
- ログで検知できるか
- 誰が確認するか
- 何日以内に気づけるか
- アラートがあるか
- 手作業確認に依存していないか

### 是正容易性
- すぐ直せるか
- 設定変更で済むか
- 規程変更が必要か
- 業務プロセス変更が必要か
- 予算・外部委託が必要か
```

NIST SP 800-30 は、リスク評価を、組織の活動・資産・個人・他組織等へのリスクを識別・推定・優先順位付けし、経営層が対応方針を決めるためのプロセスとして位置づけている。([NIST Computer Security Resource Center][5])

## Phase 6. 是正タスク化

発見事項は、必ず業務改善タスクに変換する。

```md
## 是正タスクテンプレート

- リスクID：
- 発見事項：
- 影響：
- 根本原因：
- 暫定対応：
- 恒久対応：
- 対応責任者：
- 協力部門：
- 期限：
- 必要予算：
- 必要な規程変更：
- 必要な教育：
- 完了条件：
- 再検証方法：
- 経営判断が必要な事項：
```

### 是正の分類

```md
## 是正分類

### 1. 即時対応
- 危険な外部共有を停止
- 退職者アカウント停止
- 不要な管理者権限削除
- 公開リンク削除
- パスワード再設定
- 2段階認証強制

### 2. 短期対応
- 権限棚卸し
- 共有ドライブ整理
- 端末台帳整備
- SaaS台帳整備
- インシデント連絡網作成

### 3. 中期対応
- 規程改定
- 入退社プロセス改定
- 外部共有承認フロー導入
- ベンダー契約条項見直し
- バックアップ・復旧訓練
- 管理者教育

### 4. 長期対応
- MDM導入
- SIEM / ログ監視導入

- EDR展開
- ゼロトラスト設計
- サプライチェーン管理高度化
```

## Phase 7. 再検証

是正完了の自己申告だけでは不十分。

```md
## 再検証項目

- 設定が実際に変更されたか
- 権限が削除されたか
- ログで確認できるか
- 同じ問題が別部門に残っていないか
- 手順書が更新されたか
- 担当者が理解しているか
- 例外運用が残っていないか
- 次回以降、自動検知できるか
```

再検証後、以下のどれかに分類する。

```md
## 再検証結果

- 完了
- 一部完了
- 暫定対応のみ
- 未対応
- 経営判断待ち
- 対応不能
- リスク受容
```

# 6. 重点テーマ候補

## 第1優先：Google Workspace統制

```md
## 検証テーマ

- 外部共有
- 共有ドライブ
- 管理者権限
- OAuthアプリ
- 2段階認証
- 退職者アカウント
- Gmail転送
- 監査ログ
```

理由：自社の情報資産がGoogle Workspaceに集中している場合、ここが最も高頻度・高影響のリスク領域になる。

---

## 第2優先：入退社・異動プロセス

```md
## 検証テーマ

- 入社時アカウント発行
- 部門異動時の権限変更
- 退職時アカウント停止
- Drive所有権移管
- SaaSアカウント削除
- 店舗・拠点権限削除
```

理由：人事イベントと権限管理のズレは、内部不正・情報漏えい・退職者アクセスの原因になる。

---

## 第3優先：要配慮個人情報の取扱い

```md
## 検証テーマ

- どこに保存されているか
- 誰が閲覧できるか
- 外部共有されていないか
- ダウンロード可能か
- 印刷可能か
- 保存期限があるか
- 削除手順があるか
- 委託先が扱っているか
```

理由：障害福祉・人事・顧客対応などで要配慮個人情報を扱う場合、単なるITリスクではなく、法務・信用・事業継続リスクになる。

---

## 第4優先：端末紛失・ランサムウェア

```md
## 検証テーマ

- PC暗号化
- 画面ロック
- OS更新
- EDR / antivirus
- バックアップ
- 復旧手順
- 共有フォルダへの影響
- 初動報告フロー
```

理由：中小企業では、端末管理の不備がそのまま全社情報資産への侵入口になる。

# 12. 実装上の重要論点

## 12.1 「診断」より「統制改善」を主目的にする

脆弱性診断だけを行うと、発見事項が技術部門に閉じる。今回の目的は、経営管理・情報システム・事業部門が連動して、リスクを潰し続ける仕組みを作ること。

---

## 12.2 「部門を責める仕組み」にしない

現場が報告しなくなると、ホワイトハックパイプラインは機能しない。

```md
## 運用原則

- 発見した人を責めない
- 隠れていたリスクを可視化したことを評価する
- 個人ではなくプロセスを改善する
- 例外運用は責任追及ではなく設計不備として扱う
- 是正不能なものは経営判断に上げる
```

---

## 12.3 「月1回のイベント」ではなく「常時パイプライン」にする

理想形は、以下が自動的に流れる状態。

```text
ログ・台帳・申請・共有設定
        ↓
AI / スクリプトで異常候補抽出
        ↓
人間が確認
        ↓
リスク台帳化
        ↓
是正タスク化
        ↓
再検証
        ↓
経営報告
```

---

# 13. このパイプラインの最小実行形

最初から高度なセキュリティ組織を作る必要はない。初期形は以下でよい。

```md
## 最小実行形

### 毎週

### 年1回
- 外部専門家による診断
- 規程改定
- 復旧訓練
- セキュリティ投資計画
```

# 14. 結論

自組織ホワイトハックパイプラインは、単なる脆弱性診断ではなく、次の構造で設計するべき。

```text
資産を把握する
 ↓
リスク仮説を作る
 ↓
安全に検証する
 ↓
重大度を評価する
 ↓
是正タスクに落とす
 ↓
再検証する
 ↓
経営判断に接続する
 ↓
毎週・毎月・四半期で回し続ける
```

最初の重点対象は、以下の順番が妥当。

```md
## 初期優先順位

1. Google Workspace外部共有・管理者権限
2. 入退社・異動時の権限管理
3. 要配慮個人情報の保存場所とアクセス権
4. 端末紛失・ランサムウェア対策
5. 委託先・ベンダー管理
6. インシデント対応演習
```

この設計により、セキュリティを「事故対応」ではなく、**経営管理部門が主導する継続的なリスク顕在化・是正・統制強化プロセス**として運用できる。
