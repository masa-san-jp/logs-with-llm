# 自組織ホワイトハックパイプライン設計

## 概要

自組織のセキュリティ・ガバナンスリスクを「意図的に顕在化」させ、攻撃者視点でリスクを先取りして潰す内製型の継続的ホワイトハックパイプライン。外部ペネトレーションテストの補完ではなく、**組織の文化・プロセス・技術の三層を常時検証する仕組み**として設計する。

---

## 全体アーキテクチャ

```
[ リスクモデリング ] → [ 攻撃シナリオ設計 ] → [ 実行 ] → [ 評価・報告 ] → [ 是正・制度化 ]
         ↑                                                                        ↓
         └──────────────────── フィードバックループ ──────────────────────────────┘
```

---

## Phase 1｜リスクモデリング

### 1-1. 対象ドメインの三層定義

| 層 | 対象 | 代表リスク例 |
|---|---|---|
| **技術層** | インフラ、アプリ、SaaS設定、APIキー管理 | 設定ミス、脆弱性、権限過多 |
| **プロセス層** | 承認フロー、変更管理、インシデント対応手順 | 手順の形骸化、例外処理の放置 |
| **文化層** | 社員行動、ソーシャルエンジニアリング耐性、報告文化 | フィッシング耐性不足、問題の隠蔽 |

### 1-2. 脅威モデル構築（STRIDE × 自組織文脈）

各ドメインに対してSTRIDEフレームワークを適用し、自組織固有の文脈（業種、規制環境、組織構造）で重み付けする。

- **S**poofing（なりすまし）
- **T**ampering（改ざん）
- **R**epudiation（否認）
- **I**nformation Disclosure（情報漏洩）
- **D**enial of Service（サービス拒否）
- **E**levation of Privilege（権限昇格）

### 1-3. リスクバックログの作成

```
形式: [ID] | [層] | [脅威カテゴリ] | [想定経路] | [ビジネスインパクト] | [検証優先度]
例: R-042 | 文化層 | Spoofing | 経営役員を偽装したメール承認依頼 | 不正送金・情報漏洩 | Critical
```

優先度付けにはCVSSではなく**ビジネスコンテキストスコア**（財務影響 × 実現可能性 × 検出困難度）を使用する。

## Phase 2｜攻撃シナリオ設計

### 2-1. シナリオテンプレート

```markdown
## シナリオID: WH-[YYQ]-[連番]
- **目的**: 検証したいリスク仮説
- **攻撃者ペルソナ**: 内部犯行者 / 外部攻撃者 / 取引先関係者 etc.
- **攻撃経路**: 具体的なステップ（Kill Chainに沿って記述）
- **成功条件**: 何が達成されたらシナリオ成功か
- **検知期待値**: 既存統制で何が検知されるべきか
- **スコープ外**: 絶対に触れない範囲の明記（本番DB直接操作等）
- **実行承認者**: [氏名・役職]
- **緊急停止トリガー**: 実行を中断する条件
```

### 2-2. シナリオ類型

#### 技術層シナリオ例
- **クラウド設定ドリフト検証**: IAMポリシーの過剰権限・S3バケット公開設定の意図的な再現と検出確認
- **SaaSシャドーIT侵入**: 未承認SaaS経由でのデータ持ち出し経路のシミュレーション
- **サプライチェーン攻撃模倣**: 依存パッケージの脆弱性注入シミュレーション

#### プロセス層シナリオ例
- **緊急変更手順の迂回**: 緊急対応を口実にした変更管理プロセスの無効化試行
- **デュアルコントロール回避**: 承認権限の集中を突いた単独処理の実行試行
- **インシデント対応遅延**: 検知から報告までのSLO未達シナリオの意図的再現

#### 文化層シナリオ例
- **標的型フィッシング**: 役職・業務文脈に特化したメール・Slack偽装
- **物理侵入テスト**: テールゲーティング、不審USBの放置による挙動確認
- **内部告発チャネルの機能確認**: 匿名報告が実際に適切に処理されるかの検証

## Phase 3｜実行管理

### 3-1. ガバナンス体制

```
┌─────────────────────────────────────────────┐
│            ホワイトハック委員会               │
│  (CISO / 法務 / 経営代表 / 現場責任者)       │
│  → シナリオ最終承認・エスカレーション判断    │
└────────────────┬────────────────────────────┘
                 │
    ┌────────────┼────────────┐
    ▼            ▼            ▼
 Red Team    Blue Team   Purple Team
 (攻撃実行)  (防御・監視)  (知識共有促進)
```

**重要原則**: Red TeamとBlue Teamは実行前の情報共有を原則禁止（ブラインドテスト）。Purple Teamは事後のみ関与。

### 3-2. 実行ルールブック

- 全実行にはタイムスタンプ付きの証跡ログを残す
- 本番環境への影響が生じた時点で即時停止（緊急停止手順を事前に整備）
- 法的・コンプライアンス上のリスクがあるシナリオは法務レビューを必須とする
- 「どこまでやるか」の合意を書面で取得してから実行する（スコープ合意書）

### 3-3. ツールスタック例

| 用途 | OSS / ツール例 |
|---|---|
| 偵察・OSINT | Maltego, theHarvester, Shodan |
| インフラ脆弱性 | Nessus / OpenVAS, Nuclei |
| クラウド設定監査 | Prowler (AWS), Checkov, ScoutSuite |
| フィッシングシミュレーション | GoPhish, Lucy |
| 内部権限分析 | BloodHound (AD), Lynis |
| ログ・検知確認 | Splunk / Elastic SIEM との連携確認 |
| シナリオ管理 | Jira / Notion (チケット化必須) |

## Phase 4｜評価・報告

### 4-1. 評価指標（KPI）

| 指標 | 説明 | 目標値例 |
|---|---|---|
| **検知率 (DR)** | シナリオ成功のうち既存統制が検知できた割合 | ≥ 85% |
| **平均検知時間 (MTTD)** | 攻撃行為発生から検知までの時間 | ≤ 30分（Critical） |
| **平均対応時間 (MTTR)** | 検知からインシデント封じ込めまでの時間 | ≤ 2時間（Critical） |
| **是正完了率** | 前回報告の指摘事項の是正完了割合 | ≥ 90% (四半期後) |
| **文化層スコア** | フィッシング報告率、不審行為通報率 | 四半期比改善 |

### 4-2. 報告書構成

```
1. エグゼクティブサマリー（経営向け：1ページ）
   - 今回の検証スコープ
   - Critical発見事項（件数・概要）
   - 前回比トレンド

2. 技術詳細レポート（現場向け）
   - シナリオ別実行結果
   - 攻撃経路の再現図
   - 検知・未検知の分岐点分析

3. リスク是正計画
   - 優先度別アクションアイテム
   - 担当・期限・完了条件の明記

4. ガバナンス所見
   - プロセス・文化層の観察事項
   - 制度変更の提言
```

## Phase 5｜是正・制度化

### 5-1. 是正トラッキング

発見事項はすべてリスクバックログに戻し、是正完了を次サイクルの検証シナリオで確認する（是正の検証）。

```
発見 → チケット起票 → 担当アサイン → 是正実施 → 再テスト → クローズ → バックログ更新
```

### 5-2. 学習ループの仕組み

- **ポストモーテム**: シナリオ実行後72時間以内に実施。非難なし（blameless）が原則
- **ナレッジベース**: 攻撃手法・防御パターンを社内Wikiに蓄積（Red Team Playbook化）
- **訓練への転用**: 文化層の発見事項は社員セキュリティ教育に反映

### 5-3. 実行サイクル設計

```
継続的（自動）: クラウド設定監査、依存関係スキャン、SIEM検知ルール検証
月次:           フィッシングシミュレーション、権限棚卸
四半期:         シナリオベース総合演習、ガバナンス評価
年次:           外部Red Teamとのジョイント演習、脅威モデルの全面見直し
```

## 段階的導入ロードマップ

### Step 1｜基盤整備（1〜2ヶ月）
- ガバナンス体制の設立（委員会・ルールブック策定）
- リスクバックログの初版作成（上位20件）
- ツールスタックの選定・環境構築

### Step 2｜技術層ファースト（3〜4ヶ月）
- クラウド設定・IAM・SaaS設定の自動監査を先行導入
- 発見サイクルを回して報告フォーマットを確立

### Step 3｜プロセス・文化層の追加（5〜6ヶ月）
- フィッシングシミュレーション開始
- 変更管理・承認プロセスの迂回テスト
- Blue Teamの検知能力評価を開始

### Step 4｜統合・最適化（7ヶ月〜）
- 三層統合シナリオの実行
- KPIダッシュボードの整備
- 年次外部演習の組み込み

---

## 法的・倫理的留意点

- **書面による承認**: 全シナリオは実行前に権限者の書面承認を取得
- **スコープ明文化**: 「やらないこと」を「やること」と同等に詳細に定義する
- **データ保護**: テスト中に取得した実データは即時削除・厳格に管理
- **従業員への配慮**: 文化層テスト（フィッシング等）の結果は人事評価に使用しない旨を方針に明記
- **法務確認**: 不正アクセス禁止法・個人情報保護法等との整合性を定期レビュー
