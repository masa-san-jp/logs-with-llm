# 補助金・助成金獲得パイプライン｜構築ロードマップ

## 前提：構築の全体像

```
Week 1-2        Week 3-4        Month 2         Month 3         Month 3以降
[基盤整備]  →  [仕組み設計]  →  [試験運用]  →  [本格稼働]  →  [継続改善]
```

構築は**一度にすべてを完成させない**。最小構成で動かし、申請サイクルを回しながら改善する。

---

## Step 1｜現状把握と目標設定（Week 1）

### 1-1. 補助金獲得の現状棚卸し
**アウトプット：現状診断シート**

- [ ] 過去3年間の補助金申請履歴を一覧化（申請名・申請額・採択可否・受給額）
- [ ] 現在進行中の申請・受給中の案件を一覧化
- [ ] 過去の申請で使った書類・事業計画書を収集・保管場所の確認
- [ ] 申請業務に関わった社内メンバー・外部専門家の確認

**確認すべき問い**
- 過去の採択率はどの程度か
- どの補助金カテゴリに強みがあるか
- 申請業務のボトルネックはどこにあったか（情報収集・書類作成・承認フロー など）

---

### 1-2. 年次目標の設定
**アウトプット：補助金獲得目標シート**

- [ ] 当期・来期の投資計画（設備・IT・人材）を経営計画から抽出
- [ ] 投資計画のうち補助金で賄いたい金額を設定
- [ ] 補助金カテゴリ別の目標金額に分解（設備系・雇用系・IT系・自治体系）
- [ ] 目標達成に必要な申請件数をシミュレーション（目標額 ÷ 平均採択額 ÷ 想定採択率）
- [ ] 申請工数の年間キャパシティ確認（社内リソース + 外部専門家の活用範囲）

---

## Step 2｜Phase 0 自組織マスタの整備（Week 1-2）

**アウトプット：自組織マスタ台帳（初版）**

優先度順に整備する。高優先から着手し、未整備のまま申請が来たら都度補完する。

### 優先度A：申請資格判定に直結（最初に整備）
- [ ] 法人名・法人番号・設立年月日・資本金・本社所在地・代表者名
- [ ] 主たる業種・JSICコード
- [ ] 従業員数（正規・非正規別、直近3年の推移）
- [ ] 直近3期の売上高・経常利益・付加価値額
- [ ] 納税状況の確認（滞納なし）
- [ ] 過去の補助金受給履歴（補助金名・年度・金額・事業内容）

### 優先度B：申請書記述の質に直結（次に整備）
- [ ] 主力事業の概要（200字・400字・800字の3バージョン）
- [ ] ターゲット市場・顧客の記述
- [ ] 競合優位性・独自技術・強みの記述
- [ ] 現在の経営課題トップ3（定量データ付き）
- [ ] 中期経営計画の方向性（3年後のありたい姿）
- [ ] 賃上げ計画・実績（加点要件として重要）

### 優先度C：書類整備（申請発生時に随時補完可）
- [ ] 登記簿謄本・印鑑証明（取得フロー確認）
- [ ] 直近2〜3期の決算書（税務署受付印付）の保管場所確認
- [ ] 納税証明書（その1・その2）の取得フロー確認
- [ ] 会社概要・パンフレットの最新版確認
- [ ] 許認可・資格の一覧整理

### 優先度D：テンプレート化（過去書類があれば対応）
- [ ] 過去の採択申請書を匿名化・汎用化してテンプレート化
- [ ] 不採択時の審査講評があれば改善コメントと合わせて保管
- [ ] 補助金カテゴリ別（設備投資系・IT導入系・雇用系）のひな形作成

---

## Step 3｜情報収集の仕組み化（Week 2）

**アウトプット：補助金情報収集ルーティン**

### 3-1. ソース登録（1回設定）
- [ ] ミラサポplus のメールアラート設定（業種・地域・金額でフィルタ）
- [ ] J-Net21 のメールマガジン登録
- [ ] 地元商工会議所・商工会のメール案内登録
- [ ] 都道府県・市区町村の産業振興課メルマガ登録
- [ ] 業界団体の補助金情報メーリングリスト確認
- [ ] 顧問税理士・社労士・中小企業診断士への情報共有依頼

### 3-2. 情報整理のルーティン設定
- [ ] 週次で新規公募をチェック・一次整理する担当者を決める
- [ ] 公募情報の収集フォーマットを作成（補助金名・主管機関・対象者・補助率・上限額・締切）
- [ ] 収集した情報を案件管理台帳（Step 4で作成）のS0に登録するフローを確立

---

## Step 4｜ポートフォリオ管理台帳の構築（Week 2-3）

**アウトプット：案件管理台帳（運用可能な状態）**

### 4-1. 管理ツールの選定
目的別に選択する。複雑なツールより**使い続けられるシンプルさ**を優先。

| ツール | 向いているケース |
|---|---|
| Googleスプレッドシート | 担当者1〜2名・シンプルに始めたい |
| Notion | チームで情報共有・ステージ管理を可視化したい |
| Airtable | フィルタ・ビュー切り替えを柔軟に使いたい |
| Excelローカル管理 | 社外共有不要・情報管理を厳格にしたい |

### 4-2. 台帳に持たせる項目
- [ ] 補助金名・主管機関
- [ ] ステージ（S0〜S8）
- [ ] 申請額（見込み）
- [ ] 採択確度（%）
- [ ] 期待値（= 申請額 × 採択確度）← 自動計算
- [ ] 公募締切日・申請準備開始日
- [ ] 担当者（社内・外部）
- [ ] 入金予定時期
- [ ] 備考・リスクメモ

### 4-3. 集計ビューの設定
- [ ] **パイプライン集計**：ステージ別の申請額合計・期待値合計
- [ ] **目標達成見通し**：期待値合計 vs 目標金額（RAGステータス）
- [ ] **締切カレンダー**：向こう3ヶ月の締切一覧
- [ ] **入金CF予測**：月別の入金見込み額

---

## Step 5｜申請プロセスの標準化（Week 3-4）

**アウトプット：申請標準プロセス（チェックリスト＋テンプレート）**

### 5-1. 適格性評価チェックリストの作成
- [ ] 申請資格確認項目（業種・規模・設立年数・重複制限）
- [ ] 事業適合度スコアリングシート（審査項目 × 自社の強み）
- [ ] 費用対効果試算フォーマット（申請工数コスト vs 獲得見込み額）

### 5-2. 申請準備チェックリストの作成
- [ ] 補助金カテゴリ別の必要書類一覧
- [ ] 提出前最終確認リスト（記載漏れ・数値の整合・添付ファイル確認）
- [ ] 申請書レビュー基準（審査項目ごとの記述要件）

### 5-3. 採択後管理チェックリストの作成
- [ ] 交付決定前発注禁止の確認フロー
- [ ] 証憑管理ルール（請求書・領収書・振込記録のセット保管）
- [ ] 実績報告スケジュールの自動リマインド設定
- [ ] 事後モニタリング報告の期間・内容の記録

---

## Step 6｜役割・体制の確立（Week 3-4）

**アウトプット：RACI表（誰が何をするか）**

### 6-1. 社内体制の設計

| 役割 | 担当範囲 | 目安工数 |
|---|---|---|
| **オーナー**（経営者・責任者） | 目標設定・最終承認・戦略判断 | 月2〜4時間 |
| **申請担当**（実務推進） | 情報収集・書類作成・進捗管理 | 月10〜20時間 |
| **財務担当** | 経費管理・証憑整理・CF計画 | 申請中・精算時に集中 |
| **外部専門家**（必要に応じて） | 申請書レビュー・複雑案件の代行 | 案件依存 |

### 6-2. 外部専門家との連携設計
- [ ] 中小企業診断士・行政書士・社労士の活用範囲を決める
  - 相談のみ（低コスト・社内工数大）
  - 申請書レビューのみ
  - 特定補助金の全工程委託（成功報酬型が多い）
- [ ] 複数の専門家候補をリストアップ・関係構築
- [ ] 成功報酬率の相場確認（採択額の5〜15%が一般的）

### 6-3. 承認フローの設計
- [ ] 申請判断（適格性評価通過）の承認フロー
- [ ] 申請書提出前の最終承認フロー
- [ ] 交付決定後の発注承認フロー（交付決定確認を必須ステップに組み込む）

---

## Step 7｜試験運用（Month 2）

**アウトプット：初回申請の完了 + 運用上の課題リスト**

### 7-1. パイロット申請の選定
- [ ] 採択難易度が中程度（採択率40〜60%）の案件を1〜2件選ぶ
- [ ] 締切まで十分な準備期間がある案件を優先
- [ ] 作成したチェックリスト・テンプレートを実際に使って申請する

### 7-2. プロセスの観察・記録
- [ ] 各ステップの実工数を記録（計画との乖離を把握）
- [ ] チェックリストの漏れ・不足項目を記録
- [ ] 書類収集・承認フローのボトルネックを記録

### 7-3. 台帳・ツールの動作確認
- [ ] ステージ更新・期待値集計が正しく機能するか確認
- [ ] RAGステータスの判定ロジックが意図通りか確認
- [ ] 締切アラート・リマインダーが機能するか確認

---

## Step 8｜本格稼働・定期レビューの開始（Month 3）

**アウトプット：月次・四半期レビューの定例化**

### 8-1. 月次レビューの開始
- [ ] 月1回、台帳の期待値合計とRAGステータスを確認
- [ ] 翌月の締切案件・準備開始案件を確認
- [ ] 新規S0案件の追加・優先順位の更新

### 8-2. 四半期レビューの設計
- [ ] KPI進捗（申請件数・採択件数・採択率・採択総額）を確認
- [ ] パイプラインの期待値合計が目標の1.5倍以上あるか確認
- [ ] ポートフォリオの分散（規模・主管機関・時期）に偏りがないか確認

### 8-3. 年次レビュー（Phase Sの更新）
- [ ] KPI実績の評価と翌年度目標の設定
- [ ] Phase 0マスタの年次更新（財務データ・賃上げ実績・受給履歴）
- [ ] 申請書テンプレートの改訂（採択・不採択を踏まえた改善）

## 構築スケジュールサマリー

| 週 | 主な作業 | 主なアウトプット |
|---|---|---|
| Week 1 | 現状棚卸し・目標設定・Phase 0 優先度A整備 | 現状診断シート・目標シート・マスタ台帳（基本情報） |
| Week 2 | Phase 0 優先度B-C整備・情報収集仕組み化・台帳設計 | マスタ台帳（事業情報）・情報収集ルーティン・台帳初版 |

| Month 2 | パイロット申請・運用上の課題収集 | 初回申請・課題リスト |
| Month 3 | 本格稼働・月次レビュー開始 | 定例レビュー運用開始 |

## 最小構成で始める場合（リソースが限られる場合）

「完璧な仕組みを作ってから動く」より「動きながら整える」を優先する。

| 優先度 | 最初に最低限やること |
|---|---|
| 1 | Phase 0 優先度Aのみ整備（法人基本・財務・受給履歴） |
| 2 | スプレッドシートで案件台帳を作る（7列で十分：補助金名・ステージ・申請額・確度・期待値・締切・担当） |
| 3 | 年間カレンダーに主要補助金の締切を登録する |
| 4 | 1件申請して、プロセスの問題点を洗い出す |

> 上記4つが完了すれば、最小限の「計画的に動けるパイプライン」は稼働する。残りは運用しながら追加する。
