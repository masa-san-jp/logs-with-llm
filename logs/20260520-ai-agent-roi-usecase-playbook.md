# AIエージェントROI実現のためのユースケース選定・実行プレイブック

作成日: 2026-05-20  
対象: 経営幹部、事業責任者、DX/AI推進責任者、業務部門責任者  
前提: IBMの記事「誇大宣伝を大きな影響力へ：ビジネス・リーダーがAIエージェントでROIを実現する方法」を起点に、AIエージェント導入で最初に取り組むべきユースケースと、明日から始められる具体的プラクティスを整理する。

---

## 1. エグゼクティブサマリー

### 1.1 結論

最初に最も取り組むべきユースケースは、**社内問い合わせ・申請処理エージェント**である。

その中でも、明日から始める具体ケースとしては、**経費精算問い合わせAIエージェント**を第一候補にする。

狙いは、最初から完全自動化することではない。  
初期ゴールは、**担当者が確認して使える回答案をAIで作成し、1件あたりの処理時間を30%以上削減すること**である。

### 1.2 もっとも重要な判断

AIエージェント導入で最初に避けるべきなのは、次の発想である。

```text
既存業務
+
チャットボット
=
少し便利だがROI不明
```

成功確率が高いのは、次の発想である。

```text
業務分解
+
AIが担う処理の明確化
+
人間承認
+
システム連携
+
KPI計測
=
ROIが見える業務変革
```

### 1.3 推奨する初期スコープ

最初のスコープは、**自動回答Bot**ではなく、**回答案作成アシスタント**に限定する。

```text
社員から問い合わせが来る
↓
AIが問い合わせ内容を分類する
↓
AIが経費規程・FAQ・過去回答を検索する
↓
AIが回答案を作る
↓
AIが根拠箇所を示す
↓
経理担当者が確認する
↓
担当者が送信する
↓
修正内容をログ化する
```

この形にすれば、AIの価値を出しながら、誤回答・誤処理・過剰自動化のリスクを抑えられる。

## 2. IBM記事に対する評価

### 2.1 見立て

IBMの記事の主張は、実務的にはかなり妥当である。  
ただし、IBM watsonx Orchestrateなどへの製品導線を含むコンテンツであるため、**「オープン・オーケストレーション・レイヤーが重要」という主張は、製品ポジショニングを含めて読むべき**である。

記事の骨子は以下である。

- AIエージェントは実験段階を超えているが、期待ROIを出せていない企業が多い。
- 問題はAIそのものではなく、デプロイ方法である。
- 初期は売上成長よりも、測定しやすいコスト削減ユースケースから始めるべきである。
- 適切なユースケースを選び、ベースラインを測り、アーキテクチャを整え、ROIを測定すべきである。
- 全社にAIをばらまくのではなく、明確な高インパクト領域から小さく始めるべきである。

IBMの記事では、2025年のIBM Institute for Business Valueの経営幹部調査として、AIイニシアチブのうち期待ROIを達成したものは25%、企業全体に拡大したものは16%にとどまると紹介されている。出典: [IBM「ビジネス・リーダーがAIエージェントでROIを実現する方法」](https://www.ibm.com/jp-ja/think/insights/realize-roi-ai-agents)

### 2.2 良い点

IBM記事の良い点は、AIエージェント導入を**技術導入ではなく業務設計の問題**として扱っている点である。

特に有効な指摘は以下である。

| IBM記事の論点 | 実務上の意味 |
|---|---|
| コスト削減から始める | ROIを測定しやすく、経営承認を得やすい |
| 明確なユースケースを選ぶ | PoC乱立を避けられる |
| ベースラインを取る | 導入効果を説明できる |
| アーキテクチャを考える | ベンダーロックインと個別最適を避けられる |
| ROIを速度・コスト・新機能で見る | 人件費削減以外の効果を評価できる |

IBM記事は、AI対応に向いたシグナルとして、明確に定義された反復的・単純なタスク、手動の引き継ぎ、高品質なデータ、複雑なポリシー解釈、高コスト・非効率性を挙げている。出典: [IBM「ビジネス・リーダーがAIエージェントでROIを実現する方法」](https://www.ibm.com/jp-ja/think/insights/realize-roi-ai-agents)

### 2.3 注意点

一方で、以下の点は割り引いて読むべきである。

1. IBMの営業文脈が含まれている。  
   記事は最終的にwatsonx Orchestrateなどへの導線を持つため、「オーケストレーション基盤が重要」という主張と「IBMが最適」という主張は分けて評価する必要がある。

2. エージェント化すべきでない業務までエージェント化するリスクがある。  
   ルールベース自動化、RPA、通常のLLMプロンプト、検索拡張生成で十分な業務も多い。

3. 初期段階では「自律性」より「制御性」を優先すべきである。  
   とくに顧客送信、金額変更、契約条件変更、支払・返金・与信、個人評価などは、最初からAIに完全委任すべきではない。

### 2.4 外部環境から見た補足

PwCの2025年AI Agent Surveyでは、AIエージェントを採用している企業の66%が生産性向上、57%がコスト削減、55%が意思決定の高速化、54%が顧客体験改善を報告している。一方で、初期価値は内部ユースケースから出やすいともされている。出典: [PwC AI Agent Survey](https://www.pwc.com/us/en/tech-effect/ai-analytics/ai-agent-survey.html)

Gartnerは、2027年末までにエージェント型AIプロジェクトの40%以上が、コスト増、不明確な事業価値、不十分なリスク管理を理由に中止されると予測している。また、従来のAIアシスタントやRPA、チャットボットをエージェントと呼び替える「agent washing」にも警鐘を鳴らしている。出典: [Gartner Press Release, 2025-06-25](https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027)

McKinseyの2025年調査では、AI活用は広がっているものの、企業レベルでEBITインパクトを報告している企業は39%にとどまり、AI高業績企業はワークフロー再設計に取り組んでいるとされている。出典: [McKinsey, The State of AI: Global Survey 2025](https://www.mckinsey.com/capabilities/quantumblack/our-insights/the-state-of-ai)

McKinseyは別記事で、複雑なマルチステップ業務では、エージェントそのものではなくワークフローに焦点を当て、ルールベース、分析AI、生成AI、エージェントを適切に組み合わせるべきだと述べている。出典: [McKinsey, The six key elements of agentic AI deployment](https://www.mckinsey.com/capabilities/quantumblack/our-insights/one-year-of-agentic-ai-six-lessons-from-the-people-doing-the-work)

## 3. もっとも取り組むべきユースケース

### 3.1 最優先ユースケース

最初に取り組むべきユースケースは、**問い合わせ・申請・照会業務を、AIエージェントで調査・判断補助・起案・システム登録までつなぐ業務**である。

具体領域は以下である。

| 領域 | 例 |
|---|---|
| 社内問い合わせ | 人事、経理、法務、情シス、総務 |
| 顧客接点 | カスタマーサポート、保険金請求、契約照会、注文・請求・返品対応 |
| 営業支援 | 顧客情報調査、提案書下書き、CRM更新、次アクション整理 |
| 管理部門 | 稟議チェック、契約条項確認、請求書処理、支払条件照合 |

### 3.2 その中で最初に選ぶべきもの

最初の推奨は、**社内問い合わせ・申請処理エージェント**である。

理由は以下である。

| 判断軸 | 評価 |
|---|---|
| 頻度 | 毎日発生しやすい |
| 工数 | 人間が調査・転記・確認に時間を使っている |
| ルール化 | FAQ、規程、契約、マニュアル、過去履歴に基づきやすい |
| 測定可能性 | 処理時間、一次解決率、エスカレーション率、ミス率を測れる |
| リスク制御 | 最終承認を人間に残しやすい |
| 横展開性 | 経理、人事、IT、総務、法務、購買へ展開しやすい |

### 3.3 最初の候補順位

| 優先度 | ユースケース | 理由 |
|---:|---|---|
| 1 | 経費精算問い合わせ | 規程があり、頻度が高く、リスク制御しやすい |
| 2 | ITヘルプデスク一次対応 | FAQ化しやすく、処理時間削減が見えやすい |
| 3 | 勤怠・休暇問い合わせ | 規程ベースで回答しやすい |
| 4 | 購買申請問い合わせ | ワークフロー接続まで発展しやすい |
| 5 | 契約レビュー受付 | 効果は大きいが初期難度が高い |

### 3.4 避けるべき初期ユースケース

以下は、初期PoCには不向きである。

| ユースケース | 初期に不向きな理由 |
|---|---|
| 完全自律の営業代行 | 成果責任・ブランド毀損リスクが高い |
| 採用候補者評価 | バイアス・説明責任・法務リスクが高い |
| 懲戒・人事評価 | 人的影響が大きく、AI単独判断に不向き |
| 与信判断 | 金融・法務・説明責任リスクが高い |
| 契約条件の自動変更 | 法務リスクが高い |
| 支払・返金・金額変更の自動実行 | 財務インパクトが直接発生する |
| 顧客への完全自動返信 | 誤回答時の顧客毀損が大きい |

## 4. 具体ケース: 経費精算問い合わせAIエージェント

### 4.1 対象業務

社員から来る、以下のような問い合わせを対象にする。

```text
この領収書は経費になりますか？
接待交際費で処理してよいですか？
タクシー代は申請できますか？
出張精算の締切はいつですか？
領収書をなくした場合はどうすればいいですか？
この勘定科目は何ですか？
承認者は誰にすればいいですか？
```

### 4.2 なぜ経費精算問い合わせが最初に向いているか

| 理由 | 内容 |
|---|---|
| 件数が多い | 多くの会社で毎月発生する |
| ルールがある | 経費規程、出張規程、FAQに基づける |
| 回答案を作りやすい | 判断根拠を文書から引ける |
| リスクを制御しやすい | 最終申請・承認は人間に残せる |
| ROIを測りやすい | 処理時間、問い合わせ件数、差し戻し率を測れる |
| 横展開しやすい | 購買、勤怠、ITヘルプデスクへ転用できる |

### 4.3 初期ゴール

初期ゴールは、以下に限定する。

```text
経理担当者が確認して送信できる回答案をAIが作る。
回答案には、分類、回答文、根拠、不足情報、人間確認が必要な理由を含める。
2週間で平均処理時間30%以上削減を検証する。
```

初期段階では、AIが社員に直接回答しない。  
社員への送信、支払可否の確定、例外承認、金額判断は人間に残す。

## 5. 明日から始める実行手順

## Day 1: 対象を1つに絞る

明日はAIシステムを作らない。  
最初にやるべきことは、対象業務の棚卸しである。

経理・総務・情シス・人事の中から、以下の条件に合う問い合わせを1つ選ぶ。

### 5.1 ユースケース選定基準

| 条件 | 判断 |
|---|---|
| 月50件以上ある | できれば対象 |
| 1件あたり5分以上かかる | できれば対象 |
| 回答に参照する文書がある | 必須 |
| 誤回答しても人間が止められる | 必須 |
| 回答パターンがある程度繰り返される | 必須 |
| 個別判断・法的判断・高額承認が少ない | 望ましい |

### 5.2 午前にやること

```text
1. 経理・情シス・人事・総務の問い合わせ件数をざっくり確認する
2. 月50件以上ある業務を抽出する
3. 規程・FAQがある業務を優先する
4. 失敗時リスクが低い業務を選ぶ
5. 最初の1業務を決める
```

おすすめ順は以下である。

```text
第1候補：経費精算問い合わせ
第2候補：ITヘルプデスク一次対応
第3候補：勤怠・休暇問い合わせ
```

### 5.3 午後にやること

```text
1. 過去問い合わせを30件集める
2. 問い合わせ内容と実際の回答を表にする
3. 分類ラベルを10個以内で付ける
4. 参照文書を3つだけ用意する
5. 10件だけAIに回答案を作らせる
6. 担当者が採点する
```

## 7. 過去問い合わせ30件の整理方法

Slack、メール、Teams、チケットシステムから、直近の問い合わせを30件集める。  
全部を見る必要はない。まずは30件で十分である。

最低限、以下の表に整理する。

| No | 問い合わせ内容 | 担当者の回答 | 参照した規程・FAQ | 処理時間 | 分類 |
|---:|---|---|---|---:|---|
| 1 | タクシー代は経費になりますか？ | 深夜・緊急時など条件付きで可 | 経費規程 第X条 | 7分 | 交通費 |
| 2 | 領収書をなくしました | 紛失届と承認が必要 | 経費FAQ | 5分 | 証憑 |
| 3 | 接待費の上限はいくらですか？ | 役職・相手先により異なる | 交際費規程 | 10分 | 交際費 |

この30件が、最初の評価データになる。

## 9. 最初に用意する参照文書

最初から全社文書を入れない。  
まずは以下の3つでよい。

```text
1. 経費精算規程
2. 出張旅費規程
3. よくある質問と回答
```

文書がPDFでもWordでもよいが、最初はMarkdownまたはテキスト化した方が扱いやすい。

### 9.1 FAQ文書の例

```markdown
# 経費精算FAQ

## Q. 領収書を紛失した場合はどうするか？
A. 原則として領収書の提出が必要。ただし、紛失した場合は、支払内容、日付、金額、利用目的を記載した申請メモを添付し、上長承認を得る。

## Q. タクシー代は経費精算できるか？
A. 原則として公共交通機関を利用する。ただし、深夜、緊急対応、荷物運搬、顧客同行など合理的理由がある場合は、理由を記載して申請できる。
```

## 10. AIへの指示文テンプレート

明日から使うなら、最初はこのプロンプトで十分である。

```markdown
あなたは社内経費精算問い合わせの回答案を作成するアシスタントです。

# 目的
社員からの問い合わせに対して、経費規程・FAQ・過去回答に基づき、経理担当者が確認して送信できる回答案を作成してください。

# 禁止事項
- 規程にないことを断定しない
- 税務・法務判断を最終判断として書かない
- 支払可否を確定しない
- 金額承認を代行しない
- 根拠がない場合は「確認が必要」と明記する

# 出力形式
## 分類
問い合わせ分類を1つ選ぶ。

## 回答案
社員に送る文面の下書きを作る。

## 根拠
参照した規程・FAQ・過去回答を箇条書きで示す。

## 不足情報
回答に必要だが不足している情報を示す。

## 人間確認が必要な理由
人間確認が必要な場合、その理由を示す。

# 問い合わせ
{社員からの問い合わせ}

# 参照情報
{経費規程・FAQ・過去回答}
```

## 11. AI回答案の出力例

### 11.1 入力例

```text
昨日、終電を逃したのでタクシーで帰りました。これは経費で申請できますか？
```

### 11.2 AIの理想出力

```markdown
## 分類
交通費

## 回答案
終電後のタクシー利用については、業務上やむを得ない理由がある場合、経費申請の対象となる可能性があります。

申請時には、以下を記載してください。

- 利用日
- 利用区間
- 金額
- 終電後の利用となった理由
- 業務上必要だった理由
- 領収書

ただし、最終的な承認可否は、申請内容と上長承認に基づき判断されます。

## 根拠
- 経費精算規程：交通費は業務遂行上必要な支出を対象とする
- 経費FAQ：深夜・緊急対応など合理的理由がある場合はタクシー利用理由を記載する

## 不足情報
- 終電を逃した理由
- 業務終了時刻
- 利用区間
- 金額
- 領収書の有無

## 人間確認が必要な理由
タクシー利用が業務上やむを得ないものだったかは、個別事情により判断が必要です。
```

## 12. 採点表

AI回答案は、以下で採点する。

| No | 問い合わせ | 分類正確性 | 回答案品質 | 根拠正確性 | 不足情報の指摘 | そのまま使えるか |
|---:|---|---:|---:|---:|---:|---|
| 1 | タクシー代 | 5 | 4 | 4 | 5 | 一部修正 |
| 2 | 領収書紛失 | 5 | 5 | 5 | 4 | そのまま可 |
| 3 | 接待費 | 4 | 3 | 4 | 5 | 要修正 |

採点基準は5点満点とする。

```text
5：そのまま使える
4：軽微な修正で使える
3：骨子は使える
2：大幅修正が必要
1：使えない
```

## 16. エージェント設計の具体プラクティス

### 16.1 業務を処理単位に分解する

悪い切り方は以下である。

```text
経理問い合わせをAI化する
```

良い切り方は以下である。

| 処理単位 | AI化方針 |
|---|---|
| 問い合わせ内容の分類 | 自動化 |
| 関連規程・過去回答の検索 | 自動化 |
| 必要情報の不足確認 | 自動化 |
| 回答案の作成 | 自動化 |
| 例外判断 | 人間確認 |
| 会計システム登録 | 承認後に半自動化 |
| 最終送信 | 初期は人間承認 |

### 16.2 エージェントは状態遷移で設計する

AIエージェントを「自由に考えるAI」として設計しない。  
最初は、明確な状態遷移で設計する。

```text
Intake Agent
問い合わせを分類し、必要情報を抽出する

Retrieval Agent
社内規程、FAQ、過去対応、契約、マニュアルを検索する

Policy Agent
適用されるルール・例外条件を整理する

Draft Agent
回答案、処理案、次アクションを作る

Action Agent
チケット更新、CRM登録、ワークフロー申請などを実行する

Supervisor
信頼度、リスク、例外条件を判定し、人間確認に回す
```

ポイントは、**AIエージェントを1体の万能担当者にしないこと**である。  
役割を分け、各ステップでログを残し、どこで誤ったか追跡できる構造にする。

### 16.3 初期の人間承認境界

| 処理 | 初期方針 |
|---|---|
| FAQ検索 | 自動 |
| 関連文書検索 | 自動 |
| 要約 | 自動 |
| 回答案作成 | 自動 |
| 社内チケット更新 | 半自動 |
| 顧客・社員への送信 | 人間承認 |
| 金額変更 | 人間承認 |
| 契約条件変更 | 人間承認 |
| 個人評価・採用・懲戒 | 原則AI単独不可 |
| 支払・返金・与信 | 原則AI単独不可 |

## 18. 90日での進め方

### 18.1 Phase 1: 2週間 — 対象業務の選定

やることは以下である。

- 月間件数トップ20の問い合わせ・申請を洗い出す
- 現在の処理時間を測る
- 参照している文書・システムを特定する
- 例外パターンを分類する
- 失敗時リスクを評価する

成果物は以下である。

```text
ユースケース候補一覧
業務分解表
ROI仮説
リスク分類
初期KPI
```

### 18.2 Phase 2: 4週間 — 人間承認付きプロトタイプ

対象は1業務に絞る。

例は以下である。

- 経費精算問い合わせ
- 契約書レビュー受付
- ITヘルプデスク一次対応
- 請求書不備確認
- 顧客サポートの返品・交換案内

この段階でAIが担当するのは以下だけである。

```text
分類
検索
要約
回答案作成
根拠提示
チケット下書き
```

送信、承認、金額変更、契約判断は人間に残す。

### 18.3 Phase 3: 4週間 — 現場運用テスト

見るべき指標は以下である。

| 指標 | 合格ラインの目安 |
|---|---:|
| 回答案利用率 | 50%以上 |
| 平均処理時間削減 | 30%以上 |
| 明確な誤回答率 | 3%未満 |
| 人間承認後の送信率 | 70%以上 |
| 現場継続利用率 | 60%以上 |

ここで重要なのは、モデル精度だけを見ないことである。  
現場が実際に使い、処理時間が減り、品質が落ちないかを見る。

### 18.4 Phase 4: 4週間 — システム登録・ワークフロー接続

最後に、以下へ広げる。

```text
チケットシステム更新
CRM更新
申請ワークフロー起票
Slack / Teams通知
メール下書き
FAQ更新提案
管理ダッシュボード出力
```

ここで初めて「エージェントらしい」価値が出る。  
単なるチャットボットではなく、**業務を前に進める実行系**になる。

## 20. 担当者への依頼文

明日、担当者に依頼するものはこれだけでよい。

```text

3. 経費規程・FAQ・マニュアル
4. 1件あたりの処理時間の概算
5. よくある例外パターン
```

依頼文は以下である。

```text
AIエージェント化の初期検証のため、直近の経費精算問い合わせを30件ほど確認したいです。

- 問い合わせ本文
- 実際の回答
- 参照した規程・FAQ
- だいたいの処理時間
- 判断に迷ったケース

目的は自動回答ではなく、担当者が確認して使える回答案をAIで作れるかを検証することです。
```

## 22. 最終判断

明日から始めるなら、次の手順に固定する。

```text
1. 経費精算問い合わせを第一候補にする
2. 過去30件の問い合わせと回答を集める
3. 経費規程・FAQ・過去回答を3点だけ用意する
4. AIに回答案、根拠、不足情報を出させる
5. 経理担当者が5点満点で採点する
6. 2週間で処理時間30%削減を目標にする
```

最初から「自律型エージェント」を作らない。  
**担当者の回答案作成を30%以上速くすることだけを狙う**のが、最も成功確率が高い。

## 26. 付録D: 初期プロンプト完全版

```markdown
あなたは社内経費精算問い合わせの回答案を作成するアシスタントです。

# 目的
社員からの問い合わせに対して、経費規程・FAQ・過去回答に基づき、経理担当者が確認して送信できる回答案を作成してください。

# 前提
- あなたの回答は最終判断ではありません。
- 最終判断は経理担当者または承認者が行います。
- 規程・FAQ・過去回答に根拠がある範囲で回答してください。
- 根拠が不足している場合は、断定せず、確認が必要と明記してください。

# 禁止事項
- 規程にないことを断定しない
- 税務・法務判断を最終判断として書かない
- 支払可否を確定しない
- 金額承認を代行しない
- 例外承認を代行しない
- 根拠がない場合に推測で回答しない
- 社員に直接送る前提で断定的に書きすぎない

# 出力形式
## 分類
問い合わせ分類を1つ選ぶ。

## 回答案
社員に送る文面の下書きを作る。

## 根拠
参照した規程・FAQ・過去回答を箇条書きで示す。

## 不足情報
回答に必要だが不足している情報を示す。

## 人間確認が必要な理由
人間確認が必要な場合、その理由を示す。

## リスク判定
以下から1つ選ぶ。
- 低：FAQ・規程に明確な回答がある
- 中：個別事情の確認が必要
- 高：税務・法務・例外承認・高額支出に関係する

# 問い合わせ
{社員からの問い合わせ}

# 参照情報
{経費規程・FAQ・過去回答}
```
