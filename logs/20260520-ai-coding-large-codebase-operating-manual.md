# 大規模コードベースでAIコーディング支援を再現可能にする手順書

作成日: 2026-05-20  
対象: Claude Codeを含む、コードベースを読みながら修正するAIコーディング支援ツール  
目的: 「AIが賢いかどうか」ではなく、「AIが迷わず、安全に、同じ手順で作業できる状態」を作る

---

## 0. この手順書で作るもの

この手順書では、1つのリポジトリに対して、以下を順番に作る。

```text
repository-root/
├── CLAUDE.md                         # AIが毎回読む、短い案内書
├── CODEBASE_MAP.md                   # 人間にもAIにも分かるコードベース地図
├── .claude/
│   ├── settings.json                 # チームで共有するAI利用設定
│   ├── rules/
│   │   ├── testing.md                # テスト方針
│   │   ├── security.md               # セキュリティ注意点
│   │   └── api-design.md             # API設計の局所ルール
│   └── skills/
│       ├── review-diff/
│       │   └── SKILL.md              # 差分レビューの手順
│       ├── update-docs/
│       │   └── SKILL.md              # ドキュメント更新の手順
│       └── investigate-bug/
│           └── SKILL.md              # バグ調査の手順
└── service-a/
    └── CLAUDE.md                     # service-aだけの作業ルール
```

この構成はClaude Code用の名前を使っているが、考え方は他のAIコーディング支援にも転用できる。  
転用する場合は、以下のように読み替える。

| この手順書の名前 | 一般化した意味 |
|---|---|
| `CLAUDE.md` | AIが毎回読む、短いプロジェクト案内 |
| `.claude/rules/` | 特定の場所やファイル種別にだけ効かせる作業ルール |
| `.claude/skills/` | 必要なときだけ呼び出す、繰り返し使う作業手順 |
| hooks | AIに任せず、機械的に必ず実行する処理 |
| LSP / code intelligence | 関数・型・参照関係を正確にたどる仕組み |
| MCP / external connector | 社内ツール、チケット、ログ、監視、仕様書に接続する仕組み |
| plugin | チームや複数リポジトリに配るためのひとまとまり |

## 1. 最初に決めること

### 1.1 成功条件

以下を満たしたら、最初の導入は成功とする。

- 新しく入った開発者が、AIに「このリポジトリの構造を説明して」と聞いたとき、主要ディレクトリと責務を正しく説明できる
- AIが修正対象のサブディレクトリから起動される
- AIが修正後に、該当範囲のテストまたは静的チェックを実行できる
- AIが生成ファイル、依存パッケージ、秘密情報ファイルを読みに行かない
- AIに毎回同じ注意を口頭で説明しなくてよい
- AIの失敗が起きたとき、次回から防げる形でルールまたは手順に残せる

### 1.2 最初の対象リポジトリの選び方

最初から最大・最難関のリポジトリを選ばない。次の条件を満たすものを選ぶ。

#### 選ぶべきリポジトリ

- 直近30日以内に開発がある
- テストまたは静的チェックが1つ以上存在する
- 主要な責任者が1人以上いる
- 変更範囲を1つのディレクトリ、1つのサービス、1つのパッケージに絞れる
- 秘密情報や顧客データがリポジトリ内に直接置かれていない

#### 最初は避けるリポジトリ

- テストが存在しない
- ビルド手順を誰も説明できない
- 生成ファイルが大半を占める
- バイナリファイルが多い
- 本番障害に直結するが、検証環境が弱い
- 所有者が不明
- 変更時のレビュー体制がない

### 1.3 作業者を決める

最低限、以下の役割を置く。

| 役割 | 人数 | 責任 |
|---|---:|---|
| AI設定責任者 | 1 | `CLAUDE.md`、ルール、設定、導入判断を最終決定する |
| リポジトリ責任者 | 1 | 構造、テスト、ビルド、危険箇所を説明する |
| セキュリティ確認者 | 1 | 読ませてはいけないファイル、実行させてはいけないコマンドを決める |
| 利用者代表 | 1〜3 | 実際にAIを使い、失敗例と改善点を出す |

小規模なら1人が複数役を兼ねてもよい。  
ただし、AI設定責任者だけは必ず明示する。

## 2. 全体手順

作業は以下の順で行う。

```text
1. 対象範囲を選ぶ
2. リポジトリを棚卸しする
3. AI向け地図を作る
4. 最小のCLAUDE.mdを作る
5. サブディレクトリ別のルールを作る
6. 読ませないもの・触らせないものを設定する
7. テスト・チェックの実行単位を小さくする
8. 関数や型を正確にたどれる状態にする
9. 機械的に必ず行う処理を自動化する
10. 繰り返し作業を手順化する
11. 1週間試す
12. 失敗例を修正し、再度試す
13. 他リポジトリへ展開する
```

この順番を入れ替えない。  
理由は、後ろの仕組みほど前提条件に依存するからである。

## 3. 判断の型

書く場所に迷ったら、この順に判定する。

### 3.1 情報の置き場所を決める判定表

| 質問 | Yesの場合 | Noの場合 |
|---|---|---|
| すべての作業でAIに知っていてほしいか | ルートの`CLAUDE.md`へ | 次の質問へ |
| 特定ディレクトリだけで必要か | そのディレクトリの`CLAUDE.md`またはpath ruleへ | 次の質問へ |
| 特定のファイル種別だけで必要か | `.claude/rules/`のpath ruleへ | 次の質問へ |
| 複数ステップの作業手順か | `.claude/skills/`へ | 次の質問へ |
| AIの判断に任せず必ず実行したいか | hookまたはCIへ | 次の質問へ |
| チーム全員に配りたいか | pluginまたは共有テンプレートへ | 次の質問へ |
| 自分だけの設定か | `CLAUDE.local.md`または個人設定へ | 書かない |

### 3.2 `CLAUDE.md`に書いてよいもの

書いてよいものは、以下だけ。

- リポジトリの目的
- 主要ディレクトリの責務
- よく使うテスト・ビルド・lintコマンド
- 絶対に守る安全ルール
- よくある落とし穴
- 変更前後に確認すべき短い手順
- AIが毎回迷う場所への案内

### 3.3 `CLAUDE.md`に書いてはいけないもの

以下は原則として書かない。

- 長い設計史
- 1回しか使わない作業メモ
- 特定タスク専用の長い手順
- 個人の好み
- 古い移行手順
- すでにコードや設定ファイルから明らかな情報
- 「ちゃんとやる」「いい感じにする」など検証不能な指示

### 3.4 曖昧な指示の変換規則

| 悪い書き方 | 良い書き方 |
|---|---|
| ちゃんとテストする | `npm test -- --runInBand`を実行し、失敗した場合は最初の失敗テスト名と原因仮説を書く |
| コードをきれいにする | 既存ファイルの命名、インデント、エラー処理の形に合わせる |
| 影響範囲を確認する | 変更した関数の呼び出し元を検索し、直接の呼び出し元を一覧化する |
| セキュリティに注意する | `.env*`、`secrets/**`、`credentials/**`を読まない。ログにトークン、メール、顧客IDを出さない |
| 小さく変更する | 1回のPRでは、1つの目的に関係するファイルだけを変更する |

## 4. Phase 1: リポジトリを棚卸しする

### 4.1 実行するコマンド

リポジトリのルートで以下を実行する。

```bash
pwd
git rev-parse --show-toplevel
git status --short
git branch --show-current
find . -maxdepth 2 -type d | sort | sed 's#^\./##' | head -200
find . -maxdepth 3 -name "package.json" -o -name "pyproject.toml" -o -name "go.mod" -o -name "pom.xml" -o -name "build.gradle" -o -name "Cargo.toml" -o -name "Makefile" | sort
find . -maxdepth 3 -name ".env*" -o -path "*/secrets/*" -o -path "*/credentials/*" | sort
```

`cloc`が使える場合は以下も実行する。

```bash
cloc . --exclude-dir=node_modules,vendor,dist,build,.git
```

`tree`が使える場合は以下も実行する。

```bash
tree -L 3 -a -I "node_modules|vendor|dist|build|.git"
```

### 4.2 棚卸し表を作る

`CODEBASE_MAP.md`を作り、以下の表を埋める。

```md
# CODEBASE_MAP

## このリポジトリの目的

- 

## 主要ディレクトリ

| パス | 役割 | 主な言語 | 触る人 | テストコマンド | 注意点 |
|---|---|---|---|---|---|
| `src/` |  |  |  |  |  |
| `tests/` |  |  |  |  |  |

## 生成物・読まないもの

| パス | 理由 |
|---|---|
| `dist/` | ビルド生成物 |
| `vendor/` | 外部依存 |

## 秘密情報・触らないもの

| パス | 理由 |
|---|---|
| `.env` | 秘密情報 |
| `secrets/` | 秘密情報 |

## よく使うコマンド

| 目的 | コマンド | 実行場所 | 目安時間 |
|---|---|---|---|
| テスト |  |  |  |
| lint |  |  |  |
| format |  |  |  |
| build |  |  |  |
```

### 4.3 完了条件

以下を満たしたらPhase 1完了。

- 主要ディレクトリの役割が1行で説明されている
- 生成物と外部依存の場所が分かる
- 秘密情報の可能性がある場所が分かる
- テスト・lint・buildの入口が分かる
- 「AIをどこから起動するべきか」が分かる

## 5. Phase 2: AI向け地図を作る

AIは最初に「どこを見ればよいか」を知らない。  
そのため、地図は短く、目次として使える形にする。

### 5.1 地図の書き方

`CODEBASE_MAP.md`の「主要ディレクトリ」を完成させる。  
各パスは1行で書く。

悪い例:

```md
`src/`: いろいろなアプリケーションコードが入っています。古いものも新しいものもあり、ケースによって違います。
```

良い例:

```md
`src/api/`: HTTP APIの入口。各ハンドラは`src/services/`を呼び出す。DBへ直接アクセスしない。
```

### 5.2 ディレクトリの分類

各ディレクトリに、以下のどれかを付ける。

| 分類 | 意味 |
|---|---|
| 入口 | API、CLI、画面、ジョブなど外部から呼ばれる場所 |
| 中核 | 主要な業務ロジック |
| 接続 | DB、外部API、キュー、認証、決済など |
| 表示 | UI、テンプレート、画面部品 |
| テスト | テストコード |
| 生成 | 自動生成物 |
| 外部 | vendor、third_party、node_modulesなど |
| 設定 | build、CI、lint、deploy設定 |
| 危険 | 秘密情報、移行スクリプト、本番操作に近いもの |

### 5.3 地図からAI用の起動場所を決める

作業を依頼するときは、原則としてリポジトリルートではなく、対象ディレクトリで起動する。

例:

| 作業 | 起動場所 |
|---|---|
| フロントエンドのボタン修正 | `apps/web/` |
| 決済APIの修正 | `services/payments/` |
| 共通ライブラリの型修正 | `packages/shared/` |
| リポジトリ全体の依存更新 | ルート |

判断に迷ったら、次の基準で決める。

```text
変更対象ファイルの80%以上が1つのディレクトリ配下にある
→ そのディレクトリで起動する

変更対象が複数サービスにまたがる
→ まず調査だけをルートで行い、実装はサービスごとに分ける

影響範囲が読めない
→ ルートで調査し、CODEBASE_MAPを更新してから実装する
```

## 6. Phase 3: ルートのCLAUDE.mdを作る

### 6.1 基本方針

ルートの`CLAUDE.md`は短くする。  
目安は200行未満。  
目的は「AIが最初の5分で迷わないこと」であり、「社内Wikiを全部詰め込むこと」ではない。

### 6.2 ルート用テンプレート

以下をそのまま使い、空欄を埋める。

```md
# CLAUDE.md

## このリポジトリの目的

このリポジトリは、<何をするシステムか>のコードを管理する。

## 最初に読むもの

- `CODEBASE_MAP.md`: 主要ディレクトリと責務
- `<必要ならdocs/...>`: 設計の入口
- `<必要ならREADME.md>`: 開発環境の入口

## 作業開始前の確認

- `git status --short`で未コミット変更を確認する
- 変更対象が1つのサブディレクトリに絞れる場合は、そのディレクトリで作業する
- 生成物、外部依存、秘密情報ファイルは読まない

## 主要ディレクトリ

| パス | 役割 |
|---|---|
| `src/` |  |
| `tests/` |  |
| `docs/` |  |

## よく使うコマンド

| 目的 | コマンド | 実行場所 |
|---|---|---|
| テスト | `<command>` | `<path>` |
| lint | `<command>` | `<path>` |
| format | `<command>` | `<path>` |
| build | `<command>` | `<path>` |

## 変更時の原則

- 1回の変更は1つの目的に限定する
- 既存の命名、エラー処理、ディレクトリ構造に合わせる
- 変更したファイルに対応するテストまたはチェックを実行する
- テストを実行できない場合は、理由と代替確認方法を書く
- 本番データ、秘密情報、認証情報を読まない

## 読まない・触らない場所

- `node_modules/`
- `vendor/`
- `dist/`
- `build/`
- `.env`
- `.env.*`
- `secrets/`
- `credentials/`

## よくある落とし穴

- `<例: APIハンドラからDBへ直接アクセスしない>`
- `<例: generated配下は手編集しない>`
- `<例: migrationは必ず既存形式に合わせる>`
```

### 6.3 書いた後の検査

`CLAUDE.md`を作ったら、以下を確認する。

```text
□ 200行未満である
□ 1文が長すぎない
□ 「ちゃんと」「適切に」「必要に応じて」など曖昧語が少ない
□ コマンドに実行場所が書いてある
□ 触らないファイルが書いてある
□ 新人エンジニアが読んでも意味が分かる
□ 古い事情や議論ではなく、今の作業に必要な情報だけがある
```

### 6.4 AIに自己点検させる依頼文

Claude Codeに以下を依頼する。

```text
CLAUDE.mdを点検してください。
次の観点で、削るべき行、曖昧な行、別ファイルに移すべき行を一覧にしてください。

1. 毎回読む必要があるか
2. 具体的で検証可能か
3. 特定ディレクトリだけの話ではないか
4. 複数ステップの作業手順になっていないか
5. 古くなっていないか

修正案は、差分形式ではなく、表で出してください。
```

## 7. Phase 4: サブディレクトリ別のルールを作る

### 7.1 作るべき場所

以下の条件に当てはまるディレクトリには、サブディレクトリ用の`CLAUDE.md`または`.claude/rules/`を作る。

- テストコマンドがルートと違う
- 設計ルールが違う
- 触ってはいけないファイルがある
- 特定のチームだけが管理している
- AIが過去に同じ勘違いをした
- 変更時の確認手順が違う

### 7.2 サブディレクトリ用CLAUDE.mdテンプレート

例: `services/payments/CLAUDE.md`

```md
# services/payments/CLAUDE.md

## このディレクトリの役割

決済関連API、決済状態管理、外部決済サービスとの接続を扱う。

## このディレクトリで守ること

- 金額は整数の最小通貨単位で扱う
- 外部決済APIへのリトライは既存の`PaymentRetryPolicy`を使う
- APIハンドラから外部決済APIを直接呼ばない
- ログにカード情報、トークン、メールアドレスを出さない

## よく使うコマンド

| 目的 | コマンド |
|---|---|
| 単体テスト | `npm test -- services/payments` |
| lint | `npm run lint -- services/payments` |
| 型チェック | `npm run typecheck -- services/payments` |

## 変更後に確認すること

- 変更した関数の直接の呼び出し元を確認する
- 既存のテストがある場合は該当テストを実行する
- テストがない場合は、追加すべきテスト名を提案する
```

### 7.3 ファイル種別で分けるルール

例: TypeScript APIにだけ適用したい場合。

`.claude/rules/api-design.md`

```md
---
paths:
  - "src/api/**/*.ts"
  - "services/*/api/**/*.ts"
---

# API設計ルール

- 入力値はハンドラの入口で検証する
- エラー形式は`ApiErrorResponse`に合わせる
- APIハンドラからDBへ直接アクセスしない
- 業務処理は`services/`配下へ置く
- 新しいAPIを追加した場合は、対応するテストを追加する
```

### 7.4 サブディレクトリルールの増やしすぎを防ぐ

以下の条件を満たさないルールは作らない。

```text
□ そのルールがないとAIが間違える可能性が高い
□ そのルールを人間のレビューでも確認している
□ ルールが3か月後にも有効である
□ ルールを1文で検証できる
```

## 8. Phase 5: 読ませないもの・触らせないものを設定する

### 8.1 まず分類する

ファイルを以下に分ける。

| 分類 | AIに読ませるか | 理由 |
|---|---|---|
| ソースコード | 読ませる | 作業対象 |
| テストコード | 読ませる | 仕様と確認に必要 |
| README・設計資料 | 読ませる | 文脈に必要 |
| 生成ファイル | 原則読ませない | ノイズが多い |
| 外部依存 | 原則読ませない | 自社コードではない |
| ビルド成果物 | 読ませない | 作業対象ではない |
| 秘密情報 | 絶対に読ませない | セキュリティ上危険 |
| 本番操作スクリプト | 原則触らせない | 事故リスクがある |

### 8.2 `.claude/settings.json`の最小例

`.claude/settings.json`を作る。

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./credentials/**)",
      "Read(./node_modules/**)",
      "Read(./vendor/**)",
      "Read(./dist/**)",
      "Read(./build/**)",
      "Bash(curl *)",
      "Bash(wget *)",
      "Bash(terraform apply*)",
      "Bash(kubectl delete*)",
      "Bash(kubectl apply*)"
    ]
  }
}
```

この設定は出発点であり、各社の開発環境に合わせて必ず見直す。

### 8.3 判断基準

| 判断 | 置く場所 |
|---|---|
| 絶対に禁止したい | `settings.json`の`permissions.deny` |
| AIに注意してほしい | `CLAUDE.md` |
| チーム全員に強制したい | 管理設定またはリポジトリの共有設定 |
| 自分だけ例外にしたい | ローカル設定 |

### 8.4 完了条件

```text
□ 秘密情報ファイルがdenyされている
□ 生成物がdenyまたは明確に除外されている
□ 外部依存がdenyまたは明確に除外されている
□ 危険コマンドがdenyされている
□ 例外が必要な開発者の扱いが決まっている
```

## 9. Phase 6: テスト・チェックの実行単位を小さくする

AIに「全テストを実行して」と毎回言うと、時間がかかり、出力も長くなり、途中で失敗しやすい。  
そのため、変更範囲ごとの小さい確認コマンドを作る。

### 9.1 コマンドを探す

以下を確認する。

```bash
find . -maxdepth 3 -name "package.json" -exec sh -c 'echo "### $1"; cat "$1" | grep -n "\"scripts\"" -A 30' _ {} \;
find . -maxdepth 3 -name "Makefile" -exec sh -c 'echo "### $1"; grep -n "^[a-zA-Z0-9_-].*:" "$1" | head -50' _ {} \;
find . -maxdepth 3 -name "pyproject.toml" -o -name "tox.ini" -o -name "pytest.ini"
find . -maxdepth 3 -name "go.mod" -exec dirname {} \;
```

### 9.2 コマンドを3段階に分ける

| 段階 | 目的 | 目安時間 | 例 |
|---|---|---:|---|
| 最小確認 | 変更箇所だけ確認 | 30秒〜2分 | 1テストファイル、1パッケージ |
| 通常確認 | サブディレクトリ単位 | 2〜8分 | 1サービス、1アプリ |
| 全体確認 | PR前またはCI | 8分以上可 | 全テスト、全build |

### 9.3 `CLAUDE.md`へ書く形式

```md
## 確認コマンド

| 範囲 | 目的 | コマンド | 実行場所 | 目安 |
|---|---|---|---|---|
| 変更ファイル単位 | 最小テスト | `npm test -- <test-file>` | 変更対象パッケージ | 2分以内 |
| service-a | 通常テスト | `npm test -- services/service-a` | リポジトリルート | 5分以内 |
| 全体 | CI相当 | `npm test` | リポジトリルート | 10分以上 |
```

### 9.4 コマンドが遅すぎる場合

次の順で絞る。

```text
1. 変更したファイルに対応するテストだけ実行する
2. 変更したディレクトリ配下のテストだけ実行する
3. 関連パッケージだけ実行する
4. lintまたは型チェックだけ先に実行する
5. 全体テストはCIに任せる
```

### 9.5 AIへの依頼文

```text
この変更に対して、最小確認・通常確認・全体確認の3段階で実行すべきコマンドを提案してください。
各コマンドについて、実行場所、目安時間、確認できること、確認できないことを書いてください。
```

## 10. Phase 7: 関数・型・参照を正確にたどれる状態にする

### 10.1 目的

AIが文字列検索だけで探すと、同じ名前の関数やクラスに間違って飛ぶ。  
そのため、IDEと同じように「この関数の定義」「この型の参照元」「このエラーの場所」をたどれる状態にする。

### 10.2 言語ごとの準備例

| 言語 | 必要なものの例 |
|---|---|
| TypeScript | `typescript-language-server` |
| Python | `pyright-langserver` |
| Go | `gopls` |
| Rust | `rust-analyzer` |
| Java | `jdtls` |
| C/C++ | `clangd` |
| C# | `csharp-ls` |
| PHP | `intelephense` |
| Swift | `sourcekit-lsp` |

### 10.3 確認手順

1. 開発者のIDEで「定義へ移動」が効くか確認する
2. 1つの主要関数で「参照を検索」が効くか確認する
3. 意図的に型エラーを入れ、エラーが表示されるか確認する
4. Claude Code側で対応するcode intelligence pluginを有効化する
5. AIに「この関数の定義と呼び出し元をたどって」と依頼し、結果が妥当か確認する

### 10.4 完了条件

```text
□ 主要言語のlanguage serverが開発者環境に入っている
□ AIが定義・参照・型情報を使える
□ 文字列検索だけに依存していない
□ 同名関数が複数ある場合に、正しい方をたどれる
```

## 11. Phase 8: 機械的に必ず行う処理を自動化する

### 11.1 自動化すべきもの

AIに「忘れないで」と頼むのではなく、必ず実行する。

| やりたいこと | AIへの指示ではなく使うもの |
|---|---|
| 編集後にformatする | hook |
| 特定ファイルの編集を止める | permission deny |
| 危険コマンドを止める | permission deny |
| 作業終了時に差分要約を出す | hookまたはskill |
| AIが入力待ちになったら通知する | hook |
| 毎回実行する品質確認 | hookまたはCI |

### 11.2 例: 編集後にformatする

TypeScript/JavaScriptでPrettierを使っている場合の例。

`.claude/settings.json`

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}
```

### 11.3 例: 入力待ちを通知する

macOSの場合。

```json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude Code needs your attention\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
```

### 11.4 hookに入れてはいけないもの

以下は入れない。

- 10分以上かかる全体テスト
- ネットワークに依存する不安定な処理
- 本番環境を変更する処理
- 失敗時の原因が分かりにくい巨大スクリプト
- 人間の判断が必要なレビュー

### 11.5 完了条件

```text
□ AIに覚えさせるだけのルールが減っている
□ 自動化した処理は手元で単独実行できる
□ 失敗時の出力が短く、原因が分かる
□ 危険な処理は自動実行されない
```

## 12. Phase 9: 繰り返し作業を手順化する

### 12.1 skillにするべき作業

以下に当てはまるものは、`CLAUDE.md`ではなくskillにする。

- 毎回同じ依頼文を貼っている
- 手順が3ステップ以上ある
- 使う場面が限定される
- 長いチェックリストがある
- 作業の途中でコマンド結果を読む必要がある
- 人によってやり方がぶれる

### 12.2 skillにしないもの

- 全作業で毎回必要な基本方針
- 1行で済むルール
- まだ試していない思いつき
- 1回限りの作業
- ツールやコマンドだけで強制できるもの

## 取得する情報

!`git status --short`

!`git diff HEAD`

## 13. Phase 10: 外部情報への接続を後回しにする

### 13.1 原則

最初から社内ツール連携を作らない。  
まずはコードベース内で、AIが正しく読む・直す・確認する状態を作る。

### 13.2 接続してよいタイミング

以下を満たしたら、外部ツール接続を検討する。

```text
□ CLAUDE.mdが短く保たれている
□ 主要ディレクトリにルールがある
□ テスト・lintが小さく実行できる
□ 秘密情報と危険コマンドがdenyされている
□ 1週間以上、実務で使われている
□ 同じ情報を毎回AIに貼っている
```

| 接続先 | 使う理由 |

| Datadog / Grafana | メトリクス、ログ |
| Figma | UI仕様 |
| CI | ビルド、テスト結果 |

以下に当てはまる場合は接続しない。

- 接続先の情報が古い
- 誰が管理しているか分からない
- 秘密情報の取り扱いが不明
- 権限範囲を絞れない
- AIに渡す必要がある情報とない情報を分けられない

## 19. 内容を絞れない場合の絞り込み方

### 19.1 情報を4分類する

迷った情報は、以下の4つに分ける。

| 分類 | 意味 | 置き場所 |
|---|---|---|
| 地図 | どこに何があるか | `CODEBASE_MAP.md` |
| ルール | 守るべきこと | `CLAUDE.md`または`.claude/rules/` |
| 手順 | 何をどの順でやるか | `.claude/skills/` |
| 強制 | AIの判断に任せず止める/実行する | `settings.json`またはhook |

### 19.2 削る基準

以下のどれかに該当するものは削る。

```text
□ 3か月以内に使われる見込みがない
□ 新人に説明しても役に立たない
□ コードを読めば分かる
□ READMEや設計書に既にある
□ 今の実装と違う
□ 「場合による」だけで判断基準がない
□ 失敗を防がない
```

### 19.3 残す基準

以下のどれかに該当するものは残す。

```text
□ AIが過去に間違えた
□ 人間のレビューでも頻繁に指摘される
□ 事故時の影響が大きい
□ コードから読み取れない暗黙知である
□ 新人が必ず迷う
□ テスト・ビルド・起動に必要である
```

### 19.4 優先順位

情報が多い場合、以下の順で残す。

```text
1. 安全に関わること
2. 実行コマンド
3. 主要ディレクトリの責務
4. よくある落とし穴
5. テスト方針
6. 設計判断
7. 歴史的経緯
```

歴史的経緯は原則として残さない。  
残す場合は、「今の作業で何をしてはいけないか」に変換する。

## 20. 典型的な失敗と修正

| 失敗 | 原因 | 修正 |
|---|---|---|
| AIが関係ないファイルを大量に読む | 地図がない、起動場所が広すぎる | `CODEBASE_MAP.md`を作り、サブディレクトリで起動する |
| AIが古いルールに従う | `CLAUDE.md`が古い | 月次で削除・更新する |
| AIがformatを忘れる | 指示だけで運用している | hookで自動実行する |
| AIが全テストを走らせて止まる | 小さい確認コマンドがない | 最小・通常・全体の3段階に分ける |
| AIが同名関数を間違える | 文字列検索だけに頼っている | language serverを入れる |
| AIが秘密情報を読もうとする | deny設定がない | `permissions.deny`に追加する |
| `CLAUDE.md`が巨大化する | 手順や局所ルールを詰めている | skillまたはpath ruleへ移す |
| 各チームでやり方が違いすぎる | 所有者がいない | AI設定責任者を置く |
| 良い設定が広がらない | 個人環境に閉じている | 共有設定またはplugin化する |

## 21. 最小構成

時間がない場合は、まずこれだけ作る。

```text
□ CODEBASE_MAP.md
□ CLAUDE.md
□ .claude/settings.json
□ 対象サブディレクトリ/CLAUDE.md
□ 変更範囲ごとのテストコマンド表
```

最小構成の合格基準:

```text
□ AIがどこを読めばよいか分かる
□ AIがどこを読んではいけないか分かる
□ AIがどのコマンドで確認すればよいか分かる
□ 人間が同じ説明を毎回しなくてよい
```

## 22. 推奨する導入スケジュール

### Day 1

- 対象リポジトリを選ぶ
- 棚卸しする
- `CODEBASE_MAP.md`を作る
- 最小の`CLAUDE.md`を作る
- `settings.json`で危険箇所を止める

### Day 2

- サブディレクトリ用`CLAUDE.md`を1つ作る
- テスト・lintコマンドを3段階に分ける
- 実作業1件で試す

### Day 3

- 失敗を記録する
- ルールまたはskillへ反映する
- hooksを1つ入れる

### Day 4〜5

- 実作業を3〜5件試す
- 不要ルールを削る
- 使えるskillを1つ作る

### Week 2

- 他メンバーに使わせる
- 設定の差分をレビューする
- 2つ目のディレクトリへ展開する

### Month 1

- 成功・失敗を集計する
- 共有できる設定を固定する
- 他リポジトリ展開の可否を判断する

## 23. 導入判断の最終チェックリスト

```text
□ AI設定責任者がいる
□ リポジトリ責任者がいる
□ CODEBASE_MAP.mdがある
□ ルートCLAUDE.mdが200行未満である
□ サブディレクトリ用ルールが最低1つある
□ 秘密情報を読ませない設定がある
□ 生成物・外部依存を読ませない設定がある
□ 危険コマンドを止める設定がある
□ 最小確認コマンドがある
□ 通常確認コマンドがある
□ 全体確認はCIまたは明示的手順に分離されている
□ language serverまたは同等のコード参照機能がある
□ 繰り返し作業が最低1つskill化されている
□ 失敗記録の場所がある
□ 月次見直しの担当者が決まっている
```

## 24. この手順の本質

この手順は、特定のAIツールをうまく使うためだけのものではない。  
本質は、コードベースを次の状態にすることである。

```text
1. 地図がある
2. 局所ルールがある
3. 機械的な確認がある
4. 危険な行為が止まる
5. 失敗が次回の設定に反映される
6. 個人の工夫がチームの仕組みになる
```

AIが失敗する原因の多くは、AIが弱いことではなく、作業環境が人間の暗黙知に依存していることである。  
したがって、導入の中心は「AIに賢く振る舞わせること」ではなく、「AIが迷わない環境を作ること」である。
