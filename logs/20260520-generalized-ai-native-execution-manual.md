# AIを使った不確実な仕事の実行手順書

## 0. この文書の位置づけ

この文書は、Anthropicのブログ記事「The founder's playbook: Building an AI-native startup」を、起業だけでなく、新規事業、社内改善、個人サービス開発、制作活動にも使える実行手順に変換したものです。

元記事は、起業の流れを次の4段階で整理しています。

1. Idea：何を扱うかを決める
2. MVP：最小の試作品を作る
3. Launch：外に出す
4. Scale：繰り返し作業を仕組みにする

この文書では、上の4段階を専門用語ではなく、次の言葉に置き換えます。

| 元の言葉 | この文書で使う言葉 | 意味 |
|---|---|---|
| Idea | 何に取り組むか決める | 誰の、どの困りごとを扱うか決める |
| MVP | 最小の試作品を作る | すべて作らず、反応を見るための最小版を作る |
| Launch | 外に出す | 実際の相手に見せて、使うか、払うか、紹介するかを見る |
| Scale | 仕組みにする | 毎回人が頑張らなくても回る形に直す |

この文書の目的は、次の1点です。

> 読んだ人が同じ順番で作業すれば、似た品質の判断材料、試作品、公開手順、改善記録を作れるようにする。

---

## 1. まず徹底的に抽象化する

### 1.1 どんな仕事にも共通する形

新規事業、社内改善、個人プロジェクト、制作活動は、表面上は違って見えます。

しかし、実際にやっていることは共通しています。

> まだ分からないことを、確かめられる小さな作業に分ける。

たとえば、次の仕事はすべて同じ型で扱えます。

| 仕事 | 分からないこと | 確かめる小さな作業 |
|---|---|---|
| 新しいアプリを作る | そのアプリを本当に使う人がいるか | 5人に画面案を見せる |
| 社内業務を改善する | その業務が本当に面倒か | 担当者3人の作業時間を測る |
| 有料講座を作る | お金を払って学びたい人がいるか | 申込フォームを出して反応を見る |
| YouTube企画を作る | 視聴者が見たい内容か | タイトル案を10個作り、反応を見る |
| 営業資料を直す | 相手に伝わっていない箇所はどこか | 商談後に「分かりにくかった点」を聞く |

したがって、最初にやることは「良い案を出すこと」ではありません。

最初にやることは、次の7つに分けることです。

| 番号 | 分けるもの | 問い | 作るもの |
|---:|---|---|---|
| 1 | 相手 | 誰の話か | 対象者メモ |
| 2 | 場面 | いつ、どこで困るのか | 場面メモ |
| 3 | 困りごと | 何が面倒、不安、損、遅いのか | 困りごとメモ |
| 4 | 今のやり方 | 今はどうしているのか | 代替手段メモ |
| 5 | 小さい解決策 | 最小で何を見せればよいか | 試作品メモ |
| 6 | 反応 | 相手は実際に動いたか | 反応ログ |
| 7 | 判断 | 続けるか、変えるか、やめるか | 判断メモ |

この7つが空欄のまま進めると、AIを使っても失敗しやすくなります。

理由は単純です。

AIは文章、表、画面案、コード、手順を速く作れます。ですが、「誰が本当に困っているか」や「相手がお金や時間を使うか」は、実際の相手に確かめないと分かりません。

---

### 1.2 すべての作業を同じ形にそろえる

再現性を上げるために、すべての作業を次の形で書きます。

```md
## 作業名

### 目的
何のためにやるか。

### 入力
作業前に必要な情報。

### 手順
1.
2.
3.

### 出力
作業後に残すもの。

### 合格条件
どの状態なら次に進んでよいか。

### やり直し条件
どの状態なら戻るべきか。

### 記録先
どのファイルに残すか。
```

この形を使うと、次の利点があります。

- AIに依頼しやすい
- 他人に引き継ぎやすい
- 後で見返したときに判断理由が分かる
- 途中で迷ったとき、どこに戻ればよいか分かる

## 2. 作業開始前に作るフォルダ

最初に、1つのフォルダを作ります。

```text
project-name/
  00-readme.md
  01-decision-log.md
  02-candidate-list.md
  03-score-table.md
  04-target-person.md
  05-problem-notes.md
  06-interview-plan.md
  07-interview-log.md
  08-problem-summary.md
  09-solution-outline.md
  10-prototype-plan.md
  11-prototype-log.md
  12-test-log.md
  13-launch-plan.md
  14-operation-rules.md
  15-weekly-review.md
  16-ai-prompts.md
```

各ファイルの役割は次の通りです。

| ファイル | 役割 |
|---|---|
| `00-readme.md` | このプロジェクトの前提を書く |
| `01-decision-log.md` | いつ、何を、なぜ決めたかを書く |
| `02-candidate-list.md` | 扱う候補を全部書き出す |
| `03-score-table.md` | 候補を点数で比べる |
| `04-target-person.md` | 対象者を1種類に絞る |
| `05-problem-notes.md` | 困りごとの仮説を書く |
| `06-interview-plan.md` | 誰に何を聞くかを書く |
| `07-interview-log.md` | 聞いた話をそのまま残す |
| `08-problem-summary.md` | どの困りごとを扱うか決める |
| `09-solution-outline.md` | 解決策の1枚説明を書く |
| `10-prototype-plan.md` | 試作品の作り方を書く |
| `11-prototype-log.md` | 作ったものと変更履歴を書く |
| `12-test-log.md` | 相手に見せた反応を書く |
| `13-launch-plan.md` | 外に出す手順を書く |
| `14-operation-rules.md` | 繰り返し作業のルールを書く |
| `15-weekly-review.md` | 毎週の振り返りを書く |
| `16-ai-prompts.md` | 実際に使ったAIへの依頼文を保存する |

## 3. 最初に書く3つのファイル

### 3.1 `00-readme.md`

最初に次を書きます。

```md
# プロジェクト名

## 一言で言うと
-

## いま考えている対象者
-

## いま考えている困りごと
-

## なぜ今やるのか
-

## 使える時間
- 1日あたり：
- 1週間あたり：

## 使えるお金
-

## すぐ話を聞ける相手
-

## 使ってよい道具
-

## 使ってはいけない道具・情報
-

## 期限
-
```

#### 合格条件

次のすべてが書けていれば合格です。

- 対象者が1種類以上書かれている
- 困りごとが1つ以上書かれている
- 期限がある
- すぐ話を聞ける相手が1人以上いる、または「いない」と明記されている

#### やり直し条件

次のどれかに当てはまる場合は書き直します。

- 「みんな」「企業」「若者」など広すぎる言葉だけで書いている
- 困りごとが「便利にしたい」「効率化したい」だけで終わっている
- 期限がない
- 誰にも話を聞く予定がない

---

### 3.2 `01-decision-log.md`

判断したことはすべてここに残します。

```md
## YYYY-MM-DD 判断

### 決めたこと
-

### 決めた理由
-

### 迷った選択肢
-

### 捨てた選択肢
-

### 捨てた理由
-

### まだ分からないこと
-

### 次に確かめること
-
```

#### 書くタイミング

次のときは必ず書きます。

- 対象者を決めたとき
- 扱う困りごとを決めたとき
- 試作品の形を決めたとき
- 外に出す場所を決めたとき
- 続ける、変える、やめるを決めたとき

#### 合格条件

「なぜそうしたか」を、翌週の自分が読んでも分かる状態なら合格です。

---

### 3.3 `16-ai-prompts.md`

AIに依頼した文章を保存します。

```md
## YYYY-MM-DD 依頼文

### 目的
-

### 依頼文
```text
ここに実際の依頼文を貼る
```

### 返ってきた内容のうち使ったもの
-

### 返ってきた内容のうち使わなかったもの
-

### 次回の改善点
-
```

AIを使うと作業が速く進みます。しかし、あとで「なぜこの案になったか」が分からなくなりやすいです。依頼文を残すことで、再現しやすくなります。

## 4. 書く内容を絞り込めない場合の手順

### 4.1 絞り込みの原則

絞り込めないときは、考え続けてはいけません。

次の順番で機械的に絞ります。

1. 候補を出す
2. 広すぎるものを消す
3. 点数をつける
4. 下位を捨てる
5. 上位3つだけを比べる
6. 1つだけ選ぶ
7. 7日間だけ進める

ここで重要なのは、「正解を選ぶ」ことではありません。

重要なのは、「早く確かめられる1つを選ぶ」ことです。

---

### 4.2 候補を30個出す

`02-candidate-list.md` に、候補を30個書きます。

1個につき、次の形で書きます。

```md
## 候補001

### 対象者
-

### 困っている場面
-

### 困りごと
-

### 今のやり方
-

### こちらが出せそうな助け
-

### すぐ話を聞けそうな人
-
```

#### AIへの依頼文

```text
以下の情報をもとに、取り組み候補を30個出してください。

目的は、良い案を当てることではなく、比較できる候補を増やすことです。

条件：
- 1つの候補につき、対象者を1種類にする
- 「みんな」「企業」「個人」など広すぎる言葉を避ける
- 困りごとは、実際の場面で書く
- 解決策はまだ詳しく書かなくてよい
- 表で出す

私が詳しい領域：
-

すぐ話を聞ける相手：
-

最近、自分が面倒だと感じた作業：
-

使える時間：
-

使える道具：
-
```

---

### 4.3 広すぎる候補を消す

候補を30個出したら、次に広すぎるものを消します。

次のどれかに当てはまるものは削除します。

| 消す条件 | 例 | なぜ消すか |
|---|---|---|
| 対象者が広すぎる | 会社員、学生、経営者 | 誰に聞けばよいか分からない |
| 場面がない | 仕事を効率化したい | いつ困るか分からない |
| 困りごとが弱い | 便利だったら嬉しい | 行動や支払いにつながりにくい |
| 相手に会えない | 海外の大企業CEO | 確認できない |
| 2週間で試せない | 完全自動運転を作る | 最初の反応を見る前に時間がかかりすぎる |

#### 書き換え例

悪い例：

```text
中小企業の業務効率化
```

良い例：

```text
従業員30〜100人の卸売会社で、毎週月曜に営業別の売上報告をExcelで作っている営業管理担当者が、データの転記に2時間以上かかっている。
```

悪い例：

```text
クリエイター向けAIツール
```

良い例：

```text
週3本以上ショート動画を作る個人クリエイターが、投稿前にタイトル案を20個作る作業で毎回30分以上悩んでいる。
```

---

### 4.4 点数をつける

残った候補を `03-score-table.md` に貼り、次の基準で点数をつけます。

各項目は0点、1点、2点でつけます。

| 項目 | 0点 | 1点 | 2点 |
|---|---|---|---|
| 困りごとの強さ | あれば便利 | ないと少し面倒 | 放置すると損、遅れ、不安、失注、手戻りが起きる |
| 起きる頻度 | 年数回 | 月1回程度 | 週1回以上 |
| 今のやり方の不満 | 今のままでよい | 少し不満 | 明確に面倒、高い、遅い、ミスが出る |
| 相手に会えるか | 会えない | 紹介なら会える | 7日以内に3人以上会える |
| 行動が見えるか | 口で感想を聞くしかない | クリックや返信は見える | 申込、購入、紹介、継続利用が見える |
| 2週間で試せるか | 試せない | 一部だけ試せる | 紙、手作業、簡単な画面で試せる |
| 自分が続けられるか | 興味が薄い | 条件付きで続けられる | 半年以上続けてもよい |
| 失敗時の危険 | 法務、健康、金銭被害の危険が高い | 注意すれば扱える | 危険が小さい |

合計16点満点です。

#### 判定ルール

| 合計点 | 扱い |
|---:|---|
| 13〜16点 | 最優先で検証する |
| 10〜12点 | 比較候補として残す |
| 9点以下 | 今は扱わない |

ただし、次の項目が0点なら、合計点が高くても今は扱いません。

- 相手に会えるか
- 2週間で試せるか
- 失敗時の危険

理由は、確かめられないもの、試せないもの、危険が高いものは、初期段階の題材に向かないからです。

---

### 4.5 最後に1つへ絞る

上位3つが残ったら、次の質問に答えます。

```md
## 最終比較

### 候補A
- 7日以内に話を聞ける人数：
- 2週間以内に見せられるもの：
- 相手が行動したと判断できる証拠：
- 自分が続けたい理由：
- やめるべき理由：

### 候補B
- 7日以内に話を聞ける人数：
- 2週間以内に見せられるもの：
- 相手が行動したと判断できる証拠：
- 自分が続けたい理由：
- やめるべき理由：

### 候補C
- 7日以内に話を聞ける人数：
- 2週間以内に見せられるもの：
- 相手が行動したと判断できる証拠：
- 自分が続けたい理由：
- やめるべき理由：
```

#### 最終決定ルール

次の順で選びます。

1. 7日以内に話を聞ける人数が多いもの
2. 2週間以内に見せられるものが明確なもの
3. 相手の行動を見られるもの
4. 自分が半年続けてもよいもの

それでも迷う場合は、最も小さく試せるものを選びます。

## 5. 手順1：対象者を1種類にする

### 5.1 目的

「誰のために作るか」を1種類に絞ります。

対象者を絞らないと、聞く相手、作るもの、文章、価格、見せる場所がすべて曖昧になります。

---

### 5.2 悪い対象者と良い対象者

| 悪い例 | 良い例 |
|---|---|
| 経営者 | 従業員10〜50人の制作会社で、営業も兼務している代表 |
| マーケター | BtoB SaaS企業で、月4本の導入事例記事を作る担当者 |
| クリエイター | 週3本以上ショート動画を投稿している個人クリエイター |
| 学生 | 就活で週5社以上に応募している大学3年生 |
| 営業 | 新規開拓メールを毎週100件以上送る法人営業担当者 |

良い対象者は、次の4つが分かります。

1. どんな立場か
2. どんな作業をしているか
3. どれくらいの頻度でしているか
4. どこに行けば会えるか

---

### 5.3 作業手順

#### 手順

1. 候補の対象者を10種類書く
2. それぞれについて「会える場所」を書く
3. それぞれについて「毎週やっている作業」を書く
4. それぞれについて「困っていそうな場面」を書く
5. 7日以内に3人以上話を聞ける対象者だけ残す
6. 1種類に絞る

#### 記入テンプレート

```md
## 対象者候補

| 番号 | 対象者 | 会える場所 | 毎週している作業 | 困っていそうな場面 | 7日以内に話せる人数 |
|---:|---|---|---|---|---:|
| 1 |  |  |  |  |  |
| 2 |  |  |  |  |  |
| 3 |  |  |  |  |  |
```

#### 合格条件

次の形で1文にできれば合格です。

```text
私は、［立場］で、［頻度］、［作業］をしていて、［場面］で困っている人を対象にする。
```

例：

```text
私は、従業員30〜100人の卸売会社で、毎週月曜に営業別の売上報告をExcelで作っていて、転記ミスと集計時間に困っている営業管理担当者を対象にする。
```

---

## 6. 手順2：困りごとを1つにする

### 6.1 目的

対象者が「いつ」「何に」「どれくらい」困っているかを1つに絞ります。

困りごとが複数あると、試作品が大きくなり、何に反応したのか分からなくなります。

---

### 6.2 困りごとの書き方

困りごとは、次の形で書きます。

```text
［対象者］が、［場面］で、［今のやり方］をしているため、［時間・お金・不安・ミス・手戻り］が発生している。
```

例：

```text
BtoB SaaS企業のマーケティング担当者が、導入事例記事を作るとき、営業担当者から顧客情報を何度も聞き直しているため、1本の記事を作るまでに3週間かかっている。
```

例：

```text
個人クリエイターが、ショート動画を投稿する前に、タイトル案と冒頭の一言を毎回自分で考えているため、1本あたり30分以上悩んでいる。
```

---

### 6.3 困りごとの強さを確認する質問

次の質問に答えます。

```md
## 困りごとの確認

### その困りごとは、いつ起きるか
-

### どれくらいの頻度で起きるか
-

### 1回あたり、どれくらい時間がかかるか
-

### 放置すると何が悪くなるか
-

### 今はどうやって対処しているか
-

### その対処に何が不満か
-

### すでにお金を払っているものはあるか
-

### すでに人に頼んでいる作業はあるか
-

### すでに自分で工夫していることはあるか
-
```

#### 強い困りごとの特徴

次のどれかがあれば、強い困りごとの可能性があります。

- すでにお金を払っている
- すでに人に頼んでいる
- Excel、メモ、手作業で無理やり対応している
- ミスが起きると損をする
- 遅れると売上や信用に影響する
- 毎週または毎日起きている
- 担当者が「本当はやりたくない」と言っている

#### 弱い困りごとの特徴

次の状態なら、今は扱わない方がよいです。

- 「あったら便利」だけ
- 困っている本人が何も対処していない
- 年に数回しか起きない
- 解決しても相手の行動が変わらない
- 誰に聞けばよいか分からない

---

## 7. 手順3：面談で確認する

### 7.1 目的

自分の想像ではなく、実際の相手の経験を聞きます。

面談の目的は、褒めてもらうことではありません。

目的は、次を確認することです。

1. 本当にその場面があるか
2. 本当に困っているか
3. 今どうしているか
4. どれくらい時間やお金を失っているか
5. 解決したら行動が変わるか

---

### 7.2 聞く相手の条件

最初は5人に聞きます。

5人の条件は次の通りです。

- 対象者の条件に合っている
- 直近30日以内にその作業をしている
- 自分の知人でもよいが、遠慮なく話してくれる人が望ましい
- できれば役職や会社規模が少しずつ違う

#### 避ける相手

次の相手は、最初の5人には向きません。

- その作業をしたことがない人
- 立場が違いすぎる人
- 応援したいだけの友人
- こちらの案を褒めるだけの人
- 決裁権だけあり、実作業を知らない人

---

### 7.3 依頼文

#### 知人に送る場合

```text
少し相談です。

いま、［対象者］が［作業］で困る場面について調べています。
売り込みではなく、実際にどう作業しているかを聞きたいです。

15〜20分だけ、最近その作業をしたときの流れを教えてもらえませんか。
こちらの案を説明するより、普段どうしているかを聞く時間にしたいです。
```

#### 知人ではない相手に送る場合

```text
突然の連絡失礼します。

［対象者］の［作業］について調べています。
特に、［場面］でどのような手間や困りごとがあるかを知りたいです。

売り込みではありません。
15〜20分ほど、実際の作業の流れを教えていただけないでしょうか。
お話いただいた内容は、個人名や会社名が分からない形で整理します。
```

---

### 7.4 面談で聞く質問

質問は、未来の希望ではなく、過去の具体的な経験に寄せます。

#### 最初の質問

```text
直近で、その作業をしたのはいつですか。
```

この質問に答えられない場合、その人は対象者ではない可能性があります。

#### 基本質問

```text
そのとき、最初に何をしましたか。
次に何をしましたか。
どの道具を使いましたか。
どこで時間がかかりましたか。
どこで迷いましたか。
どこでミスが起きやすいですか。
誰に確認しましたか。
何分くらいかかりましたか。
終わった後、何を見て「終わった」と判断しましたか。
```

#### 困りごとの強さを測る質問

```text
その作業が遅れると、何が困りますか。
その作業でミスが起きると、何が起きますか。
その作業を減らすために、すでに試したことはありますか。
その作業のために、すでにお金を払っているものはありますか。
誰かに頼んでいる部分はありますか。
できれば避けたい作業はどこですか。
```

#### 聞いてはいけない質問

次の質問は、相手が気を使ってよいことを言いやすいため避けます。

```text
こういうサービスがあったら使いますか。
いくらなら払いますか。
このアイデア、どう思いますか。
便利だと思いますか。
```

代わりに、次のように聞きます。

```text
今、その作業にいくら払っていますか。
今、その作業に何時間かかっていますか。
その作業が半分の時間になったら、空いた時間で何をしますか。
同じ問題を解決するために、過去に何を試しましたか。
```

## 9. 手順5：最小の試作品を決める

### 9.1 目的

いきなり完成品を作らず、相手の反応を見るための最小のものを作ります。

試作品の目的は、完成させることではありません。

目的は、次を確かめることです。

- 相手が理解するか
- 相手が使いたいと言うだけでなく、実際に行動するか
- 相手が今のやり方から変えるほど困っているか
- こちらが作る価値があるか

---

### 9.2 試作品の種類

最初は、できるだけ軽い試作品を選びます。

| 種類 | 作るもの | 向いている場合 | 作成時間の目安 |
|---|---|---|---:|
| 文章 | 説明文、提案文、手順書 | サービス内容が言葉で説明できる | 1〜3時間 |
| 画像 | 画面案、スライド、図 | 見た目や流れを見せたい | 2〜6時間 |
| 手作業代行 | 裏側は人間が作業する | 自動化前に価値を確かめたい | 1〜2日 |
| 申込ページ | 予約、事前登録、問い合わせ | お金や時間を使う意思を見たい | 半日〜1日 |
| 簡単な画面 | 入力と出力だけの小さい画面 | 実際の操作感を見たい | 2〜5日 |
| 動画 | 使い方の短い説明 | 操作前に理解させたい | 半日〜1日 |

#### 選び方

次の順に選びます。

1. 文章だけで反応が見られるなら、文章にする
2. 文章で伝わらないなら、画像にする
3. 価値があるか分からないなら、手作業代行にする
4. 行動を見る必要があるなら、申込ページにする
5. 操作感が重要なら、簡単な画面にする

---

### 9.3 試作品で作らないもの

最初の試作品では、次を作りません。

- ログイン機能
- 決済機能
- 管理画面
- 高度な自動化
- 完璧なデザイン
- すべての例外対応
- 大量データ処理
- 細かい権限設定
- 長期運用のための複雑な仕組み

ただし、個人情報、医療、金融、法務、人事評価、未成年、機密情報に関わる場合は、最初から安全確認が必要です。

## 11. 手順7：試作品を見せて反応を見る

### 11.1 目的

相手が実際に行動するかを見ます。

「いいですね」は証拠として弱いです。

強い証拠は、相手が時間、お金、情報、紹介、日程のいずれかを出したことです。

---

### 11.2 反応の強さ

| 反応 | 強さ | 判断 |
|---|---|---|
| いいですね | 弱い | 参考程度 |
| また見ます | 弱い | ほぼ未反応 |
| 詳しく知りたい | 中 | 追加説明へ進む |
| 自分のデータで試したい | 強 | 試用へ進む |
| 次の日程を決めたい | 強 | 継続確認へ進む |
| 他の人を紹介する | 強 | 需要ありの可能性 |
| 有料で試したい | とても強い | 価格確認へ進む |
| 今は不要 | 明確 | 理由を聞いて記録する |

---

### 11.3 見せるときの流れ

1人あたり20分以内にします。

```text
1分：今日の目的を説明する
3分：相手の現在のやり方を聞く
5分：試作品を見せる
5分：相手に使う場面を想像してもらう
3分：次にどうするかを聞く
3分：記録を確認する
```

#### 最初の説明

```text
今日は売り込みではなく、これが実際の作業に役立つかを確認したいです。
褒めてもらうより、使わない理由や分かりにくい箇所を知りたいです。
最後に、次に試すかどうかだけ確認させてください。
```

### 11.4 最後に必ず聞く質問

次のどれかを必ず聞きます。

#### 業務向けの場合

```text
次にこの作業が発生するのはいつですか。
そのとき、これを使って試してみますか。
試すなら、いつまでに何を送ればよいですか。
```

#### 有料の可能性がある場合

```text
もし次回こちらが手作業で同じ結果を作る場合、いくらなら試してもよいですか。
```

#### 紹介が欲しい場合

```text
同じ作業で困っていそうな人を1人だけ紹介してもらうことはできますか。
```

#### 継続利用を見たい場合

```text
次回も使うとしたら、どのタイミングで思い出せる必要がありますか。
```

## 12. 手順8：続ける、変える、やめるを判断する

### 12.1 目的

感情ではなく、記録を見て次を決めます。

判断は次の3種類だけです。

| 判断 | 意味 |
|---|---|
| 続ける | 同じ対象者、同じ困りごとで進む |
| 変える | 対象者、困りごと、見せ方のどれかを変える |
| やめる | いったんこの題材を止める |

---

### 12.2 判断基準

5人に見せた後、次で判断します。

| 結果 | 判断 |
|---|---|
| 5人中3人以上が次の行動を取った | 続ける |
| 5人中1〜2人が次の行動を取った | 変える |
| 5人中0人が次の行動を取った | やめる、または困りごとに戻る |
| 反応は良いが次の行動がない | 見せ方か相手の選び方を変える |
| 不要理由が全員同じ | その理由を解く案に変えるか、やめる |

次の行動とは、次のいずれかです。

- 次回試す日程を決める
- 自分のデータや資料を送る
- 他の人を紹介する
- 有料試用に同意する
- 継続して使うための条件を具体的に話す

## 13. 手順9：外に出す

### 13.1 目的

知人以外にも見せて、反応が続くかを確認します。

知人だけの反応は、優しさや関係性が混ざります。

外に出すことで、より厳しい反応を見ます。

---

### 13.2 外に出す場所の選び方

| 対象者 | 出す場所の例 |
|---|---|
| 仕事で使う人 | LinkedIn、X、業界コミュニティ、知人経由、既存顧客への連絡 |
| 社内担当者 | 社内Slack、部署会議、個別DM、業務改善提案会 |
| クリエイター | X、YouTube概要欄、Discord、note、イベント |
| 学習者 | X、note、勉強会、学校・コミュニティ |
| 経営者 | 紹介、経営者会、業界団体、直接メール |

最初から大きく出す必要はありません。

最初の公開は、10〜30人に届けば十分です。

## 14. 手順10：繰り返し作業を仕組みにする

### 14.1 目的

同じ作業を毎回手でやらないようにします。

ただし、最初から完全自動化してはいけません。

先に手作業で価値を確認し、その後で繰り返し部分だけを仕組みにします。

---

### 14.2 仕組みにする順番

次の順で進めます。

1. まず手作業でやる
2. 毎回同じ作業を書き出す
3. 人間が判断する部分と、機械的にできる部分を分ける
4. 機械的にできる部分だけAIに任せる
5. 人間が確認する場所を決める
6. 失敗時の戻し方を決める
7. 手順書にする

---

### 14.3 作業分解テンプレート

`14-operation-rules.md` に書きます。

```md
# 繰り返し作業の分解

-

## 作業の開始条件
-

## 作業の終了条件
-

## 手順
| 番号 | 作業 | 人間がやる | AIに任せる | 確認方法 | 失敗時の対応 |
|---:|---|---|---|---|---|
| 1 |  |  |  |  |  |
| 2 |  |  |  |  |  |
| 3 |  |  |  |  |  |

## AIに渡してよい情報
-

## AIに渡してはいけない情報
-

## 人間が必ず確認する箇所
-

## 記録するもの
-
```

---

### 14.4 AIに任せてよい作業

| 任せてよい作業 | 条件 |
|---|---|
| 議事録の要約 | 元の記録が残っている |
| 質問案の作成 | 人間が確認してから使う |
| 候補の洗い出し | 最終判断を人間がする |
| 表の整理 | 元データを確認できる |
| 文章の下書き | 公開前に人間が読む |
| 簡単なコードの初版 | テストと確認をする |
| 作業手順の下書き | 実際に人が試して直す |

### 14.5 AIに任せてはいけない作業

| 任せてはいけない作業 | 理由 |
|---|---|
| 最終的な採否判断 | 責任を取れない |
| 契約・法務判断 | 専門確認が必要 |
| 医療・健康判断 | 人に害が出る可能性がある |
| 採用・評価の最終判断 | 不公平や差別につながる可能性がある |
| お金の最終判断 | 損失が出る可能性がある |
| 機密情報の無加工投入 | 情報漏えいの可能性がある |
| セキュリティ確認なしのコード公開 | 脆弱性が残る可能性がある |

### 15.1 候補を出す

- 困っている場面

## 18. よくある失敗と対処

### 18.1 失敗：最初から大きく作る

#### 状態

- 機能を増やしている
- 公開前に何週間も作っている
- まだ誰にも見せていない

#### 対処

次の質問に戻ります。

```text
この試作品で、相手のどの行動を見たいのか。
```

行動が1つに絞れない場合、作る範囲を削ります。

---

### 18.2 失敗：相手が広すぎる

#### 状態

- いろいろな人に使えそう
- 説明文がぼんやりする
- 面談相手の話がばらばら

#### 対処

次の文に直します。

```text
［立場］で、［頻度］、［作業］をしている人
```

例：

```text
従業員30〜100人の会社で、毎週月曜に売上集計をしている営業管理担当者
```

---

### 18.3 失敗：感想を証拠だと思う

#### 状態

- いいですねと言われた
- 便利そうと言われた
- でも次の約束がない

#### 対処

最後に必ず次を聞きます。

```text
次にこの作業が発生するのはいつですか。
そのとき試しますか。
試すなら、何をいつ送ればよいですか。
```

---

### 18.4 失敗：AIの出力をそのまま信じる

#### 状態

- AIが市場があると言った
- AIが良い案だと言った
- AIの作った文章をそのまま公開した

#### 対処

AIの出力を次に分けます。

```md
## AI出力の確認

### 事実として確認できること
-

### 推測にすぎないこと
-

### 実際の相手に聞くこと
-

### 公開前に人間が確認すること
-
```

---

### 18.5 失敗：絞り込みすぎて市場が小さすぎる

#### 状態

- 1人には刺さるが、他に同じ人がいない
- 対象者を探すのに時間がかかりすぎる

#### 対処

対象者を1段階だけ広げます。

例：

狭すぎる：

```text
東京都渋谷区の従業員20人の動画制作会社で、月曜朝にだけ請求書を作る担当者
```

広げた形：

```text
従業員10〜50人の制作会社で、毎月請求書を作る担当者
```

広げるときは、次のうち1つだけ広げます。

- 地域
- 業種
- 会社規模
- 作業頻度
- 役職

一度に全部広げてはいけません。

## 20. まとめ：この手順書の使い方

この手順書の中心は、次の考え方です。

> 大きな仕事を、相手、場面、困りごと、今のやり方、小さな試作品、相手の行動、次の判断に分ける。

AIを使うと、候補出し、整理、下書き、試作品作りは速くなります。

しかし、AIを使っても、次は人間が確認する必要があります。

- その相手は本当にいるか
- その困りごとは本当に起きているか
- 相手は今どう対処しているか
- 相手は時間、お金、情報、紹介、日程のどれかを出すか
- 続ける価値があるか

最初から正しい案を作る必要はありません。

必要なのは、次の順番を崩さないことです。

```text
候補を出す
↓
絞る
↓
相手に聞く
↓
小さく作る
↓
見せる
↓
行動を見る
↓
判断する
↓
直す
```

この順番を守れば、新規事業、社内改善、個人サービス、制作活動のどれにも適用できます。

## 21. 最小実行版

時間がない場合は、次の10手順だけ実行します。

1. 候補を10個書く
2. 7日以内に話を聞けない候補を消す
3. 2週間で試せない候補を消す
4. 1つ選ぶ
5. 対象者を1文で書く
6. 困りごとを1文で書く
7. 3人に過去の作業を聞く
8. 文章だけの試作品を作る
9. 5人に見せる
10. 次の行動を取った人数で、続ける、変える、やめるを決める

これで最低限の検証はできます。
