# 意思決定メモ：AI開発で握るもの／手放すもの

対象：事業オーナー（方針決定者）
用途：AIエージェント中心の開発で、何を委任し何を握るかの判断基準

-----

## 結論

**握るべきものは開発工程の中に無い。握るのは「自分が最初のユーザーであり続ける」という立ち位置だけ。**
実装・テスト・言語化・文脈化は全てAIに委任してよい。

-----

## 委任してよいもの（人間が手を出す価値が無い）

- 実装：既知パターンの組み合わせが大半。AIの得意領域そのもの。
- テスト・検証：書く工程はAIが優れる。人間が手で書くのは自己満足。
- 欲求の言語化：潜在欲求の事前言語化は誰も得意でない。AIに任せて構わない。
- 文脈化：散在する事象に文脈を与え関連づけるのはAIの得意分野。

-----

## 握るもの（立ち位置）の定義

「使いながら作る」段階では、生成を駆動するのは**使用の中の反応**。
決定的な問いは一つ：**その反応は誰の使用から出たか。**

- あなたが本物のユーザーとして使う → あなたの文脈が沈殿 → 事業の固有性が保たれる
- AIが平均ユーザーを模擬使用 → 一般解が沈殿 → 誰のものでもないものになる

これは作業でも判定工程でもなく、**立ち位置**。最初のユーザーであり続けること。

-----

## なぜ手放せないか

実務の難所はロジック（既知・AIが書ける）ではなく、**文脈**にある。
業務ルール・顧客固有の例外・規制下の扱い——世界に存在しない、言語化されていない研鑽の集積。
これは事前に仕様へ落とせず、**使って「これじゃない」と漏れ出た瞬間にだけ形になる暗黙知**。
その暗黙知は、使う主体があなた自身であるときにだけ沈殿する。
立ち位置を手放す＝事業の固有性（競争優位の源泉）をAIの平均に均されて失う。

-----

## 工程をどう振り分けるか（補助的な判定軸）

委任した工程の中で「事前に検査で固定する／事後に結果で選ぶ」を分ける軸。判定は2問で切れる。

|問い                                 |Yes     |No     |
|-----------------------------------|--------|-------|
|正しさの条件を言語化できるか（値は不明でも、満たすべき条件は言えるか）|事前検査で囲う |事後選別へ  |
|良し悪しを事後に安く判定できるか                   |作って選ぶが成立|人間が直接判断|

- 条件を言語化できる（昇順である・合計が一致する・負にならない等）→ 検査で固定し、それを壊さない範囲でAIに量産させる
- 条件自体が定義できない（いい感じのUX・価値判断・探索）→ 作り続けて結果で選ぶ
- 「ロジックが新しいか」は判定軸にならない。軸は常に「条件を言語化できるか」

-----

## 固有性が失われる条件（アラート）

以下に該当したら、立ち位置を手放しかけている。

- [ ] 自分はもうこのプロダクトを本物のユーザーとして使っていない
- [ ] 使用データの大半がAIの模擬使用・合成データになっている
- [ ] 「適当に使って育てておいて」とAIに使用ループごと委ねた
- [ ] 「これじゃない」という自分の反応が、最近一度も生成入力になっていない

ひとつでも該当 → 沈殿しているのはあなたの文脈でなくAIの平均。

-----

## オーナーの行動原則（1行）

**作るな、判定もしなくていい。ただし、最初のユーザーであり続けろ。**
