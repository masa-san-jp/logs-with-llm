# オフィス会話録音から暗黙知を抽出するパイプライン設計案

## 1. 設計思想

このシステムの本質は、最初から高精度な分類・要約・マニュアル化を狙うことではなく、**将来のAIモデルで再処理可能な形で、現場会話データを時系列・多層構造で蓄積すること**にある。

重要なのは、次の順序である。

1. 生音声を欠損なく保存する
2. 音声に時刻・場所・録音デバイスなどのメタデータを付与する
3. Speech-to-Textで文字起こしする
4. 文字起こし結果を意味単位に分割する
5. 業務ナレッジ候補だけを抽出する
6. 抽出結果を人間が読めるカード・手順・FAQ・マニュアル断片に変換する
7. 週次・月次で人間がレビューする
8. 将来のモデル改善時に、生音声から再処理できる状態を維持する

最初から分類ラベルや業務体系を作り込むと、現場運用が重くなり失敗しやすい。初期段階では「とにかく録る」「壊さず貯める」「後で再解釈できる形にする」を優先する。

---

## 2. 全体アーキテクチャ

```text
[録音デバイス]
    ↓
[音声収集エージェント]
    ↓
[生音声ストレージ]
    ↓
[文字起こしジョブ]
    ↓
[タイムスタンプ付きテキスト]
    ↓
[会話セグメント化]
    ↓
[ナレッジ候補抽出]
    ↓
[知識カード生成]
    ↓
[マニュアル・FAQ・業務知識ベース]
    ↓
[週次レビュー]
    ↓
[承認済みナレッジ / 差し戻し / 再処理対象]
```

この構成では、処理を一枚岩にしない。録音、文字起こし、分類、抽出、統合、レビューをそれぞれ独立した工程として扱う。

## 3. レイヤー設計

## Layer 0: 録音基盤

### 目的

オフィス内の会話を、一定単位で自動録音し、扱いやすいファイル構造で保存する。

### 基本方針

- 生音声は原則として削除しない
- 編集済み音声ではなく、元データを保存する
- 1時間ごとにファイルを分割する
- ファイル名に日付・時間・場所・デバイスIDを含める
- 録音失敗・無音・ノイズもメタデータとして記録する

### 推奨ファイル構造

```text
data/
  raw_audio/
    2026/
      07/
        10/
          office-main/
            20260710-090000-100000-office-main-device-a.wav
            20260710-100000-110000-office-main-device-a.wav
            20260710-110000-120000-office-main-device-a.wav
          office-sub/
            20260710-090000-100000-office-sub-device-b.wav

  manifests/
    2026/
      07/
        10/
          20260710-office-main-manifest.jsonl
```

### 録音メタデータ例

```json
{
  "recording_id": "20260710-090000-100000-office-main-device-a",
  "date": "2026-07-10",
  "start_time": "2026-07-10T09:00:00+09:00",
  "end_time": "2026-07-10T10:00:00+09:00",
  "location": "office-main",
  "device_id": "device-a",
  "file_path": "data/raw_audio/2026/07/10/office-main/20260710-090000-100000-office-main-device-a.wav",
  "format": "wav",
  "duration_seconds": 3600,
  "sample_rate": 16000,
  "channels": 1,
  "status": "recorded",
  "checksum": "sha256:...",
  "notes": null
}
```

### 重要な判断

最初の段階では、話者分離やノイズ除去を録音段階で無理に行わない。録音基盤は「できるだけ素の事実を保存する」役割に限定する。

## Layer 1: 前処理

### 目的

文字起こししやすい状態に音声を整える。ただし、元音声は必ず残す。

### 処理内容

- 無音区間の検出
- 音量正規化
- ノイズ傾向の推定
- 音声品質スコアの算出
- 必要に応じた短尺チャンク化

### 出力例

```text
data/
  processed_audio/
    2026/07/10/office-main/
      20260710-090000-091500-office-main-device-a.chunk.wav
      20260710-091500-093000-office-main-device-a.chunk.wav
```

### 品質メタデータ例

```json
{
  "recording_id": "20260710-090000-100000-office-main-device-a",
  "chunk_id": "20260710-090000-091500-office-main-device-a",
  "start_offset_seconds": 0,
  "end_offset_seconds": 900,
  "speech_ratio": 0.42,
  "estimated_noise_level": "medium",
  "overlap_risk": "high",
  "transcription_priority": "normal"
}
```

### 重要な判断

音声の「改善」は行ってもよいが、元音声を上書きしてはいけない。後から別の前処理方式で再処理できる状態を維持する。

---

## Layer 2: Speech-to-Text

### 目的

音声をタイムスタンプ付きテキストに変換する。

### 必須要件

- 発話単位の開始時刻・終了時刻を持つ
- 文字起こしモデル名・バージョンを保存する
- 信頼度スコアを保存する
- 聞き取り不能箇所を無理に補完しない
- 話者分離は可能なら行うが、必須にしない

### 出力例

```json
{
  "transcript_id": "tr-20260710-090000-office-main-device-a",
  "recording_id": "20260710-090000-100000-office-main-device-a",
  "model": "stt-model-name",
  "model_version": "2026-07-xx",
  "language": "ja",
  "segments": [
    {
      "segment_id": "seg-000001",
      "start_time": "2026-07-10T09:03:12+09:00",
      "end_time": "2026-07-10T09:03:26+09:00",
      "start_offset_seconds": 192,
      "end_offset_seconds": 206,
      "speaker": "unknown",
      "text": "この申請、前回と同じパターンなら先に添付書類を確認した方が早いです。",
      "confidence": 0.81
    }
  ]
}
```

### 重要な判断

この段階では「正しい業務知識かどうか」は判断しない。あくまで後段のAIが読める一次テキストを作る。

---

## Layer 3: 会話セグメント化

### 目的

文字起こしされた発話を、後から扱いやすい意味単位にまとめる。

### セグメント単位

発話単位では細かすぎるため、次のような「会話断片」にまとめる。

- 1つの質問と回答
- 1つのトラブル相談
- 1つの判断理由
- 1つの手順説明
- 1つの顧客対応のコツ
- 1つの雑談

### 出力例

```json
{
  "conversation_unit_id": "cu-20260710-090312-office-main-001",
  "source_segments": ["seg-000001", "seg-000002", "seg-000003"],
  "start_time": "2026-07-10T09:03:12+09:00",
  "end_time": "2026-07-10T09:04:40+09:00",
  "location": "office-main",
  "speaker_status": "unknown",
  "raw_text": "この申請、前回と同じパターンなら先に添付書類を確認した方が早いです。...",
  "summary": "特定申請では、添付書類の確認を先に行うと手戻りが減るという実務上の助言。",
  "topic_guess": "申請処理",
  "business_relevance_score": 0.74,
  "noise_risk": "low"
}
```

### 分類は粗くてよい

初期分類は次の程度で十分。

| 分類 | 意味 |
|---|---|
| business_candidate | 業務ナレッジ候補 |
| operational_chat | 日常業務連絡 |
| customer_case | 顧客・利用者対応 |
| troubleshooting | 問題解決・例外処理 |
| personal_chat | 雑談・私的会話 |
| unclear | 判定不能 |
| noise | ノイズ・聞き取り不能 |

細かい業務分類は後から作る。初期段階では、分類精度よりも「ナレッジ候補を取りこぼしすぎないこと」を優先する。

---

## Layer 4: ナレッジ候補抽出

### 目的

会話断片から、将来の業務マニュアル・FAQ・教育資料・改善提案につながる候補を抽出する。

### 抽出対象

- 手順
- 判断基準
- 例外対応
- よくある失敗
- 顧客対応のコツ
- 暗黙の優先順位
- 熟練者の経験則
- 属人的な工夫
- 現行マニュアルとの差分
- 現場でしか共有されていない注意点

### 出力する知識カード

```json
{
  "knowledge_candidate_id": "kc-20260710-001",
  "source_conversation_unit_id": "cu-20260710-090312-office-main-001",
  "title": "申請処理では添付書類確認を先に行うと手戻りが減る",
  "knowledge_type": "workaround",
  "domain": "申請処理",
  "content": "前回と同じパターンの申請では、入力内容の詳細確認より先に添付書類の有無・不備を確認すると、後工程の手戻りを減らせる可能性がある。",
  "evidence_text": "この申請、前回と同じパターンなら先に添付書類を確認した方が早いです。",
  "confidence": 0.76,
  "requires_human_review": true,
  "risk_flags": ["needs_validation"],
  "source_audio_ref": {
    "recording_id": "20260710-090000-100000-office-main-device-a",
    "start_offset_seconds": 192,
    "end_offset_seconds": 206
  }
}
```

### 重要な判断

AIはこの段階で「正式ナレッジ」として登録しない。あくまで「候補」として蓄積する。根拠となる音声位置・文字起こし位置に戻れることが必須。

---

## Layer 5: 知識化・ドキュメント化

### 目的

ナレッジ候補を、人間が読める業務知識に変換する。

### 出力形式

初期段階では、巨大なマニュアルではなく、1件1ページ程度の小さな知識カードにする。

```markdown
# 申請処理では添付書類確認を先に行う

## 概要

特定パターンの申請では、入力内容の詳細確認より先に添付書類を確認すると、手戻りを減らせる可能性がある。

## 適用場面

- 過去に同種の不備が発生した申請
- 添付書類の有無が後工程に影響する申請
- 確認項目が多く、後戻りコストが高いケース

## 実務上の手順

1. 申請種別を確認する
2. 過去に同様の不備があったか確認する
3. 添付書類の有無を先に確認する
4. 不備があれば詳細審査に入る前に差し戻す
5. 添付書類が問題なければ通常確認に進む

## 根拠

2026-07-10 09:03頃のオフィス会話から抽出。

## ステータス

未レビュー
```

### 重要な判断

完成マニュアルをいきなり作らない。まずは小さな知識カードを蓄積し、後から統合する。

---

## Layer 6: 知識統合AI

### 目的

蓄積された知識カードを横断的に読み、意味的に近いものを統合する。

### 処理内容

- 類似ナレッジのクラスタリング
- 重複候補の検出
- 既存マニュアルとの差分検出
- 矛盾する知識の検出
- 頻出テーマの抽出
- 業務プロセス全体像へのマッピング
- 教育資料化できるテーマの抽出

### 出力例

```json
{
  "cluster_id": "cluster-application-attachments",
  "title": "申請処理における添付書類確認の実務知",
  "related_candidates": [
    "kc-20260710-001",
    "kc-20260712-014",
    "kc-20260718-006"
  ],
  "synthesis": "複数の会話から、申請処理では添付書類の確認を早期に行うことで、後工程の手戻りを減らせる可能性が示唆される。",
  "manual_update_suggestion": "申請処理マニュアルの初期確認手順に、添付書類確認を先に行う条件を追加する。",
  "confidence": 0.83,
  "review_priority": "high"
}
```

### 重要な判断

このAIは「現場会話を読むAI」ではなく、「蓄積された知識候補を編集するAI」として位置づける。編集長AIに近い役割である。

---

## Layer 7: 人間レビュー

### 目的

AIが生成した知識候補を、週次または月次で人間が確認する。

### レビュー単位

- 1日単位ではなく、週次でまとめて確認する
- すべてを読むのではなく、AIが優先度を付けた候補だけを見る
- 正しいかどうかだけでなく、使えるかどうかを見る

### レビューステータス

| ステータス | 意味 |
|---|---|
| approved | 正式ナレッジ化 |
| needs_edit | 内容修正が必要 |
| duplicate | 既存ナレッジと重複 |
| wrong | 誤り |
| sensitive | 機密・個人情報を含む |
| irrelevant | 業務ナレッジではない |
| pending | 保留 |

### 重要な判断

人間は全量確認しない。レビューすべきものをAIが絞り、人間は最終判断だけを行う。

## 4. データ保存ポリシー

### 生データは不変にする

```text
raw_audio: immutable
processed_audio: reproducible
transcripts: versioned
conversation_units: versioned
knowledge_candidates: versioned
approved_knowledge: curated
```

### モデルバージョンを必ず保存する

同じ音声でも、モデルが変われば文字起こし・要約・分類結果は変わる。したがって、すべてのAI処理結果に以下を保存する。

- モデル名
- モデルバージョン
- プロンプトバージョン
- 処理日時
- 入力データID
- 出力データID
- 信頼度
- エラー情報

### 再処理可能性を担保する

将来モデルが改善したとき、次の単位で再処理できる必要がある。

- 特定日の全音声
- 特定部屋の全音声
- 特定テーマに関係する会話
- 信頼度が低かった文字起こし
- 人間レビューで差し戻された候補
- 古いモデルで処理されたデータ

## 5. セキュリティ・プライバシー設計

この種のシステムでは、技術的に可能かどうかよりも、運用上許容されるかが重要になる。

### 必須論点

- 録音していることの明示
- 録音対象エリアの明確化
- 私的会話の扱い
- 個人情報・顧客情報のマスキング
- 閲覧権限
- 削除依頼への対応
- 退職者・異動者の会話データの扱い
- 音声データと抽出ナレッジの保存期間
- 法務・労務・情報セキュリティ観点の確認

### 推奨方針

- 生音声を見られる人は最小限にする
- 文字起こしテキストも全員公開しない
- 抽出済みナレッジだけを広く共有する
- 個人名は原則としてマスキングする
- 顧客名・個人情報・機微情報は自動検出して隔離する
- 私的会話と判定されたものはナレッジ処理対象から外す
- アクセスログを保存する

## 6. MVP設計

最初のMVPでは、以下だけを実装すればよい。

### MVPスコープ

1. 1部屋・1デバイスで録音
2. 1時間ごとに音声ファイルを保存
3. 日付・時刻・デバイスID付きでフォルダ整理
4. 自動でSpeech-to-Textを実行
5. タイムスタンプ付きテキストを保存
6. AIが業務ナレッジ候補を1日分抽出
7. 1件1ページのMarkdownカードを生成
8. 週次で人間が確認する

### MVPでやらないこと

- 完璧な話者分離
- 完璧なノイズ除去
- リアルタイム処理
- 全社展開
- 高度な業務分類体系
- 自動で正式マニュアルを書き換えること
- 人事評価への利用
- 個人別の発話分析

### MVPの成功条件

- 毎日録音が欠損なく蓄積される
- 毎日文字起こしが自動生成される
- 週に数件でも「これは使える」と思える知識カードが出る
- 元音声と根拠テキストに戻れる
- 人間レビューが30分以内に終わる

## 8. 処理ジョブ設計

### 常時稼働ジョブ

```text
recording-daemon
  - 音声入力を受け取る
  - 1時間ごとにファイルを保存
  - manifestを書き出す
```

### 定期バッチ

```text
transcription-worker
  - 未処理音声を検出
  - 前処理
  - 文字起こし
  - transcript保存
```

```text
knowledge-extraction-worker
  - transcriptを読む
  - 会話単位に分割
  - 業務ナレッジ候補を抽出
  - Markdownカードを生成
```

```text
weekly-synthesis-worker
  - 1週間分の知識候補を読む
  - 類似テーマを統合
  - 重要候補を優先順位付け
  - レビュー用レポートを生成
```

## 9. 初期KPI

精度指標よりも、運用継続性と再処理可能性を重視する。

| 指標 | 目標 |
|---|---|
| 録音欠損率 | 5%未満 |
| 文字起こし完了率 | 90%以上 |
| 1日あたりナレッジ候補数 | 3〜20件 |
| 週次レビュー時間 | 30分以内 |
| 週次で承認される知識カード | 1〜5件 |
| 元音声への参照成功率 | 100% |
| 個人情報検出漏れ | 低く抑える必要あり |

重要なのは、初期から高精度なナレッジ抽出を求めないこと。週に1件でも本当に使える暗黙知が出れば、長期的には価値が積み上がる。

## 10. 実装順序

### Phase 1: 録音・保存基盤

- 録音デバイス選定
- 1時間分割保存
- ファイル命名規則
- manifest生成
- 欠損監視

### Phase 2: 文字起こし基盤

- Speech-to-Textジョブ
- タイムスタンプ保存
- 失敗時リトライ
- モデルバージョン管理
- transcript保存

### Phase 3: ナレッジ候補抽出

- 会話セグメント化
- 業務関連度スコアリング
- 知識カード生成
- 根拠音声へのリンク保持

### Phase 4: 週次レビュー

- レビュー用一覧生成
- 承認・差し戻し・削除
- 承認済みナレッジ保存
- 差し戻し理由の保存

### Phase 5: 知識統合

- 類似カード統合
- マニュアル更新候補生成
- FAQ生成
- 教育資料化

## 11. 最初に作るべきもの

最初に作るべきものは、高度なAIではなく、以下の3つである。

1. **録音データを安全に蓄積する仕組み**
2. **録音データと文字起こしを時系列で紐づける仕組み**
3. **AIが抽出した知識候補から元音声に戻れる仕組み**

これがあれば、AIモデルが未熟でもデータ資産は残る。逆に、これがない状態で分類やマニュアル化を作り込んでも、後から検証・再処理できない。

---

## 12. 結論

この構想は、単なる自動議事録システムではない。  
本質は、**現場会話を企業の知識資産に変換するためのデータパイプライン**である。

初期段階では、精度よりも以下を優先する。

- 生音声を捨てない
- 時系列で整理する
- 文字起こしを残す
- AI処理結果をバージョン管理する
- 根拠に戻れる
- 人間レビューは軽くする
- 将来モデルで再処理できる

この設計にすると、今すぐ完璧な暗黙知抽出ができなくても、蓄積されたデータが将来の競争優位になる。
