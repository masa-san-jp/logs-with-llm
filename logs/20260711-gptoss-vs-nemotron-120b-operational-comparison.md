# ローカルLLM実タスク比較：gpt-oss:120b vs Nemotron-120b（運用タスク編）

- 実施: 2026-07-11 / GB10（Grace Blackwell・121GB 統合メモリ）上で実測
- ねらい: 合成ベンチではなく、AIエージェント群が日々ローカルLLMに投げている「本番タスク」で 2 つの 120B 級モデルを同一条件で突き合わせ、常用モデルを乗り換える価値があるかを判断する。

## 対象モデル

| 名前 | backend | 量子化 | 種別 |
|------|---------|--------|------|
| gpt-oss:120b | Ollama | Q4系 | 現行の常用 120B 枠 |
| Nemotron-120b | vLLM (OpenAI互換API) | NVFP4 | NVIDIA-Nemotron-3-Super-120B-A12B・**推論モデル** |

量子化が非対称（Q4 vs NVFP4）＝純粋なモデル能力差ではない、という但し書き付き。両モデルとも ~65-70GB 級で同時常駐が不可能なため、逐次実行（Nemotron を serve → 停止 → gpt-oss）で計測した。

## 測定条件

- `temperature=0.1`、出力上限は判定・構造化タスクで `max_tokens=4096`（推論モデルの枠切れ対策）。
- 生応答を全件 raw 保存し、採点は後から再現・修正できるようにした。
- 客観タスクは「出力 JSON が仕様スキーマに適合するか」で自動採点。独立した正解ラベル（人手 gold）が無いタスクは精度を測らず、両モデルの出力を並置して目視 + 速度 + 空応答率で比較した（gold を捏造しない方針）。

## 結果（運用 4 タスク）

いずれもエージェント群が本番で実際に走らせている処理を流用した。

| タスク | gpt-oss:120b | Nemotron-120b | 備考 |
|---|---|---|---|
| URL 分類（JSON 構造化出力） | 30/30 スキーマ適合・空0・err0・44s | 28/30 適合・**2件 HTTP400**・87s | カテゴリ判定は**両モデル 28/28 完全一致**。Nemotron の 400 は長文入力で context 長（8192）超過 |
| 監査ジャッジ（構造化 verdict） | 妥当な verdict を出力・20s | 応答はあるが形式差・29s | ※自動採点器にスキーマ不一致のバグがあり数値は無効化。実出力自体は両モデルとも妥当 |
| 記事要約（品質・目視） | 8/8・空0・忠実で詳細・46s | 6/8（**2件が空応答**）・167s | gpt-oss が安定。Nemotron は長プロンプトで一部 floor 化 |
| 作業ログ要約（構造化出力） | 5/5 スキーマ適合・空0・138s | **0/5**（2件空＋残りは JSON 不適合）・**480s** | Nemotron は推論に 4096 トークンを使い切り本文が空・1問 8 分。gpt-oss は全件構造化成功 |

## 重要な発見：推論モデルの落とし穴

- **Nemotron（推論モデル）は、長いプロンプトの構造化出力タスクで詰まる**。作業ログ要約 0/5（4096 トークンを「思考（reasoning）」で使い切り、最終的な本文（content）を 1 文字も返さない・1問 8 分）、記事要約 2/8 空、URL 分類で 2 件が context 長超過エラー。全タスクで gpt-oss の 2〜4 倍遅い。
- **gpt-oss:120b は全タスクで空応答ゼロ・エラーゼロ・有効 JSON・高速**。要約・分類・構造化・監査といった「日次自動処理のパンとバター」で運用信頼性が明確に高い。
- なお、別途走らせた「難度の高い多条件の判断推論」ベンチでは Nemotron がわずかに上回った。Nemotron の強みはそうした専門特化の局面に限られる。

## 結論（乗り換え判断）

- **常用モデルは gpt-oss:120b 継続を推奨。** 日次自動処理では速度・空応答ゼロ・JSON 安定性が効き、gpt-oss が総合で明確に上。
- **Nemotron は専門特化の呼び出し**に留めるのが妥当。重い多条件推論が要る局面だけ、`max_tokens` を 8k–16k+ に上げて floor 化を防いだ上で使う（その分さらに遅い）。
- 量子化非対称の但し書きは維持。gold が無いタスクの「精度」は未取得（別途の人手ラベリングが必要）。

## 学び（横展開できる gotcha）

推論モデル（Nemotron 系など）を「長いプロンプト × 構造化 JSON 出力」タスクに使うと、reasoning にトークン予算を食われて最終出力が空になることがある。対策は (1) `max_tokens` を大きめに取る、(2) API ルーター側で `reasoning_content` も回収して最終回答を拾う、の 2 点。単発の短い対話では表面化せず、長文の本番タスクで初めて露呈するため、導入判断は必ず実運用タスクで検証すること。
