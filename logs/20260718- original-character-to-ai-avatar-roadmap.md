# オリジナルキャラクター → AIアバター化 実装ロードマップ（ARDY参照）

## 前提
- 手元にあるのは「キャラクター設定」であり **VRMはまだ存在しない**
- モーション生成エンジンとして NVIDIA **ARDY**（テキスト→リアルタイム人体モーション生成）を利用する
- 最終形は、コメントやスクリプトに反応して動く配信用AIアバター
- 方針決定はオーナー、本ロードマップは判断材料・再現手順・テスト設計を提供する

前回整理した通り、ARDY本体は「テキスト→モーション」までしか担わない。**VRMの用意・スケルトンのリターゲット・配信ランタイム・音声/口パク・コメント取り込みはすべて自作の統合層**になる。本書はその統合層を最小工数で組む順序を定義する。

---

## 全体アーキテクチャ

処理フロー（推奨構成）:

```
[配信コメント/操作] 
      │  (YouTube/Twitch API → プロンプト整形)
      ▼
[ARDY 推論プロセス(Python, GPU)]  ── autoregressive_step でモーション生成
      │  posed_joints / joint rotations / root / foot contacts を毎フレーム出力
      ▼
[リターゲット・ブリッジ(自作)]  ── ARDYスケルトン → VRM Humanoidボーン回転に変換
      │  VMC(OSC) or WebSocket で送出
      ▼
[VRM表示ランタイム]  ── VRMを読み込みボーン駆動＋口パク＋まばたき
      ▼
[OBS]  ── 配信/録画
```

### 構成の3択（Phase 0で選ぶ）

| 案 | VRM表示ランタイム | 特徴 | 工数 | 向く人 |
|---|---|---|---|---|
| **A. 既製VTuberアプリ経由（推奨/低工数）** | VSeeFace / VirtualMotionCapture 等（VMCプロトコル受信・VRM表示・口パク・まばたき内蔵） | レンダラ/OBS連携/口パクを既製に丸投げ。自作はブリッジのみ | 小〜中 | まず動かしたい |
| **B. Web オーバーレイ** | Three.js + `@pixiv/three-vrm`、OBSはブラウザソース | Web完結・軽量・カスタムUI自由。表示層も自作 | 中 | Web資産を活かす |
| **C. Unity + UniVRM 自作** | Unity実装、Spout/仮想カメラでOBSへ | 最も自由度が高く演出しやすい。実装量は最大 | 大 | 作り込む前提 |

**推奨は A**。理由は、レンダリング・OBS出力・口パク・まばたき・視線といった「ARDYが一切面倒を見ない周辺機能」を実績あるツールに肩代わりさせ、自作範囲を **リターゲット・ブリッジ1点** に絞れるため。まずAで最小構成を通し、演出要求が固まってからB/Cへ移行する判断でよい。

## Phase 0: 意思決定・確認事項（着手前）

### オーナー判断ポイント
1. **表示ランタイム**: 上記 A / B / C のどれで始めるか（推奨A）
2. **キャラの現状素材**: (a)2Dイラスト/設定のみ (b)3Dモデルあり(Blender等) (c)これから外注 — Phase 1の分岐に直結
3. **画風**: アニメ調（VRoidで自作可）か、リアル/独自造形（外注前提か）
4. **商用範囲**: 収益化配信・グッズ等の想定有無（ライセンス確認事項に影響）

### 技術的な要確認事項（先に潰す）
- ARDYのスケルトンは **Core / G1 / SOMA**。G1はロボット骨格でVRM Humanoid化に不向き。**Coreを第一候補**とし、人体寄りのSOMA（“Coming soon”表記）の提供状況も確認する
- ARDY出力は「ジョイント位置＋回転＋ルート＋足接地」。**指ボーン・表情・視線・まばたきは出力に含まれない** → これらは表示ランタイム側で別途生成
- 最大の不確実性は Phase 3（リターゲット）。ここは着手前に **スパイク（小さな検証実装）** でGo/No-Goを判断する

**Phase 0 完了判定**: 構成案（A/B/C）決定・キャラ素材の区分確定・ARDY Coreで手元GPUが動く見込みが立っている。

## Phase 1: キャラクターのVRM化

### 目的
配信で使えるVRM 1.0（または0.x）ファイルを1つ用意する。

### 手順（素材別分岐）
- **(a) 2Dイラスト/設定のみ、アニメ調でよい** → **VRoid Studio**（無償）でモデリングし、VRM書き出し。最短ルート。原案イラストを見ながら顔・髪・衣装を寄せる
- **(b) Blender等に3Dモデルがある** → **UniVRM**（Unity）または Blender用VRMアドオンで Humanoidボーンを割り当ててVRMエクスポート
- **(c) 独自造形/リアル寄り** → 3Dモデラーへ外注。**納品要件に「Unity Humanoidリグ準拠のVRMエクスポート」を明記**（後工程の前提）

### 品質チェック
- VRM必須ボーン（Hips/Spine/Chest/Neck/Head/両肩・上腕・前腕・手・上下脚・足）が揃い、Tポーズで破綻しない
- 表情ブレンドシェイプ（あいうえお＋まばたき）が入っている（口パク用）
- VSeeFace等の実機ビューアで読み込み確認

**Phase 1 完了判定**: 表示ランタイム（Phase 0で選んだもの）にVRMを読み込み、手付けモーション/デフォルトポーズで正しく表示・まばたきする。

## Phase 2: ARDY環境構築と生成検証

### 目的
ARDYを単体で動かし、テキスト→モーションが手元GPUでリアルタイム生成できることを確認する。

### 手順
1. Ubuntu 22.04 / RTX GPU、Python 3.11環境を用意
2. CUDA対応PyTorch（>=2.4）を先に導入 → `pip install -e ".[all]"`（C++拡張ビルド用に cmake / build-essential も）
3. Hugging Faceで Meta-Llama-3-8B-Instruct のアクセス許可を取得しトークン設定（LLM2Vecテキストエンコーダの背後で使用）
4. `python scripts/run_demo.py` でデモ起動、ブラウザでテキストプロンプト→モーションを確認
5. **低遅延重視でHorizon8（core8）モデル**を選択。再起動を繰り返すなら `run_text_encoder_server.py` を常駐

### テスト設計
- 「手を振る」「歩く」「ジャンプ」等の代表プロンプトで意図通りのモーションが出るか
- 1フレームあたりの生成時間を計測し、目標FPS（例: 20FPS→50ms/フレーム）に収まるか
- アイドル用デフォルトプロンプト（例: “立ってゆっくり呼吸”）で自然に継続生成されるか

**Phase 2 完了判定**: デモ上でリアルタイム生成が破綻なく回り、目標フレーム時間内に収まる。

## Phase 3: リターゲット・ブリッジ（最大の技術リスク）

### 目的
ARDY出力（Coreスケルトンの位置・回転）を **VRM Humanoidボーンのローカル回転**へ変換し、表示ランタイムへ毎フレーム送出する。**プロジェクトの成否を分ける核**。

### まずスパイク（検証実装）で潰すこと
1. リポジトリ内の **Coreスケルトン定義**（関節名・階層・レストポーズ）を読み、VRM Humanoid各ボーンとの **対応表** を作る
2. 座標系・軸の違い（up軸・左右手系・VRM 1.0のZ前方など）とFPS差を吸収する変換を定義
3. 1本の生成モーションをオフライン（`scripts/generate.py` の `.npz`）で変換し、VRMに焼いて破綻しないか目視 → ここがNGなら構成/スケルトンを見直す（No-Go判断点）

### 本実装
- ARDY推論ループ（`autoregressive_step`）の各ステップ出力を、上記変換でVRMボーン回転へマッピング
- 送出は **VMCプロトコル(OSC)**（案A：既製アプリが受信）または **WebSocket**（案B/C）
- ARDYが出さない要素の扱いを決める: 指＝ニュートラル固定or手続き的、足接地(foot contacts)＝フットロック/IK補正に活用、表情・視線・まばたき＝表示ランタイム側の自動機能に任せる

### テスト設計
- 静止プロンプトで手足が滑らない・地面にめり込まない（足接地情報の反映確認）
- プロンプト切替時にワープ/破綻せず遷移する
- エンドツーエンド遅延（コメント→動作反映）を計測、目標内か

**Phase 3 完了判定**: 任意テキストプロンプトに対し、VRMアバターがリアルタイムかつ破綻なく追従して動く。

---

## Phase 4: 配信ランタイム統合

### 目的
コメント・音声・映像を配線し、配信として成立させる。

### 手順
- **コメント→プロンプト**: YouTube Live Chat API / Twitch IRC を購読 → フィルタ/整形（NGワード除去・指示抽出）→ ARDYのアクティブプロンプトを更新する薄いスクリプト（`load_text_encoder` で埋め込み生成し差し替え）。安全のため許可リスト/クールダウンを入れる
- **音声**: Voisonaで生成した音声を仮想オーディオデバイス経由で流し、表示ランタイムの **音声ベース口パク**（VSeeFace等が対応）に入力。台詞テキストとモーションプロンプトを同期させる制御を1枚噛ませる
- **映像出力**: 表示ランタイムの映像を **OBS**（案A/C＝仮想カメラ/Spout、案B＝ブラウザソース）へ。背景・オーバーレイはOBS側で合成

### テスト設計
- コメント投入→数百ms以内にモーション反映される
- 不正/悪意あるコメントがそのままプロンプト化されない（フィルタが効く）
- 音声とモーションのタイミングがずれない

**Phase 4 完了判定**: 実コメントに反応して動き・喋り、OBSに配信可能な絵が出る。

---

## Phase 5: アイドリング・運用・全体テスト

- **アイドリング**: 入力が無い間はデフォルトプロンプトで自然な待機モーションを継続（Auto-Replan活用）。一定時間ごとに小芝居プロンプトをローテーションさせると“生きている感”が出る
- **フェイルセーフ**: 生成が詰まった/GPU高負荷時に待機モーションへフォールバック
- **通し負荷試験**: 1〜2時間の連続配信を模擬し、メモリリーク・遅延増大・破綻がないか
- **再現性**: 環境構築〜起動を手順化（スクリプト/README化）し、seed固定での挙動確認手順を残す

## リスク一覧

| リスク | 影響 | 対策 |
|---|---|---|
| ARDYスケルトン→VRMリターゲットが破綻 | 致命（成否の核） | Phase 3を**先行スパイク**しGo/No-Go判断。Core優先、SOMA提供待ちも選択肢 |
| 指/表情/視線が生成に含まれない | アバターの表現力低下 | 表示ランタイムの自動機能＋手続き的補完で補う |
| エンドツーエンド遅延が大きい | 配信のテンポ悪化 | Horizon8選択・History Crop短縮・ローカルGPU・ブリッジ軽量化 |
| コメントの悪用（不適切プロンプト） | 配信事故 | 許可リスト・NGフィルタ・クールダウン・人手モデレーション |
| 独自造形の外注品質 | 後工程の手戻り | 発注要件にHumanoidリグ準拠VRMを明記、中間レビュー |
