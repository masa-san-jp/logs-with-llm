# AI駆動型1人スクワッド開発プロトコル (AI-Augmented One-Person Squad Development Protocol)

本プロトコルは、熟練したエンジニア1名が複数のAIエージェントと協働し、大規模エンタープライズの既存システム（ブラウンフィールド環境）において、少人数チームと同等以上の成果を最高速度かつ高品質で達成するための標準手順書である。

---

## 1. 原則 (Core Principles)
1. **仕様ファースト (Specification-Driven):** 実装コードをいきなり書かない、書かせない。人間が完全に言語化した「仕様」が、AIエージェントのすべての行動と成果物を支配する。
2. **非対称な役割分担:** 人間は「高リスクな意思決定」「仕様定義」「評価」に集中し、AIは「定型処理」「ボイラープレート生成」「自動テスト実行」に徹する。
3. **調整コストの排除:** 人間同士の合意形成やミーティングをゼロにし、思考から実行までのリードタイムを極小化する。

---

## 2. フェーズ1：ナレッジの先行投入 (Knowledge Ingestion)
AIエージェントが既存の暗黙知やドメインルールに抵触して誤った出力をするのを防ぐため、開発開始前にコンテキストを徹底的にインプットする。

### 2.1. インプット対象の定義
* 連携先となる既存システムのインターフェース仕様（API仕様書、データモデル）
* 組織固有のセキュリティ規制、アーキテクチャ規約、コーディング標準
* 業務ドメイン固有のロジック、用語集（Ubiquitous Language）

### 2.2. 手順
1. **ナレッジベース構築:** 適切なAIツール（例: 組織特化型AIプラットフォーム）へ上記ドキュメント群を学習またはコンテキストとして集約する。
2. **コンテキスト適合性の確認:** テストプロンプトを投げ、AIが組織特有のルール（例: 認証方式、ログ出力規約）を正しく理解しているか検証する。

---

## 3. フェーズ2：仕様駆動開発 (Spec-Driven Development: SDD)
人間が記述する仕様のクオリティが、AIによるコード生成成功の主たる決定要因（Binding Constraint）となる。

### 3.1. 厳密なインターフェース契約 (Interface Contract)
仕様書には以下の要素を完全に明文化し、AIに「推測」の余地を与えないこと。
* **入力と出力:** 厳密なデータ型、バリデーションルール、エラーハンドリング。
* **前提条件 (Pre-conditions):** 当該機能が呼び出される前に満たされているべき状態。
* **事後条件 (Post-conditions):** 実行後にシステムやデータベースが満たしていなければならない状態。
* **受け入れ基準 (Acceptance Criteria):** どのような動作を持って「完了」とするかの具体的なシナリオ。

### 3.2. 実行手順
1. **仕様書の共同作成:** 人間が定義した要件をベースに、仕様書作成支援AI（例: Devin等）を用いて、構造化された詳細な仕様書（Spec）のドラフトを作成する。
2. **仕様のレビューと凍結:** 人間が仕様書をレビューし、曖昧な記述やレガシー連携の考慮漏れがないかを確認して確定させる。
3. **テストの自動生成:** 確定した仕様書を基に、AIにまず統合テスト・ユニットテストのコードを自動生成させる。

---

## 4. フェーズ3：コアと非コアの動的切り分け (Core/Non-core Partitioning)
開発対象の機能を、人間の判断が必要な「コア（ドメインロジック）」と、AIに完全自動化させられる「非コア（インフラ・定型処理）」に動的に分類・処理する。

### 4.1. 分類基準
* **コア (Core):** 業務ルール、高度なビジネス判断、データ整合性に直結するロジック。
* **非コア (Non-core):** DB接続処理、APIのボイラープレート、メッセージキュー設定、定型UI、テストコード生成。

### 4.2. 動的スクリーニングと切り替え手順

```
[開発タスクの開始]
       │
       ▼
【初期割当】
タスクをAIエージェント（例: Devin）へ自律実行としてアサイン
       │
       ▼
【エージェントによる実装試行】
       │
       ├─► [成功] 一発でテスト通過・コード採択 ──► 【非コア】として確定（自動化完了）
       │
       └─► [手戻り発生] AIがエラーを繰り返す、または人間による複数回の修正・指示が必要
                 │
                 ▼
           【シグナル検知】
           高度な業務コンテキストや人間の暗黙的判断が必要な「コア領域」と判定
                 │
                 ▼
           【有人ループ（Supervised Loop）への移行】
           AIの自律実行を即座に停止
                 │
                 ▼
           【人間＋インラインAIでの実装】
           エンジニアが主導し、エディタ支援型AI（例: GitHub Copilot）と
           対話しながらペアプログラミング形式で慎重に実装・修正を行う
```

1. **AIファーストでのアサイン:** 新機能やタスクは、まず自律型AIエージェントに「非コア」の仮定で投げ、実装を試みさせる。
2. **手戻りシグナルの監視:** AIがテストを通過できない、あるいはコードレビューにおいて人間が何度も手作業で修正・指示出しを行う必要が発生したタスクを検知する。
3. **有人ループ (Supervised Loop) への切り替え:** シグナルを検知したタスクは、即座に自律型AIから剥がし、人間が主導する「有人ループ」へ移動する。
4. **ペアプログラミングの実行:** 人間がコードの主導権を握り、開発環境に統合されたインラインAI（例: GitHub Copilot）を伴走者として活用しながらコアロジックを実装する。

---

## 5. チーム体制とAIエージェントの役割定義 (Agent Topology)
1人スクワッドにおいては、4つの役割（AIエージェント）をパッチワークのように組み合わせ、パイプライン化する。

| 担当ロール / ツール例 | 主な責務・役割 | 運用の位置づけ |
| :--- | :--- | :--- |
| **ドメイン知識・調査 (StackSpot等)** | 組織内ルール、レガシー仕様の検索、コンテキストの提供 | **ナレッジレイヤー** (開発前のインプット) |
| **仕様策定・非コア自律開発 (Devin等)** | 仕様書ドラフト作成、テストコード生成、非コア機能の自律実装 | **自律ループ** (人手を介さない高速処理) |
| **インライン実装支援 (GitHub Copilot等)**| 人間が主導するコアコード実装時のリアルタイム補完、ペアプロ | **有人ループ** (人間の判断と直結) |
